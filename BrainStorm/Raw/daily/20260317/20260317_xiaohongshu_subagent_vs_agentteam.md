## 原文摘要（正文）

最近在本地 AI 桌面应用里同时实现了 Sub Agent（子代理）和 Agent Team（智能体团队）两套多智能体方案，踩了不少坑，分享了一些心得：

- **Sub Agent：轻量快速**
  - 进程内运行，毫秒级启动，适合简单后台任务。
  - 用 `Map` 管理生命周期，通过 `EventEmitter` 投递结果。
  - 缺点是子代理之间无法直接通信，更像一堆「工具小工」。

- **Agent Team：重量级协作**
  - 基于 SDK，每个成员是独立进程，通过 mailbox 通信。
  - AI 可以自主决定团队结构、分配任务、汇总结果。
  - 适合复杂研究和分析任务，更偏向「项目组」而不是单人工具。

- **核心架构亮点**
  - **双引擎设计**：用 `/team` 命令在两套执行引擎之间切换。
  - 两套系统共享 **Provider 配置** 和 **记忆系统**，通过适配层打通。
  - 解决了：
    - SDK stream 生命周期管理（resume 循环 + 空闲检测）；
    - ESM / CJS 混合加载带来的模块边界问题。

- **行业趋势判断**
  - 2025 年 MCP 协议成为标准，多智能体框架百花齐放（CrewAI、AutoGen、LangGraph 等）。
  - 预计 2026 年会出现 **自组织 Agent 团队** 和 **跨平台协作**。
  - 面向未来的桌面 Agent 应该预留「多引擎、多团队」的扩展位。

## Butler 视角下的要点线索（简版）

- **1. Sub Agent = 进程内轻量工兵**
  - 适合 Butler 内部的「一次性子任务」：如单次 web 抓取、单个文件分析、轻量重试等。
  - 生命周期最好与「当前任务」强绑定，结束就回收，不强求横跨多轮记忆。

- **2. Agent Team = 跨进程项目组**
  - 适合「跨工具 + 跨阶段」的研究、复盘、系统设计任务。
  - 对 Butler 来说，更像调度外部「研究小组 /运维小组 /写作小组」，需要明确角色与交接点。

- **3. 双引擎 + 统一记忆，是未来桌面 Butler 的自然形态**
  - 一套是本地身体里的 SubAgent 引擎，一套是挂在外部 SDK / 云端的 Agent Team 引擎。
  - 上层通过统一的任务/记忆接口（如 `task_ledger` + 统一 Provider 配置）打通，避免两套系统各自为政。
