 ## 推断式结构化理解（需后续校对）

 > 本节是结合标题、简介和你最近一系列「Agent 架构 / 自律系统 / Vibe Coding」材料做的概念整理，用来先搭一个可讨论的框架，方便后续对照原文修正。

 ### 1. 关键词拆解

 - **SDD**：这里可以理解为一种「结构化设计文档 / System Design Doc」——在动手写代码前，把系统边界、模块职责、关键数据流、约束条件写清楚。
 - **SSD 方法论**：更偏「Specification-first / 规范先行」的开发方式，强调先把规范、约束、接口和行为预期写成可以被人和 AI 共同遵循的文档或模板。
 - **Vibe Coding**：不是完全无设计的「即兴乱写」，而是借助大模型在交互式对话里逐步成形代码；风险在于容易「越写越散」、难以长期维护。

 ### 2. 这篇笔记大概率想说什么

 - **用 SDD 把 vibe coding「拢起来」**：先写一份足够具体、对齐共识的 SDD，让每一段 vibe coding 都有「参照物」和「回家路径」。
 - **用 SSD 做「护栏」**：把规范（命名约定、模块边界、错误处理、日志/监控要求等）固化成模板 / checklist / LLM prompt，让每次改动都自动对照这些规范自检。
 - **把「写新代码」改成「在 SDD/SSD 约束下重构」**：不是让模型随手起一个实现，而是围绕 SDD 里已经给出的职责和接口，做局部实现 +重构，减少结构性偏航。

 ### 3. 对 Butler / 你当前主线的直接启发

 - **从「一次性聊天 prompt」升级到「稳定的 SDD/SSD 文档」**  
   - 给 Butler 的每个核心子系统（心跳、记忆、工具编排、web 抓取等）都写一份轻量 SDD：问题、目标、约束、边界、关键数据结构、关键流程。
   - 再从这些 SDD 抽一份「SSD 级规范」：命名、目录、依赖、错误处理、日志、测试习惯，把它变成 LLM 可读的 checklist。
 - **Vibe Coding = 按 SDD 走流程的「带感实现」**  
   - 每次让模型改代码前，先显式贴：相关 SDD 片段 + SSD checklist，再下达「只在这些边界内重构」的指令。
   - 把「随缘改哪都行」收缩为「围绕某个用例 / 某个数据流做一次重构 session」。
 - **把这篇笔记当成一种「session 模板」**  
   - 一次重构 session = 读 SDD → 对齐 SSD 规范 → 划定本次改动边界 → 由 LLM 带着 vibecoding → 最后对照 SDD/SSD 做一次自检与小结。
