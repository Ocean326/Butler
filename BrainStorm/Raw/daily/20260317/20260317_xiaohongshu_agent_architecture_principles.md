## Butler / Agent 结构化要点（架构设计原则 · 摘要）

1. 单次「感知-规划-执行-反馈」循环只适合一次性 Demo 任务，现实场景需要面向**多轮迭代**的架构。  
2. Agent 的价值在于自主性，当前模型能力不足以完全自律，因此必须用**架构与机制**来弥补大模型的短板。  
3. 设计时应优先给 Agent 提供足够的自由度与自主条件，再通过 Human-in-the-loop 做必要的约束与纠偏。  
4. 工具应保持**原子性与通用性**，不要在单个工具中硬塞特定业务流程逻辑，以免无法复用。  
5. 业务适配应通过**适配器层 / 外围配置**承接，避免把具体场景逻辑写死在规划、执行、反馈等通用模块里。  
6. 当需要多 Agent 协作时，要自底向上设计：从消息/状态/工具管理，到协调层调度策略，再到应用层业务流程。  
7. 单 Agent → Multi-Agent 的跨越，本质是在管理「多轮迭代 + 多主体协作」，需要清晰的**任务调度、状态管理与协作协议**。  
8. 对 Butler 来说，上述原则直接映射为：把主 Agent 的自由度、工具原子性、业务适配层和多 Agent 协作协议分层设计，避免把所有逻辑堆在单轮对话与长上下文里。

## 一手正文（按网页文本直译记录）

> Agent要实现好效果，架构设计原则有哪些？  
>  
> Agent的基本流程设计就是这个模式:感知-规划-执行-反馈，要实现这样的一个agent，接入大模型做规划和反馈，集成几个工具，再创建一个记忆模块管理上下文和个性化信息，这样一个agent就基本搭出来了，而且用langraph,crewal,owl这样的框架会更容易实现。  
>  
> 但是！这只是完成一次任务的执行过程，比如制定一个简单旅行计划，规划调用搜索工具搜索目的地，机票信息，然后llm来结合搜索信息来生成一个计划。这种只需要一次迭代就完成的任务，只能说做demo可以，在大部分现实场景基本不可用。  
>  
> 那要实现一个好的agent，难度在哪里，结合上面描述，就是从单轮迭代到多轮迭代的跨越。什么意思呢？就是大部分任务不是一轮规划-执行-反馈的迭代就能完成的，而这个多轮不能靠写规则去驱动，否则就变成预定义工作流了。  
>  
> Agent的价值在于智能性和自主性，要做什么，用什么做，该怎么做，有没有做完都应该由agent本身来判断决策。如果大模型能力足够强，那似乎都交给大模型就行，然而现实是大模型还不具备这种完全自主能力。  
>  
> 那自主能力不够，就需要靠架构和设计来弥补。关于Agent架构设计原则，个人认为有四点供大家参考:  
>  
> [一R]以提供给agent自由，自主条件为首要设计原则，让架构对agent更大的作用是发挥能力，而不是限制，然后再添加必要的human in loop;  
>  
> [二R]保持工具的原子性和通用性，不要为了完成复杂任务，给一个工具注入完成特定场景任务的业务逻辑  
>  
> [三R]在规划，执行，反馈的线性依赖之外，保持各模块之间解耦，业务适配不要直接硬编码到对应的通用模块，通过适配器解决适配和外部约束  
>  
> [四R]对于场景有更多自定义需求(不依赖第三方框架)，而且场景还需要使用multi-agent，那架构设计就需要放在技术方案设计的首位。而multi-agent架构设计的难度体现在，首先每个子agent已经有一个迭代循环，然后多agent之间通信该如何进行，多agent间的状态，迭代如何管理，这需要自底向上设计，从基础层的消息，状态，工具管理，到协调层的任务调度，交互策略管理，再到应用层的场景适配，业务流程管理，需要不断的设计-实践-优化来搞出一套生产可用的agent框架。  
>  
> 当然agent在现实场景要发挥比较好的作用，除了架构问题，还包括工具链构建，模型使用策略(大，小模型，多模型集成)，模型优化(强化微调)，自动测评等等。  
>  
> 关于agent架构设计的问题，也欢迎大家交流，指出观点不足之处[害羞R]
