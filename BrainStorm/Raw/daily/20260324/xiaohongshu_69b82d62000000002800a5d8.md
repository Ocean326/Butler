## Content

1. 构建 research agent 本质上是在同时构建两个东西：agent 基础设施和 research 工作流。也对应了俩问题：无法正确实现创新点和创新性不足。解决方案为 harness 工程以及 subagent 架构
2. 创新问题：简单几个论文一丢，快速攒完了代码和实验，创(胡)意(编)写(乱)作(造)的问题早解决了。但我感觉 idea 过于直给，于是问了一句：你俩觉得这能投 cvpr 吗？glm-5 支支吾吾(thinking)，claude 先没绷住：这只能拿 C，并且委婉的指出甚至没啥 rebuttal 的必要。
3. claude 的解法很简单， harness 工程。强制进行更细粒度的重复性检查 skill 以及作为 reviewer 给出必拒理由让 glm-5 提前想好这个 idea 最后有没有可能 rebuttal 赢。到这一步就完成了简单的 harness，agent 能自己独立执行并验证。这里的 novelty check 也不是单纯的 prompt 或 skill，而是复现了某篇论文(agent transformer还是啥的)的基于前置任务的有向无环图和日志录制/回放测试。并且只在 arxiv 上搜索有点太 toy 了，还是会误判，于是把各种会议期刊都加了进去。
4. 这套范式下独立迭代了四十几轮文献搜索和头脑风暴，想出来的 idea 连 claude 都直呼 nb，研究工作流的问题基本通过 harness 工程解决。同时，claude和我都担心实现会对不齐 idea。
5. 那么对不齐的问题怎么解呢？前几篇有点尬黑 glm-5 的意思，是为了节目效果。实际它只比 claude 低 3 个点，不会差太多。于是按着这个思路，结合了一些 agent engineering 的论文和 claude 技术博客，我发现了之前的设计下存在严重的上下文腐败(context rot)，即 agent 被技能描述，工具返回值，debug 日志淹没了上下文，最终忘记关键信息。因此跑开源代码库的 baseline 很快，需要自己调试的时候容易越跑越远。
5~6：我一直有个不完全正确的认知，即多智能体合作容易切断或污染 context 造成 1+1<2的效果，所以设计之初希望一个 agent 串到底。但单智能体上下文腐败同样是致命的，上下文隔离是必要的。正确的做法是保证主 context 干净，子任务开 subagent，做完了 compact 一轮回主 context。成功可静默。同时把 skill 变成了动态注入，避免越积越多的技能淹没任务本身。并且在代码实现层面上也做了一轮 harness 工程
7～8. 创新点，指标略好，待ablation
