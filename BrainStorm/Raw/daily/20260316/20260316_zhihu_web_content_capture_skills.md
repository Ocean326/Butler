 ### 2.1 关键概念与术语
 
 - **Harness Engineering（马具工程）**：`把 agent 外面那一整圈「运行环境」当成核心工程对象来设计：工具调度、上下文压缩、状态持久化、安全门控、观测与评测，而不是只盯着模型本身。`
 - **MAS（multi-agent system，多智能体系统）**：`由多个具备一定自治能力的 agent 组成的协作系统，通过协调、路由和门控把各自的推理能力汇聚成整体能力。`
 - **MAS Harness 四层架构**：`知识供给层（Knowledge）、执行编排层（Orchestration）、风险门控层（Policy / Guardrail）、治理运营层（Governance），三层由经验沉淀飞轮串起来。`
 - **Context Rot / Lost-in-the-middle**：`上下文越长性能反而不稳定、关键信息被埋在中段准确率显著下降，需要通过 context engineering、分段重启等手段治理。`
 - **Ralph Loop**：`一种强调「每轮全新 context + 文件持久化进度 + 自动验收」的长程任务治理机制，用门控和 checkpoint 替代无限累积的对话历史。`
 
 ### 2.2 文章结构 / 模块拆解
 
 - **模块 1：龙虾热潮与 Harness 问题的提出**
   - OpenClaw 在 72 小时拿到 6 万 star、现在 30 万+，代表高授权 agent 真正跑进用户桌面。
   - 高度自主的 agent 带来了恶意 skill、市级 token 泄露、CVE 攻击链等安全事件，本质是 harness 缺位而非模型能力错误。
   - 结论：能力越大，责任越大，agent 真正落地的主战场在模型外侧的工程层。
 - **模块 2：从单 agent 到多 agent 的演进与边界**
   - 回顾 ReAct、Plan-and-Execute、ReWOO、异构模型分工、Ralph Loop 等单 agent 设计范式。
   - 指出单 agent 在 context 污染、并行性不足、知识深度和 Maker-Checker、错误隔离上的结构性天花板。
   - 引出多 agent 场景下 Subagents / Skills / Handoffs / Router 等拓扑，以及 Anthropic 的 Subagents vs Agent Teams 区分。
 - **模块 3：MAS 失败模式与协调复杂度**
   - 复盘「Why Do Multi-Agent LLM Systems Fail?」中的 MAST 框架：系统设计问题（FC1）、协调失败（FC2）、验证缺失（FC3）。
   - 强调多数失败不是 prompt 写得不够好，而是协调拓扑、角色设计、验证路径在系统层面有缺陷。
   - 引入「协调税」和 4+ agent 后收益递减甚至下降的实证结果，说明盲目堆 agent 只会制造复杂度。
 - **模块 4：Harness Engineering 的定义与工程定位**
   - 区分 scaffolding（首轮组装）和 harness（长期运行环境），强调 harness 与推理逻辑解耦。
   - 用马具类比：马决定往哪跑、多快跑，马具负责把力量安全地传导到车上并防止脱轨。
   - 总结：MAS harness = 多 agent 的运行基础设施，涵盖工具调度、context 压缩、安全门控、状态持久化、全链路观测。
 - **模块 5：四层架构——知识供给 / 执行编排 / 风险门控 / 治理运营**
   - 知识供给层：区分参数化知识、非参数化知识（RAG）、经验知识三类资产，以及跨 agent 知识一致性问题。
   - 执行编排层：Orchestrator + Stateful Workflow + Policy Runtime 三分，解决「任务怎么拆、谁来做、怎么协调」。
   - 风险门控层：作为推理链条之外的安全中间件，负责权限、预算、动作拦截、prompt injection 防御等。
   - 治理运营层：把复杂的 MAS 运行轨迹从日志提炼成可检索的协调模式库、任务案例库、失败模式库。
 - **模块 6：经验闭环与三阶段演化路径**
   - 阶段一：先跑通闭环，重在全量记录和可追溯，不急于引入复杂自适应调度。
   - 阶段二：经验开始反哺，Orchestrator 与门控开始引用历史成功路径和自动归纳的规则，运营成本与成功率共振改善。
   - 阶段三：飞轮显现，新任务越来越能从历史经验中获益，系统表现出「越跑越强」而非「跑越多越乱」的特征。
 
 ### 2.3 核心论点 / 方法论
 
 - **核心论点 1：agent 项目的主战场在模型外，而不在模型里**
   - 结论：生产级 MAS 的决定性差异不在于调用了哪家 frontier 模型，而在于是否有一套完整的 harness 架构与治理闭环。
   - 支撑：几乎所有高分 agentic benchmark 成绩都伴随明确的 harness 条件说明；OpenClaw 与 Anthropic 16-agent 写编译器实验都表明，绝大部分工程精力花在测试体系、任务锁、反馈循环等 harness 侧。
 - **核心论点 2：知识—编排—门控—治理四层解耦是 MAS 的基础结构**
   - 结论：把这四层混在一个 orchestrator 或 workflow 里，只会在迭代中不断堆规则，最后滑向「more rules, less autonomous」。
   - 支撑：作者自己的 MAS 项目踩坑复盘 + 业界实践（OpenAI Codex、Anthropic、Azure Connected Agents 等）的架构拆分经验。
 - **核心论点 3：经验资产是 MAS 的长期护城河**
   - 结论：任务级案例库、协调模式库、失败模式库，是不会因模型升级而归零的长期优势。
   - 支撑：这些资产只能从真实运行中积累，无法直接购买或从公开 benchmark 学到；投入时间越长、运行次数越多，优势越大。
