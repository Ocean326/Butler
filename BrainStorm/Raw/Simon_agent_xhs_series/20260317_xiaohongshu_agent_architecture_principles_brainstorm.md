## 这篇在讲什么问题 / 场景？

1. **核心问题**：现实业务里要让 Agent「真正好用」，不能只停留在一次性 Demo 式的「感知-规划-执行-反馈」单轮循环，而需要为多轮迭代和复杂场景设计一套合适的架构原则。  
2. **典型场景**：从简单旅行计划这类一次性任务，走向需要多轮试探、反复修正、多人 / 多系统协作的真实用例（如复杂方案制定、研发协作、运营活动执行等）。

---

## 文中的关键设计原则 / 方法（3–5 条要点）

- **从「单轮迭代」跨越到「多轮迭代」的思维转换**  
  - 简单 Demo 可以用一次规划-执行-反馈完成，但大多数现实任务需要多轮尝试与纠错。  
  - 若用规则和固定流程硬写所有多轮逻辑，就退化成预定义工作流，失去 Agent 自主性。

- **原则一：先给 Agent 足够的自由度与自主条件，再用 Human-in-the-loop 做必要约束**  
  - 架构的首要任务不是限制 Agent，而是为其提供发挥空间（工具、记忆、知识、上下文）。  
  - 在此基础上通过人类审核 / 关键节点确认来兜底，而不是事前写死所有路径。

- **原则二：保持工具的原子性与通用性**  
  - 工具职责应单一、可复用，不在单个工具中硬塞完整业务流程。  
  - 复杂任务的业务逻辑通过组合与编排实现，而不是让一个「超级工具」承担全部责任。

- **原则三：模块解耦 + 通过适配器承接业务约束**  
  - 规划、执行、反馈模块不直接绑死具体场景逻辑，保持通用性。  
  - 具体业务规则和外部系统约束通过适配器 / 配置层承接，避免将业务写死在通用模块里。

- **原则四：multi-agent 架构需要自底向上的系统设计**  
  - 先设计消息、状态、工具管理等基础层，再设计协调层的任务调度与交互策略，最后才是应用层的场景适配与业务流程。  
  - 多个子 Agent 各自有迭代循环，之间如何通信与共享状态，是需要反复「设计-实践-优化」的工程问题。

## 核心观点（压缩小结）

- 从一次性 Demo 的「单轮感知-规划-执行」升级为**默认多轮迭代**的 Agent 架构，才能覆盖真实业务的复杂度。
- 通过**原子化工具 + 编排层组合**承接业务流程，而不是把所有逻辑塞进一个「超级工具」或一个大 Prompt。
- 用**模块解耦 + 适配器层**承接具体业务约束，让通用 Agent 循环保持干净、可迁移。
- 为 future multi-agent 预留「消息 / 状态 / 工具管理 → 协调策略 → 应用场景」三层，避免一开始就把多 Agent 写死成复杂 if-else。

## 对 Butler 架构 / 心跳 / 自律 hooks 的可借鉴点

- **在 Butler 的任务系统中内建「多轮迭代」假设**  
  - `task_ledger` 与 heartbeat 流程不应假设一次规划就能完成任务，而是默认允许多轮重试 / 微调。  
  - 在执行链路里，为「重试原因」「调整点」等信息预留结构化字段，方便后续经验沉淀。

- **将工具设计收敛为「原子能力层 + 适配器层」**  
  - skills 与工具实现尽量保持原子、通用，把特定业务逻辑和外部系统差异压在适配器 / 配置层（例如 workspace 规则、项目特定 adapter）。  
  - 在 `./工作区` 或项目级目录中维护清晰的适配器说明，而不是把业务细节写进通用执行逻辑。

- **为未来 multi-agent / AgentTeam 提前打好「基础层 → 协调层 → 应用层」分层**  
  - 在设计新能力（如子 agent、协作流程）时，先问清：当前改动属于消息 / 状态 / 工具管理基础层，还是协调层策略，还是应用场景层。  
  - 把这一分层显式体现在 Butler 的目录与文档中（例如在 BrainStorm / docs 中维护对应的三层映射），避免未来多 Agent 演进时出现「所有逻辑挤在一个大循环里」的反模式。

## 潜在实验想法（Butler 内部可落地）

- **task_ledger 增加「重试原因 / 调整点」字段试点**  
  - 选取 1–2 条当前仍在迭代中的任务，在 `task_ledger` 或工作区任务描述中显式记录每次重试的触发原因与改动点，观察对后续复盘与自我改进是否有帮助。
- **为 1 个典型工作流拆分「原子工具 + 适配器」**  
  - 以现有某个 Feishu / 文档处理类 skill 为例，将其中的场景特定逻辑下沉到 `./工作区` 或项目级配置，保留一层相对通用的原子操作函数，并在文档中对齐本篇的「原子能力 + 适配器」分层。
- **在 Heartbeat 报告中增加「本轮多轮迭代视角」小节**  
  - 在某次 heartbeat 总结中，用 1–2 行方式记录：本轮有哪些任务经历了多轮迭代、主要调整了哪些点，对后续自律与架构调优是否产生了可见影响。

## Butler 可借鉴要点（聚焦 MAS / Harness）

- **用「单主循环 + Git 状态」管理多轮迭代**  
  - 将当前 Butler 的执行主循环（chat / heartbeat / executor）视为单一「Harness Shell」，所有任务状态统一落在 `task_ledger` + Git 工作区，而不是多个各自为政的小循环。  
  - 每次迭代都要求能在 Git diff 与任务记录中看到「这一步改了什么、为什么改」，方便事后回溯与经验抽取。

- **用「死循环壳」保护核心 Agent 循环**  
  - 对高价值长任务（例如持续多轮的架构演进、长期研究专题），可以考虑用「死循环壳 + 明确中止条件」包起内部 Agent 循环，把异常 / 超时 / 外部错误都收敛为「这一轮失败原因」。  
  - 壳层只负责兜底与重试策略，不直接参与业务逻辑，确保内部 Agent 逻辑相对干净。

- **为上下文腐烂预留主动 reset 机制**  
  - 在多轮任务中，约定明确的「上下文 reset 节点」，例如：关键里程碑、方案基本定型、或历史对话噪音累积过多时，主动切断旧上下文、只保留结构化纪要。  
  - 心跳与任务总结应鼓励把「什么时候 reset、保留了什么、丢弃了什么」写成显式决策，而不是隐性漂移。

- **把自律 hooks 设计成「可插拔」而非散落 if-else**  
  - 将诸如「强制自我评估」「风险自检」「预算提醒」「结果对照预期」这类自律行为，收拢为统一的 hooks 机制（如 before/after-execute、before-commit 等），集中配置生效范围。  
  - 在多 Agent / 多任务场景下，可以按任务类型或风险级别开启不同组合的 hooks，而不是在每个 skill 里手写一遍自检逻辑。

- **区分「原子工具」与「Harness 策略」避免混杂**  
  - 让工具层专注于「能不能做到」、Harness 与心跳层专注于「该不该做、做到什么程度」，避免在工具内部写死策略。  
  - 对于后续 MAS 演进，这种分层能减少「多个 Agent 争抢同一片 if-else」的混乱，便于统一调整策略与约束。
