## 简短结构化摘要（Simon · Agent 架构设计原则）

- **主题**：从一次性 Demo 式「单轮感知-规划-执行」升级到面向真实业务的「多轮迭代 Agent 架构」，强调工具原子化与模块解耦。
- **关键方法**：先假定任务天然需要多轮试探与纠错，再通过任务系统与日志承载「重试原因 / 调整点」，而不是把流程写死在单次对话里。
- **关键方法**：将工具压缩为职责单一、可组合的原子能力，把具体业务约束和外部系统差异放到适配器 / 配置层承接。
- **对 Butler 的启发**：在 `task_ledger` 与 heartbeat 流程中内建「多轮迭代」假设，并为每次重试结构化记录原因与调整，作为后续自我改进的数据资产。
- **对 Butler 的启发**：在 skills 与脚本设计上维持「原子能力层 + 适配器层」分层，把项目/工作区特有规则落在 `./工作区` 或项目目录，而非写进通用执行循环。
- **对 Butler 的启发**：在规划新子 Agent 或协作流程时，先判断当前改动属于基础能力、协调策略还是具体场景适配，并在文档与目录结构中保持这一分层的一致性。
