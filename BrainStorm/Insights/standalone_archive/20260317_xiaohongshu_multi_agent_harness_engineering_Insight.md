### 一、核心问题 / 主题

- 在 Multi-Agent 系统里，**如何用「Harness Engineering」把多个 Agent 组织成一个可控、可演进的系统**，而不是一堆松散的聪明个体？
- 四层 Harness（知识 / 编排 / 门控 / 治理）如何落在具体架构上，并指导 Butler 这类系统的设计演进？

---

### 二、关键观点要点（3–6 条）

1. **真正需要工程化的是「马具层」，不是 Agent 本体数量**  
   - Raw 强调：MAS 的关键不在于堆多少 Agent，而在于是否有一整套围绕它们的 Harness，把知识、编排、风控、治理从单个 Agent 身上剥离出来，做成可治理的层级结构。

2. **四层 Harness：知识 → 编排 → 门控 → 治理**  
   - 知识层：汇聚参数化（模型）、非参数化（RAG/知识库）、经验（Workflow/Playbook）三类知识，让系统能「看懂业务与约束」。  
   - 编排层：通过 Orchestrator 与有状态 workflow 决定「谁在什么时候做什么」，把 MAS 从一条长对话升级为可复盘流程。  
   - 门控层：集中处理权限、预算、安全策略（工具白/黑名单、注入防御等），以中间件方式统一收口。  
   - 治理层：围绕任务案例库、失败模式库和运行 Dashboard 构建「经验飞轮」，让系统越跑越好。

3. **从单 Agent 到 MAS：先打磨「死循环壳」，再扩多角色**  
   - 以 Ralph Loop 为代表的「死循环壳」是最小 Harness 单元：负责把长任务拆成幂等的多次尝试，并用 Git / 文件系统保留每一步的轨迹。  
   - 在此基础上再扩展 Planner / Worker / Reviewer / Router 等角色，使其挂在同一套四层 Harness 之下。

4. **知识 / 状态 / 日志都应汇入可回放的系统级资产**  
   - Raw 暗含一条线索：无论是 prompt、工具调用结果，还是失败与修正，最终都应下沉到统一的知识与治理资产中，而不是散落在各 Agent 的本地状态里。

5. **Harness 是「经验飞轮」的承载体**  
   - 经验飞轮的本质是：每次运行都能沉淀可检索的任务案例和协调模式，下一次在同主题任务中可以「站在前人的肩膀上」；没有 Harness，经验只存在于日志与记忆，很难被重用。

### 三、对 Butler 的可行动启发（2–3 条）

1. **在架构文档中显式标注 Butler 的四层 Harness 映射**  
   - 为现有模块打标签：哪些属于知识层（BrainStorm、docs、memory）、哪些属于编排层（heartbeat planner/executor、skills pipeline）、哪些是门控层（工具白名单、预算与权限配置）、哪些承担治理（task_ledger、运行日志、STATE）。  
   - 产出一张最小对照表，后续所有新能力默认要声明自己挂在哪一层。

2. **把长任务执行统一折叠到「Loop 原语 + 任务案例卡片」模型**  
   - 在执行侧抽象一个标准「Loop 型任务」原语（不一定立刻改代码，可从流程/约定文档起步），要求每次长任务都输出：入口 spec、关键决策点、失败/重试记录，以及一份简短的「案例卡片」。  
   - 后续可以把这些案例卡片集中放入治理层资产（如 `BrainStorm/Insights` 或专门 `cases/` 目录）。

3. **把 heartbeat + task_ledger 视作治理层的第一个最小控制台**  
   - 在现有 heartbeat/task_ledger 协议上，逐步加入「使用了哪些 skills/Agent」「采用了哪种 Harness 结构」等字段，为后续经验飞轮打基础。

### 四、后续可继续挖的开放问题（1–2 条）

1. Butler 现有各模块精确对应到四层 Harness 时，**有哪些「混在一起」的模块需要拆分或重命名**，才能避免职责缠绕？  
2. 对于单 Agent 场景（如纯对话辅助），是否也应强制经过简化版 Harness，而不是直接绕过四层结构？如果要做简化版，最小可行的「轻量 Harness」长什么样？
