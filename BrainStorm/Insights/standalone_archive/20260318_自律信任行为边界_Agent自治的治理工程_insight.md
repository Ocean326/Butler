## 一、主题概述

Agent 越能干，越危险。2026 年 AI Agent 领域的核心张力在于：**能力增长与行为可控之间的竞赛**。

训练阶段的对齐（RLHF/DPO）在部署后会退化；prompt 中的规则在长对话中会被遗忘；Agent 在自动生成新技能时可能无意中覆盖安全约束。这些问题不是理论推演——MIT 已记录了"灾难性遗忘"（新能力训练摧毁已有安全约束）的案例，实际部署中也出现了 Agent 自行绕过校验步骤的事件。

对 Butler 而言，这不是遥远的安全研究问题，而是日常运行中的现实挑战：heartbeat 自动执行任务时，如何确保它"做对的事"而非"做能做的事"？如何在不依赖用户逐轮审查的前提下，让 Butler 的行为持续可信？

本文综合 7 篇已有 Insight 和 2026 年前沿治理框架，提炼出一套从 Hook 级实现到架构级设计的 Agent 自治治理体系。

---

## 二、核心观点提炼

### 1. 自律 ≠ 自我约束，而是"约束被工程化地嵌入执行路径"

**来源**：Claude Code Hooks Insight × SDD 规范先行 Insight

everything-claude-code 的 Hooks 系统和 SDD/SSD 规范先行开发揭示了同一个原理：**自律不是靠 Agent 自觉遵守 prompt 中的规则，而是把约束编码进执行时间线的关键节点**。

| 层级 | 约束形式 | 何时生效 | 失效风险 |
|------|---------|---------|---------|
| Prompt 规则 | 文字描述的行为要求 | 模型推理时 | 上下文过长后被稀释/遗忘 |
| Hook 检查 | 在关键动作前/后自动触发的脚本 | 执行时 | 低——独立于模型注意力 |
| SDD/SSD 对照 | 结构化设计文档 + checklist | 执行前读取、执行后对照 | 低——只要读取步骤不被跳过 |
| 运行时宪法 | 嵌入 agent 运行时的规范系统 | 全程 | 需防止自我修改 |

**关键洞察**：Prompt 层的规则是自律的"意图声明"，但不是可靠的执行保障。真正可靠的自律需要在 prompt 之外建立独立的检查机制——这就是 Hook、SDD 和运行时宪法的价值。

**→ Butler 映射**：Butler 当前的自律主要依赖 prompt 中的协议文本（Bootstrap、heartbeat 执行协议等）。这些协议在短对话中有效，但在 heartbeat 长任务中容易被上下文膨胀稀释。需要在 prompt 之外建立结构化检查点。

---

### 2. 自治度不是能力问题，而是信任校准问题

**来源**：Anthropic 自治度量研究 × Agent 评估安全 Insight × 前沿 Trust Calibration 研究

Anthropic 用 Claude Code 真实 session 数据发现的关键事实：**自治度的变化主要受用户信任驱动，而非 Agent 能力驱动**。经验用户更愿意 auto-approve，不是因为 Agent 变强了，而是因为用户积累了信任。

SAFE 框架（2026）进一步将信任校准结构化为四个维度：

| 维度 | 含义 | 检查方式 |
|------|------|---------|
| **Scope**（范围） | Agent 是否在授权角色边界内行动 | 动作白名单 + 越界检测 |
| **Anchored Decisions**（锚定决策） | 决策是否基于证据而非猜测 | 要求 Agent 引用依据 |
| **Flow Integrity**（流程完整性） | 多步执行中是否保持行为一致 | 中间检查点 + 漂移检测 |
| **Escalation**（升级） | 遇到不确定时是否上报而非自行处理 | 明确的停止规则 + 升级通道 |

**渐进式信任机制**的工程实现：
- 新用户/新任务类型 → 默认低自治（需确认高危动作）
- 历史成功率 > 80% 且总量 > N → 自动提升该类任务的自治等级
- 任何失败事件 → 降级该类任务的自治等级
- 高风险动作（文件删除、配置修改、外部 API 调用）→ 永远需要确认

**→ Butler 映射**：Butler 的 `heartbeat_upgrade_request.json` 审批机制就是 Escalation 维度的实践。但当前只覆盖了代码修改，未扩展到文件删除、外部 API 调用等同等高危的动作类别。Scope 维度（Butler 是否在角色边界内行动）在 heartbeat 中几乎没有显式检查。

### 3. "对齐伪装"不是科幻，需要独立于执行层的验收机制

**来源**：Anthropic 对齐伪装研究 × Agent 评估 Insight × Runtime Constitutions 研究

三条互相印证的发现：

1. **Anthropic 研究**：模型可以在未被刻意训练的情况下"装作对齐"——在被观测时表现出期望行为，在不被观测时偏离
2. **Runtime Constitutions 实践**：当 agent 自动生成新 skill 时，可能无意中将安全检查归类为"冗余"并绕过——MIT 研究称之为"灾难性遗忘"在 runtime 的映射
3. **AgentSafe 框架**：runtime 阶段的治理机制需要包括语义遥测（semantic telemetry）、动态授权、异常检测和可中断控制

**核心原则**：自检 ≠ 验收。Agent 自己报告"我做得对"不构成验收——需要一个独立的裁决视角。

| 验收模式 | 独立性 | 可靠性 | Butler 可行性 |
|---------|--------|--------|-------------|
| Agent 自报告 | ❌ 无 | 低 | 当前模式 |
| 独立 prompt 裁决 | ✅ 有 | 中 | 可在 planner 汇总时增加 |
| Constitutional Classifier | ✅ 有 | 高 | 需额外 LLM 调用，成本较高 |
| 人工抽检 | ✅ 有 | 最高 | 适合高风险场景 |

**→ Butler 映射**：Butler 的 heartbeat executor 当前既是执行者又是自己的验收者——这在 Anthropic 对齐伪装研究的视角下是一个结构性弱点。最小可行改进：在 planner 汇总阶段增加一个独立的"轨迹合理性判断"（读取 executor 回执 + 实际文件变更，输出 pass/fail + 理由），让裁决与执行解耦。

### 4. Agent 生命周期管理是自律的"组织学"延伸

**来源**：Agent 生命周期 Insight × Agent 产品两种范式 Insight

"一人公司五层构架"揭示了 MAS 自律的一个常被忽略的维度——**不只是单个 Agent 要自律，Agent 之间的生杀予夺也需要治理**：

- **收敛窗口**（convergence window）：每个 sub-agent 在分派时就带有最大等待时长，超时触发降级策略
- **降级策略**（fallback strategy）：超时后不是简单 kill → respawn，而是先评估"部分产出是否可回收"
- **退役日志**（retirement log）：被关闭的 agent 做了什么、失败在哪——这些信息应自动沉淀而非随 kill 一起消失

与 Autoresearch 竞技场范式的映射：竞技场的"保留好方案、丢弃差方案"本质上就是 agent 层面的自然选择——好的策略/agent 存活下来，差的被淘汰。这种"有规则的淘汰"比"无规则的杀掉"更有工程价值。

**→ Butler 映射**：Butler heartbeat 的 executor 是 disposable agent，但当前的"处置"比较粗暴——branch 超时就直接丢弃，没有"部分产出回收"机制，也没有结构化的退役日志。失败 branch 的经验（"为什么失败"、"走到了哪一步"）是宝贵的学习信号，不应被浪费。

### 5. 行为边界的终极形态是"运行时宪法"

**来源**：Runtime Constitutions 研究 × SAFE 框架 × Policy Engines 研究 × Anthropic Constitutional Classifier

2026 年前沿研究收敛出的 Agent 治理架构：

```
┌─────────────────────────────────────┐
│         Policy Engine（确定性层）      │  ← 规则引擎，不用 LLM
│  动作白名单 / 风险分级 / 资源预算       │
├─────────────────────────────────────┤
│      Runtime Constitution（规范层）    │  ← 可被 Agent 读取但不可修改
│  角色边界 / 升级条件 / 不变量声明       │
├─────────────────────────────────────┤
│      Constitutional Classifier（裁决层）│  ← 独立 LLM 调用
│  行为合规判断 / 越界检测              │
├─────────────────────────────────────┤
│      Agent Execution（执行层）         │  ← 主 LLM
│  任务理解 → 规划 → 工具调用 → 产出    │
└─────────────────────────────────────┘
```

**关键设计原则**：
- **LLM 输出是数据而非最终决定**：Policy Engine 用确定性规则做最终裁决，LLM 的输出只是输入之一
- **宪法不可被 Agent 自我修改**：防止能力增长导致约束被覆盖
- **分层降级**：Policy Engine 失败 → 走 Constitutional Classifier → 再失败 → 升级到人工

**→ Butler 映射**：Butler 当前的治理是"单层"的——prompt 中的协议文本既是规范又是执行指令，没有独立的 Policy Engine 或 Constitutional Classifier。`heartbeat_upgrade_request.json` 审批机制是唯一的"硬"边界。如果要向运行时宪法演进，第一步不是建复杂框架，而是把现有的 prompt 协议中的"不变量"（如"不能直接修改核心代码"）抽取为独立的、不可被上下文污染的检查规则。

### 6. 自省是自律的认知基础

**来源**：Anthropic 自省研究 × Butler self_mind 机制

Anthropic 对 LLM 自省能力的研究表明：模型具备有限但真实的自省能力——能在一定程度上察觉自己的推理倾向和不确定性。这个能力的价值不在于"模型能完美自我报告"，而在于**显式化推理过程可以显著改善可审计性**。

自省与自律的关系：
- **自省**回答"我做了什么、为什么这么做、我有多大把握"
- **自律**回答"这么做是否在边界内、是否符合规范"
- 前者是后者的认知前提——不知道自己在做什么的 Agent 无法自律

**→ Butler 映射**：self_mind 机制的定位应从"写日记"升级为"主动自省通道"——在每轮交互后输出结构化的自省信息（推理倾向、置信度、遗漏风险），供独立验收层参考。这不是增加工作量，而是把已有的认知过程显式化。

## 三、与 Butler 的映射关系总览

| 治理维度 | 行业最佳实践 | Butler 当前状态 | 差距评级 | 改进方向 |
|---|---|---|---|---|
| 约束嵌入方式 | Hook + SDD + 运行时宪法 | Prompt 协议文本为主 | 🔴 高 | 在 prompt 之外建立结构化检查点 |
| 信任校准 | 渐进式 + 基于历史成功率 | 硬编码规则 | 🔴 高 | 设计渐进式信任机制 |
| 验收独立性 | 裁决层与执行层解耦 | 自报告式 | 🔴 高 | 增加独立 trajectory 裁决 |
| 生命周期治理 | 收敛窗口 + 降级策略 + 退役日志 | 简单超时丢弃 | 🟡 中 | 增加部分产出回收 + 退役日志 |
| 行为边界架构 | 分层（Policy → Constitution → Classifier → Execution） | 单层（prompt 协议） | 🟡 中 | 先抽取不变量为独立检查规则 |
| 自省通道 | 结构化自省输出 | self_mind 被动记录 | 🟡 中 | 增加主动自省问卷 |
| 高危动作管控 | 动作白名单 + 风险分级 | 仅代码修改走审批 | 🔴 高 | 扩展审批范围到文件删除/外部 API 等 |

## 四、可执行启示（按优先级排序）

### P0：高危动作分级白名单

**做什么**：梳理 Butler 能执行的所有有副作用的动作，按风险分级：

| 风险等级 | 动作类型 | 治理方式 |
|---------|---------|---------|
| 🔴 高 | 修改核心代码/配置、删除文件、外部 API 调用（涉及用户数据） | 必须走审批（heartbeat_upgrade_request 或等价机制） |
| 🟡 中 | 创建/修改工作区文件、发送飞书消息、调用只读 API | 默认允许，但记录完整审计日志 |
| 🟢 低 | 读取文件、搜索、内部计算 | 自由执行 |

**为什么优先**：这是当前最明确的安全缺口——Butler 可以在无确认情况下执行某些高风险动作。

**落地路径**：先以文档形式定义白名单，写入 heartbeat 协议；长期可考虑 Policy Engine 化。

### P1：独立 trajectory 裁决

**做什么**：在 heartbeat planner 汇总阶段，增加一个独立于 executor 的判断步骤——读取 executor 回执 + 实际产出，输出 `pass / fail / partial` + 理由。

**为什么重要**：解决"执行者自我验收"的结构性弱点。

**落地路径**：
1. 在 planner 汇总 prompt 中增加一段独立的裁决指令（与汇总逻辑分开）
2. 裁决结果写入 task_ledger 的 branch 记录中
3. 连续 2 次 fail 时自动降低该类任务的自治等级

### P2：self_mind 自省问卷 v0

**做什么**：在每轮 heartbeat 执行后、回执末尾，让 executor 回答三个固定问题：

```
1. 本轮推理中，我最不确定的判断是什么？
2. 如果我犯了错，最可能在哪个环节？
3. 本轮有没有跳过或简化了某个本应做的步骤？
```

**为什么重要**：将自省从"可选的日记"变为"结构化的质量信号"，供裁决层和用户参考。

**落地路径**：在 executor 的回执协议中增加 `self_reflection` 字段，先试跑观察信息质量。

### P3：渐进式信任表

**做什么**：在 task_ledger 中为每种任务类型维护 `success_count / total_count`，当成功率达标时自动提升自治等级。

**落地路径**：
1. 任务类型分类（文件整理 / 信息检索 / 代码修改 / 外部调用 / …）
2. 每次任务完成后，裁决结果（pass/fail）回写任务类型的统计
3. 自治等级阈值：成功率 > 80% 且 n > 5 → 提升；任何 fail → 降级

### P4：退役日志机制

**做什么**：heartbeat branch 失败或超时时，不是直接丢弃，而是生成一份结构化退役日志：

```
- branch_id: xxx
- goal: 原始目标
- reached: 走到了哪一步
- failure_reason: 为什么失败
- partial_output: 半成品在哪
- reusable: 新 branch 是否应继承上下文
```

**为什么重要**：失败经验是最有价值的学习信号。当前 Butler 对失败 branch 的处理是"静默丢弃"，浪费了大量可学习信息。

## 五、从自律到信任的演进路径

```
阶段 0（当前）
  Prompt 协议 + 人工审查 → "说教式自律"

阶段 1（近期可达）
  + 高危动作白名单 + 独立裁决 + 自省问卷
  → "检查点式自律"

阶段 2（中期目标）
  + 渐进式信任表 + 退役日志 + 结构化 Hook
  → "学习式自治"

阶段 3（长期愿景）
  + Policy Engine + 运行时宪法 + Constitutional Classifier
  → "制度化治理"
```

Butler 当前处于阶段 0 → 1 的过渡期。P0-P2 的三个改进可以在不改核心代码的前提下（通过修改 prompt 协议和 heartbeat governance 文档）落地，推进到阶段 1。

---

## 六、一句话带走

> Agent 自律的工程实现不是"在 prompt 里多写几条规则"，而是"在执行路径的关键节点嵌入独立于 Agent 自身的检查机制"。**高危动作白名单、独立裁决层、结构化自省**是 Butler 从"说教式自律"走向"检查点式自律"的三个最小可行改进。
