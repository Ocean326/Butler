## 核心观点

### 1. Vibe Coding 的真正风险不是"写得快"，而是"散得快"

Vibe Coding（在 LLM 对话中逐步成形代码）的效率优势毋庸置疑，但其固有弱点是**结构性漂移（structural drift）**：每一轮对话都可能把代码带向与整体架构不一致的方向。写得越多、越快，偏离初始设计的概率越大。

### 2. SDD 是 Vibe Coding 的"回家路径"

SDD（System Design Document）在这里不是传统瀑布式的重文档，而是一份**轻量但明确的参照物**：系统边界、模块职责、关键数据流、约束条件。它的作用是让每一段 vibe coding 都有一个"对照基准"——随时可以检查"我有没有偏出边界"。

### 3. SSD（规范先行）是更细粒度的自动化护栏

在 SDD 之下，SSD（Specification-first / 规范先行）进一步把约束具体化为 LLM 可读的 checklist：

- 命名约定
- 模块边界与接口契约
- 错误处理模式
- 日志/监控要求
- 测试覆盖预期

关键区别：SDD 回答"这个系统长什么样"，SSD 回答"写这个系统的代码时必须遵守哪些规矩"。两者结合才能有效约束 LLM 的自由发挥空间。

### 4. 重构 Session 应成为标准化工作单元

一次有效的 vibe coding 重构不应是"打开文件随便改"，而应遵循可复制的流程：

```
读 SDD → 对齐 SSD 规范 → 划定本次改动边界 → LLM 实现 → 对照 SDD/SSD 自检 → 产出小结
```

这个流程可以被模板化、被 agent 自动执行、被 harness 层监督。

### 5. Agent 的"自律"本质上就是内化了 SDD/SSD 的行为约束

当一个 agent 在执行前自动读取规范、在执行后自动对照 checklist 自检——它就已经在做 SDD/SSD 驱动的开发了。Agent 自律系统（self-discipline）的工程实现，就是把 SDD/SSD 注入到 agent 的 prompt 模板和验收钩子中。

## 与 Butler 当前架构的映射

| 原文概念 | Butler 对应 | 当前状态 | 改进方向 |
|---------|-----------|---------|---------|
| SDD (系统设计文档) | 架构文档 + SOUL 文件 | 有架构说明但分散 | 为每个核心子系统产出轻量 SDD |
| SSD (规范先行 checklist) | `.cursor/rules/` 规则文件 | 已有部分规则 | 可抽取为 LLM 可读的统一 checklist |
| 重构 Session 模板 | heartbeat executor 的 branch 流程 | branch 有目标和回执 | 可在 branch 协议中嵌入 SDD 引用和自检步骤 |
| 结构性漂移风险 | 多轮迭代后代码偏移 | 依赖人工 review | 可引入自动化"架构对齐检查"钩子 |
| 自检/验收 | executor 首轮自验收 | 已要求但未标准化 | 可基于 SSD checklist 做结构化自检 |

## 可行动建议

1. **为 Butler 核心子系统各写一份轻量 SDD**：心跳调度、记忆系统、工具编排、飞书适配层——每份 SDD 控制在 1-2 页，聚焦于"边界、职责、关键数据流、不变量"。这些 SDD 应放在 `butler_main/` 下可被 agent 自动引用的位置。

2. **把 `.cursor/rules/` 中的规则整理为 SSD checklist 格式**：当前规则文件偏叙述体，可以同步维护一份结构化 checklist 版本，方便 executor 在自检步骤中逐项核对。

3. **在 heartbeat branch 协议中增加可选的 `sdd_ref` 字段**：planner 分派 branch 时，如果该任务涉及某子系统的改动，可以指定对应 SDD 路径；executor 在执行前先读取该 SDD，执行后对照它做架构对齐检查。

4. **设计一个"架构漂移检测"的轻量钩子**：不需要复杂的 AST 分析，只需在每次代码改动后，让 executor 回答"本次改动是否改变了 SDD 中定义的模块边界或接口契约？"并记录答案。

## 关键引用

> "以 SSD 为方法论，先做好规范，再开始开发，是现阶段 vibe coding 提升稳定性和可控性的一种有效方式。" —— 原文副标题

> "不是让模型随手起一个实现，而是围绕 SDD 里已经给出的职责和接口，做局部实现 + 重构，减少结构性偏航。" —— Raw 推断式整理
