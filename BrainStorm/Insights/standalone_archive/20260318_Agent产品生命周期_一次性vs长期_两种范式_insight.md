## 核心观点

### 1. Agent 世界存在两种截然不同的生命周期范式

**"杀人如麻"范式（Disposable Agent）**：

母本 1 展示了一个鲜活场景——在一人公司五层构架中，上层 agent 给下属起花名、分配任务，五分钟后因为"办事不利"直接 kill 掉换新的。终端截图里，Dalton 和 Helmholtz 因未及时收敛被关闭，立刻 spawn 了 Kepler 和 Pauli 替代。

这不是 bug，而是一种有意的设计选择：
- worker agent 是无状态的、可随时替换的执行单元
- 上层 orchestrator 保持记忆和决策权，下层 agent 只负责短程执行
- 复用的是角色定义和能力模板，不是具体的 agent 实例

**"持续进化"范式（Persistent Agent Product）**：

母本 2 的科研龙虾则完全相反——同一个 agent 产品在 72 小时内持续迭代：
- Day 1：修 bug 到凌晨（标签对不上、暗色主题白弹窗、删论文标签未同步）
- Day 2：16 个新功能上线（全来自用户评论和私信）
- Day 3：测试用例从 318 → 1029 个
- 核心身份不变，能力持续增长，数据持久化

**关键区分**：
- Disposable Agent 的价值在于**并行性和隔离性**——每个 agent 失败不影响整体
- Persistent Agent 的价值在于**积累性和信任度**——越用越懂用户，数据越安全

---

### 2. Butler 不是 disposable agent，但需要管理好自己的 disposable 下属

母本 1 的作者边看终端边"狠狠共情"——大模型下面挂一群小 agent，一小时内二十几个同学"毕业"。这种场景在 Butler 的 heartbeat sub-agent 层已经是现实：

- heartbeat 每轮 spawn executor agent，执行完就释放
- 但 Butler 自身是 persistent 的——它保持记忆、维护关系、沉淀偏好
- 关键矛盾：**Butler 是长期的，但它调度的执行层是一次性的**

这种"长期核心 + 一次性外围"的架构模式，恰好就是 MAS Harness 架构的实践形态：
- Orchestration 层（Butler 主体）→ 持久化、有记忆、有人格
- Worker 层（heartbeat executor）→ 无状态、可替换、按需 spawn

---

### 3. 产品级 agent 的质量工程不是可选项

科研龙虾的 72 小时实录给出了一个清晰的质量工程范本：

| 时间线 | 动作 | 工程意义 |
|---|---|---|
| Day 1 | 连夜修 bug | 发布后第一时间修复用户反馈的数据一致性问题 |
| Day 2 | 16 个功能 | 全部来自用户反馈，不是自嗨开发 |
| Day 3 | 测试 318→1029 | 每个操作、状态变更、文件写入都有自动化测试守着 |
| 持续 | Git 自动提交 | 所有工作区文件变更可回滚 |
| 持续 | 清理 57 个低质 skill | 431 个 skill 全经人工遴选，不是 AI 批量生成 |

**关键数字**：431 个技能 · 28 个本地工具 · 13 个 API 工具 · 150 个 MCP 配置 · 6 个 Dashboard · 21 种卡片 · 1029 个测试

这对 Butler 的启发非常直接：
- **测试覆盖**：Butler 的 heartbeat / memory / prompt 管线当前测试覆盖还较薄
- **skill 质量管理**：不是 skill 越多越好，需要有遴选和淘汰机制
- **变更可回滚**：agent 对文件系统的所有操作都应有回滚能力

---

### 4. "一人公司"想象的真实挑战：心智负担管理

母本 1 描述的"一人公司五层构架"（我管 agent 管 agent 管 agent 管……）在理想中很美，但实际中最大的挑战不是技术，而是**心智负担**：
- 用户需要监控多层 agent 的状态
- 需要判断哪些 agent 该留、哪些该杀
- 需要理解每层 agent 的能力边界和失败原因

母本 1 的作者提示了 Butler 可以做的事：帮用户管"杀与留"的决策负担——
- 主动提示"这条线该复用 / 这条线可以退休了"
- 记录每层 agent 的职责与评估标准
- 减少用户"疯狂招/杀下属"的心智开销

## 与 Butler 架构的映射

| 观察点 | Butler 对应 | 当前状态 | 改进方向 |
|---|---|---|---|
| 持久核心 + 一次性外围 | Butler + heartbeat executor | 已有雏形 | 明确两层的状态边界和交接协议 |
| 测试覆盖 | tests/ 目录 | 部分覆盖 | 参考科研龙虾的"每个操作都有测试"标准 |
| Skill 质量管理 | skills/ 目录 | 有增无减 | 增加 skill 遴选/淘汰/评分机制 |
| 变更可回滚 | 工作区文件操作 | 无系统性保障 | 对关键文件操作增加 Git 自动快照 |
| 心智负担管理 | heartbeat 报告 | 信息偏多偏散 | 优化为"决策建议"而非"状态罗列" |

---

## 可执行的下一步

1. **Butler 双层生命周期文档**：明确 Butler 主体（persistent）和 heartbeat executor（disposable）的状态边界——哪些状态必须回写主体、哪些随 executor 销毁
2. **Skill 健康度仪表板**：为现有 skills 建立一个简单的评分/使用频率/最后维护日期索引，作为遴选和淘汰的依据
3. **heartbeat 报告精简**：从"发生了什么"转向"建议做什么"，减少用户的认知负担
4. **关键操作 Git 快照**：在 heartbeat 修改工作区文件前，自动 commit 当前状态作为回滚点
