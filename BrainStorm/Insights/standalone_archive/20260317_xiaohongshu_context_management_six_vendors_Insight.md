### 一、核心问题 / 主题

- 不同厂商（Manus、Cursor、Anthropic、OpenAI、Google、LangChain）在 **Agent 上下文管理** 上分别采用了什么策略？  
- 这些策略在「上下文窗口管理 / 信息检索 / 规划与一致性 / 多 Agent 与隔离 / 记忆与健壮性」五个维度上的共识与分歧是什么？

---

### 二、关键观点要点（3–6 条）

1. **共识：聪明的上下文管理比单纯放大窗口更关键**  
   - 各家都承认上下文窗口有限且成本高；长任务易产生 Context Rot。  
   - 与其一味追求更大窗口，不如通过 KV-cache、文件系统扩展记忆、动态检索等方式，把「什么时候给模型看什么信息」做精细设计。

2. **Manus：围绕 KV-cache 的生产原则**  
   - 把 KV-cache 视为「神圣资源」：缓存比未缓存便宜一个数量级，prompt 前缀要尽可能稳定。  
   - 通过「日志只追加、不重排」「Mask 而不是移除工具」等策略稳定上下文；大观察写入文件系统，只在上下文中保留轻量引用。

3. **Cursor：动态上下文发现 + 文件系统式扩展记忆**  
   - 倾向于减少预先塞入的细节，改为让 Agent 自己按需拉取上下文（有 A/B 数据支持）。  
   - 把聊天历史、技能、终端会话等都当成「可搜索文件」，并通过懒加载 MCP 工具显著降低总 token 使用（文中列出 ~46.9% 的 token 降幅）。

4. **Anthropic：注意力预算与循环内检索**  
   - 提出「注意力预算」框架，把 System Prompt 从过度工程化或过度模糊拉回一个结构化的「金发女孩区」。  
   - 主张从「推理前 RAG」转向「循环内检索」，在运行过程中不断按需取回信息，并通过代码执行 + MCP 组合调度外部工具。

5. **OpenAI：会话记忆即基础设施**  
   - 提出截断、压缩、基于状态的长期记忆三种模式，强调通过结构化状态对象（profile + notes）跨会话持久化 Agent 的「信念」。  
   - 通过文件系统中的规划文件与 Context Summary 类型组件，让长对话在多轮压缩后仍可继续保持一致性。

6. **Google 与 LangChain：极长上下文与分类学视角**  
   - Google 下注 2M–10M token 长上下文，但发现上下文变长性能仍可能下降，需要 ReadAgent/Gist Memory 等组合方案。  
   - LangChain 提供「写-拉-压缩-隔离」四操作的分类法，并强调上下文工程的评估与对比矩阵，而不是单点技术。

### 三、对 Butler 的可行动启发（2–3 条）

1. **显式把「写-拉-压缩-隔离」四操作映射到 Butler 的上下文策略**  
   - 为对话/任务执行链路补一份简短设计稿：哪些信息写入长期状态（STATE/MEMORY/task_ledger）、哪些通过检索/搜索「拉」回来、在何处做摘要压缩、哪些状态需要在多 Agent 之间隔离。  
   - 将现有实践（如 BrainStorm、self_mind、任务回执）对齐到这四个操作，减少隐式约定。

2. **借鉴 Cursor/Manus，把「文件系统作为扩展记忆」做成一等公民约定**  
   - 对长任务与多轮对话，制定默认策略：大体量信息优先落盘为文件（Raw/Working/日志），在对话与执行时通过文件引用与搜索取回，而不是死撑在单轮上下文中。  
   - 同时约定「尽量不改动历史片段，只追加新内容」，为未来的 KV-cache/Prompt-caching 优化预留空间。

3. **引入「注意力预算/上下文预算」概念到 heartbeat 规划**  
   - 在 planner 侧增加一层轻量的「上下文预算」意识：规划长任务时显式考虑每步可用的上下文与记忆预算，鼓励通过拆步、落盘与检索来控制成本，而不是无意识地累加历史。  
   - 在任务回执中记录本轮大致 token 开销与关键上下文决策，便于后续做自己的「上下文工程」评估。

### 四、后续可继续挖的开放问题（1–2 条）

1. Butler 应该优先向哪一家方案对齐（例如更像 Cursor/Manus 的「文件+检索优先」流派，还是更偏 OpenAI/Anthropic 的状态与注意力预算流派），或者需要按场景混合？  
2. 如何在本地构建一套轻量的「上下文工程评估指标」（如 token 成本、错误率、对任务完成率的影响），以便像 Manus/Cursor 那样用数据驱动上下文策略演进？
