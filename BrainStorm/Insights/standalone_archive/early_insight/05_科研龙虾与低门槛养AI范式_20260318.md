## 二、"养龙虾"现象：2026 年的养成式 AI 第一浪

### 2.1 OpenClaw 的爆发

2026 年 Q1 出现了一股席卷全网的"养龙虾"热潮。OpenClaw 是一个开源的本地优先 AI Agent 框架，突破了 AI 只能聊天的局限——它能接管键鼠权限、调用系统 API 完成复杂任务，被称为"拥有最高权限、24 小时不睡觉、会自己写代码、能自己学新技能的超级员工"。

**关键数据**：
- 深圳腾讯公司门口千人排队参与免费安装
- 用户跨越 9 岁到 70 岁——真正的"零门槛"
- 超过 4 万个 OpenClaw 实例暴露于公网（63% 存在可利用漏洞——安全隐患严峻）
- 从安装到运行仅需四步：安装 Docker → 下载程序 → 接入 AI → 调试指令

**这意味着什么**：AI Agent 的用户群不再局限于工程师和技术爱好者。当部署门槛降到"装 App"的水平时，用户基数将扩大 1-2 个数量级——但安全和信任问题也将同步放大。

### 2.2 MetaClaw：技能进化框架

MetaClaw（AIMING Lab，UNC 计算机系）是 OpenClaw 生态中最重要的进化引擎，实现了**无 GPU 集群的在线强化学习**：

```
用户对话 → MetaClaw 透明代理 → 大模型
   ↑                  ↓
   │           ┌──────────────────┐
   │           │ SkillBank        │
   │           │ ├ 技能注入（对话前）│
   │           │ └ 技能提取（对话后）│
   │           ├──────────────────┤
   │           │ Evolver LLM      │
   │           │ (分析失败对话,     │
   │           │  提取新技能)       │
   │           ├──────────────────┤
   │           │ 在线 RL          │
   │           │ (PRM 打分 →       │
   │           │  LoRA 微调)       │
   │           ├──────────────────┤
   │           │ 智能调度          │
   │           │ (检测空闲窗口,     │
   │           │  仅在睡眠/键盘     │
   │           │  闲置时更新模型)   │
   │           └──────────────────┘
   └──────────── 进化后的更好回答
```

**MetaClaw 的四个设计决策**（对 Butler 有直接借鉴价值）：

| 设计决策 | 做法 | 工程意义 |
|---------|------|---------|
| 透明代理架构 | 作为 OpenAI-compatible proxy 插入，不改底层模型 | 零侵入——现有系统架构不变 |
| 分层技能库 | 对话前注入相关技能，对话后提取新技能 | 技能是**活的**，随使用演化 |
| 空闲时更新 | 检测睡眠/键盘闲置/日历会议，仅在空闲窗口做模型更新 | 不影响用户实时体验 |
| 多 Agent 变体 | 支持 7 种 Agent 类型（OpenClaw/CoPaw/IronClaw/...） | 同一进化引擎服务不同角色 |

### 2.3 傅盛的 14 天养成日记

傅盛公开的"AI 龙虾三万养成日记"是"用户-AI 共同演化"的最佳实证：

| 指标 | 数值 |
|------|------|
| 调教周期 | 14 天 |
| 消息量 | 1,157 条，约 22 万字 |
| 起点 | 1 个单功能 Agent |
| 终点 | 8 Agent 团队（内容、排版、审核、发布...） |
| 业务成果 | 公众号日更、涨粉 5,000、百万阅读爆款 |

**关键洞察**：傅盛的"调教"过程本质上是一个**双向适应**——用户在教 AI 的同时也在改变自己的工作方式。14 天后，不只是 AI 变强了，**用户的工作模式也被重塑了**。这就是"共同演化"的含义。

## 三、科研龙虾：Skill 型 Agent 产品的质量工程标杆

### 3.1 72 小时迭代战报

刘思源的科研龙虾（学术 AI 助手）在上线后 72 小时内的迭代轨迹，是 Agent 产品质量工程的教科书级案例：

| 时间 | 动作 | 工程规律 |
|------|------|---------|
| Day 1 | 连夜修 bug（标签数量不对、暗色主题白弹窗、删论文标签未同步） | Agent 管理高价值数据时容错空间极小——一次错误可能丢失用户文献 |
| Day 2 | 16 个新功能上线 | 全部来自用户评论和私信——用户反馈是 Skill 迭代的最高效信号源 |
| Day 3 | 测试用例 318 → 1,029 | 从"能 demo"到"可日用"的分水岭——自动化测试是信任基础设施 |
| 持续 | Git 自动提交 | 所有工作区文件变更可回滚——将不可逆风险降级为可控风险 |
| 持续 | 清理 57 个低质 Skill | Skill 价值不在数量，在于可信度——人工遴选是当前阶段的必要保障 |

**核心数字**：431 个学术技能 · 28 个本地工具 · 13 个 API 工具 · 150 个 MCP 配置 · 6 个 Dashboard · 21 种消息卡片 · 1,029 个单元测试

### 3.2 从科研龙虾提炼的 Skill 管理三原则

1. **遴选优于堆量**：431 个 Skill 每个经人工遴选，三天又清理 57 个。AI 生成 Skill 的时代，人工把关不是低效，而是信任的唯一可信路径
2. **测试粒度对齐到 Skill 级别**：每个 Skill 的每个操作路径都有自动化验证，不只测端到端场景
3. **退化是持续风险**：Skill 质量不是"上线即定型"，而是持续监控、持续淘汰

### 3.3 Cron 预设：从"工具"到"持续服务"的关键设计

科研龙虾的预设任务模板——arXiv 日扫、引用追踪、截止提醒、组会准备、周报——揭示了 Agent 产品的关键跃迁：

> **Agent 不只是被动响应用户指令，而是主动在正确的时间做正确的事。**

这与 Butler 的 heartbeat 理念高度共振：heartbeat 本质上就是"Agent 主动在后台持续工作"的基础设施。

## 四、两种生命周期范式的碰撞

### 4.1 Disposable vs Persistent：根本分野

| 维度 | Disposable Agent（用完即弃） | Persistent Agent（持续存活） |
|------|---------------------------|---------------------------|
| 生命期 | 分钟到小时 | 天到月 |
| 状态 | 无状态，每轮重置 | 有状态，跨会话持久化 |
| 记忆 | 无 | 长期记忆 + 偏好累积 |
| 核心价值 | 并行性、隔离性、低风险 | 累积性、信任度、连续性 |
| 失败模式 | 失败即弃 | 失败需恢复 |
| 代表 | heartbeat executor / worker agent | Butler 主体 / 科研龙虾 |

**2026 量化证据**（Computer Agents Blog）：

| 任务类型 | Disposable 成功率 | Persistent 成功率 | 差距 |
|---------|-----------------|------------------|------|
| 单次研究 | 92% | 94% | ≈持平 |
| 多文件代码重构 | 68% | 91% | +34% |
| 7 天连续日报 | 22% | 87% | **≈4×** |
| 错误自恢复率 | 41% | 78% | +90% |

**关键结论**：对需要跨轮次上下文保持的任务，Persistent 架构的优势是压倒性的。但简单的一次性任务两种架构等效。**系统需要同时支持两种模式。**

### 4.2 "持久核心 + 一次性外围"双层架构

```
┌──────────────────────────────────────────────────────┐
│              Persistent Core（持久核心）                │
│  ┌──────────────────────────────────────────────┐    │
│  │ · 长期记忆      · 用户偏好    · 人格与价值观   │    │
│  │ · 关系连续性    · 任务台账    · 经验资产库     │    │
│  └──────────────────────────────────────────────┘    │
│                     ↕ 状态交接协议                      │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐                │
│  │Worker│ │Worker│ │Worker│ │Worker│  ← 按需 spawn   │
│  │  A   │ │  B   │ │  C   │ │  D   │     / kill      │
│  └──────┘ └──────┘ └──────┘ └──────┘                │
│          Disposable Workers（一次性执行层）              │
└──────────────────────────────────────────────────────┘
```

**Butler 的定位断言**：

> Butler 不是 Disposable Agent，而是 Persistent Partner。它的核心差异化在于跨会话记忆、关系连续性和累积学习——这不是附加功能，是存在基础。但 Butler 同时需要管理好自己的 Disposable 下属（heartbeat executor、sub-agent）。

## 五、用户-AI 共同演化的四个层次

### 5.1 共同演化的递进模型

```
L0  工具适配        用户学会使用 AI → AI 被动服务
    ↓
L1  习惯塑造        AI 持续提供价值 → 用户工作模式开始改变
    ↓
L2  双向适应        用户调教 AI → AI 记住偏好 → 用户信任加深 → AI 获得更多自主权
    ↓
L3  协同进化        AI 从历史中提取最优策略 → 用户获得越来越好的服务 → 新的协作模式涌现
```

| 层次 | 典型案例 | 关键机制 | Butler 当前位置 |
|------|---------|---------|---------------|
| L0 工具适配 | ChatGPT 初次使用 | 用户学习 prompt 技巧 | 已超越 |
| L1 习惯塑造 | 科研龙虾 Cron 预设 | 定时任务改变用户日常流程 | heartbeat 已具雏形 |
| L2 双向适应 | 傅盛 14 天养成 | 记忆 + 偏好沉淀 + 信任升级 | self_mind + memory 已有基础 |
| L3 协同进化 | MetaClaw SkillBank 进化 | 经验提取 → 策略优化 → 行为改善 | **尚未实现** |

### 5.2 共同演化中的去能化风险

主线④（Anthropic 去能化研究）警示：共同演化的阴暗面是**用户主体性的侵蚀**。

| 演化层次 | 积极面 | 去能化风险 |
|---------|--------|-----------|
| L1 习惯塑造 | 效率提升 | 用户可能对"AI 没提醒我"形成依赖 |
| L2 双向适应 | 个性化服务 | AI"太懂你"导致信息茧房 |
| L3 协同进化 | 持续改善 | 用户从决策者退化为审批者 |

**Butler 的平衡策略**：在用户的**学习区间**内，优先解释而非代劳；在用户的**重复劳动区间**内，优先自动化。`deliver-not-guide` 规则需要与去能化防护协调：交付优先 ≠ 剥夺用户理解权。

## 六、"低门槛"的三重含义

### 6.1 部署门槛低

| 传统 AI Agent | 养成式 AI Agent |
|-------------|----------------|
| 需要编程经验 | 零编码部署 |
| 需要 GPU 集群 | 消费级硬件 / 纯 API |
| 需要理解模型原理 | "安装 → 对话 → 自动进化" |
| 面向工程师 | 面向所有人 |

科研龙虾、OpenClaw、MetaClaw 都在降低部署门槛。但门槛降低带来的安全问题不容忽视——4 万+公网暴露实例中 63% 存在漏洞。

### 6.2 使用门槛低

**autoresearch 的启示**：作者背景是 AI 产品经理（零 ML 理论），在 RTX 5060 Ti（16GB，实际占用 6.1GB）上成功运行。自驱实验场类工具的用户群可以远超传统 ML 工程师。

**核心模式**：搭建自动试验场 → 定义评估指标 → 让 AI 在约束内自主搜索最优策略 → 人类审查结果。

### 6.3 认知门槛低

**"养龙虾"的隐喻力量**：把复杂的 AI Agent 养成过程包装成"养宠物"的认知框架——降低的不是技术门槛，而是**心理门槛**。用户不需要理解 LoRA 微调或在线 RL，只需要"和 AI 聊天"就能让它变强。

## 七、autoresearch 范式：竞技场思维的工程化

### 7.1 核心抽象

```
目标（What to optimize）
  + 规则（Constraints & safety bounds）
    + 搜索空间（What can be varied）
      + 自动评估（How to score results）
        = 自主进化循环
```

**autoresearch 实证数据**：

| 指标 | 数值 |
|------|------|
| 核心代码 | ~630 行 |
| 实验频率 | 每 5 分钟一轮 |
| AI 重发现人类技术 | ~17 小时 vs 人类 ~8 年 |
| 改善幅度 | val_bpb 1.506 → 0.975（~35%） |
| 项目 Stars | 37,300+ |

### 7.2 对 Butler 的映射

| autoresearch 机制 | Butler 可映射方向 |
|------------------|-----------------|
| 5 分钟自动实验周期 | heartbeat 周期循环已具备基础设施 |
| 自动保留好方案 / 丢弃差方案 | skill / prompt 策略的 A/B 评估 |
| 人类只设目标和搜索空间 | 用户表达意图，Butler 在工具空间内自主寻找最优路径 |
| 630 行极简核心 | 复杂度应在 skill/配置层，不在核心循环 |
| results.tsv 结果留痕 | task_ledger 理念一致——每轮结果必须持久化 |

### 8.1 现状对标

| 跨会话记忆 | memory_manager + self_mind | 基础实现 | ★★★☆☆ |
| 偏好累积 | L1/L2 偏好 + 用户模型 | 有沉淀通道 | ★★☆☆☆ |
| Skill 进化 | skills/ 目录 | 有增无减，无淘汰 | ★☆☆☆☆ |
| 在线学习 | 无 | 不存在 | ☆☆☆☆☆ |
| 经验驱动调度 | task_ledger | 有记录，无策略提取 | ★☆☆☆☆ |
| 用户信任度追踪 | 无 | 不存在 | ☆☆☆☆☆ |
| Cron 预设服务 | heartbeat 周期循环 | 已有基础设施 | ★★☆☆☆ |
| 测试覆盖 | tests/ 目录 | 部分覆盖 | ★★☆☆☆ |
| 变更可回滚 | 工作区文件操作 | 无系统性保障 | ★☆☆☆☆ |
| 共同演化意识 | self_mind current_context | 有，但未结构化利用 | ★☆☆☆☆ |

### 8.2 Butler 的养成式 AI 路线图

#### 阶段 0：经验沉淀基础（现在 → 2 周）

**不改代码，只加数据记录**

1. **task_ledger 增加经验字段**：`outcome_score`（0-1）+ `strategy_used`，executor 回执中自评。积累 50+ 条后有数据基础做分析
2. **Skill 使用日志**：哪个 skill、什么场景、成功/失败、耗时，存入 `工作区/skill_usage_log.jsonl`
3. **失败模式归档**：结构化记录失败类型、根因、补救措施，存入 `工作区/failure_patterns.jsonl`

#### 阶段 1：微实验试点（2-4 周）

**在 heartbeat 非关键路径引入受控实验**

4. **Prompt 变体 A/B**：选一类重复性任务（如 BrainStorm 提炼），同一任务用 2 种策略执行，人类审查保留更好的
5. **Skill 选择优化**：任务可被多个 skill 覆盖时，记录选择理由和结果，建立"场景 → 最优 skill"映射

#### 阶段 2：养成闭环（1-2 月）

**引入自动化的策略评估与选择**

6. **经验导航器**：新任务到来时自动检索历史相似任务的执行记录，推荐历史最优策略
7. **Skill 健康度仪表板**：评分 / 使用频率 / 最后维护日期索引，作为遴选和淘汰依据
8. **渐进式信任机制**：基于历史成功率自动调整任务类型的自治等级

#### 阶段 3：协同进化（季度级）

**Butler 和用户一起变强**

9. **用户行为模式追踪**：识别用户的"学习区"和"重复劳动区"，动态调整辅助策略
10. **Skill 达尔文进化**：同一功能的 skill 允许 2-3 个版本并存，按使用效果自动排序
11. **Cron 预设服务模板**：设计 Butler 版的"开箱即用周期性服务"

## 九、开放问题

1. **养成的边界在哪**？MetaClaw 允许 AI 在对话中自动学习进化——但学到了"坏习惯"怎么办？Butler 的自我进化是否需要一个"品格锁"（SOUL 不可变量）来防止养偏？

2. **低门槛 × 高风险**：OpenClaw 4 万+公网实例中 63% 有漏洞。当 Agent 能力增长但安全意识不同步增长时，低门槛可能变成高风险。Butler 如何在降低使用门槛的同时维持安全基线？

3. **谁在养谁**？傅盛 14 天 22 万字的"调教"——是他在训练 AI，还是 AI 在重塑他的工作习惯？共同演化的权力动态如何在产品设计中显式化？

4. **Persistent Partner 的退出成本**：用户在 Disposable Agent 上没有沉没成本，但在 Persistent Agent 上投入了大量时间和数据。这种粘性是好的（关系连续性）还是坏的（去能化）？

5. **测试的可扩展性**：科研龙虾 1,029 个测试覆盖 431 个 Skill。Butler 当前 skill 数量少但复杂度高——测试策略应该是"少而全"还是"多而轻"？

---

## 十、关键论点总结

### 三句话定位本主线

1. **养成式 AI 是 2026 年正在发生的范式转移**——从"工具"到"伙伴"不是隐喻，OpenClaw/MetaClaw/科研龙虾已经在数百万用户中验证了可行性。

2. **Butler 天然处于养成式 AI 的定位**——跨会话记忆、关系连续性、heartbeat 持续服务，这些已有机制是养成式 AI 的工程基础。但"养成闭环"（经验积累 → 策略优化 → 行为改善）尚未形成。

3. **养成式 AI 的核心风险不是技术失败，而是去能化和安全**——越是"懂你"的 AI，越可能在不知不觉中替代你的判断力。Butler 必须在"越来越强"的同时，显式保护用户的主体性。
