## 核心观点

### 1. 上下文管理才是 Agent 真正的主战场，不是模型能力

六家公司（Manus、Cursor、Anthropic、OpenAI、Google、LangChain）都在解同一道题：**LLM 应该什么时候看到什么信息，信息应该如何组织**。做出最好 Agent 的团队都在不断简化而非堆叠复杂度——Manus 重写了五遍框架，每一遍在删东西。

**密度判断**：这不是一个"未来可能重要"的方向，而是当下所有头部 Agent 产品已经在做的核心工程。

### 2. 四个行业共识正在形成

| 共识 | 具体表现 |
|------|---------|
| **文件系统 = 扩展记忆** | 大观察写入文件、上下文只留轻量引用；Manus/Cursor/Anthropic/OpenAI 均采用 |
| **动态优于静态检索** | 从预填 RAG 转向循环内按需拉取；Cursor 的懒加载 MCP 使 token 减少 46.9% |
| **长任务用持久化计划文件** | 跨窗口的结构化 spec/plan 作为"锚点"，防止上下文腐烂后目标漂移 |
| **错误保留不清理** | 保留错误轨迹而非清理，便于 LLM 隐式信念更新、避免重蹈覆辙 |

### 3. KV-Cache 是被低估的成本杠杆

Manus 将"KV-Cache 神圣"列为第一原则。缓存 token 与未缓存成本差约 **10 倍**（0.3 vs 3/MTok）。工程启示：prompt 前缀保持稳定、日志只追加不重排、即使 JSON key 顺序变化也会击穿缓存。

### 4. 长上下文 vs 精简压缩仍是核心争议

Google 押注"富足"（2M token 窗口），其余五家押注"精简工程"。Google 自己的研究也显示，上下文变长时性能仍可能下降 15-47%。这意味着：**更大的窗口不能替代更聪明的上下文管理**，两条路径的张力将长期共存。

### 5. 工具过载问题有两条对立解法

| 方案 | 代表 | 思路 |
|------|------|------|
| Logit 屏蔽 | Manus | 永久加载所有工具定义到上下文，通过解码时 logit mask 控制每步可用性 |
| 懒加载 | Cursor | 只预加载工具名，按需取完整定义；A/B 证实 token 减少近半 |

两种方案的取舍取决于工具数量级和 KV-Cache 策略。Butler 当前工具规模偏小，短期可用懒加载，长期若工具扩展到百级别需重新评估。

## 对 Butler 的启发

### 启发 1：将"上下文工程"作为 Butler 架构的一等公民

当前 Butler 的上下文策略散布在对话管理、心跳编排、memory_manager 等多个模块中，缺乏一个统一的"上下文预算"概念。可参考 Anthropic 的"注意力预算框架"，为 Butler 定义：
- 每类任务的 token 预算（对话/心跳/self_mind 各多少）
- 何时触发压缩/摘要
- 哪些信息用文件系统承载、哪些保留在窗口内

### 启发 2：skill/tool 发现机制应从静态加载走向按需拉取

Cursor 的经验最直接可用：skill 定义存为文件、通过搜索发现而非全量注入 system prompt。Butler 当前的 skills 目录已经有这个雏形（`skills.md` 索引 → 按需读取 `SKILL.md`），但心跳和对话的 prompt 组装中仍有大量预填信息可以精简。
