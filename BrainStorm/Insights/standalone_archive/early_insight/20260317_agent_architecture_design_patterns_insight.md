## 核心观点

### 1. Agent 四件套是唯一的底层范式

10 个不同的 AI Agent 项目拆开后，核心架构高度同构：**LLM + Tool System + Agent Loop + Memory**。这不是某种"最佳实践推荐"，而是工程上的必然收敛——要让 LLM 自主完成任务，就必须有工具、有循环、有记忆，形态上没有第二条路。

**密度判断**：四件套是描述 Agent 的最小完备集。理解这一点后，关注点应该从"用什么框架"转向"四件套中每一件做到什么深度"。

### 2. 真正的差异在工具生态和任务规划能力

项目之间的差距不在于架构是否精巧，而在于两个维度：

| 差异维度 | 含义 | 实际影响 |
|---------|------|---------|
| **工具生态丰富度** | Agent 能调用的工具数量与质量 | 决定能力边界——你有什么工具，Agent 就能做什么事 |
| **任务规划能力** | 是否支持多步拆解、子任务委派、多 Agent 协作 | 决定复杂度天花板——单步能做的，多步才做得完 |

### 3. 单轮迭代到多轮迭代是关键跨越

单次「感知→规划→执行→反馈」只能完成 Demo 级任务。真实场景几乎都需要多轮迭代，且不能靠硬编码规则驱动（否则退化为预定义工作流）。Agent 的价值正是"自主决策下一步做什么"。

**当前大模型的现实**：完全自主能力不足 → 必须用架构和机制弥补。设计策略应该是：先给足自由度和自主条件，再通过 Human-in-the-Loop 做必要约束。

### 4. 四条架构设计原则

| 原则 | 要求 | 反面教材 |
|------|------|---------|
| **自由优先** | 先为 Agent 提供自主条件，架构的首要作用是发挥能力而非限制；再补 Human-in-the-Loop | 一上来就堆规则，让 Agent 退化为工作流执行器 |
| **工具原子化** | 每个工具只做一件事、保持通用性，不在工具内塞特定业务流程 | 把"查价格 + 比三家 + 生成报告"写成一个大工具 |
| **适配器解耦** | 通用模块（规划/执行/反馈）保持解耦，业务适配通过外围配置/适配器层承接 | 把某个客户的特殊规则硬编码进通用 planner |
| **MAS 自底向上** | 多 Agent 架构从消息/状态/工具管理开始，到协调层调度策略，再到应用层业务流程 | 先画好"5 个 Agent 的职责"，再发现底层连消息怎么传都没想清楚 |

### 5. 最小可用 Agent 远比想象中简单

一个最小 Agent 系统几百行代码就可以实现。"觉得 Agent 很复杂"通常是被框架层和配置层的表面厚度吓到了。理想的 Agent 框架应把通用部分（循环、工具封装、上下文管理）打包好，让开发者把精力放在**写好工具与定义清晰任务**上。

## 对 Butler 的映射

### 映射 1：四件套在 Butler 中的落点

| 四件套 | Butler 当前实现 | 改进方向 |
|--------|---------------|---------|
| **LLM** | 飞书对话 + 心跳的模型调用 | 多模型策略（大/小模型分工）尚未显式设计 |
| **Tool System** | `skills/` 目录 + MCP 集成 | 已有原子化雏形，需持续维护工具索引与发现机制 |
| **Agent Loop** | heartbeat 循环 + 对话主循环 | 多轮迭代假设已内建于 heartbeat，但对话侧还偏单轮 |
| **Memory** | `memory_manager` + `local_memory` + `BrainStorm` | 分层已有，但"注意力预算"和动态压缩策略还在探索 |

### 映射 2：工具原子化的实践检查

Butler 的 `skills/` 设计已符合"原子能力 + 适配器"分层精神：每个 skill 有独立的 `SKILL.md`（定义 + 用法）和 `scripts/`（原子脚本）。需要持续警惕的是：

- 不要在单个 skill 脚本内塞入多步业务编排逻辑
- 业务特化规则应放在 `./工作区` 或项目目录的配置文件中，而非 skill 内部

### 映射 3：多轮迭代在 task_ledger 中的体现

`task_ledger.json` 已支持任务状态追踪。下一步应结构化记录每次重试/调整的**原因与变更点**，使之成为"经验数据资产"而非仅仅是状态标记。这直接对接 MAS Harness Insight 中"经验飞轮"的概念。

### 映射 4：MAS 自底向上 → Butler 演进路线

当前 Butler 处于"主 Agent + sub-agent 层（heartbeat-executor）"阶段。要向多 Agent 协作（AgentTeam / 专家 Agent）演进，应按照"自底向上"原则：

1. 先夯实消息传递与状态管理（当前 heartbeat 回执格式已是雏形）
2. 再定义协调策略（planner 与 executor 的分工协议已存在）
3. 最后接入应用层业务流程（具体场景的 skill 编排）
