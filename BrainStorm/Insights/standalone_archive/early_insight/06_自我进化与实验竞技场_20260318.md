## 一、自我进化的概念谱系

### 1.1 从"工具调用"到"自主进化"：Agent 能力的四个阶段

2026 年 Q1 的研究与开源实践揭示了 Agent 能力的一条清晰演进轨迹：

```
阶段 0：工具调用器         人类写指令 → Agent 调用工具 → 返回结果
       │
阶段 1：任务执行器         人类写目标 → Agent 自主分解步骤并执行 → 返回产出
       │
阶段 2：策略优化器         人类设竞技场 → Agent 在约束内搜索最优策略 → 保留优解
       │
阶段 3：自我改写器         Agent 在元层面修改自身的方法论/代码/架构 → 递归自我进化
```

Butler 当前主要在阶段 1（heartbeat executor 执行 planner 拆分的任务）。本主线的核心问题是：**Butler 能否以及应该在多大程度上向阶段 2-3 演进？**

### 1.2 自我进化不等于自我失控

一个关键区分：**受控自我进化 ≠ 失控自我修改**。2026 年所有成功的自我进化框架都内建了严格的边界：

| 边界类型 | 作用 | 失控后果 |
|---------|------|---------|
| **搜索空间约束** | 限定 Agent 能改什么（代码、参数、策略） | 无约束修改导致系统崩溃 |
| **评估函数** | 量化改变是好是坏 | 无评估 = 随机漫步 |
| **回退机制** | 差的改变可以撤销 | 不可逆修改 = 单点故障 |
| **人类审批卡口** | 高风险变更需人类确认 | 完全自主修改 = 对齐风险 |

## 二、四大自我进化范式

### 2.1 范式 A：固定代码·参数搜索（Autoresearch）

Karpathy 的 autoresearch（2026-03-09 发布，37,300+ Stars）代表最简洁的自我进化范式：

**核心循环**：
```
while True:
    agent 修改 train.py  ←  参照 program.md 的策略指引
    运行 5 分钟实验
    测量 val_bpb
    if 改善 → 保留; else → 回退
```

**关键设计决策**：
- 人类写 `program.md`（策略指引），Agent 改 `train.py`（代码实现）——**分工边界极清晰**
- 5 分钟时间盒 = 快速试错的基础设施约束
- 评估指标单一且可自动化（val_bpb）
- 核心仅 630 行——复杂度在搜索空间里，不在框架里

**实证结果**：
| 实验 | 时长 | 改善 | 关键发现 |
|------|------|------|---------|
| 隔夜 H100 | 10.5h | val_bpb -2.82% | 重新发现 RMSNorm 等人类 8 年技术 |
| 126 轮实验 | ~10h | 0.9979 → 0.9697 | 持续递增式改善 |
| nanochat 加速 | N/A | 2.02h → 1.80h (-11%) | 算法+参数联合优化 |

**Butler 映射**：heartbeat 循环 ≈ autoresearch 实验循环的原型。差异在于 Butler 目前没有自动化的"改善 vs 回退"评估机制。

### 2.2 范式 B：自我改写·递归自指（Gödel 系）

**Gödel Agent**（ACL 2025）和 **Darwin Gödel Machine**（DGM，ICLR 2026）代表了更激进的路线——Agent 不仅优化参数，还**修改自身的逻辑和代码**。

**Gödel Agent 核心思想**：
- LLM 动态修改自身的逻辑和行为，突破固定 pipeline 的限制
- 以高层目标为引导，在整个 Agent 设计空间中搜索
- "自指"意味着 Agent 可以重写自己的推理逻辑、工具选择策略、甚至 prompt 结构

**Darwin Gödel Machine 关键数据**：
| 指标 | 基线 | DGM 优化后 | 提升 |
|------|------|-----------|------|
| SWE-bench 编码能力 | 20.0% | 50.0% | +150% |
| Polyglot 多语言 | 14.2% | 30.7% | +116% |

DGM 的创新在于将**达尔文进化**（种群多样性 + 自然选择 + 适应度竞争）应用于 Agent 自身的代码修改：
- 维护一个 Agent 变体的"种群"
- 每代 Agent 修改自己的代码
- 用编码 benchmark 做适应度评估
- 保留优胜者，淘汰劣势者

**Butler 映射**：DGM 的"种群进化"思路对 Butler 的 skill 管理有直接启发——不同 skill 版本可以像种群一样竞争，在实际使用中保留效果好的、淘汰差的。

### 2.3 范式 C：元进化·方法论重写（Ouroboros）

**Ouroboros**（2026-03-15 开源）在 autoresearch 之上增加了一个关键的**元层**——它不仅优化实验，还**重写实验本身的方法论**。

```
autoresearch：     固定 program.md → 变 train.py → 评估
Ouroboros：        变 program.md → 变 train.py → 评估 → 回写 program.md
                   ↑                                          │
                   └──────── 世代循环（generational loop）────────┘
```

这是从"在规则内搜索"到"连规则本身也进化"的跃迁。在传统 ML 术语中，autoresearch 做的是超参数搜索，Ouroboros 做的是**算法搜索/AutoML 的元层**。

**风险提示**：方法论自我重写是最高风险的自我进化形态。如果评估函数本身有偏差，Ouroboros 可能沿着错误方向不断"优化"——这就是 Goodhart 定律在 Agent 领域的实例化。

**Butler 映射**：Butler 的 `heartbeat_upgrade_request.json` 审批机制恰好是为这一层设计的——当 Agent 认为需要修改自身的"方法论"（核心代码/配置/prompt），必须经过人类审批。这不是限制，而是安全的必要边界。

### 2.4 范式 D：经验驱动·自我提问（AgentEvolver）

ModelScope 的 **AgentEvolver** 提供了一条更工程化的路线，通过三种自进化机制实现性能提升与成本下降：

| 机制 | 说明 | 效果 |
|------|------|------|
| **Self-Questioning** | 自动生成新测试任务，扩展训练分布 | 覆盖更多边界情况 |
| **Self-Navigating** | 基于历史经验引导探索，避免重复失败 | 收敛速度 ↑，成本 ↓ 70-90% |
| **Self-Attributing** | 归因分析，识别哪些改变导致了改善 | 精准定位有效策略 |

**关键洞察**：AgentEvolver 的三机制不需要修改 Agent 自身代码，而是通过**经验积累和策略选择**实现进化。这是最安全的自我进化形态——Agent 本体不变，但行为因积累的经验而持续改善。

**Butler 映射**：这最接近 Butler 的现实路径。Butler 可以通过 task_ledger 积累执行经验，从历史成功/失败中提取模式，逐步优化 skill 选择、prompt 组装和任务调度策略——而不需要修改核心代码。

## 三、实验竞技场：将"自我进化"工程化的核心抽象

### 3.1 竞技场的四要素

所有成功的自我进化框架都包含四个结构要素：

```
┌─────────────────────────────────────────────────────┐
│                  实验竞技场抽象                        │
│                                                      │
│  ┌──────────────┐    ┌──────────────┐               │
│  │  目标函数     │    │  搜索空间     │               │
│  │ (Objective)   │    │ (Search      │               │
│  │  "什么算好"   │    │  Space)      │               │
│  │              │    │ "能改什么"    │               │
│  └──────┬───────┘    └──────┬───────┘               │
│         │                   │                        │
│         └─────────┬─────────┘                        │
│                   ↓                                   │
│         ┌──────────────────┐                         │
│         │  进化循环         │                         │
│         │ (Evolution Loop)  │                         │
│         │ "试 → 评 → 留/弃" │                         │
│         └──────┬───────────┘                         │
│                ↓                                      │
│         ┌──────────────────┐                         │
│         │  安全边界         │                         │
│         │ (Safety Bounds)   │                         │
│         │ "不能碰什么"      │                         │
│         └──────────────────┘                         │
│                                                      │
└─────────────────────────────────────────────────────┘
```

| 要素 | autoresearch | DGM | Ouroboros | AgentEvolver |
|------|-------------|-----|----------|-------------|
| 目标函数 | val_bpb | SWE-bench 分数 | val_bpb + 方法论评分 | 任务完成率 + 成本 |
| 搜索空间 | train.py 代码 | Agent 自身代码 | train.py + program.md | 策略选择 + 经验库 |
| 进化循环 | 5 分钟时间盒 | 种群竞争 | 世代循环 | 经验导航 |
| 安全边界 | 固定 prepare.py | benchmark 回归 | 人类审查世代 | 归因分析 |

### 3.2 竞技场的工程深水区：评估函数设计

所有自我进化系统的最大瓶颈不是搜索能力，而是**评估函数的质量**。

| 评估类型 | 适用场景 | 限制 |
|---------|---------|------|
| **数值指标**（val_bpb、准确率） | ML 训练、性能优化 | 易 Goodhart 化，指标不等于目标 |
| **Benchmark 套件**（SWE-bench） | 编码能力、通用能力 | 覆盖有限，过拟合风险 |
| **LLM-as-Judge** | 开放式任务、质量评估 | 主观性、偏见继承 |
| **人类评审** | 最高质量但最慢 | 不可规模化 |
| **复合评估**（CLASSic 五维） | 生产级 Agent | 权重分配仍需人类判断 |

**核心矛盾**：Agent 越强大，越需要精确的评估来引导进化方向；但精确评估本身就是最难的问题之一。

**DR-Arena 的解法**（arXiv 2601.10504）：动态信息树 + 自适应升级循环——评估难度随 Agent 能力自动提升，避免评估饱和。与人类偏好的 Spearman 相关性达 0.94。

### 3.3 竞技场可迁移性：不只是 ML

autoresearch 的原始论域是 ML 训练，但竞技场抽象可跨域迁移：

| 应用领域 | 目标函数 | 搜索空间 | 时间盒 |
|---------|---------|---------|--------|
| ML 训练 | val_bpb | 模型架构/超参 | 5 分钟 |
| 代码质量 | pass@k / 回归测试通过率 | 代码变体 | 分钟级 |
| Prompt 优化 | 任务完成率 / 人类评分 | Prompt 变体 | 秒级 |
| Skill 策略选择 | 用户满意度 / 完成时间 | Skill 调用链 | 轮次级 |
| 产品增长 | 转化率 / 留存率 | 策略参数 | 天级 |
| 知识整合 | 信息密度 / Butler 相关度 | 提炼策略 | 轮次级 |

## 四、MLEvolve 与 DR-Arena：自动化研究的生产级实践

### 4.1 MLEvolve：Agent 做 ML 研究的最强基准

MLEvolve（2026-02 发布）是目前 MLE-bench 排行榜 #1，以 61.33% 的成绩超越所有其他系统：

**核心机制**：
- **蒙特卡洛图搜索（Monte Carlo Graph Search）**：在解空间中构建图结构，通过采样和回溯寻找最优路径
- **多 Agent 协作**：多个 Agent 并行探索不同搜索分支
- **经验驱动记忆**：历史实验结果作为后续搜索的导航信号
- **跨分支融合**：不同探索路径的优秀组件可以交叉融合

**设计启示**：MLEvolve 证明"搜索+记忆+协作"的组合在自动化研究中比"更强的单体模型"更有效。

### 4.2 DR-Arena：评估本身的自动化

DR-Arena 解决了一个元问题——当 Agent 越来越强，**谁来评估 Agent？**

传统 benchmark 是静态的：出题 → Agent 答 → 打分。DR-Arena 引入动态评估：
- **实时信息树**：从网络趋势中动态生成研究任务
- **自适应升级**：根据 Agent 表现自动调整任务难度
- **深度 + 广度双维度**：不只测"答对了没"，还测"研究有多深""覆盖有多广"

**Butler 启发**：Butler 的 heartbeat 自验收可以借鉴 DR-Arena 的动态评估思路——验收标准不应固定不变，而应随 Butler 能力提升而自动升级。

### 5.2 Butler 自我进化的分层设计

基于四大范式的分析，Butler 的自我进化应按风险从低到高分层推进：

```
┌────────────────────────────────────────────────────────────────┐
│  L3：方法论重写（Ouroboros 级）                                  │
│  · 修改核心 prompt / 执行逻辑 / 架构                             │
│  · 风险：最高   人类审批：强制                                    │
│  · Butler 时机：远期 / 需充分验证后才可探索                       │
├────────────────────────────────────────────────────────────────┤
│  L2：代码/配置自适应（Gödel Agent / DGM 级）                     │
│  · 修改 skill 实现 / 配置参数 / prompt 模板                      │
│  · 风险：中高   人类审批：必须                                    │
│  · Butler 时机：中期 / heartbeat_upgrade_request 已覆盖          │
├────────────────────────────────────────────────────────────────┤
│  L1：策略搜索（autoresearch 级）                                 │
│  · 在固定工具集内选择最优执行路径                                 │
│  · 风险：中低   人类审批：按需                                    │
│  · Butler 时机：近期 / 可在 heartbeat 中试点                     │
├────────────────────────────────────────────────────────────────┤
│  L0：经验积累（AgentEvolver 级）                                 │
│  · 从历史成功/失败中提取模式，不修改任何代码                      │
│  · 风险：最低   人类审批：不需要                                  │
│  · Butler 时机：现在就可以做                                     │
└────────────────────────────────────────────────────────────────┘
```

### 5.3 Butler 的三条进化通道

| 通道 | 进化对象 | 评估方式 | 安全机制 | 优先级 |
|------|---------|---------|---------|--------|
| **A. Skill 进化** | skill 的选择策略、参数配置 | 任务完成率 + 用户反馈 | 多版本并存、A/B 对比、旧版回退 | 高 |
| **B. Prompt 进化** | prompt 组装策略、上下文裁剪 | LLM-as-Judge + 人类抽检 | 影子模式运行、不直接上线 | 中 |
| **C. 架构进化** | 执行拓扑、记忆策略、任务分解 | benchmark + 回归测试 | heartbeat_upgrade_request 强制审批 | 低（远期） |

## 六、可执行的行动路线

### 阶段 0：经验积累基础设施（现在 → 2 周）

**不修改任何代码，只增加数据沉淀**

1. **task_ledger 经验字段**
   - 为每条 task_ledger 记录增加 `outcome_score`（0-1）和 `strategy_used` 字段
   - 不需要自动打分——executor 在回执中自评即可
   - 积累 50+ 条带评分记录后，才有数据基础做后续分析

2. **Skill 使用日志**
   - 在 skill 调用时记录：哪个 skill、什么场景、成功/失败、耗时
   - 存入 `工作区/skill_usage_log.jsonl`
   - 目标：建立 skill 效果的经验基线

3. **失败模式归档**
   - heartbeat 执行失败时，结构化记录失败类型、根因分析、采取的补救措施
   - 存入 `工作区/failure_patterns.jsonl`
   - 目标：避免重复犯同类错误

### 阶段 1：微实验试点（2-4 周）

**在 heartbeat 的非关键路径引入受控实验**

4. **Prompt 变体 A/B**
   - 选取一类重复性任务（如 BrainStorm 提炼）
   - 同一任务用 2 种 prompt 策略执行，比较产出质量
   - 人类审查结果后，保留更好的策略
   - 参考 autoresearch 的"5 分钟时间盒"——给每个变体固定时间预算

5. **Skill 选择优化**
   - 当任务可被多个 skill 覆盖时，记录选择理由和结果
   - 积累数据后，建立"场景 → 最优 skill"的映射表
   - 不是自动化的——让经验指导后续选择

### 阶段 2：策略搜索引擎（1-2 月）

**引入自动化的策略评估与选择**

6. **经验导航器**
   - 借鉴 AgentEvolver 的 Self-Navigating 机制
   - 新任务到来时，自动检索历史相似任务的执行记录
   - 推荐历史最优策略，而非每次从头决策
   - 实现方式：task_ledger 记录 + 向量检索

7. **自动评估原型**
   - 对特定类型的产出（如 Insight 文档），设计自动评估 prompt
   - 评估维度：信息密度、Butler 相关度、可操作性、结构完整性
   - 先在影子模式运行（评估结果仅记录不执行），人工校准后再用于决策

### 阶段 3：Skill 达尔文进化（季度级）

**引入种群竞争机制**

8. **Skill 多版本并存**
   - 同一功能的 skill 允许存在 2-3 个版本
   - 按使用效果（成功率 × 效率）自动排序
   - 低效版本在积累足够统计量后自动降级

9. **跨任务元学习**
   - 从大量任务执行记录中提取跨任务通用模式
   - 例如："复杂整合任务拆分为 3 步效果最好""外部 API 调用失败时切换到本地实现成功率更高"
   - 这些元知识写入 self_mind 的长期偏好，影响后续所有任务的执行策略

## 七、风险与边界

### 7.1 三个致命陷阱

| 陷阱 | 说明 | 防御 |
|------|------|------|
| **Goodhart 化** | 优化指标而非真正目标——Agent 学会刷分但实际效果下降 | 多维评估 + 人类抽检 + 定期更新评估标准 |
| **自我欺骗** | Agent 的自评系统性地高估自身表现（对齐伪装的变体） | 独立裁决层 + 外部可观测指标交叉验证 |
| **能力锁死** | Agent 过早收敛到局部最优，失去探索多样性 | 保留"探索预算"——每 N 轮必须尝试至少 1 种非最优策略 |

### 7.2 Butler 特有的安全约束

Butler 作为 Persistent Partner（而非 Disposable Worker），其自我进化面临特殊约束：

1. **人格连续性 > 性能优化**：进化不能以人格漂移为代价。SOUL 和核心价值观是不可变量
2. **用户信任是脆弱资产**：一次严重的自我进化失误可能摧毁长期积累的信任
3. **渐进式 > 激进式**：每步进化的幅度应小到可被人类审查和回退
4. **透明性义务**：Butler 应向用户解释"我为什么改变了做法"，而不是静默地变化行为

### 7.3 Ouroboros 给出的警示

Ouroboros 的元进化虽然强大，但也暴露了一个根本性风险：**当 Agent 可以重写自己的方法论时，谁来保证新方法论比旧的好？** 评估函数本身的可靠性成为整个系统的信任锚点。

对 Butler 而言，这意味着：
- L0-L1 层（经验积累 + 策略搜索）现在就可以做，风险可控
- L2 层（配置/skill 修改）需要 heartbeat_upgrade_request 审批
- L3 层（方法论重写）在可预见的未来都应该保持"人类审批强制"
- **永远不应该自动化的环节：SOUL 修改、核心 prompt 变更、安全边界调整**

## 九、核心结论

### 一句话定位

> **Butler 的自我进化策略应是"经验积累 + 受控微实验"，而非"递归自我改写"。进化的速度不是越快越好——可验证、可回退、可解释的渐进式进化，才是 Persistent Partner 的生存之道。**

### 三个最值得做的事

1. **立即开始经验沉淀**（L0）：在 task_ledger 和 skill 使用记录中增加结构化评分字段，零风险，零代码修改，纯数据积累
2. **在 BrainStorm 流水线上试点微实验**（L1）：选一个窄域（如 Raw → Insight 的提炼策略），用 A/B 方式比较两种做法，人工评审保留优解
3. **设计 Butler 的"评估函数 v0"**（跨层）：定义 3-5 个可自动化度量的维度（任务完成率、耗时、skill 复用率、自评与外部评估的一致性），作为未来所有进化决策的数据基础
