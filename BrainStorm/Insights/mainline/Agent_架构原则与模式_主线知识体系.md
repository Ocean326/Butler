# Agent 架构原则与模式：从 Demo 到生产的知识体系

> **主线定位**：整合 14 篇 Raw/Working/Insight 文件 + 网络最新研究。从"10 个项目长一样"到"如何选架构、如何做多 Agent"，系统化梳理 Agent 架构的设计原则和工程模式。
>
> **整合来源**：小红书 Simon 系列（架构原则/10 个项目/死循环/SubAgent vs AgentTeam）、知乎深度拆解、Butler 对标分析、2026 年最新产业研究

## 一、所有 Agent 长一样——四件套收敛

### 1.1 核心发现

看完 10 个不同的 Agent 项目后，架构几乎完全一致：

```
┌──────────────────────────────┐
│       LLM（推理引擎）         │
├──────────────────────────────┤
│     工具系统（Tool System）    │
├──────────────────────────────┤
│   Agent 循环（Agent Loop）    │
├──────────────────────────────┤
│       记忆（Memory）          │
└──────────────────────────────┘
```

这不是巧合，而是**范式级收敛**——在当前约束下构建自主 AI 系统的唯一方式。

### 1.2 两个真正的差异化维度

| 维度 | 决定了什么 |
|------|-----------|
| **工具生态丰富度** | Agent *能做*什么——工具越多，能力边界越宽 |
| **任务规划能力** | Agent *做得多好*——多步分解、并行执行、多 Agent 协作 |

### 1.3 实际含义

- 最小可行 Agent 可以用**几百行代码**搭建。入门门槛远低于直觉。
- **真正的护城河是工程打磨**，不是架构发明。
- 好的框架应该把通用部分（循环、工具注册、上下文管理）打包好，让开发者只关注**写工具**和**定义任务**。

## 二、单次循环 vs 多轮迭代——Demo 到生产的分水岭

### 2.1 Demo 级 vs 生产级

| 特征 | Demo 级（单循环） | 生产级（多轮迭代） |
|------|-------------------|-------------------|
| 流程 | Perceive → Plan → Execute → Done | while(未完成) { Perceive → Plan → Execute → 自评 → 决定 } |
| 终止 | 固定步骤完成即止 | Agent 自己决定是否继续、调整、停止 |
| 失败处理 | 报错即崩 | 重试、回滚、升级、降级 |
| 适用 | 旅行规划器 demo | 真实代码修改、研究分析、长时任务 |

### 2.2 关键判断

> **多轮迭代能力是 Agent 架构成熟度的第一指标。** 设计系统时应**默认多轮**，而非把多轮当作特殊处理。

### 2.3 陷阱

如果多轮逻辑是硬编码的规则，"Agent" 就退化成**预定义工作流**——失去了自主性这个核心价值。Agent 的价值 = **智能 + 自主**：做什么、用什么工具、怎么做、是否做完——都应该由 Agent 自己决定。

## 三、四条架构设计原则（Simon 提炼）

### 原则 1：先自由，后护栏

> 架构应该**放大 Agent 能力，而非限制它**。

先给 Agent 自由和自主条件，再加 Human-in-the-loop 作为必要约束和纠偏。反过来做（先限制、再尝试解锁）会把 Agent 退化成流水线脚本。

### 原则 2：工具原子化与通用性

- 工具必须是**小的、原子的、单一职责的、可复用的**组件
- 绝不把整个业务流程塞进一个工具——这会导致工具层的"业务锁定"
- 复杂任务通过**组合和编排**处理，而不是通过"超级工具"

### 原则 3：模块解耦 + 适配器

- 在线性 Plan→Execute→Feedback 链之外，保持所有模块**解耦**
- 业务适配逻辑**不应**硬编码到通用模块中
- 使用**适配器**处理场景差异、外部约束和集成细节
- 目标：即使场景变化，核心 Planner / Executor / Feedback 模块保持不变

### 原则 4：多 Agent 自底向上三层设计

当多个 Agent 协作时，难点转移到：
- Agent 间**通信**
- **状态共享** / 分区
- 在全局任务下**协调**各自的迭代

三层设计：

```
┌─────────────────────────────────┐
│  应用层: 场景适配、业务流程管理   │
├─────────────────────────────────┤
│  协调层: 任务调度、交互策略管理   │
├─────────────────────────────────┤
│  基础层: 消息、状态、工具管理     │
└─────────────────────────────────┘
```

## 四、Ralph Loop——"最好的架构是死循环"

### 4.1 范式转变

- **旧假设**：模型不够聪明 → 需要复杂多轮编排 + 工具调用 + 记忆管理（AutoGPT, BabyAGI, LangGraph, ReAct, Plan-and-Execute, Reflection）
- **新现实**：Claude / Cursor / Gemini / Qwen 3.5 等模型能在**单次推理调用**中处理极长上下文和复杂指令。一个好的 spec + 一个长上下文可以替代整条流水线。

### 4.2 核心理念

一行定义：`while true; do claude-code < prompt.md; done`

哲学：**"脏了就杀掉重来"**——不要试图抢救一个已经腐烂的上下文；清场、重新加载 spec。

### 4.3 记忆外置到 Git

不要强行把长期记忆塞进正在腐烂的上下文。记忆 = 版本化的 prompt、配置、日志——而非不断增长的聊天历史。

### 4.4 Runtime Shell > 推理策略

Ralph Loop 是一个 **runtime shell**，不是新的推理算法。推理本身不变；外层壳提供无限重试 + 快速重置 + 版本管理。

### 4.5 "无架构"是一种新架构

这意味着：
1. 大部分逻辑直接进入 **prompt/spec**，而非拆分到子模块
2. 复杂性存在于**运行时约束和重试策略**中，而非流程图
3. 模型在一次调用中自己做规划 + 自我反思，而非引入独立的 Planner 模块

### 4.6 关键洞察

> **"更多层" ≠ "更聪明"**——在强模型时代，多层嵌套（AutoGPT, BabyAGI, LangGraph）变得**昂贵、缓慢、脆弱**。关键是**缩短状态链**，减少中间层，把复杂性推回"一次强推理 + 外层死循环"。

## 五、SubAgent vs AgentTeam——双引擎架构

### 5.1 两种多 Agent 模式

| 维度 | SubAgent（子代理） | AgentTeam（智能体团队） |
|------|-------------------|----------------------|
| **运行时** | 进程内，毫秒启动 | 独立进程，邮箱通信 |
| **协作** | 无 Agent 间通信；像"工具工人" | 成员通信、分工、汇总结果 |
| **适用** | 简单后台任务、一次性子任务 | 复杂研究、跨阶段协作 |
| **生命周期** | Map 管理，轻量销毁 | SDK 管理，需要 stream 生命周期处理 |
| **比喻** | 一群随叫随到的工具操作员 | 有组织的项目团队 |

### 5.2 关键洞察：不是二选一

> **在同一系统中保留两个引擎**，根据任务复杂度动态切换。

- 统一入口切换：`/team` 命令切换引擎；用户无感
- 共享 Provider + Memory：两个系统通过适配器层共享模型配置和记忆系统

### 5.3 记忆是胶水

如果两种 Agent 模式各自维护独立状态，系统退化成两个不相关的工具集。**统一记忆**是让双引擎输出叠加而非覆盖的关键架构决策。

## 六、五大协调模式（2026 产业研究补充）

来源：Zylos Research、AI Academy、Anthropic、Medium

### 6.1 模式对照表

| 模式 | 结构 | 优势 | 风险 | 适用场景 |
|------|------|------|------|---------|
| **层级制 (Manager-Worker)** | 主管路由到专业子 Agent | 复杂度从 O(n²) 降到 O(n) | 主管成为瓶颈 | 通用多任务系统 |
| **编排器-工人 (Orchestrator-Worker)** | 编排器维持策略，工人在隔离上下文执行 | 上下文清洁（Anthropic 实测 90% 性能提升） | 编排器不能参与执行 | 代码生成、研究分析 |
| **规划器-执行器 (Planner-Executor)** | 规划器生成任务清单，执行器从队列处理 | 清晰的任务边界 | 需要"计划修复"机制 | 确定性工作流 |
| **群体/P2P (Swarm)** | Agent 自主交接，无中心主管 | 极高灵活性 | 消息风暴风险 | 探索性任务 |
| **DAG/状态机** | 硬编码有向无环图 | 最健壮的合规保证 | 缺乏灵活性 | 合规审批流程 |

### 6.2 核心最佳实践

- **编排器绝不执行**：只做分解、委派、验证、升级。编排器执行会导致上下文污染和系统崩溃。
- **迭代审查循环**：比单次审查多捕获 3-5 倍缺陷，3-4 次迭代后收益递减。
- **委派控制应为 opt-in**：CrewAI 已收紧为 `allow_delegation=True` + 精细控制。

### 6.3 生产现实（2026 数据）

> **41-87% 的多 Agent LLM 系统在生产中失败**，79% 的失败根源在**规格和协调问题**，而非代码 Bug。

任务分解是关键枢纽——分解错误会在整个团队中级联。

## 七、上下文腐烂与记忆策略

### 7.1 上下文腐烂的症状

- Prompt 越来越长，但质量下降
- 记忆无差别积累（"这个也记一下，那个也记一下"）
- 对同一问题的回答变得不一致和自相矛盾

### 7.2 三层记忆架构

```
┌──────────────────────────┐
│ 短期上下文: 当前对话窗口  │ ← 最新、最相关
├──────────────────────────┤
│ 长期文件记忆:             │ ← Git 版本化
│ BrainStorm / MEMORY / docs│
├──────────────────────────┤
│ 可恢复 Raw: 原始素材      │ ← 可重新摄入
└──────────────────────────┘
```

### 7.3 主动重置机制

在关键里程碑处，刻意切断旧上下文，只保留结构化摘要和关键信号，然后进入下一个迭代周期。让"何时重置、保留了什么、丢弃了什么"成为**显式决策**而非隐式漂移。

### 7.4 产业共识

六大厂商 + 独立研究者都收敛到：
- **上下文 = 稀缺资源**
- **文件系统作为扩展记忆**（Manus, Cursor, Anthropic, LangChain 全部如此）
- **动态上下文发现 + 懒加载工具**：启动时不注入所有工具；使用 skill 索引 + 按需加载

## 八、2026-03-18 新增：两个开源项目的架构模式补充

> 来源：`Insights/20260318_AutoResearchClaw_全自主科研管线架构拆解_insight.md` + `Insights/20260318_agency_agents_Persona框架与Swarm编排_insight.md`

### 8.1 三种 Agent 定义粒度——从 Persona 到全管线

AutoResearchClaw 与 agency-agents 分别代表了 Agent 系统的两个极端，结合 autoresearch 形成完整光谱：

| 粒度 | 代表项目 | 定义的是什么 | 不定义什么 | 复杂度 |
|------|---------|------------|-----------|--------|
| **Persona 粒度** | agency-agents (52K★) | "谁来做"——身份、规则、交付规范 | 运行时、记忆、编排 | 极低（1 个 MD 文件） |
| **约束粒度** | autoresearch (Karpathy) | "在什么框里做"——搜索空间、指标、预算 | 谁做、具体流程 | 极低（3 文件 / 630 行） |
| **流程粒度** | AutoResearchClaw (4.3K★) | "谁做 × 怎么做 × 做到什么程度才过门" | — | 高（23 Stage / 8 Phase） |

**设计启示**：大多数系统应该从 Persona 或约束粒度起步，只在领域任务的时序依赖确实复杂时才升级到流程粒度。Butler 当前在约束粒度和流程粒度之间——skill 体系可以往 Persona 粒度借鉴定义规范。

### 8.2 Agent 定义六要素模型

agency-agents 的 112+ Persona 沉淀出的通用 Agent 定义结构：

```
┌─────────────────────────────────────────┐
│  1. Identity Traits   — 谁？性格/思维方式  │
│  2. Core Mission      — 干什么？职责边界    │
│  3. Success Metrics   — 做到什么算好？      │
│  4. Critical Rules    — 绝对不能做什么？     │
│  5. Workflow Process  — 怎么做？决策链      │
│  6. Deliverables      — 交什么？示例/模板    │
└─────────────────────────────────────────┘
```

与 Butler sub-agent 对照：Identity、Mission、Rules、Workflow 已有覆盖；**Success Metrics**（可衡量的完成判据）和 **Deliverables**（显式交付规范 + 示例）普遍缺失——这两项是 Agent 自验收和 planner 验收的基础。

### 8.3 Stage-Gated Pipeline + 三路决策分支

AutoResearchClaw 的 23-Stage 管线不是随意堆砌，而是映射真实科研工作流的时序依赖（Scoping → Literature → Synthesis → Design → Execution → Analysis → Writing → Finalization）。三个关键设计选择构成通用模式：

1. **Gate Stages**：在高风险节点设人类审批点（Stage 5/9/20），但允许 `--auto-approve` 全跳过——自治度是分级控制而非二元开关
2. **Decision Loop (Stage 15)**：PROCEED / REFINE / PIVOT 三路分支 + 产物自动版本化——管线内的自适应重规划
3. **Stage 级幂等重试**：单 Stage 失败不导致整条管线从头开始

**与第二节"Demo vs 生产"的呼应**：PROCEED/REFINE/PIVOT 三路决策是"多轮迭代"理念的细化——不只是"继续/停止"，而是"继续/微调/换方向"。

### 8.4 按产出物类型划分 Agent（而非按流程阶段）

AutoResearchClaw 的三个专业 Agent 按"产出物质量域"切分：

| Agent | 产出物 | 硬验证门控 |
|-------|--------|-----------|
| CodeAgent | 实验代码 | AST 验证、禁止硬编码指标、跨文件导入检查、≤3 轮修复 |
| BenchmarkAgent | 基准测试 | 导入验证、预训练模型兼容性 |
| FigureAgent | 学术图表 | 色盲安全配色、300 DPI、误差棒+置信区间 |

**原则**：每个 Agent 对自己产出的质量负全责。验证条件是硬门控（不满足则重试或报错），而非软建议。

**与第三节"工具原子化"的呼应**：Agent 按产出物类型切分 = 更粗粒度的"原子化"——每个 Agent 就是一个有自检能力的大工具。

### 8.5 正交横切的 Sentinel 层

AutoResearchClaw 的 Sentinel Watchdog 独立于管线的 23 个 Stage，作为后台横切层持续巡检：

- NaN/Inf 快速失败检测
- 论文-证据一致性核查
- 4 层引用验证（arXiv → CrossRef → Semantic Scholar → LLM）
- 反编造 + 反 AI-slop 守卫

**通用模式**：质量守护不嵌入任何单一执行阶段，而是作为正交横切关注点跨 Stage 运行。这与第三节"模块解耦 + 适配器"互补——Sentinel 是"解耦的质量适配器"。

### 8.6 Multi-Agent Debate > Multi-Agent Voting

AutoResearchClaw 在假说生成、结果分析、Peer Review 三处使用多视角辩论：

- 不是多个 Agent 独立产出然后投票选最优
- 而是**结构化多轮辩论 + 强制对立视角 + 仲裁者综合**
- 规避群体极化和回声室效应

这是"Debate / Critique"附加模式（第五节原 6.1 表中列出但未展开）的生产级实现。

## 九、十二条可执行设计原则（全主线浓缩）

1. **默认多轮迭代**——永远不假设任务在一个 plan-execute 周期完成。把迭代作为基线，而非例外。
2. **先自由后护栏**——让 Agent 有最大自主空间，在关键决策门控处加 Human-in-the-loop。
3. **原子工具 + 适配器层**——工具保持小而通用；业务逻辑在适配器/配置中，可更换不动核心。
4. **少数强主循环**——运行时收敛到极少数高信任主循环。其他一切都是由这些循环分发的"挂载任务"。
5. **先持久化，别死在上下文上**——长任务把中间结果存文件/Git。上下文只携带必要引用和当前目标。宁可重建上下文，不堆积在腐烂的上面。
6. **单一真源 + 自律钩子**——影响任务决策的任何状态必须先落到统一来源，再被消费。自律钩子（完成标准检查、上下文预算检查、长尾异常回滚）是每轮强制的。
7. **双引擎就绪**——同时为轻量 SubAgent 和重量 AgentTeam 模式设计，规划器中有基于复杂度的路由器。
8. **四层 Harness 分离**——知识、编排、门控、治理显式分层。不要混在一个 prompt 或循环里。
9. **上下文腐烂作为一等指标**——跟踪上下文健康度，定义重置触发器，让重置成为正常操作事件。
10. **经验飞轮**——每次任务执行自动产出结构化摘要和治理指标，反馈到案例库供未来复用。
11. **Agent 定义六要素完备**——每个 Agent/Skill 须具备 Identity、Mission、Success Metrics、Critical Rules、Workflow、Deliverables 六要素，缺项即补。（v1.1 新增，来自 agency-agents）
12. **按产出物类型划分 Agent + 硬验证门控**——专业 Agent 按"产出质量域"切分，每个 Agent 自带不可绕过的验证条件。（v1.1 新增，来自 AutoResearchClaw）

## 九、Butler 对标分析

### 9.1 四件套现状

| 组件 | Butler 实现 | 状态 |
|------|------------|------|
| **LLM** | Claude (主) + 其他模型 (辅) | 对齐主流 |
| **工具系统** | `skills` 目录 + MCP 工具 | 就位，扩展中 |
| **Agent Loop** | heartbeat planner → executor 循环 | 功能可用；需要显式迭代状态机 |
| **记忆** | local_memory + self_mind + task_ledger | 存在；需要统一读/写/压缩 API |

### 9.2 关键差距

| 差距 | 现状 | 目标 |
|------|------|------|
| 迭代控制 | planner prompt 驱动 | 显式 `{continue\|terminate\|escalate\|retry}` 状态机 |
| 工具原子性 | 大部分合规，部分耦合平台假设 | 提取适配器 |
| 上下文预算 | 隐式经验 | 可见的 token/轮次预算模型 |
| Skill 懒加载 | 启动时一次注入 | skill 索引 + 按需注入 |
| 经验飞轮 | 有 task_ledger 和 BrainStorm | 缺系统化的案例/模式/失败提取复用 |

## 十、学习路径建议

### 从零开始的阅读顺序

1. **共性认知**：四件套收敛 + 两个差异化维度（第一节）→ 理解"Agent 为什么都长一样"
