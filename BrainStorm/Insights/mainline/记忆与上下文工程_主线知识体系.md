## 一、什么是上下文工程

### 1.1 从 Prompt Engineering 到 Context Engineering

2025 年之前，AI 工程的重心是**怎么写提示词**（Prompt Engineering）。2025-2026 年，随着 Agent 从单轮推理走向多步自主执行，瓶颈转移到了**怎么管信息**——上下文工程（Context Engineering）应运而生。

> "Context engineering is the delicate art and science of filling the context window with just the right information for the next step."
> —— Andrej Karpathy

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| **关注点** | 单轮指令的措辞与结构 | 多轮交互中完整的信息状态 |
| **频率** | 写一次、调几次 | 每轮推理前都要做一次"策展" |
| **管理对象** | 系统提示词 | 系统指令 + 工具 + 消息历史 + 外部数据 + MCP + 记忆 |
| **核心挑战** | 找到"对的措辞" | 在有限注意力预算中放入"最高信号密度的 token" |

### 1.2 为什么上下文是有限资源

Anthropic 在官方博客中明确指出：**上下文不是免费的，它是一种注意力预算（attention budget）**。

- **Context Rot**：随着 context window 中的 token 数量增加，模型准确回忆信息的能力会下降（Chroma Research 实证）
- **n² 注意力关系**：Transformer 中每个 token 要 attend 到所有其他 token，n 个 token 产生 n² 个关系——窗口越大，注意力越"稀薄"
- **更大的窗口不是解药**：Google 押注 2M-10M token 长上下文，但仍然发现性能随窗口增大而下降，需要 ReadAgent/Gist Memory 等组合方案补偿

### 1.3 核心问题的统一表述

所有六家厂商（Manus、Cursor、Anthropic、OpenAI、Google、LangChain）都在回答同一个问题：

> **LLM 应该什么时候看到什么信息、信息应该如何组织？**

答案的差异不在于"要不要管"，而在于"怎么管"——这就是上下文工程。

### 1.4 关键认知转向：Agent 的失败是上下文的失败

2026 年行业共识正在收敛到一个判断：

> "Most agent failures are not model failures anymore, they are context failures."

一项标志性研究发现：**被给予 300 token 高质量聚焦上下文的模型，表现优于拥有 113,000 token 未过滤对话的模型**。这彻底颠覆了"给得越多越好"的直觉——质量远比数量重要。

企业级实证数据：2025 年约 **65% 的企业 AI 失败**源于上下文漂移或记忆丢失，而非原始上下文容量耗尽（Zylos Research 2026）。系统性的上下文管理可防止约 **30% 的信息丢失**，实施有效管理的系统报告成本降低 **60-80%**（Meta-Intelligence 2026 综述）。

## 二、上下文的三种退化模式

上下文管理不仅要防"溢出"，更要防"腐烂"。行业对上下文问题的认知已从"窗口不够大"迁移到"窗口里的信息在退化"：

| 退化模式 | 定义 | 典型场景 | 后果 |
|---------|------|---------|------|
| **Context Rot**（上下文腐化） | 长对话中早期信息逐渐失去影响力，模型对中段信息的注意力显著下降 | 多步 Agent 执行到第 5-6 轮时，早期指令被淹没 | "Lost in the Middle"——关键约束被遗忘 |
| **Context Poisoning**（上下文投毒） | 错误信息一旦进入上下文，会被模型当作事实反复强化 | 工具返回错误结果后，后续推理全部基于错误前提 | 错误级联放大 |
| **Context Distraction**（上下文干扰） | 过量的正确但无关信息稀释了关键信息的注意力份额 | 把大段搜索结果全部塞入上下文 | 模型"走神"，忽略真正关键的指令 |

**类比**：上下文窗口像人的工作记忆（7±2 个信息块）。上下文腐化类似"记忆模糊"，投毒类似"错误记忆"，干扰类似"注意力分散"。三种退化需要不同的应对策略。

**→ Butler 映射**：Butler 的 heartbeat 多轮执行是 Context Rot 的高发场景——executor 的回执、planner 的分析、历史 branch 的摘要逐轮累积，到第 5-6 轮时早期信息几乎被淹没。当前没有主动的"上下文卫生"机制来清理过时或冗余信息。

## 三、四种操作原语——上下文工程的统一语言

LangChain 在 2025-2026 年间提出了一套被广泛接受的分类法，将所有上下文管理策略归纳为四种操作原语：

```
┌─────────────────────────────────────────────┐
│           Context Engineering               │
│                                             │
│   ┌──────────┐  ┌──────────┐               │
│   │  Write   │  │  Select  │               │
│   │ 写入外部  │  │  按需检索 │               │
│   │  存储     │  │  上下文   │               │
│   └──────────┘  └──────────┘               │
│                                             │
│   ┌──────────┐  ┌──────────┐               │
│   │ Compress │  │ Isolate  │               │
│   │ 摘要压缩  │  │ 子上下文 │               │
│   │ 降精度    │  │  隔离    │               │
│   └──────────┘  └──────────┘               │
│                                             │
└─────────────────────────────────────────────┘
```

### 3.1 Write（写入）

将信息持久化到上下文窗口之外的存储中（文件系统、数据库、结构化 notes）。

| 厂商 | 实现方式 |
|------|---------|
| Manus | 大观察（工具输出）写入文件系统，上下文仅保留轻量引用 |
| Cursor | Rules (.cursor/rules/) 作为系统级指令持久化；Agent Skills 按需加载 |
| Anthropic | Memory Tool（beta）：在上下文窗口外做持久存储，提供 view/create/str_replace/insert/delete/rename 等 CRUD 操作，案例中 token 使用减少约 84% |
| OpenAI | Agents SDK Session 管理多轮对话历史，支持 SQLiteSession 持久化 |
| Claude Code | NOTES.md / 结构化笔记，Agent 主动写"日记"记录进度和决策 |

### 3.2 Select / Pull（选择/检索）

在需要时从外部存储中拉回相关信息，而非提前预装所有内容。

**关键范式转变**：从"推理前 RAG"（Pre-inference retrieval）转向"循环内检索"（In-loop / Just-in-time retrieval）。

Anthropic 在官方博客中将此描述为**渐进式披露（progressive disclosure）**：
- Agent 维护轻量标识（文件路径、存储的查询、URL）
- 运行时通过工具动态加载数据
- 每次交互产生的上下文指引下一步决策——文件大小暗示复杂度，命名约定暗示用途，时间戳代理相关性

**Claude Code 实践**：`CLAUDE.md` 文件预加载进上下文，而 `glob` 和 `grep` 等工具则在运行时按需检索文件——混合策略兼顾速度与灵活性。

**Cursor 实践**：
- A/B 测试证实：减少预先塞入的细节、让 Agent 自己按需拉取上下文，效果更好
- 懒加载 MCP 工具显著降低总 token 使用（约 46.9% 的 token 降幅）

### 3.3 Compress（压缩）

在保留任务关键信息的前提下，减少上下文中的 token 数量。

Anthropic 提出三种递进式压缩技术：

**① 工具结果清理（Tool Result Clearing）**——最轻量
- 历史深处的工具调用结果，Agent 不太可能再需要看原始输出
- 直接清除或替换为占位符
- Anthropic 已将此作为 Claude Developer Platform 的正式功能发布（Context Editing API，2025-09）

**② Compaction（压缩续写）**——中等
- 对话接近窗口上限时，由模型总结对话内容，用压缩版开启新窗口
- 保留架构决策、未解决的 bug、实现细节；丢弃冗余工具输出
- Claude Code 实践：压缩后保留 5 个最近访问的文件，维持连续性
- **关键技巧**：先最大化召回（确保不漏关键信息），再优化精确度（去除冗余）
- **Anthropic 产品化**：`compact-2026-01-12` compaction API 已支持 Claude、AWS Bedrock、Google Vertex AI、Microsoft Foundry
- **⚡ 2026-03 重大进展**：Claude Opus 4.6 将 compaction 从 API 工具提升为**模型内置能力**——对话接近窗口上限时 API 自动触发压缩，无需开发者手动调用。MRCR v2 基准（1M tokens 多针检索）达 76% 准确率，是 Sonnet 4.5（18.5%）的 **4 倍**。同时最大输出倍增至 128K tokens，1M 上下文窗口进入 beta。这意味着 Compress 原语正在从"工程师必须自建"转变为"模型原生能力"

**③ Summarization（摘要化）**——最重
- LLM 生成结构化摘要替换完整对话历史
- 完整消息保留在文件系统中可回溯

**Manus 的双存储 Compact 策略**（独特且值得借鉴）：
- 每个工具结果存储"完整版 + 紧凑版"
- 老旧结果自动替换为紧凑版（如文件路径引用）
- 新近结果保留完整版以指导下一步决策
- 典型任务需约 50 次工具调用，此策略可有效防止窗口退化

**LangChain Deep Agents 的分层压缩**（2026 年 3 月）：
- >20,000 token 的工具返回 → 自动写入文件系统 + 保留预览引用
- 85% 上下文容量 → 截断文件写/编辑操作的输入，替换为磁盘指针
- 仍不够 → 触发 LLM 摘要化
- **2026 新增**：自主压缩工具（Autonomous Context Compression）——模型可以在任务边界、大上下文消费前等"恰当时机"主动触发压缩，而非被动等待阈值

**2026 前沿压缩技术**：

- **锚定迭代摘要（Anchored Iterative Summarization）**：不每次从头重建摘要，而是将新摘要合并到一个持久化的锚定状态（意图、变更、决策、下一步）。Factory 对 36,000 个真实工程 session 的评估显示，此方法在准确性、完整性和连续性上始终优于全量重建。
- **ACON（失败驱动的压缩优化）**：无梯度方法，通过分析"压缩后上下文导致失败"的案例来迭代优化压缩 prompt——可将记忆使用降低 26-54%，同时保持 95%+ 的任务准确率。兼容闭源模型。
- **三层独立压缩叠加**：系统提示词架构（60-70% 缩减）+ MCP 输出压缩（通过相关性排序实现 94% 缩减）+ 知识囤积策略。三层独立优化后可叠加。

### 3.4 Isolate（隔离）

将子任务放入独立的上下文窗口，防止信息交叉污染，保护主对话上下文。

| 实现 | 说明 |
|------|------|
| Cursor Subagents | 高上下文负载任务进入独立窗口；支持并行多线（Explore / Bash / Browser） |
| Anthropic 子 Agent 架构 | 主 Agent 协调高层计划，子 Agent 做深度技术工作；子 Agent 可能消耗数万 token，但只返回 1,000-2,000 token 的精炼摘要 |
| Manus | 多 Agent 各自维护独立上下文窗口，separation of concerns |
| LangChain Deep Agents | 通过 LangGraph 实现的子 Agent 沙箱环境 + Task tool 生成子 Agent |

**Anthropic 的选择建议**：
- 多 Agent 架构：适合需要并行探索的复杂研究与分析
- 结构化笔记：适合有明确里程碑的迭代开发
- Compaction：适合需要大量来回的对话式任务

**No-op 规划工具作为隐性上下文策略**（LangChain / Anthropic 共识）：
- Claude Code 的 `write_todos` 工具表面是"规划"，实质是**上下文策略**——强制 Agent 将当前理解结构化地写出来，等效于把松散的推理状态"固化"到上下文中高注意力区域
- Manus 的"通过背诵操作注意力"也是同一思路：每步更新并重读活的待办列表，把当前目标放在上下文末尾（高注意力区域）

## 四、多层记忆架构——Agent 的长期"大脑"

### 4.1 三层基础模型

成熟的 Agent 记忆系统至少需要三层，对应认知科学中的经典分类：

| 层级 | 特征 | 典型内容 | 生命周期 |
|------|------|---------|---------|
| **工作记忆**（Working Memory） | 当前任务的即时上下文 | 当前对话、当前 branch 目标、工具调用结果 | 单轮/单 branch |
| **情景记忆**（Episodic Memory） | 过去交互的结构化回忆 | 历史任务的回执摘要、用户偏好变更记录 | 跨轮/跨天 |
| **语义记忆**（Semantic Memory） | 沉淀的知识和规则 | 用户画像、SOUL 设定、skill 能力描述 | 长期稳定 |

### 4.2 七层精细模型（2026 前沿）

2026 年的研究将三层模型进一步细化为七层：

```
┌────────────────────────────────────────┐
│  Layer 7: Reflective Memory            │
│  性能评估 + 策略改进 + 自省            │
├────────────────────────────────────────┤
│  Layer 6: Episodic Memory              │
│  特定事件和任务执行结果的记录          │
├────────────────────────────────────────┤
│  Layer 5: Long-Term Memory             │
│  跨会话的用户偏好和行为持久化          │
├────────────────────────────────────────┤
│  Layer 4: Retrieval Memory             │
│  通过 RAG 和语义搜索访问外部知识      │
├────────────────────────────────────────┤
│  Layer 3: Short-Term Memory            │
│  对话历史和叙事上下文                  │
├────────────────────────────────────────┤
│  Layer 2: Working Memory               │
│  中间推理步骤和活跃计划                │
├────────────────────────────────────────┤
│  Layer 1: Input Memory                 │
│  原始 prompt、指令和会话状态           │
└────────────────────────────────────────┘
```

**关键洞察**：在标准的上下文管理模型中，系统/用户 prompt 仅占约 **5%** 的 token 预算，剩余 **95%** 需要在状态、记忆、检索文档和工具输出之间做有意识的设计分配。

### 4.3 Anthropic Memory Tool 的产品化实践

Anthropic 将多层记忆思路产品化为 Memory Tool：

- **客户端文件目录系统**：在 `/memories` 目录下提供 view/create/str_replace/insert/delete/rename 操作
- **即时检索而非预装**：信息按需拉取而非提前加载，保持活跃上下文聚焦于当前相关内容
- **客户端控制**：开发者自选存储位置（本地、数据库、加密等），保持数据主权
- **效果**：实测 token 消耗降低约 84%

### 4.4 跨轮知识迁移的工程范本：MetaClaw

> *来源：`Insights/20260318_AutoResearchClaw_全自主科研管线架构拆解_insight.md`*

AutoResearchClaw v0.3.0 的 MetaClaw 子系统是目前最完整的 **Agent 跨轮知识迁移** 工程实现，它把 §4.1-4.3 中的记忆分层理论落地为一条可运行的管线：

```
Run N 执行 → 失败/警告 → 结构化 Lessons（情景记忆）
                ↓
    MetaClaw: Lesson → Skill 转化（文件级 SKILL.md）（语义记忆）
                ↓
    Run N+1 → build_overlay() 注入全部 23 个 Stage 的 LLM prompt
                ↓
    LLM 避开已知坑 → 更高质量、更少重试
```

**关键设计选择**：

| 设计决策 | 选择 | 理由 |
|---------|------|------|
| Skill 粒度 | 文件级（独立 SKILL.md） | 可单独管理/删除，不污染全局配置 |
| 触发条件 | 失败/警告 → 结构化提取 | 从错误中学习，而非从成功中学习 |
| 注入方式 | overlay 到所有 Stage 的 prompt | 全局可用，不需要路由 |
| **时间衰减** | **30 天**未触发自动降权 | 防止过时 lessons 永久污染 |
| 启用策略 | 默认关闭，opt-in | 不增加核心管线复杂度 |

**实测效果**：stage 重试率 -24.8%，refine 周期 -40%，鲁棒性 +18.3%。

**与 Butler 记忆体系的映射**：

| MetaClaw 概念 | Butler 对应 | 当前差距 |
|-------------|------------|---------|
| Lesson（结构化失败记录） | task_ledger 失败回执 | Butler 失败记录缺乏结构化提取 |
| Skill（可复用经验） | skills/ + MEMORY.md | 缺少从失败到 skill 的自动转化管线 |
| build_overlay（注入） | prompt 组装 Block 5 | Butler 的 local_memory 命中是被动检索，非主动 overlay |
| **时间衰减** | **无** | **关键差距**——Butler 的长期记忆无衰减机制，可能累积过时信息 |

**→ 可执行项**：
1. 为 task_ledger 的失败回执增加结构化 lesson 提取（做了什么 / 失败原因 / 可避免条件）
2. 设计 lesson → skill/rule 的阈值触发机制（同类失败 ≥3 次 → 候选 skill）
3. 为 MEMORY.md / local_memory 引入时间衰减打分——长期未被检索命中的条目自动降低优先级

### 4.5 Butler 的记忆层映射

| 记忆层级 | Butler 当前实现 | 成熟度 | 差距 |
|---------|---------------|--------|------|
| 工作记忆 | 对话上下文 + branch prompt + 工具返回 | ★★★ | 缺显式的"工作记忆边界"声明 |
| 情景记忆 | recent（摘录）+ task_ledger + local_memory | ★★☆ | recent 摘录质量依赖截断而非语义；task_ledger 缺检索接口 |
| 语义记忆 | SOUL + 用户画像 + MEMORY.md + STATE.md | ★★★ | 基本够用，但缺少自动从情景记忆中提炼到语义记忆的升级通道 |

## 五、Token 预算管理——从直觉到工程

### 5.1 注意力预算概念

Anthropic 提出的"注意力预算"框架将上下文管理从"尽量多给信息"转向"精确分配有限资源"：

- **System Prompt 有"金发女孩区"**：过度工程化（塞太多规则、硬编码 if-else）和过度模糊（只给"be helpful"）都不好，存在一个最优信息密度区间
- **Cursor A/B 验证**：减少预先塞入的细节、改为让 Agent 按需拉取，在多个指标上优于"全量预加载"
- **可量化标志**：Cursor 的 MCP 工具懒加载实现了约 46.9% 的 token 降幅

```
过度工程化                    金发女孩区                    过于模糊
(硬编码复杂 if-else)    ←→   (具体到足够引导行为,          ←→   (高层概念、假设共享上下文)
                              灵活到提供启发式)
```

**Anthropic 实操建议**：
1. 先用最好的模型测试最小 prompt
2. 根据失败模式逐步增加明确指令和示例
3. 用 XML 标签或 Markdown 标题分隔不同 section
4. 追求**最小但充分**的信息集——"最小"不等于"短"，而是"每个 token 都在工作"

### 5.2 Token 预算分配策略

2026 年的前沿实践已从"凭感觉分配 token"演进为显式的预算框架：

**四大预算支柱**（Meta-Intelligence / paperclipped.de 综述）：
1. **Token 预算编制**：为指令、知识、历史、推理等预分配百分比
2. **激进的检索结果压缩**：将检索结果压缩为高密度事实
3. **结构化架构**：信息的组织结构本身传递意义
4. **动态上下文优化**：运行时根据任务阶段动态调整分配比例

**预算感知的实践信号**：当任务执行到第 10 步（共 14 步）时，上下文窗口通常已充满陈旧的工具输出、无关历史和冗余指令——这就是需要主动压缩或刷新的时机。

## 六、KV-Cache——上下文工程的性能基石

### 6.1 为什么 KV-Cache 命中率是第一指标

Manus 团队将 KV-cache 命中率定义为**生产 Agent 最重要的单一指标**：

- Agent 工作流的 token 比例极度偏斜——典型输入输出比为 **100:1**（每次工具调用都在增长上下文，而输出只有短短几行）
- 缓存 vs 未缓存的成本差异高达 **10 倍**（以 Claude Sonnet 为例：缓存 $0.30/MTok vs 未缓存 $3.00/MTok）
- 通过前缀共享（prefix-sharing），相同前缀的上下文可以直接复用 KV-cache，显著降低 TTFT（Time-to-First-Token）

### 6.2 Cache 友好设计的四条共识

跨 Manus、OpenAI、Cursor、Anthropic 四家的工程实践，收敛出一致的设计原则：

| # | 原则 | 说明 | 反模式 |
|---|------|------|--------|
| 1 | **前缀稳定** | 系统指令/规则/示例/角色放 prompt 最前面，格式保持一致（含空白、key 顺序） | 频繁改动系统 prompt 的细节 |
| 2 | **追加优于重排** | 历史日志只追加不重排/插入 | 每轮重新组织消息顺序 |
| 3 | **Mask 优于移除** | 不用的工具用 mask 标记为不可用，保持工具列表结构稳定 | 动态增删工具列表 |
| 4 | **大观察落盘** | 工具调用的大体量返回值写文件系统，prompt 只保留轻量引用 | 把整段搜索结果塞进上下文 |

### 6.3 OpenAI Prompt Caching 的工程细节

- **自动触发**：≥1024 token 的 prompt 自动参与缓存
- **前缀完全一致**才能命中——一个字符的差异都会导致缓存失效
- **保留策略**：
  - 标准：内存 5-10 分钟（最长约 1 小时）
  - Extended：保留约 24 小时（gpt-5 / gpt-5.1-codex 等新模型支持）
- **效果**：延迟降低约 80%、输入 token 成本降低约 90%

### 6.4 Anthropic Context Editing API（2025-09 发布）

Anthropic 推出的两个互补的上下文管理能力：

- **工具结果清理（Tool Result Clearing）**：自动清除对话中陈旧的工具输出内容
- **思维块清理（Thinking Block Clearing）**：清除扩展思维（extended thinking）产生的内容
- **产品化 Compaction**：`compact-2026-01-12` API 已跨平台支持（Claude、AWS Bedrock、Google Vertex AI、Microsoft Foundry）

这些功能让 Agent 能够处理扩展任务而不耗尽上下文窗口或在会话间丢失关键信息。

## 七、长程任务的上下文持续性

### 7.1 问题定义

当任务跨度从几分钟延伸到数小时（大型代码库迁移、深度研究项目），token 总量会超过上下文窗口。这是上下文工程中最具挑战性的场景。

**关键发现**：上下文漂移先于容量限制导致失败——约 65% 的企业 AI 失败源于上下文漂移或记忆丢失，而非窗口耗尽。

### 7.2 Compaction（压缩续写）

**Claude Code 的实践**：
1. 对话接近窗口上限时，将完整消息历史传给模型做摘要
2. 模型保留：架构决策、未解决的 bug、实现细节
3. 模型丢弃：冗余工具输出、中间调试信息
4. 用压缩摘要 + 5 个最近访问文件开启新的上下文窗口
5. 用户无需担心窗口限制，获得连续体验

**调优建议**：先最大化召回、再优化精确——过度压缩可能丢失"重要性只在后续才显现"的微妙上下文。

### 7.3 结构化笔记（Agentic Memory）

Agent 主动在上下文窗口之外写笔记，后续需要时拉回窗口。

**Claude 打宝可梦的经典案例**：
- 在数千个游戏步骤中精确追踪目标："过去 1,234 步我一直在 Route 1 训练皮卡丘，已涨 8 级，目标 10 级"
- 在没有人类提示的情况下，自主开发区域地图、记录战斗策略、维护已解锁成就清单
- 上下文重置后读取自己的笔记，无缝继续多小时的任务

### 7.4 上下文检查点压缩（Checkpoint Compaction）

一种更结构化的跨实例交接机制，与 Compaction 互补——前者强调**跨 Agent 实例 handoff**，后者强调**单窗口内续写**。

**交接摘要的四要素**：
1. **当前进度与关键决策**（progress + decisions）
2. **重要约束与用户偏好**（constraints + preferences）
3. **待完成事项与下一步**（remaining + next steps）
4. **关键数据与参考资料**（critical data + references）

强制性 QA 步骤：生成摘要前必须逐字引用原始压缩指令，防止指令在传递中被篡改。

### 7.5 锚定迭代摘要（Anchored Iterative Summarization）——2026 前沿

Factory 基于 36,000 个真实工程 session 的评估揭示了一个更好的长程压缩范式：

**核心思路**：不再每次从零重建摘要，而是维护一个**持久化的锚定状态**，新摘要增量合并到该锚定上：

| 锚定状态字段 | 内容 |
|-------------|------|
| **Intent（意图）** | 任务目标和核心约束，只在用户显式修改时更新 |
| **Changes（变更）** | 本轮的实际操作和产出 |
| **Decisions（决策）** | 已做出的架构/设计决策及其理由 |
| **Next Steps（下一步）** | 待续任务的明确行动项 |

相比全量重建，锚定迭代摘要在准确性、完整性和连续性得分上始终更优——因为锚定状态中的意图和决策不会在反复摘要中被"冲淡"。

### 7.6 LangChain Deep Agents 的自主压缩

LangChain 于 2026 年 3 月发布的 Deep Agents Runtime，内建了自主压缩能力：

**Agent 主动触发压缩的四个时机**：
1. 在干净的任务边界（完成一个交付物后）
2. 从大量上下文中提取结果之后
3. 即将消费大量新上下文之前
4. 复杂多步流程开始之前

这标志着压缩策略从"被动触发"（固定阈值）到"主动策展"（Agent 自行判断时机）的范式转变。

## 九、Butler 的上下文架构解剖

### 9.1 Prompt 组装的 15 个 Block

Butler 的 `build_feishu_agent_prompt()` 逆向解剖显示出一个 **15 块层叠结构**：

| 位置 | 块 | 稳定性 | Cache 友好度 |
|------|-----|--------|-------------|
| 1-4 | 身份角色 → 场景 → Bootstrap → 基础行为 | 高（跨轮不变） | ★★★ |
| 5 | 对话上下文（主意识摘录、SOUL 摘录、用户画像、长期记忆命中、self_mind） | 中（按需加载、有长度控制） | ★☆☆ |
| 6-14 | 协议、技能目录、回复要求等（按 mode 可选注入） | 中-高（同 mode 下稳定） | ★★☆ |
| 15 | 用户消息 | 低（每轮变） | — |

**已做对的事**：
- 静态内容在前、动态内容在后——天然符合 Cache 友好原则
- Bootstrap 来自单一真源，避免 prompt 中堆叠重复规则
- SOUL/用户画像/local_memory/self_mind 均为摘录而非全量
- 按 mode（companion/execution/maintenance）控制注入量

**隐含风险**：
- 第 5 块（对话上下文）在每轮都变化且位于中段，可能切断后续 block 的 cache 命中
- Bootstrap 文件的细微改动（多加一个换行）导致整条缓存失效
- 缺乏显式的 cache 边界标记，难以诊断命中率
- 可选块（skill 目录、协议）通过 include/exclude 控制，而非 Manus 式的 mask

### 9.2 Butler 四操作覆盖度

| 原语 | 含义 | Butler 覆盖情况 | 差距 |
|------|------|----------------|------|
| **Write** | 信息写入长期存储 | ✅ STATE / MEMORY / task_ledger / 工作区文件 | 覆盖良好 |
| **Select / Pull** | 按需从存储拉回上下文 | ✅ local_memory 命中 / self_mind 摘录 / 文件引用 | 覆盖良好 |
| **Compress** | 摘要/截断/降精度 | ⚠️ 有 max_chars 截断，但缺少语义级压缩和双模存储 | **关键差距** |
| **Isolate** | 独立上下文执行子任务 | ✅ heartbeat executor / sub-agent 独立 prompt | 覆盖良好 |

### 9.3 与各家方案的逐点对照

| 维度 | Butler 当前状态 | 改进方向 | 优先级 |
|------|---------------|---------|--------|
| Prompt 前缀稳定性 | Bootstrap 在前，基本符合 | 增加 cache 边界标记 + 诊断工具 | 中 |

## 十、可执行的改进路线图

### 10.1 高优先级（立即可做）

**① handoff_summary 模板 v0**

设计一个标准化的 4 字段交接摘要结构，在 heartbeat branch executor 回执中引入：

```json
{
  "progress": "已完成/关键决策",
  "constraints": "约束/用户偏好",
  "remaining": "待完成/清晰下一步",
  "references": "关键数据路径/引用"
}
```

在 planner 汇总时做完整性校验——不再依赖自由文本回执的隐式传递。

**② 压缩策略配置化**

为 prompt 组装管线中的 self_mind / recent / heartbeat_context 增加 `compression_mode` 配置项：

| Mode | 行为 | 适用场景 |
|------|------|---------|
| `full` | 完整内容入 prompt | 近期/高相关 |
| `compact` | 结构化摘要入 prompt，完整版存盘 | 中期历史 |
| `reference_only` | 仅保留路径/ID | 远期/大体量 |

参考 Manus 的"完整版 + 紧凑版"双存储，自动按时效衰减。

**③ 上下文卫生——heartbeat 多轮的信息刷新**

在 heartbeat 执行超过 3 轮时，自动触发一次"上下文卫生检查"——标记哪些早期 branch 的结果已被后续覆盖/废弃，将其从 planner 的活跃上下文中移除。防止 Context Rot 导致的长任务质量下降。

### 10.2 中优先级（近期规划）

**④ 文件系统扩展记忆一等公民化**

制定默认约定：
- 大体量信息优先落盘为文件（Raw/Working/日志）
- 对话与执行时通过文件引用与搜索取回
- "尽量不改动历史片段，只追加新内容"——为 KV-cache 优化预留空间

**⑤ 上下文预算意识注入 planner**

在 heartbeat planner 规划长任务时，显式考虑每步可用的上下文与记忆预算：
- 通过拆步、落盘与检索控制成本
- 在任务回执中记录本轮大致 token 开销与关键上下文决策
- 便于后续做 Butler 自己的"上下文工程评估"

**⑥ 记忆层间升级通道（含 MetaClaw 启发）**

建立从情景记忆到语义记忆的自动提炼机制：
- 多次重复出现的用户偏好自动升级为 MEMORY.md 条目
- task_ledger 中反复成功/失败的模式提炼为经验规则（参考 MetaClaw 的 lesson→skill 管线）
- self_mind 的反思洞察固化为认知体系条目
- 引入**时间衰减**机制（参考 MetaClaw 30 天衰减）——长期未被检索命中的记忆条目自动降低优先级

**⑦ Prompt cache 诊断**

若使用 OpenAI 或兼容 API：
- 在日志中记录每次请求的 `cached_tokens` 字段
- 积累数据后分析命中率与前缀稳定性的关系
- 识别哪些 prompt 变动最频繁地破坏缓存

### 10.3 低优先级（长期储备）

**⑧ 工具列表 mask 化**

将当前按条件 include/exclude 的可选块（skill 目录、协议等）改为 mask 模式——保持工具列表结构稳定。

**⑨ 自主压缩能力** ← ⚡ 优先级上调信号

参考 LangChain 2026 年发布的 Autonomous Context Compression——让模型在任务边界、大上下文消费前等"恰当时机"主动触发压缩，而非等待固定阈值。

> **v2.1 更新**：Claude Opus 4.6（2026-03）已将 compaction 内置为模型原生能力（详见 §3.3②），Butler 若升级到 Opus 4.6 + 启用 API 级 compaction，此项可直接从"长期储备"提升为"近期可验证"——核心工作变为验证 Butler 多轮 heartbeat 场景下自动 compaction 的保留质量与信息丢失率。

**⑩ 锚定迭代摘要原型**

参考 Factory 的实践，在 heartbeat 多轮执行中引入锚定状态（Intent/Changes/Decisions/Next Steps），摘要增量合并而非全量重建。
