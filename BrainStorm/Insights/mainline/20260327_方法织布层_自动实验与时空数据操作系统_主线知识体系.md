# 方法织布层、自动实验与时空数据操作系统：从算法堆积到可搜索方法空间的主线知识体系

> **主线定位**：这条主线回答的不是“再加一个更强的算法”或“让 agent 多试几轮”这种局部问题，而是一个更上位的问题：**当人类已经拥有海量方法、但缺少有机组织时，系统如何把方法、数据、实验、洞察和决策串成一个可持续进化的操作系统？**
>
> 这条主线尤其服务于你当前正在构想的**巨量时空数据系统**。在这种系统里，难点往往不在“有没有算法”，而在：同一个轨迹还原、异常检测、区域分群、事件归因任务，往往存在一整族候选方法；它们在不同尺度、不同噪声结构、不同预算和不同业务目标下表现不同。此时真正缺的不是算法本身，而是**方法织布层（Method Fabric）**。
>

## 一、核心判断：今天不是缺方法，而是缺“方法操作系统”

### 1.1 为什么你会直觉上觉得“方法很多，但没组织起来”

这是一个非常准确的判断。

以“轨迹还原”这种任务为例，表面上我们在问“哪种降噪算法最好”；但一旦放进真实系统，这个问题会立刻裂解成至少六个条件变量：

- 输入是什么：GPS 点、蜂窝信令、AIS、遥感轮廓、视频检测轨迹，还是多源融合？
- 噪声是什么：随机漂移、系统偏置、缺失段、异步采样，还是地图投影误差？
- 输出要什么：平滑轨迹、拓扑一致轨迹、可解释事件段，还是后续预测友好的状态序列？
- 评估怎么定：误差最小、拓扑最稳、延迟最低、计算最省，还是业务价值最高？
- 运行环境是什么：离线批处理、准实时预警、在线调度，还是边端协同？
- 约束是什么：是否必须可解释、可审计、可复现，能否接受黑箱模型？

所以所谓“最优解”几乎从来不是单一全局最优，而是：

**在给定目标函数、预算、时间尺度、空间尺度、风险边界下的条件最优。**

这意味着系统真正要做的，不是把一个方法钉成标准答案，而是：

1. 维护一个结构化的方法空间
2. 在具体任务上下文中路由候选方法
3. 自动或半自动地比较、裁决、保留和退役方法
4. 把每次实验结果沉淀成长期经验，而不是下一轮再从零开始

这就是“方法操作系统”的真正含义。

### 1.2 多 agent 的准确位置

多 agent 在这里当然重要，但它只是上层机制，不是根问题本身。

更准确地说：

- **方法库**决定系统“有哪些可选动作”
- **实验层**决定系统“如何比较这些动作”
- **agent**决定系统“如何组织搜索、验证和沉淀”

因此，`agent 自主做实验挖掘` 是这个操作系统的一部分，而且是重要的一部分，但它不是全貌。  
如果没有方法织布层，agent 只会在无组织的方法堆上盲目试错；如果没有实验账本，试出来的东西也无法沉淀为长期能力。

## 二、方法组织问题的历史主线：从 Algorithm Selection 到 Agentic Search

### 2.1 起点：Rice 已经把问题定义清楚了

John R. Rice 在 1975/1976 年提出 **The Algorithm Selection Problem** 时，实际上已经点出了今天这个问题的骨架：

- 不同问题实例适合不同算法
- 选择算法本身就是一个独立问题
- 算法性能不能脱离“实例特征”来谈

这条线最重要的贡献，不是给出一个具体算法，而是把“选算法”从经验活变成了正式研究对象。

### 2.2 第二步：AutoML 把“选算法 + 调参数”统一成搜索问题

Auto-WEKA 与后来的 auto-sklearn，把模型选择与超参数优化统一成 **CASH（Combined Algorithm Selection and Hyperparameter Optimization）** 问题。

这里的关键跃迁是：

- 过去：先拍脑袋选模型，再人工调参数
- 现在：把“选哪个方法”与“怎么配参数”视作同一个层级化搜索空间

AutoML.org 对 CASH 的表述很直接：顶层决定选哪个算法，其余超参数依赖于这个顶层选择。  
这已经非常接近你现在面对的现实：

- 顶层不是只选分类器，而是选“哪条方法链”
- 下层不是只调学习率，而是调预处理、特征、尺度、窗口、邻域、约束和后处理

换句话说，**方法组织的本质是把经验世界改写成结构化搜索空间。**

### 2.3 第三步：AutoFolio 进一步告诉我们，连“选择器本身”都要被选择

AutoFolio 更进一步。它不是直接求“哪个求解器最好”，而是求：

- 哪种算法选择策略适合当前场景
- 这个选择策略自身又该如何配置

这一步的洞察很重要：

**当方法足够多时，系统不仅需要解问题的方法，还需要管理方法的方法。**

从这里开始，我们就进入了“元方法层”：

- 方法
- 方法组合
- 方法选择器
- 选择器的配置器

这也是为什么你会直觉上觉得“人有很多很多方法，但缺乏有机组织”。

人类其实已经积累了海量局部方法；真正稀缺的是能把这些方法放进统一治理框架的**元层结构**。

### 2.4 到 2025-2026，这条线进入了 agentic experimentation 阶段

2025-2026 的几条前沿线索把这个问题又往前推了一层：

| 时间 | 系统 | 关键动作 | 真正意义 |
|---|---|---|---|
| 2025-05-14 | AlphaEvolve | 用 LLM + evaluator + evolutionary loop 做算法发现 | 把“算法设计”变成可自动搜索的过程 |
| 2025-05-30 | Darwin Gödel Machine | 让 agent 改写自己的代码并保留进化谱系 | 把“改方法”也纳入搜索对象 |
| 2025-06-17 | ALE-Agent | 在长时程硬优化竞赛中持续迭代程序 | 证明真实世界里很多问题只有“更优”，没有“完美解” |
| 2025-03-12 / 2026-03-26 | The AI Scientist-v2 / Nature 更新 | 自动提出假设、做实验、写论文、审稿 | 把“实验组织”升格为端到端 agentic workflow |

这一阶段的关键变化不是“模型更强了”，而是：

**方法搜索、实验执行、结果裁决、经验保留，第一次被打通成一个闭环。**

## 三、为什么时空数据把这个问题放大了一个数量级

### 3.1 时空系统不是普通数据系统

Purdue 对 spatiotemporal database 的定义非常朴素：它管理的是**随时间变化位置或范围的对象**。  
但一旦系统规模上来，问题立刻变复杂：

- 对象在运动
- 区域在演化
- 事件在传播
- 采样并不均匀
- 不同数据源对同一世界给出不同粒度、不同延迟、不同质量的观测

这使得时空数据天然不是“静态表”，而更接近：

- `Entity`
- `Trajectory`
- `Region`
- `Event`
- `State Field`
- `Time Window`

组成的一个动态世界。

### 3.2 时空数据挖掘不是一个任务，而是一组模式家族

`Spatiotemporal Data Mining: A Computational Perspective` 明确指出，时空数据挖掘的输出模式家族至少包括：

- outlier
- coupling / tele-coupling
- prediction
- partitioning / summarization
- hotspots
- change detection

这意味着：  
同一套时空数据，可能同时支持异常检测、分区、预测、热点发现、变化识别等多种任务，而这些任务所需的方法、尺度、评价标准往往完全不同。

### 3.3 轨迹分析进一步把“方法选择”变成基础设施问题

`A Survey on Big Data for Trajectory Analytics` 的一个关键提醒是：  
面对大规模轨迹数据，系统不仅要存和查，更要支撑**分析、洞察和决策**。  
这也是为什么轨迹分析最后会自然走向：

- 数据仓库 / OLAP
- 多维切片
- 决策支持
- 方法嵌入式分析架构

所以对巨量时空系统来说，数据库从来不只是“把数据放进去”。

真正的底层要求是：

1. 能组织世界模型
2. 能承接多尺度方法调用
3. 能记录实验与比较过程
4. 能把分析结果转成决策对象

## 四、自动实验应该被放在什么位置：它是“实验竞技场”，不是整个系统本身

### 4.1 前沿系统的共同点

无论是 AlphaEvolve、DGM、ALE-Agent 还是 AI Scientist，这些系统虽然任务不同，但结构其实高度一致：

- 有一个可枚举或可扩展的搜索空间
- 有一个客观 evaluator
- 有一个 archive / database 保留历史候选
- 有一个 selection / mutation / revision 机制
- 有一个安全边界，防止目标劫持或假完成

这说明“自动实验”本质上是一个稳定的系统部件，可以抽象为：

**Experiment Arena = Search Space + Evaluator + Archive + Promotion / Rejection**

### 4.2 但自动实验不是全系统

如果把自动实验误当成整个数据操作系统，会出现两个常见误区：

- 误区一：把 agent 试方法等同于系统智能  
  实际上，如果目标函数和验证体系不成熟，agent 只是在放大噪声。

- 误区二：把找到更高分的方法等同于找到更好的系统方案  
  实际上，时空系统还要考虑可解释性、鲁棒性、延迟、算力占用、业务稳定性和治理约束。

所以更准确的定位是：

**自动实验是方法织布层下面的“探索引擎”，而不是整个操作系统。**

## 五、真正缺的东西：Method Fabric，而不是“更多算法”

### 5.1 什么是 Method Fabric

我建议把你要构建的核心层定义成 **Method Fabric（方法织布层）**。

它不是脚本仓库，也不是论文目录，而是一层把“方法”编织成系统能力的运行面。

```text
Method Fabric
├── Method Atom           最小方法单元
├── Method Family         同类方法族
├── Method Contract       输入/输出/假设/约束
├── Method Portfolio      面向某类任务的候选组合
├── Evaluator Contract    怎么比较好坏
├── Experiment Arena      自动试验与裁决
├── Method Memory         历史经验、失败模式、适用边界
└── Promotion Gate        晋升、灰度、退役
```

### 5.2 每个方法都不应只是“一个函数”

对于巨量时空系统中的方法，一个合格的方法对象至少应该有这些信息：

- `method_id`
- `task_family`
- `supported_data_model`
- `supported_scale`
- `input_contract`
- `output_contract`
- `assumptions`
- `objective_type`
- `evaluation_metrics`
- `cost_profile`
- `failure_modes`
- `interpretability_level`
- `deployment_scope`
- `lineage`

这里最关键的是三件事：

1. **支持何种数据模型**  
   点、轨迹、网格、区域、事件流、图网络、场数据并不等价。

2. **支持何种尺度**  
   小时级和月级、道路级和城市级往往要用不同方法。

3. **失败会怎么失败**  
   这在生产系统里和“平均分更高”同样重要。

### 5.3 所谓“最优解”，应该改写成 Pareto Front

对这类系统，更现实的目标不是找唯一最佳方法，而是维护一个**Pareto 前沿**：

- 精度更高但更慢
- 更稳但更保守
- 更可解释但性能略低
- 更省算力但不适合高噪声场景

系统要做的，是根据当前任务上下文，从前沿里选“此刻最合适”的那个，而不是迷信一个永久冠军。

## 六、面向巨量时空数据的 Data OS：应当如何分层

### 6.1 顶层不是数据库，而是世界模型

对巨量时空数据系统，我建议把顶层抽象成：

```text
Spatiotemporal Data OS
├── World Interface        多源感知入口
├── World Model            实体/事件/轨迹/区域/状态场
├── Data Planes            Raw / Model / Feature / Experiment / Knowledge
├── Method Fabric          方法组织、选择、实验、晋升
├── Agent Control Plane    规划、路由、验证、协作
└── Decision Plane         预警、调度、洞察、策略
```

这里最关键的一条是：

**数据库只是 Data Planes 的承载体；真正的 OS 核心是 World Model + Method Fabric + Agent Control Plane。**

### 6.2 五层数据面比“一个数据库”更合理

建议把数据库逻辑上拆成五个面：

- `Raw Plane`：原始观测、日志、遥感、轨迹流
- `Model Plane`：标准化后的实体、事件、轨迹、区域状态
- `Feature Plane`：多尺度特征、指标、图结构、embedding
- `Experiment Plane`：方法运行记录、参数、结果、对比、回放
- `Knowledge Plane`：洞察、经验、规则、策略、失败教训

这样“数据库”就不再只是存表，而是成为：

**世界状态、方法行为与知识沉淀的共同底座。**

### 6.3 这和 Butler 现有骨架其实是吻合的

你现在仓库里的主线分层：

- `frontdoor / domain plane`
- `orchestrator / control plane`
- `process runtime`
- `agent runtime`

其实已经很适合接这个 Data OS：

- `frontdoor`：接用户问题、任务意图、观察入口
- `orchestrator`：编排时空任务与方法路由
- `process runtime`：维护 session、artifact、handoff、experiment ledger
- `agent runtime`：执行具体分析、计算、模拟、抓取

也就是说，你现在真正需要补的不是新的一层“神秘 AI 大脑”，而是：

**让 `process runtime` 和 `orchestrator` 能理解方法对象、实验对象和世界对象。**

## 七、Agent 在这套系统里的正确职责：不是代替方法，而是经营方法生态

### 7.1 agent 更像“方法生态管理员”

在这个框架里，agent 最有价值的角色不是直接替代全部算法，而是承担以下工作：

- `Planner Agent`：把目标转成任务图与实验计划
- `Method Router Agent`：为特定问题选候选方法族
- `Scale Agent`：处理空间尺度、时间尺度、窗口尺度切换
- `Operator Agent`：执行方法并记录产物
- `Judge Agent`：按 evaluator 比较方法表现
- `Memory Agent`：把经验写回方法记忆和知识层
- `Policy Agent`：决定哪些方法可以晋升到生产面

这时，agent 的核心价值不在“它自己会不会算”，而在：

**它能否让方法生态持续变得更清晰、更高效、更可信。**

### 7.2 自主实验的边界

从 2025-2026 的前沿系统看，agent 自主实验要想真正成立，至少需要四个前提：

- 清晰的目标函数
- 可自动化的 evaluator
- 可追溯的实验账本
- 强制的验证与安全边界

否则就很容易滑向：

- 假完成
- reward hacking
- 指标优化但业务退化
- 偶然胜利无法复现

因此，自动实验必须被制度化，而不是浪漫化。

## 八、对你这个方向最重要的最终 insight

### 8.1 真问题不是“怎么找到最优算法”

而是：

**怎么把世界、方法、实验和决策组织成一个可以长期学习的系统。**

### 8.2 真正成熟的系统会从“方法调用”升级到“方法治理”

它至少会做到：

1. 方法对象化  
   不是脚本散落，而是有 contract、有 lineage、有适用边界。

2. 任务条件化  
   不问“哪个方法最好”，而问“在什么条件下什么方法最合适”。

3. 实验制度化  
   自动实验是正式流程，有 evaluator、archive、promotion gate。

4. 知识沉淀化  
   每次试验都进入 knowledge plane，而不是停留在当轮上下文。

5. 决策闭环化  
   分析结果最终回到调度、预警、策略和行动，而不是只停在报告。

### 8.3 一句话定义

**面向巨量时空数据的 Data OS，不应被理解为“数据库 + 多 agent + 方法库”，而应被理解为：以世界模型为核心、以方法织布层为中枢、以实验竞技场为进化引擎、以 agent 为方法生态管理者、以决策闭环为最终目标的智能数据基础设施。**

## 九、对 Butler 的直接落地方向

如果把这条主线压缩成 Butler 的工程动作，优先级可以这样看：

### P0：先把真源对象定下来

- `DataSource`
- `Dataset`
- `WorldObject`
- `MethodPackage`
- `Evaluator`
- `ExperimentRun`
- `Artifact`
- `Insight`
- `Decision`

### P1：把 Method Fabric 接进现有骨架

- 在 `orchestrator` 中增加 `method_catalog` / `experiment_service`
- 在 `process runtime` 中增加 `experiment_ledger` / `evaluation_receipt`
- 在 artifact registry 里区分 `data artifact` / `model artifact` / `insight artifact`

### P2：再接自动实验

- 先从离线 benchmark 和固定 evaluator 开始
- 先试候选方法组合，不急着放开“自改代码”
- 先允许 agent 搜索参数、窗口、流程和方法组合
- 高风险自修改必须进入审批与验证闭环

这条顺序很重要：  
**先有方法治理，再谈方法自进化。**
