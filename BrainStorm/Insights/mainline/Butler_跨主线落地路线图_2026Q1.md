# Butler 跨主线落地路线图（2026 Q1）

> **文档目的**：9 条主线各自提出了 Butler 的改进方向，但改进项散落在各文档的"路线图"章节中，存在大量交叉重复且缺乏统一优先级排序。本文将 9 条主线共计 50+ 条改进建议去重合并为 **24 条可执行行动项**，按紧迫度和依赖关系排列，形成一张可直接跟进的落地清单。
>
> **约定**：本文只描述"做什么"和"为什么优先"，不做"怎么做的代码设计"——那是 heartbeat_upgrade_request 或具体实施文档的事。

---

## 一、优先级框架

| 等级 | 判据 | 实施约束 |
|------|------|---------|
| **P0 — 立即可做** | 不改核心代码，仅改文档/配置/协议约定即可落地；效果跨多条主线共识 | 本周可落地 |
| **P1 — 近期** | 可能需要轻量代码改动或新建配置文件；需要 upgrade_request 走审批 | 1-2 周 |
| **P2 — 中期** | 需要系统性设计或架构调整 | 1 个月 |
| **P3 — 远期储备** | 技术预研或长期愿景，当前不急 | 季度级 |

## 二、P0 — 立即可做（不改核心代码）

### 2.1 结构化 Handoff 协议 v0

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑨MAS + ④记忆与上下文 + ③Claude Code工程化 |
| **做什么** | 在 heartbeat executor 回执中引入 4 字段交接摘要：`progress / constraints / remaining / references` |
| **为什么优先** | 3 条主线同时指出自由文本回执导致信息传递不可靠，是当前 MAS 和长任务最大的信息损耗源 |
| **落地形式** | 写入 heartbeat 协议文档 + Bootstrap 约定 |
| **验收标准** | planner 汇总时可自动检查 4 字段完整性 |

### 2.2 MAST 失败分类标签

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑨MAS + ⑤Agent评估 |
| **做什么** | task_ledger 失败回执增加 `failure_class: FC1(系统设计)/FC2(协调)/FC3(验证)` 字段 |
| **为什么优先** | 当前失败经验无结构化积累，无法驱动经验飞轮 |
| **落地形式** | 修改 task_ledger 约定文档 |

### 2.3 退役日志协议

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑨MAS + ⑥Agent产品形态 + ⑧自律系统 |
| **做什么** | 被关闭的 executor 分支自动输出 6 字段退役日志（goal/progress/failure_reason/artifacts/context_worth_inheriting/failure_class） |
| **为什么优先** | 3 条主线共识：失败 Agent 的知识随 kill 消失是最大的学习信号浪费 |
| **落地形式** | 写入 heartbeat 执行协议 |

### 2.4 高危动作分级白名单

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑧自律系统 + ⑤Agent评估 + ⑨MAS |
| **做什么** | 定义三级动作风险分类（🔴高：代码修改/文件删除/外部API → 必须审批；🟡中：工作区修改/只读API → 记录；🟢低：读取/搜索 → 自由） |
| **为什么优先** | 当前 heartbeat_upgrade_request 仅覆盖代码修改，遗漏了文件删除和外部 API 调用等高危操作 |
| **落地形式** | 以文档形式定义白名单，写入 heartbeat 协议 |

### 2.5 self_mind 自省问卷 v0

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑦Anthropic研究 + ⑧自律系统 + ⑤Agent评估 |
| **做什么** | executor 回执增加 `self_reflection` 字段，回答 3 个固定问题：本轮最不确定的判断 / 最可能犯错的环节 / 是否跳过了应做的步骤 |
| **为什么优先** | 3 条主线共识：自省是自律的认知前提，无差别记录应升级为结构化质量信号 |
| **落地形式** | 写入 executor 回执协议 |

## 三、P1 — 近期（1-2 周，可能需轻量代码改动）

### 3.1 独立 Trajectory 裁决

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑧自律系统 + ⑤Agent评估 + ⑦Anthropic研究 |
| **做什么** | heartbeat planner 汇总阶段增加一个独立于 executor 的裁决步骤，读取回执 + 实际产出，输出 pass/fail/partial + 理由 |
| **为什么优先** | 3 条主线的核心共识：自检 ≠ 验收；Agent 自报告不构成独立验收 |
| **依赖** | 需要 upgrade_request 修改 planner 逻辑 |

### 3.2 压缩策略配置化

| 属性 | 说明 |
|------|------|
| **来源主线** | ④记忆与上下文 + ③Claude Code工程化 |
| **做什么** | 为 self_mind / recent / heartbeat_context 增加 `compression_mode` 配置：full(完整) / compact(摘要) / reference_only(仅引用) |
| **为什么优先** | 当前仅有 max_chars 截断，缺少语义级压缩——是上下文工程主线识别的关键差距 |
| **依赖** | 需要 upgrade_request 修改 prompt_assembly_service |

### 3.3 任务复杂度路由

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑨MAS + ②Agent架构 |
| **做什么** | heartbeat planner 增加"任务复杂度评估 → 引擎路由"：单步→SubAgent / 多步同主题→Loop / 跨角色→多分支并行 |
| **为什么优先** | 当前简单任务和复杂任务走同一条路，效率和上下文使用不optimal |
| **依赖** | 需要 upgrade_request 修改 planner 逻辑 |

### 3.4 Agent 可读命名

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑨MAS + ⑥Agent产品形态 |
| **做什么** | 用 explorer/worker/reviewer/researcher 等可读标签替代裸 branch ID |
| **为什么优先** | 并行多 Agent 时人类无法快速区分追踪，认知负担落在用户身上 |
| **依赖** | 需要 upgrade_request 修改 planner 分支生成逻辑 |

### 3.5 heartbeat 报告精简

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑥Agent产品形态 |
| **做什么** | heartbeat 报告从"发生了什么"转向"建议做什么"，减少信息量，提升决策辅助价值 |
| **为什么优先** | 当前播报偏冗长，容易制造信息焦虑而非减轻认知负担 |
| **依赖** | 修改报告模板/协议 |

## 四、P2 — 中期（1 个月）

### 4.1 渐进式信任表

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑧自律系统 + ⑤Agent评估 + ⑦Anthropic研究 |
| **做什么** | task_ledger 中为每种任务类型维护 `success_count / total_count`；成功率 >80% 且总量 >5 时，自动提升该类任务的自治等级 |
| **价值** | 基于历史成功率的渐进式信任 > 硬编码的二元开关 |

### 4.2 Skill 健康度索引

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑥Agent产品形态 + ②Agent架构 |
| **做什么** | 为现有 skills 建立评分/使用频率/最后维护日期索引，作为遴选和淘汰依据 |
| **价值** | 科研龙虾经验证实：Skill 价值不在数量，在于可信度 |

### 4.3 角色轴锚定

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑦Anthropic研究 |
| **做什么** | 为 talk/heartbeat/self_mind 三个核心角色各定义 2-3 个人格轴的锚定值（正式度、主动度、情绪表达强度），写入角色配置 |
| **价值** | 用量化的人格约束替代纯文字 system prompt，防止跨角色漂移 |

### 4.4 文件系统扩展记忆一等公民化

| 属性 | 说明 |
|------|------|
| **来源主线** | ④记忆与上下文 + ③Claude Code + ②Agent架构 |
| **做什么** | 制定默认约定——大体量信息优先落盘、对话时通过文件引用取回、不改历史只追加 |
| **价值** | 3 条主线都认定"文件系统是一等扩展记忆"，但当前缺显式约定 |

### 4.5 上下文预算意识

| 属性 | 说明 |
|------|------|
| **来源主线** | ④记忆与上下文 + ②Agent架构 |
| **做什么** | planner 规划长任务时显式考虑可用 token 预算，通过拆步、落盘、检索控制成本 |
| **价值** | 当前无上下文预算意识，长任务容易触发 context rot |

### 4.6 Prompt 中移除工具链已覆盖的约束

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑧自律系统 |
| **做什么** | 审计现有 prompt 协议，把 L2(工具链)/L3(编排层) 已保证的约束从 L1(prompt) 中移除，释放注意力预算 |
| **价值** | Zylos 红队数据：纯 prompt 治理失败率 26.7%，重复声明是纯浪费 |

### 4.7 关键操作 Git 快照

| 属性 | 说明 |
|------|------|
| **来源主线** | ⑥Agent产品形态 |
| **做什么** | heartbeat 修改工作区文件前，自动 commit 当前状态作为回滚点 |
| **价值** | 将 Agent 操作从"不可逆"降级为"可回滚" |

### 4.8 `agent_os` 双层运行时定型

| 属性 | 说明 |
|------|------|
| **来源主线** | ①Harness + ②Agent架构 + ④记忆与上下文 |
| **做什么** | 将 `agent_os` 显式定型为两层：`Agent Runtime`（`run / session / instance / context / capability`）与 `Process Runtime`（`workflow / cursor / checkpoint / verify / approve / recover / join`），并把 `approval / verification / recovery` 的执行语义真源收回 `agent_os` |
| **价值** | 解决“`agent_os` 只等于单 agent 生命周期”这一过窄定义，避免通用 workflow 执行语义散落在 `orchestrator` 和各业务域中 |

### 4.9 `orchestrator` 控制面分层

| 属性 | 说明 |
|------|------|
| **来源主线** | ①Harness + ②Agent架构 + ⑨MAS |
| **做什么** | 将 `orchestrator` 明确重构为 `domain / application / compile / runtime bridge / infra / interfaces / frameworks / fixtures` 八类分层；保留 `Mission / Node / Branch / Ledger` 作为控制面真源，并把当前 `service.py` 按 command / use-case 拆开 |
| **价值** | 防止 `orchestrator` 继续吸收 runtime/gate/domain 语义，使目录结构和代码边界能够直接表达架构，而不是继续平铺堆叠 |

## 七、实施建议

### 7.1 P0 的实施路径

P0 的 5 条行动项全部可以通过修改协议文档和 Bootstrap 约定来落地，不需要 upgrade_request：

1. 先修改 heartbeat 执行协议文档，加入 handoff 4 字段 + 退役日志 + MAST 分类 + self_reflection
2. 修改 Bootstrap 约定文档，加入高危动作白名单
3. 在下一轮 heartbeat 运行中自然生效

### 7.2 P1 的实施路径

P1 的 5 条行动项中，3.1/3.2/3.3/3.4 需要 upgrade_request 走审批。建议：

1. 先把 4 条 upgrade_request 写入 `工作区/heartbeat_upgrade_request.json`
2. 等用户批准后由主进程接管执行
3. 3.5 可直接通过修改报告模板落地

### 7.3 度量与验收

| 改进项 | 验收信号 |
|--------|---------|
| Handoff 协议 | planner 汇总时 4 字段完整率 >90% |
| MAST 分类 | 失败回执中 failure_class 字段填写率 >80% |
| 退役日志 | kill 的分支中退役日志产出率 >80% |
| 独立裁决 | 裁决结果与 executor 自报告的一致率（预期 <100%，差距即为发现的问题） |
| 压缩策略 | 同类任务在启用 compact 模式后的 token 使用降幅 |
| 渐进式信任 | 高成功率任务类型的自治等级自动提升次数 |
