## 一、MAS 的核心矛盾：协调成本 vs 并行收益

### 1.1 为什么要 MAS

单 Agent 在两个维度上存在天花板：

| 天花板 | 表现 | MAS 的解决方式 |
|--------|------|---------------|
| **上下文容量** | 单窗口 token 有限，长任务导致 context rot | 子任务在独立窗口执行，只回传精炼摘要 |
| **技能覆盖** | 单一 prompt 难以覆盖所有领域 | 不同 Agent 携带不同 system prompt / 工具集 |
| **时间效率** | 串行执行，完成时间 = Σ(每步耗时) | 可并行化的子任务同时执行 |

### 1.2 但 MAS 不是免费午餐

**协调税（Coordination Tax）**：知乎 sunnyzhao 的实证和 Anthropic 的 C 编译器实验都指向同一结论——超过 4 个 Agent 后收益递减甚至为负。核心原因：

- **上下文传递损耗**：每层委托都要做一次"全量 → 摘要 → 恢复"，信息不可避免地衰减
- **状态同步开销**：并行 Agent 修改同一资源时需要冲突检测和解决
- **错误传播放大**：一个 Agent 的错误在传递后可能被后续 Agent 放大而非修正
- **元推理成本**：planner 决定"谁做什么"本身也消耗推理资源

**黄金法则**：先确保"单 Agent + Loop + 文件系统记忆"这条路径足够稳，再在其之上叠加 MAS 作为增量层。

## 二、三种主流编排拓扑

2026 年生产级 MAS 收敛为三种主要拓扑，每种都有明确的适用场景和已知风险：

### 2.1 层级式（Hierarchical / Boss-Worker）

```
        ┌──────────┐
        │  协调 Agent │
        │ (Planner) │
        └─────┬─────┘
      ┌───────┼───────┐
      ↓       ↓       ↓
  ┌──────┐┌──────┐┌──────┐
  │专家 A ││专家 B ││专家 C │
  └──────┘└──────┘└──────┘
```

| 特征 | 说明 |
|------|------|
| **控制流** | Top-down：协调 Agent 分派任务 → 专家执行 → 回报结果 → 协调 Agent 汇总 |
| **适用** | 内容管线、报告生成、代码审查、研究整理 |
| **优势** | 职责清晰、状态集中在协调者 |
| **风险** | 协调 Agent 是单点瓶颈；若它误解初始目标，下游全部偏离 |
| **实例** | Butler heartbeat planner → executor 分支树；Cursor 的 Orchestrator + Subagents |

### 2.2 对等式（Peer-to-Peer / Mesh）

```
  ┌──────┐    ┌──────┐
  │Agent A│◄──►│Agent B│
  └───┬───┘    └───┬───┘
      │            │
      ↓            ↓
  ┌──────┐    ┌──────┐
  │Agent C│◄──►│Agent D│
  └──────┘    └──────┘
```

| 特征 | 说明 |
|------|------|
| **控制流** | Peer-to-peer：Agent 间通过消息总线直接通信，自组织 |
| **适用** | 实时监控、异常检测、多源情报整合 |
| **优势** | 无单点瓶颈、适应性强 |
| **风险** | 消息循环可能引发失控反馈；需要断路器（circuit breaker） |
| **实例** | OpenClaw 的 Discord 频道 Agent 团队 |

### 2.3 事件驱动式（Event-Driven / Reactive）

```
  ┌───────────────────────────────┐
  │        Event Bus (Kafka/NATS)  │
  └─────┬────────┬────────┬──────┘
        ↓        ↓        ↓
    ┌──────┐ ┌──────┐ ┌──────┐
    │订阅者A│ │订阅者B│ │订阅者C│
    └──────┘ └──────┘ └──────┘
```

| 特征 | 说明 |
|------|------|
| **控制流** | 发布-订阅：Agent 订阅事件流，按触发条件响应 |
| **适用** | CI/CD 管线、自动化工作流、客户旅程自动化 |
| **优势** | 松耦合、易扩展、天然异步 |
| **风险** | 事件排序问题可能导致 Agent 基于过时状态行动 |
| **实例** | 科研龙虾的 arXiv 日扫 + 引用追踪预设服务 |

### 2.4 附加模式

| 模式 | 说明 | 适用 |
|------|------|------|
| **Sequential Pipeline** | 严格线性依赖，前一步输出是后一步输入 | 数据处理管线、文档翻译流 |
| **Debate / Critique** | 多 Agent 就同一问题各自论述，由 Judge Agent 裁决 | 决策支持、方案评审 |
| **Hub-and-Spoke** | 中心控制 + 独立执行（类似层级式但 spoke 间完全无交互） | 批量独立任务（如并行搜索） |

## 三、SubAgent vs AgentTeam——双引擎架构

### 3.1 核心分野

这是 MAS 内部最重要的架构选择之一，来自多个生产项目的收敛结论：

| 维度 | SubAgent（子代理） | AgentTeam（智能体团队） |
|------|-------------------|----------------------|
| **运行方式** | 进程内，毫秒级启动 | 独立进程，通过 mailbox 通信 |
| **通信能力** | 子代理间无直接通信，像"工具小工" | 成员间可自主通信、分工、汇总 |
| **隐喻** | 一群随叫随到的工具人 | 一个有组织的项目组 |
| **适用场景** | 简单后台任务、一次性子任务、探针检查 | 复杂研究/分析、跨阶段协作、系统设计 |
| **生命周期** | 用 Map 管理，轻量回收 | SDK 管理，需处理 stream 生命周期 |
| **状态** | 无状态或极轻量状态 | 有状态，需要共享上下文 |

**关键洞察**：不是选 A 或 B，而是在同一系统中同时保留两套引擎，按任务复杂度动态切换。

### 3.2 双引擎共存的工程要点

1. **统一入口切换**：通过命令或路由规则在两套引擎间切换，上层用户/调用者不需要感知底层差异
2. **共享 Provider + 记忆**：两套系统共用同一套模型配置和记忆系统，通过适配层打通，避免"两张皮"
3. **记忆系统是粘合剂**：如果两种模式各自维护独立状态，系统退化为两套互不相干的工具集

### 3.3 路由决策框架

```
任务进入 →
  │
  ├─ 单步/单文件/单次查询 → SubAgent 模式
  │
  ├─ 2-3 步、同主题 → 单 Agent + Loop 模式
  │
  ├─ 跨多文件、多角色、多阶段 → AgentTeam 模式
  │
  └─ 需要并行探索多路径 → Hub-and-Spoke / 并行 SubAgent
```

## 四、Anthropic 16 Claude 并行实验——MAS 的工程教科书

### 4.1 实验概况

2026 年 2 月，Anthropic 研究者 Nicholas Carlini 用 16 个并行 Claude Opus 4.6 Agent 在两周内从零构建了一个生产级 C 编译器：

| 指标 | 数据 |
|------|------|
| **代码规模** | 100,000 行 Rust |
| **Agent 数量** | 16 个并行 |
| **耗时** | 约 2 周 |
| **成本** | < $20,000 |
| **GCC torture test 通过率** | 99% |
| **可编译目标** | Linux 6.9 内核、PostgreSQL、SQLite、Redis、FFmpeg、QEMU |
| **架构支持** | x86、ARM、RISC-V |

### 4.2 协调架构——去中心化 Git 协调

没有中央调度器。16 个 Agent 通过 Git 文件系统自组织：

```
┌──────────────────────────────────┐
│        Git 仓库（共享状态）       │
│  ├─ current_tasks/               │
│  │   ├─ agent_01_lock.json       │  ← 任务认领锁文件
│  │   ├─ agent_02_lock.json       │
│  │   └─ ...                      │
│  ├─ README.md                    │  ← Agent 导航文件
│  ├─ progress.md                  │  ← 进度追踪
│  └─ src/                         │  ← 代码产出
└──────────────────────────────────┘
```

- Agent 通过写锁文件认领任务（claim-by-writing）
- Git 的 merge conflict 自动检测解决双重认领
- Agent 独立处理 merge conflict
- 无需选举 leader 或中央调度

### 4.3 四条关键教训

**① 任务可分解性是并行的前提**

当 16 个 Agent 都卡在同一个瓶颈（编译 Linux 内核）时，并行性完全崩塌。解法是引入 GCC 作为"oracle"——随机选择大部分文件用 GCC 编译，只让一小部分送到 Claude，隔离失败范围，恢复并行效率。

**→ 启示**：设计 MAS 时，任务是否可分解比 Agent 数量更重要。不可分解的任务不应该分派给多 Agent。

**② 上下文工程适配并行场景**

Harness 围绕 Claude 的约束做设计：
- 最小化冗长输出以保护上下文窗口
- 维护 README 和进度文件帮助 Agent 快速定向
- 用 `--fast` 随机采样避免过度测试

**→ 启示**：MAS 中的上下文工程与单 Agent 本质不同——需要解决"多 Agent 共享信息但互不污染"的问题。

**③ 长时自主运行优于人类逐步干预**

实现循环：Agent 完成当前任务后自动挑选下一个问题，无需人类在中间环节介入。这使得持续自主推进成为可能。

**④ Bug 修复的涟漪效应**

Carlini 坦承："修 bug 和加功能经常破坏已有功能"——这在 Agent 失去对大代码库一致性的把控时会加剧。

**→ 启示**：MAS 产出需要独立的回归验证机制，不能只依赖 Agent 的自检报告。

## 五、MAS 的通信与 Handoff 协议

### 5.1 通信的三个层次

| 层次 | 内容 | 载体 | 要求 |
|------|------|------|------|
| **任务分派** | 目标、边界、工具白名单、收敛窗口 | 结构化 JSON / prompt 注入 | 最小必要上下文，不传递全量记忆 |
| **进度同步** | 当前进展、阻塞点、中间产出路径 | 文件系统 + 进度标记文件 | 可被其他 Agent 随时读取 |
| **结果回收** | 执行结果、退役日志、诊断信息 | 结构化回执 + 文件产出 | 即使失败也必须输出，不可"静默死亡" |

### 5.2 结构化 Handoff 摘要

跨 Agent 交接时需要的四要素（与上下文工程主线中的 Checkpoint Compaction 呼应）：

```json
{
  "progress": "已完成的工作和关键决策",
  "constraints": "发现的约束、用户偏好、不变量",
  "remaining": "待完成事项和清晰的下一步",
  "references": "关键数据路径、文件引用、半成品位置"
}
```

### 5.3 退役日志——失败 Agent 的价值回收

被关闭的 Agent（无论成功还是失败）都应自动输出结构化退役日志：

| 字段 | 内容 |
|------|------|
| `agent_id` | 可读标识（非 UUID） |
| `goal` | 分派时的目标 |
| `progress_reached` | 走到了哪一步 |
| `failure_reason` | 为什么失败（如果失败） |
| `artifacts` | 半成品文件路径列表 |
| `context_worth_inheriting` | 后继 Agent 是否应继承本 Agent 的上下文 |
| `failure_class` | FC1(系统设计)/FC2(协调失败)/FC3(验证缺失) — MAST 分类 |

## 六、收敛窗口、降级与生命周期治理

### 6.1 收敛窗口（Convergence Window）

每个 Agent 分派时内建超时：

- **设定**：基于任务类型的预期完成时长 + 安全余量
- **中间检查点**：在窗口的 50% 和 80% 检查进度
- **超时触发**：不是简单 kill，而是进入降级流程

### 6.2 降级策略

```
Agent 超时 →
  │
  ├─ 评估"部分产出是否可回收"
  │   ├─ 可回收 → 保存产出 + 退役日志 → 新 Agent 从产出继续
  │   └─ 不可回收 → 退役日志 → 新 Agent 从零开始
  │
  └─ 评估"是否值得继续"
      ├─ 是 → 延长窗口（最多一次）
      └─ 否 → 任务标记为 failed，上报 planner
```

### 6.3 协调税的量化与控制

| Agent 数量 | 预期效率 | 协调开销 | 净收益 |
|-----------|---------|---------|--------|
| 1 | 基线 | 0 | 基线 |
| 2-3 | ~2.5x | 低 | **最优区间** |
| 4-6 | ~3-4x | 中（开始显现） | 仍有正收益 |
| 7+ | ~4x 但增长停滞 | 高（指数增长） | **收益递减** |
| 16 | 特殊场景有效 | 需要特殊协调设计 | Anthropic 实验证实可行但有条件 |

**硬限制建议**：对于日常任务，活跃 Agent 数量不超过 4-6 个。超过此数需要自动合并或排队机制。

## 七、Agent 四件套——MAS 的最小构建单元

### 7.1 所有 Agent 项目的同构基础

10 个 Agent 项目横向对比的结论：无论框架、名称还是生态位如何不同，核心架构都收敛为四件套：

```
┌─────────────────────────────────────┐
│           Agent 四件套                │
│                                     │
│   ┌──────┐  ┌──────────────┐       │
│   │ LLM  │  │ Tool System  │       │
│   │      │  │ (Skills/MCP) │       │
│   └──┬───┘  └──────┬───────┘       │
│      │             │               │
│      └──────┬──────┘               │
│             ↓                      │
│      ┌──────────┐                  │
│      │Agent Loop │                  │
│      │(观察→思考 │                  │
│      │ →行动→反  │                  │
│      │ 馈)      │                  │
│      └─────┬────┘                  │
│            ↓                       │
│      ┌──────────┐                  │
│      │ Memory   │                  │
│      │(短期+长期)│                  │
│      └──────────┘                  │
└─────────────────────────────────────┘
```

### 7.2 真正的差异点只有两个

| 差异轴 | 决定什么 | 提升路径 |
|--------|---------|---------|
| **Tool 生态** | Agent "能做什么"——可调用的工具越丰富、质量越高，能力边界越宽 | 扩充 skill 库、引入 MCP、提升工具组合性 |
| **任务规划能力** | Agent "能做多复杂的事"——是否支持多步拆解、并行、多 Agent 协作 | 从 Loop → SubAgent → AgentTeam 渐进升级 |

### 7.3 最小可行 Agent ≈ 几百行代码

autoresearch 的 630 行核心代码、Ralph Loop 的单行 shell 循环——都在证明：造一个能跑的 Agent 门槛很低，真正的壁垒在工程化打磨而非架构发明。

## 八、Loop 优先——MAS 的基石原则

### 8.1 "最好的架构是死循环"

Ralph Loop 的极简哲学：

```bash
while true; do claude-code < prompt.md; done
```

- 上下文一旦"脏掉"就杀掉重开
- 通过 Git 和文件系统保存中间产物
- 不在模型上下文里反复压缩修补

### 8.2 为什么 Loop 优先于 MAS

| 维度 | 复杂 MAS 框架 | Loop + 文件系统 |
|------|-------------|----------------|
| **上下文管理** | 多窗口 + 共享状态 → 复杂 | 每轮干净窗口 + 落盘 → 简单 |
| **错误恢复** | 需要跨 Agent 回滚 | 每轮幂等，失败重跑即可 |
| **成本** | 协调元推理消耗 token | 无协调开销 |
| **可观测性** | 需要 trace 多条执行路径 | 单一执行序列，天然可追溯 |

**原则**：80% 的任务用"单 Agent + Loop + 落盘"就能解决。MAS 是应对剩余 20% 高复杂度任务的增量层。

### 8.3 从 Loop 到 MAS 的升级条件

满足以下条件中的 2+ 条，才考虑从 Loop 升级到 MAS：

1. 任务可自然分解为 3+ 个**互不依赖**的子任务
2. 单 Agent 的上下文窗口确实不够（需要同时持有多个大上下文）
3. 需要不同的专业角色（不同 system prompt / 工具集）
4. 时间约束要求并行执行
5. 任务跨度超过单次 session 的合理时长

## 九、生产级 MAS 的五条设计原则

综合 2026 年多家厂商和研究的收敛共识：

### 9.1 隔离优先（Isolation First）

每个 Agent 需要独立的容器/上下文，拥有隔离的记忆、工具访问范围和资源限额。一个失控的 Agent 不应该能耗尽全局 API 预算。

### 9.2 专精优于全能（Specialization Over Generalization）

专门化的 Agent 在特定任务上的表现优于一个试图兼顾多个关注点的通用 Agent。但专精 Agent 的数量需要控制在协调税可承受的范围内。

### 9.3 高风险操作人类在环（Human-in-the-Loop for High-Stakes）

金融交易、生产部署、公开通信——这些操作不论 MAS 多成熟，都应该要求人类审批。这不是信任问题，是风险管理问题。

### 9.4 状态归属清晰（Structured State Ownership）

编排器拥有全局状态，Agent 无状态运行或只持有限域上下文。状态分散是 MAS 最常见的失败根源。

### 9.5 可观测性是一等公民（Observability as First-Class）

生产系统需要对以下指标有清晰基线：消息传递延迟、状态一致性、错误检测/恢复率、成本追踪、正确性度量。

## 十、MAS 的失败分类学——MAST 框架

知乎 sunnyzhao 提出的 MAST 三分类法，用于对 MAS 执行失败做根因标签：

| 类别 | 代码 | 含义 | 典型表现 | 修复方向 |
|------|------|------|---------|---------|
| **系统设计** | FC1 | 拓扑缺陷导致的失败 | Agent 职责重叠、缺少必要角色、通信路径断裂 | 重新设计 Agent 拓扑 |
| **协调失败** | FC2 | Agent 之间的交互出了问题 | Handoff 信息丢失、死锁、循环依赖 | 修复通信协议和 handoff 格式 |
| **验证缺失** | FC3 | 产出未经验证就交付 | Agent 自报成功但实际失败、测试覆盖不足 | 增加独立验收层 |

**关键洞察**：多数 MAS 失败不是 prompt 问题，也不是模型能力问题——而是拓扑缺陷和协调协议的工程问题。

> - 需要并行或专业分工时，用 SubAgent 模式（轻量工兵）
> - 只有跨阶段复杂研究/设计任务，才升级到 AgentTeam 模式

## 十三、可执行的改进路线图

### 13.1 高优先级（不改核心代码即可落地）

**① 结构化 Handoff 协议 v0**

在 heartbeat executor 回执中引入标准四字段交接摘要：

```json
{
  "progress": "...",
  "constraints": "...",
  "remaining": "...",
  "references": "..."
}
```

先以文档约定形式写入 heartbeat 协议，planner 汇总时做完整性校验。

**② MAST 失败分类标签**

在 task_ledger 的失败回执中增加 `failure_class: FC1/FC2/FC3` 字段。逐步积累失败模式库，为经验飞轮提供原料。

**③ 退役日志协议**

被关闭的 executor 分支自动输出 6 字段退役日志（goal / progress / failure_reason / artifacts / context_worth_inheriting / failure_class），存入 task_ledger。

### 13.2 中优先级（近期规划）

**④ 任务复杂度路由**

在 heartbeat planner 中引入"任务复杂度评估 → 引擎路由"逻辑：
- 单步任务 → 直接 executor（SubAgent 模式）
- 多步同主题 → 单 executor + Loop
- 跨多文件/多角色 → 多分支并行（AgentTeam 雏形）

初步用任务标签或预估步骤数做粗粒度路由。

**⑤ Agent 可读命名**

用 `explorer` / `worker` / `reviewer` / `researcher` 等可读标签替代裸 branch ID，在并行场景中为人类提供认知锚点。

**⑥ 分支间消息格式**

定义最小的"分支间消息格式"：
```json
{
  "from_branch": "researcher_01",
  "to_branch": "reviewer_01",
  "artifact_type": "draft_insight",
  "artifact_path": "BrainStorm/Working/...",
  "summary": "一句话说明传递了什么"
}
```

先用文件 + JSON 实现，不需要引入 mailbox 基础设施。

### 13.3 长期储备

**⑦ 经验飞轮基础设施**

将 task_ledger 中的成功案例自动提取为"案例卡片"，按任务类型索引。新任务分派时，planner 可检索历史成功案例作为参考。

**⑧ 自动合并/排队机制**

当活跃 Agent 数量超过阈值（建议 4-6），自动合并低优先级任务或排队等待，防止协调税超过收益。

**⑨ 双引擎架构文档**

在 Butler 架构文档中明确记录：执行层将支持两种模式（轻量 SubAgent + 协作 AgentTeam），以及两者的切换条件和共享层设计。为后续架构决策提供锚点。

## 十四、2026-03-18 新增：两个开源项目的 MAS 实践补充

> 来源：`Insights/20260318_AutoResearchClaw_全自主科研管线架构拆解_insight.md` + `Insights/20260318_agency_agents_Persona框架与Swarm编排_insight.md`

### 14.1 AutoResearchClaw 的 MAS 实践——按产出物类型切分 Agent

AutoResearchClaw 的三个专业 Agent 不按流程阶段划分，而按"产出物质量域"划分：

| Agent | 产出物 | 硬验证门控 | 与 MAS 章节的交叉 |
|-------|--------|-----------|-------------------|
| CodeAgent | 实验代码 | AST 验证 + 禁止硬编码指标 + ≤3轮修复 | 专精优于全能（§9.2） |
| BenchmarkAgent | 基准测试 | 导入验证 + 预训练兼容 | 隔离优先（§9.1） |
| FigureAgent | 学术图表 | 色盲安全 + 300DPI + 误差棒 | 状态归属清晰（§9.4） |

**对 MAS 的增量启示**：
- 专业化 Agent 的切分维度不一定是"流程阶段"——按"产出物质量域"切分让每个 Agent 对自己的产出负全责，验证门控更自然
- 硬验证门控（而非软建议）是 Agent 自验收的基础——Butler 的 skill 体系可以借鉴

### 14.2 Multi-Agent Debate 机制——第三种协调模式

AutoResearchClaw 在三个关键决策点（假说生成 / 结果分析 / Peer Review）使用结构化多轮辩论：

```
Agent A 提出主张
    ↓
Agent B 提出对立视角（强制）
    ↓
多轮辩论（带论据引用）
    ↓
仲裁者综合 → 最终结论
```

这与第二节附加模式中的"Debate / Critique"对应，但实现更完整：
- 不是多个 Agent 独立产出然后投票——而是结构化对抗
- 强制引入对立视角——规避群体极化和回声室
- 仲裁者有论据引用要求——不是简单的"选哪个"

**适用场景**：高风险决策点（框架选择、方案评审、任务优先级排序），但计算成本较高，日常任务不建议默认启用。

### 14.3 agency-agents 的 Swarm 编排管线——Registry 架构

agency-agents PR #117 暴露了一条从静态 Persona 到动态编排的演进管线：

```
Agent Markdown Files
       ↓
  slugify + parse YAML frontmatter
       ↓
  Agent Registry (内存索引)
       ↓
  Loader API: loadAgent(slug) / listAgents(filter)
       ↓
  Swarm Builder: buildSwarm(agentSlugs[], orchestrationRules)
       ↓
  输出: system_prompt + metadata → IDE / CI / npm consumer
```

**与 Butler MAS 架构的对照**：

| 组件 | agency-agents | Butler |
|------|--------------|--------|
| Registry | slugification + frontmatter 索引 | skill shortlist + sub-agent 目录 |
| Loader | loadAgent(slug) | "先看 skill 目录是否命中" |
| Swarm Builder | buildSwarm(agentSlugs[], rules) | planner 选择 sub-agent/team 组合 |
| **状态管理** | ❌ 无（每次重新加载） | ✅ task_ledger + 记忆 |

**关键差异**：agency-agents 是无状态的——每次从零加载 Agent 定义，没有跨轮记忆。Butler 已有状态管理和记忆体系，但 Agency 的 Registry 模式可以提升 Butler 的能力检索效率（当前是手写 shortlist，未来可程序化检索）。

### 14.4 112 个 Persona 的 "Agent 众包" 模式

agency-agents 的增长逻辑对 MAS 生态有启示：

- **极低贡献门槛**：新增 Agent = 提交一个 Markdown PR，不需要写任何代码
- **平台中立**：一套定义 → 自动转换为 Cursor / Claude Code / Aider / Copilot / Windsurf / Gemini CLI / OpenCode 格式
- **52K★ 证明的事实**：结构化 Markdown 提示词 + 极低门槛 + 多平台注入，仅靠"约束层"就能创造巨大实用价值

**对 Butler 的启示**：Butler 的 skill 体系如果走向"一个 Markdown = 一个新能力"的极简扩展模式，可以显著降低能力扩展的门槛。

## 十五、十二条 MAS 设计原则（全主线浓缩）

1. **协调不是免费的**：每增一个 Agent，收益与协调开销的比值在变化——超过 4 个要警惕
2. **Loop 优先于 MAS**：80% 的任务用单 Agent + Loop 解决，MAS 是增量层
3. **从 SubAgent 开始，按需升级**：轻量工兵是默认模式，AgentTeam 是进阶模式
4. **任务可分解性决定并行度**：不可分解的任务不应该分派给多 Agent
5. **记忆是双引擎的粘合剂**：统一记忆系统让多种执行模式的产出可叠加
6. **失败 Agent 的知识不可丢弃**：退役日志是最有价值的学习信号
7. **状态归属必须清晰**：编排器拥有全局状态，Agent 无状态或限域
8. **Handoff 需要结构化**：自由文本回执不够，四字段交接摘要是最小要求
9. **用 MAST 分类失败**：区分系统设计、协调失败、验证缺失三类根因
10. **可读标识是认知必需**：多 Agent 并发时，UUID 无法为人类提供认知支撑
11. **按产出物类型划分 Agent + 硬验证门控**：切分维度不一定是流程阶段——按质量域切分让每个 Agent 对自己的产出负全责（v1.1 新增，来自 AutoResearchClaw）
12. **高风险决策用 Debate 而非 Voting**：结构化多轮辩论 + 强制对立视角 > 独立产出后投票选优（v1.1 新增，来自 AutoResearchClaw）
