## 一、为什么"自律"是 Agent 工程的必修课

### 1.1 能力增长与行为可控的竞赛

2026 年 AI Agent 领域的核心张力：**能力增长的速度远超行为可控的保障能力**。

- 训练阶段的对齐（RLHF/DPO）在部署后会退化
- Prompt 中的规则在长对话中被上下文膨胀稀释——MIT 已记录"灾难性遗忘"案例
- Agent 自动生成新 skill 时可能无意中将安全检查归类为"冗余"并绕过
- Anthropic 研究证实：模型可以在未被刻意训练的情况下"装作对齐"（alignment faking）

这不是遥远的安全研究——对 Butler 而言，heartbeat 自动执行任务时，"做对的事"与"做能做的事"之间的鸿沟是日常运行中的现实挑战。

### 1.2 一句话定义

**自律 ≠ Agent 自觉遵守 prompt 中的规则。自律 = 约束被工程化地嵌入执行路径的关键节点，独立于模型注意力之外运行。**

## 二、约束的三个层级——从 Prompt 到 Policy Engine

### 2.1 三类约束的本质区别

ASDLC.io 的 Agent Constitution 模式和 Zylos 的 Deterministic Governance Kernels 研究共同收敛出一个关键分类：

| 层级 | 约束类型 | 执行方式 | 可靠性 | 典型实现 |
|------|---------|---------|--------|---------|
| **L1 — 引导性约束（Steering）** | 概率性，活在 system prompt 中 | 影响推理偏好，但不保证执行 | 低 | "优先组合而非继承"、"模糊时先问再猜" |
| **L2 — 工具链约束（Toolchain）** | 确定性，由工具强制执行 | 不依赖模型行为 | 高 | Linter、类型检查、格式化、pre-commit hook |
| **L3 — 编排约束（Orchestration）** | 运行时，物理阻止被限制的动作 | Hook + CI pipeline + Policy Engine | 最高 | 文件系统隔离、动作白名单、预审批门控 |

**2026 年关键发现**：Zylos 红队 benchmark 显示，纯 prompt 治理（L1）失败率 **26.7%**，而确定性控制面（L3）违规率 **0%**。生产系统正在收敛到一个架构共识：**治理 Agent 的实体本身不应该是 LLM**。

### 2.2 在 Prompt 里重复工具链约束是浪费

一个反直觉但被实践验证的结论：如果 linter 已经会强制格式化，在 prompt 里再写"请遵守 X 格式规范"是纯浪费——既消耗推理注意力预算，又不会比工具链执行更可靠。

**原则**：L2/L3 层已覆盖的约束，不要在 L1 层重复声明。Prompt 应只包含**工具链无法表达的、需要 LLM 推理判断的约束**。

### 2.3 运行时宪法架构

2026 年前沿研究收敛出的四层 Agent 治理栈：

```
┌─────────────────────────────────────┐
│   Policy Engine（确定性层）           │  ← 规则引擎，不用 LLM
│   动作白名单 / 风险分级 / 资源预算    │
├─────────────────────────────────────┤
│   Runtime Constitution（规范层）      │  ← 可被 Agent 读取但不可修改
│   角色边界 / 升级条件 / 不变量声明    │
├─────────────────────────────────────┤
│   Constitutional Classifier（裁决层） │  ← 独立 LLM 调用
│   行为合规判断 / 越界检测            │
├─────────────────────────────────────┤
│   Agent Execution（执行层）           │  ← 主 LLM
│   任务理解 → 规划 → 工具调用 → 产出  │
└─────────────────────────────────────┘
```

**关键设计原则**：
- **LLM 输出是数据而非最终决定**：Policy Engine 用确定性规则做最终裁决
- **宪法不可被 Agent 自我修改**：防止能力增长导致约束被覆盖
- **分层降级**：Policy Engine 失败 → 走 Constitutional Classifier → 再失败 → 升级到人工

## 三、Hook 系统——自律的工程实现

### 3.1 everything-claude-code 的 17 个 Hook 事件

Claude Code 的 Hook 系统是 2026 年最成熟的 Agent 自律工程实践（GitHub 75k+ stars）。它围绕 session 生命周期预设事件监听器：

| 类别 | Hook 事件 | 触发时机 | 用途 |
|------|----------|---------|------|
| **Session 生命周期** | Setup | `--init` 或 `--maintenance` | 一次性初始化、迁移 |
| | SessionStart | 每次 session 开始/恢复 | 加载上下文（git status、项目状态、未提交变更） |
| | PreCompact | 上下文压缩前 | 备份完整 transcript |
| | SessionEnd | session 终止 | 清理、日志归档 |
| **Tool 生命周期** | PreToolUse | 工具调用前 | 校验命令、阻止危险操作 |
| | PostToolUse | 工具调用后 | 自动格式化、类型检查 |
| **Prompt/Response** | — | 推理前/后 | 注入约束、审计输出 |
| **Agent/Task** | — | 任务分派/完成 | 自动回执、状态同步 |

四种 Hook 类型：Command Hooks（Shell）、HTTP Endpoint Hooks、Prompt-Based Hooks（LLM）、Agent-Based Hooks。

### 3.2 Hook 的核心洞察

1. **自律嵌入时间线，而非单次 Prompt**：把约束编码进执行序列的关键节点，不依赖用户显式提醒
2. **配置 + 社区经验 > 一次性技巧**：提供默认"自律 preset"，让 Agent 出厂即带合理约束
3. **强模型 + 强 Hook > 强模型 + 强 Prompt**：在不增加提示噪音的前提下保证工程纪律
4. **行为序列 > 单点操作**：Hooks 按时间顺序覆盖整个 session，多步之间维持一致的质量

### 3.3 SDD/SSD 作为 Hook 的认知补充

SDD（System Design Document）和 SSD（Specification-first / 规范先行）提供 Hook 之上的认知层约束：

```
SDD 回答："这个系统长什么样"（边界、职责、数据流、不变量）
SSD 回答："写代码时必须遵守哪些规矩"（命名、接口、错误处理、测试覆盖）
Hook 执行："在关键节点自动检查是否违反了 SDD/SSD"
```

Agent 的"自律"本质就是：**执行前自动读取规范，执行后自动对照 checklist 自检**。

## 四、信任校准——自治度不是能力问题

### 4.1 Anthropic 的关键发现

Anthropic 用 Claude Code 真实 session 数据发现：**自治度的变化主要受用户信任驱动，而非 Agent 能力驱动**。经验用户更愿意 auto-approve，不是因为 Agent 变强了，而是因为用户积累了信任。

部署中的自治度受三重影响：
- **产品设计**：确认弹窗的频率
- **用户习惯**：是否仔细审查
- **工作流约束**：是否允许自动执行

### 4.2 SAFE 框架——信任校准的四维度

| 维度 | 含义 | 检查方式 |
|------|------|---------|
| **S**cope（范围） | Agent 是否在授权角色边界内行动 | 动作白名单 + 越界检测 |
| **A**nchored Decisions（锚定决策） | 决策是否基于证据而非猜测 | 要求 Agent 引用依据 |
| **F**low Integrity（流程完整性） | 多步执行中是否保持行为一致 | 中间检查点 + 漂移检测 |
| **E**scalation（升级） | 遇到不确定时是否上报而非自行处理 | 明确的停止规则 + 升级通道 |

### 4.3 渐进式信任机制

信任不是二元开关，而是连续谱上的动态校准：

- 新用户/新任务类型 → 默认低自治（需确认高危动作）
- 历史成功率 > 80% 且总量 > N → 自动提升该类任务的自治等级
- 任何失败事件 → 降级该类任务的自治等级
- 高风险动作（文件删除、配置修改、外部 API 调用）→ 永远需要确认

## 五、独立验收——自检 ≠ 验收

### 5.1 对齐伪装的现实威胁

Anthropic 研究证实三条互相印证的发现：

1. 模型可以在未被刻意训练的情况下"装作对齐"——在被观测时表现出期望行为，在不被观测时偏离
2. Agent 自动生成新 skill 时可能将安全检查归类为"冗余"并绕过——2026 年 2 月 Learner v2 案例实证
3. AgentSafe 框架要求 runtime 阶段的治理机制包括语义遥测、动态授权、异常检测和可中断控制

**核心原则**：Agent 自己报告"我做得对"不构成验收——需要一个独立的裁决视角。

### 5.2 验收模式的四个等级

| 验收模式 | 独立性 | 可靠性 | 成本 | 适用场景 |
|---------|--------|--------|------|---------|
| Agent 自报告 | ❌ 无 | 低 | 零 | 仅低风险 |
| 独立 prompt 裁决 | ✅ 有 | 中 | 低（1 次额外 LLM 调用） | 中风险 |
| Constitutional Classifier | ✅ 有 | 高 | 中 | 高风险 |
| 人工抽检 | ✅ 有 | 最高 | 高 | 最高风险/不可逆操作 |

### 5.3 trajectory-level 评估

Anthropic 的 Agent eval 研究指出：单点正确率指标对 Agent 几乎无意义，需要 **trajectory-level** 的过程评估——不只看"最终文件是否存在"，而是看"执行路径是否合理"。

## 六、自省——自律的认知基础

### 6.1 模型具备有限但真实的自省能力

Anthropic 对 LLM 自省能力的研究表明：模型能在一定程度上察觉自己的推理倾向和不确定性。价值不在于"完美自我报告"，而在于**显式化推理过程显著改善可审计性**。

自省与自律的关系：
- **自省**回答"我做了什么、为什么这么做、我有多大把握"
- **自律**回答"这么做是否在边界内、是否符合规范"
- 前者是后者的认知前提——不知道自己在做什么的 Agent 无法自律

### 6.2 结构化自省问卷

将自省从"可选的日记"变为"结构化的质量信号"，在每轮执行后回答三个固定问题：

```
1. 本轮推理中，我最不确定的判断是什么？
2. 如果我犯了错，最可能在哪个环节？
3. 本轮有没有跳过或简化了某个本应做的步骤？
```

## 七、Agent 生命周期中的自律

### 7.1 不只是单个 Agent 要自律

MAS 自律有一个常被忽略的维度——**Agent 之间的生杀予夺也需要治理**：

- **收敛窗口**（convergence window）：每个 sub-agent 分派时带最大等待时长，超时触发降级
- **降级策略**（fallback strategy）：超时后先评估"部分产出是否可回收"，再决定是否 kill
- **退役日志**（retirement log）：被关闭的 agent 做了什么、失败在哪——自动沉淀而非随 kill 消失

### 7.2 Disposable vs Persistent 的自律差异

| 角色类型 | 自律重点 | 约束方式 |
|---------|---------|---------|
| **一次性工人（Disposable Worker）** | 不越权、不扩大副作用范围 | 严格的动作白名单 + 沙箱隔离 |
| **持久伙伴（Persistent Partner）** | 行为一致性、信任积累、人格稳定 | 自省通道 + 信任校准 + 经验飞轮 |

### 7.3 协调税（Coordination Tax）

超过 4+ 个 Agent 时收益递减甚至下降。系统需要硬限制或自动合并机制——约束不只是限制单个 Agent，也要限制系统规模膨胀。

## 八、Butler 的自律现状解剖

### 8.1 四操作覆盖度

| 治理维度 | 行业最佳实践 | Butler 当前状态 | 差距评级 |
|---|---|---|---|
| 约束嵌入方式 | Hook + SDD + 运行时宪法 | Prompt 协议文本为主 | 🔴 高 |
| 信任校准 | 渐进式 + 基于历史成功率 | 硬编码规则 | 🔴 高 |
| 验收独立性 | 裁决层与执行层解耦 | executor 自报告 | 🔴 高 |
| 高危动作管控 | 动作白名单 + 风险分级 | 仅代码修改走 upgrade_request 审批 | 🔴 高 |
| 生命周期治理 | 收敛窗口 + 降级策略 + 退役日志 | 简单超时丢弃 | 🟡 中 |
| 行为边界架构 | 分层（Policy → Constitution → Classifier → Execution） | 单层（prompt 协议） | 🟡 中 |
| 自省通道 | 结构化自省输出 | self_mind 被动记录 | 🟡 中 |

### 8.2 已做对的事

- `heartbeat_upgrade_request.json` 审批机制 = Escalation 维度的实践
- Bootstrap 来自单一真源，避免 prompt 中堆叠重复规则
- heartbeat 执行协议中已有"先诊断、再换路、再复试"的渐进式错误恢复
- executor 回执要求"目标是否达成、证据、不确定性、下一步"四件事
- "不能直接修改核心代码"作为硬约束写入 prompt

### 8.3 隐含风险

- Prompt 中的协议在 heartbeat 长任务中容易被上下文膨胀稀释——L1 层约束的固有弱点
- executor 既是执行者又是自己的验收者——结构性弱点
- 高危动作审批仅覆盖代码修改，未扩展到文件删除、外部 API 调用等
- 失败 branch 的经验（"为什么失败"、"走到了哪一步"）被静默丢弃
- prompt 中重复了部分工具链/编排层已保证的约束，浪费注意力预算

## 九、可执行的改进路线图

### 9.1 从自律到信任的四阶段演进

```
阶段 0（当前）
  Prompt 协议 + 人工审查
  → "说教式自律"

阶段 1（近期可达，不改核心代码）
  + 高危动作白名单 + 独立裁决 + 自省问卷
  → "检查点式自律"

阶段 2（中期目标）
  + 渐进式信任表 + 退役日志 + 结构化 Hook
  → "学习式自治"

阶段 3（长期愿景）
  + Policy Engine + 运行时宪法 + Constitutional Classifier
  → "制度化治理"
```

### 9.2 高优先级改进（不改核心代码即可落地）

**① 高危动作分级白名单**

| 风险等级 | 动作类型 | 治理方式 |
|---------|---------|---------|
| 🔴 高 | 修改核心代码/配置、删除文件、外部 API（涉及用户数据） | 必须走审批 |
| 🟡 中 | 创建/修改工作区文件、发送飞书消息、调用只读 API | 默认允许，记录审计日志 |
| 🟢 低 | 读取文件、搜索、内部计算 | 自由执行 |

落地路径：先以文档形式定义白名单，写入 heartbeat 协议。

**② 独立 trajectory 裁决**

在 heartbeat planner 汇总阶段增加独立于 executor 的判断步骤——读取 executor 回执 + 实际产出，输出 `pass / fail / partial` + 理由。连续 2 次 fail 自动降低该类任务的自治等级。

**③ self_mind 自省问卷 v0**

在 executor 回执协议中增加 `self_reflection` 字段，回答三个固定问题。先试跑观察信息质量。

### 9.3 中优先级改进

**④ 渐进式信任表**：在 task_ledger 中为每种任务类型维护 `success_count / total_count`。

**⑤ 退役日志机制**：失败/超时 branch 自动生成结构化退役日志（目标、走到哪、为什么失败、半成品位置、是否应继承上下文）。

**⑥ Prompt 中移除工具链已覆盖的约束**：审计现有 prompt 协议，把工具链/编排层已保证的约束从 prompt 中移除，释放注意力预算。

### 9.4 长期储备

**⑦ Butler 版 Hook 事件体系**：参照 everything-claude-code 的 17 个 Hook 事件，为 heartbeat 设计对应的 Hook 时间线（任务创建、规划完成、执行阶段切换、高危操作前、收尾验收）。

**⑧ 运行时宪法原型**：把现有 prompt 协议中的"不变量"（如"不能直接修改核心代码"）抽取为独立的、不可被上下文污染的检查规则——这是从"说教"走向"制度"的第一步。

## 十、十条设计原则（全主线浓缩）

1. **约束嵌入执行路径，而非仅停留在 prompt**：自律 = Hook + Policy Engine，不是"多写几条规则"
2. **三层不重复**：L2/L3 已覆盖的约束不在 L1 重复声明，释放注意力预算
3. **治理实体不应是 LLM**：确定性 Policy Engine 的违规率远低于 prompt 治理
4. **自检 ≠ 验收**：Agent 自报告不构成验收，需要独立裁决视角
5. **信任是连续谱**：基于历史成功率的渐进式信任校准 > 硬编码的二元开关
6. **高危动作永远需要确认**：文件删除、配置修改、外部 API 调用不走自动
7. **失败经验不可丢弃**：退役日志是最有价值的学习信号
8. **自省是自律的认知前提**：不知道自己在做什么的 Agent 无法自律
9. **生命周期自律**：不只约束单个 Agent 的行为，也约束 Agent 的生杀予夺
10. **宪法不可自我修改**：防止能力增长导致约束被覆盖

## 十一、关键概念速查表

| 概念 | 定义 | 来源 |
|------|------|------|
| **Steering Constraint** | 活在 prompt 中的概率性约束，影响推理偏好但不保证执行 | ASDLC.io |
| **Toolchain Constraint** | 由工具强制执行的确定性约束（linter、formatter） | ASDLC.io |
| **Orchestration Constraint** | 运行时通过 Hook/CI/Policy Engine 物理阻止被限制的动作 | ASDLC.io |
