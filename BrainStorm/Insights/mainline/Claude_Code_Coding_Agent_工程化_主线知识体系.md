## 一、Agent 成熟度的五道坎

来源：learn-claude-code 项目（s01-s12 渐进式教程）

### 1.1 核心论点

> **Agent 真正的差异化不是 Prompt 质量，而是工程能力。**

五个问题决定一个 Agent 能否从"能跑"到"能信任"：

| # | 工程问题 | 说明 |
|---|---------|------|
| 1 | **状态持久化** | 状态如何存储 |
| 2 | **上下文管理/压缩** | 上下文如何控制 |
| 3 | **任务跟踪/规划** | 任务如何追踪 |
| 4 | **错误恢复/回滚** | 失败如何处理 |
| 5 | **多 Agent 协调** | Agent 间如何对齐 |

### 1.2 十二步渐进式架构

learn-claude-code 每课加一个机制：

```
单工具 → 循环 → 显式规划 → 上下文压缩 → 子 Agent 隔离 →
Skill 懒加载 → 任务持久化 → 并发隔离 → Worktree 级协作
```

设计原则：**好的工程让系统在复杂性下变得更简单，而非更复杂。** 终态是"像乐高一样可组合"——可隔离、可替换、可组合的模块。

### 2.1 三层 Agent 架构

```
┌──────────────────────────────┐

├──────────────────────────────┤

│  共享任务队列和收件箱          │
```

### 2.2 Subagent vs Agent Team

| 维度 | Subagent | Agent Team |
|------|----------|------------|
| 上下文 | 隔离的 200K-token 窗口 | 持久共享上下文 |
| 通信 | 父子单向（Task 工具） | 磁盘 JSON 文件 + 队列 + 收件箱 |
| Token 消耗 | 基线 | ~2x（因持久上下文维护） |
| 适用 | 单次派发任务 | 复杂多阶段协作 |

### 2.3 五个扩展点（自律基础设施）

| 扩展点 | 功能 |
|--------|------|
| **CLAUDE.md** | 项目规则和约定的指令文件 |

## 三、自律系统——Hook 作为行为自动化

### 3.1 核心概念

来源：everything-claude-code（Anthropic hackathon 冠军的配置，GitHub 75k+ stars）

> **自律 = 在正确的时间触发正确的 Hook，而非期望你记得做正确的事。**

Hook 是在两个时刻触发的动作："即将做 X"和"刚完成 X"。覆盖格式化、类型检查、测试运行、commit 消息生成、会话记忆管理等。

### 3.2 会话时间线视图

不按配置文件组织，而是映射到一个编码会话的时间线：

```
会话开始 → 同步上下文，设置工作模式
    │
    ▼
保存前 → 自动格式化，类型检查
    │
    ▼
提交前 → 运行测试，lint，生成变更摘要
    │
    ▼
会话结束 → 整合对话亮点，更新任务状态记忆
```

效果：**Claude 在不被要求的情况下完成所有日常打理。**

### 3.3 三条自律设计原则

1. **Hook = 行为触发点**——把纪律嵌入时间线，而非单个 prompt
2. **自律是配置 + 社区最佳实践**，不是一次性技巧——应该随附"纪律预设"
3. **强模型 + 强 Hook > 单次提示**——把约束编码到 Hook 中避免 prompt 噪音，同时保证工程习惯

### 3.4 关键洞察

Hook 覆盖**整个**会话（分支创建 → 依赖安装 → 文件修改 → 测试 → 提交）。质量和上下文一致性在多个步骤中维持，而非仅在单个检查点。

## 四、上下文工程——六大厂方案对比

### 4.1 统一问题

所有六家（Manus / Cursor / Anthropic / OpenAI / Google / LangChain）在解决：**LLM 应该看到什么信息、何时看到、如何组织？**

最好的团队在**做减法**，不是堆叠复杂度——Manus 重写了 5 次框架，每次都在删东西。

### 4.2 四项产业共识

| 共识 | 细节 |
|------|------|
| **文件系统 = 扩展记忆** | 大型输出写入文件，上下文只持有轻量引用 |
| **动态 > 静态检索** | 从预填充 RAG 转向循环内按需拉取；Cursor 懒加载 MCP 节省 46.9% token |
| **长任务用持久化计划文件** | 结构化 spec/plan 文件作为"锚点"对抗上下文腐烂和目标漂移 |
| **保留错误不清理** | 保留错误轨迹用于隐式信念更新；避免重复犯错 |

### 4.3 厂商对比精华

**Manus** — KV-cache 视为"神圣资源"：
- 缓存 token 成本约为未缓存的 1/10（0.3 vs 3/MTok）
- 日志只追加不重排；甚至 JSON key 顺序变化都会破坏缓存
- 工具永久加载 + logit mask 控制每步可用性
- 工具输出存为 full + compact 双版本；旧结果压缩为 compact（仅路径引用）

**Cursor** — 动态上下文发现 + 文件系统记忆：
- 最小化预注入；Agent 按需拉取上下文（有 A/B 数据支撑）
- 聊天历史、skill、终端会话视为"可搜索文件"
- 懒加载 MCP 工具：只预加载工具名，按需获取完整定义
- 规则控制在 ~500 行，用 `@file` 引用而非内联复制
- Subagent 将重任务卸载到独立上下文窗口

**Anthropic (Claude)** — 注意力预算 + 循环内检索：
- "注意力预算"框架：系统 prompt 在"金发姑娘区"，既不过度工程化也不模糊
- 从"预推理 RAG"转向"循环内检索"——执行过程中持续拉取信息
- Memory Tool（beta）：上下文窗口外的持久存储；增删改查；案例显示 ~84% token 减少

**OpenAI** — 会话记忆作为基础设施：
- 三种模式：截断 / 压缩 / 基于状态的长期记忆
- 结构化状态对象（profile + notes）跨会话持久化 Agent "信念"
- Prompt Caching：相同前缀复用；≥1024 token 自动参与；~80% 延迟降低，~90% 输入成本降低

**Google** — 赌丰裕（2M-10M token 窗口）：
- 但自己的研究显示随上下文增长 15-47% 性能下降
- 需要 ReadAgent/Gist Memory 等补充方案

**LangChain** — 四种操作分类法：

| 操作 | 含义 |
|------|------|
| **Write** | 写入长期存储 |
| **Pull** | 按需检索上下文 |
| **Compress** | 摘要 / 截断 / 降低精度 |
| **Isolate** | 在独立上下文中执行子任务 |

### 4.4 核心争论：长上下文 vs 智能压缩

Google 押注"丰裕"（大窗口）。其他五家押注"智能工程"。Google 自己的数据显示更大窗口不能消除性能下降。**两条路径将在张力中共存。**

## 五、Butler Prompt 架构内部解剖

### 5.1 15 区块 Prompt 结构

`build_feishu_agent_prompt()` 通过 `"\n\n".join(blocks)` + 尾部用户消息组装：

| 区域 | 内容 | 稳定性 |
|------|------|--------|
| 1-4 | 身份/角色 → 场景/模式 → Bootstrap (SOUL/TALK/USER/TOOLS/MEMORY_POLICY) → 基线行为 | **高**（跨轮稳定） |
| 5 | 对话上下文：意识摘录、SOUL 摘录、用户画像、长期记忆命中、self_mind | **中**（每轮变化，长度受控） |
| 6-14 | 协议、skill 目录、回复要求、文件占位符（模式条件化） | **中高**（模式内稳定） |
| 15 | 用户消息 | **低**（每轮变化） |

### 5.2 缓存友好度评估

Butler 已遵循"静态前缀 + 动态后缀"——天然适配 OpenAI Prompt Caching。但风险：
- 区块 5（对话上下文）每轮变化且在 prompt 中段，可能打断其后所有内容的缓存
- Bootstrap 的细微变化（甚至多一个换行）使整个缓存失效
- 缺少显式"缓存边界标记"用于诊断

### 5.3 四条缓存友好规则

1. **前缀稳定性**：系统指令/规则/示例在前面，格式冻结
2. **仅追加**：历史日志只追加，不重排/插入
3. **掩码代替删除**：未使用的工具标记为不可用，而非从列表中删除
4. **大输出写文件**：大的工具输出写入文件，prompt 只持有引用

## 六、上下文交接协议——CONTEXT CHECKPOINT COMPACTION

### 6.1 跨 LLM 交接的四要素

```
当前进度 + 关键决策
    ↓
重要约束 + 用户偏好
    ↓
待完成事项 + 清晰的下一步
    ↓
关键数据 + 示例 + 引用
```

### 6.2 强制 QA 步骤

生成交接摘要前，必须逐字引用包含 `CONTEXT CHECKPOINT`、`handoff summary`、`concise` 或 `seamlessly` 等关键词的消息。这**防止多跳接力中的指令漂移**。

### 6.3 Butler 差距

| 交接要素 | Butler 现状 | 差距 |
|---------|------------|------|
| 进度 + 决策 | heartbeat 回执"做了什么" | 缺少显式"关键决策"记录 |
| 约束 + 偏好 | user_preferences + SOUL 摘录 | 存在但不总是带入交接 |
| 剩余 + 下一步 | task_ledger next_steps | 格式不一致 |
| 关键数据 + 引用 | 工作区文件路径 | 分散在回执中，非结构化 |
| 强制 QA | **无** | 完全缺失——无交接完整性验证 |

## 七、SDD + 规范先行开发——Vibe Coding 的护栏

### 7.1 问题：结构漂移

Vibe coding（在 LLM 对话中迭代塑造代码）高效但天然倾向于**结构漂移**：每轮对话都可能把代码带离整体架构。写得越快，偏离概率越高。

### 7.2 两层防御

| 层 | 文件 | 回答的问题 |
|---|------|-----------|
| **SDD** (System Design Document) | 系统边界、模块职责、关键数据流、约束 | "这个系统长什么样？" |
| **SSD** (Specification-first / 规范先行) | 命名约定、模块边界与接口契约、错误处理模式、日志/监控要求、测试覆盖 | "构建过程中如何约束行为？" |

### 7.3 标准化重构会话

```
读 SDD → 对齐 SSD 规范 → 定义本次变更边界 →
LLM 实现 → 自检 SDD/SSD → 产出会话总结
```

这个流程可模板化、Agent 自动化、被 Harness 层监督。

### 7.4 核心洞察

> **Agent 自律 = 内化的 SDD/SSD。**

当 Agent 自动在执行前读取规范、执行后对照清单检查时，它**就是在做 SDD/SSD 驱动的开发**。自律系统是把 SDD/SSD 注入到 prompt 模板和验收 Hook 中的工程实现。

## 八、十条可执行设计原则（全主线浓缩）

1. **工程能力 > Prompt 质量**——状态持久化、上下文压缩、错误恢复、多 Agent 协调才是真正的差异化。
2. **Hook 时间线 > 单次提示**——把自律嵌入会话生命周期（开始/保存前/提交前/结束），而非堆在 system prompt 里。
3. **文件系统是一等扩展记忆**——大输出写文件，上下文只持有引用，日志只追加不重排。
4. **懒加载工具定义**——只预加载工具名索引，按需获取完整定义（节省 ~47% token）。
5. **Full + Compact 双存储**——心跳回执、self_mind 快照等同时保留完整版和紧凑版，上下文按需选择精度。
6. **缓存边界意识**——prompt 组装中标记缓存边界，静态前缀冻结格式，动态内容尽量追加到尾部。
7. **结构化交接摘要**——用 4 字段模板（进度/约束/剩余/引用）+ QA 完整性验证替代自由文本回执。
8. **SDD/SSD 先行**——每个子系统有 1-2 页轻量设计文档，Agent 可在执行前参考、执行后自检。
9. **掩码代替删除**——工具和 skill 不用时标记为不可用，而非从上下文中移除（保持缓存稳定）。
10. **上下文腐烂是工程问题**——不是等模型更聪明，而是用压缩/隔离/重置/持久化来工程化解决。

## 九、Butler 对标与优先差距

| 差距 | 紧迫度 | 描述 |
|------|--------|------|
| **上下文压缩** | ★★★ | 从隐式 max_chars 截断升级为显式语义压缩 + 可配置模式（full/compact/reference_only） |
| **结构化交接** | ★★★ | 用 4 字段模板 + QA 验证替代自由文本回执 |
| **失败回滚** | ★★☆ | 在基本重试之上加结构化回滚点 |
| **缓存边界标记** | ★★☆ | prompt 组装中加诊断标记，量化缓存命中率 |
| **Skill 懒加载** | ★★☆ | 从启动时全量注入改为索引 + 按需注入 |
| **SDD per 子系统** | ★☆☆ | heartbeat / 记忆 / 工具编排 / 飞书适配器各一份，Agent 可引用 |
| **并发隔离** | ★☆☆ | 当前单线程，未来需要 Worktree 级隔离 |
