## 第 1 页：我们真正缺的不是算法，而是“方法操作系统”

### 核心 message

今天的问题不是“没有方法”，而是“方法太多，却缺乏有机组织”。

### 展开

- 面对一个轨迹还原任务，我们可能同时有很多降噪、插值、地图匹配、分段、约束优化方法。
- 真正困难的地方不是再找一个新算法，而是：
  - 哪类数据该用哪类方法；
  - 哪种噪声结构适合哪种方法；
  - 哪组参数、哪条方法链在当前目标下更优；
  - 试出来的经验如何沉淀，而不是下一轮再从零开始。

### 一句话结论

**问题已经从“发明方法”转向“组织方法”。**

---

## 第 2 页：为什么“最优解”通常是一个伪命题

### 核心 message

多数真实任务没有单一全局最优，只有条件最优。

### 展开

一个方法好不好，至少受这些因素影响：

- 数据类型：点、轨迹、网格、区域、事件流
- 噪声结构：漂移、缺失、异步、系统偏差
- 时间尺度：分钟、小时、天、月
- 空间尺度：道路、街区、城市、区域
- 目标函数：精度、延迟、鲁棒性、可解释性、成本
- 运行面：离线、准实时、在线调度

### 一句话结论

**所谓“最优方法”，本质上是“在特定约束下的局部最优方法”。**

---

## 第 3 页：这条问题线的学术起点很早

### 核心 message

“选算法”本身就是一个老问题，不是今天才出现。

### 展开

- Rice 在 `The Algorithm Selection Problem` 里已经指出：
  - 不同问题实例适合不同算法；
  - 算法性能不能脱离实例特征讨论；
  - 选择算法本身是一个独立问题。
- 后来的 AutoML 把这个问题推进为：
  - 不只是选算法；
  - 还要一起调参数；
  - 最终形成 `CASH` 问题。
- AutoFolio 进一步说明：
  - 连“算法选择器本身”都需要被自动配置。

### 一句话结论

**方法组织的本质，是把经验判断改写成结构化搜索问题。**

---

## 第 4 页：为什么时空数据把这个问题放大了一个数量级

### 核心 message

时空数据系统不是普通数据库，它面对的是一个持续演化的世界。

### 展开

- 时空对象不是静态记录，而是动态存在：
  - `Entity`
  - `Trajectory`
  - `Region`
  - `Event`
  - `State Field`
- 同一世界会被不同来源、不同粒度、不同时间延迟重复观测。
- 同一份数据可支持多类任务：
  - 异常检测
  - 分群与分区
  - 热点发现
  - 变化识别
  - 预测
  - 传播关系分析

### 一句话结论

**时空系统天然需要多方法、多尺度、多目标联合存在，因此更需要“方法织布层”。**

## 第 5 页：多 agent 很重要，但它不是根问题

### 核心 message

多 agent 的价值不在“多找几个人一起干活”，而在“经营方法生态”。

### 展开

在这类系统里：

- 方法库负责提供候选动作
- 实验层负责比较方法好坏
- agent 负责组织搜索、验证、沉淀、晋升

更合理的 agent 角色是：

- `Planner Agent`
- `Method Router Agent`
- `Scale Agent`
- `Operator Agent`
- `Judge Agent`
- `Memory Agent`
- `Policy Agent`

### 一句话结论

**agent 不是方法的替代品，而是方法生态的管理者。**

---

## 第 6 页：自动实验应该放在哪个位置

### 核心 message

自动实验是“实验竞技场”，不是整个系统本身。

### 展开

前沿系统如 `AlphaEvolve`、`Darwin Gödel Machine`、`ALE-Agent`、`The AI Scientist` 共同说明了一件事：

- 可以把候选方法放入搜索空间；
- 用 evaluator 比较结果；
- 用 archive 保留历史；
- 用 selection / promotion 机制持续优化。

它们抽象出来就是：

```text
Experiment Arena
= Search Space
+ Evaluator
+ Archive
+ Promotion / Rejection
```

### 一句话结论

**自动实验是方法层的进化引擎，不是数据操作系统的全部。**

---

## 第 7 页：真正缺的是 Method Fabric

### 核心 message

系统需要的不是“方法仓库”，而是“方法织布层”。

### 展开

`Method Fabric` 至少应包含：

- `Method Atom`：最小方法单元
- `Method Family`：方法族
- `Method Contract`：输入/输出/假设/约束
- `Method Portfolio`：候选方法组合
- `Evaluator Contract`：评价协议
- `Experiment Arena`：自动试验场
- `Method Memory`：历史经验和失败模式
- `Promotion Gate`：晋升、灰度、退役

每个方法对象至少要带：

- 适用数据模型
- 适用尺度
- 目标函数
- 成本画像
- 失败模式
- 可解释性等级
- lineage

### 一句话结论

**Method Fabric 是把“很多方法”变成“系统能力”的关键中枢。**

---

## 第 8 页：面向巨量时空数据的 Data OS 应该怎么分层

### 核心 message

数据库只是底座，真正的 Data OS 是一个分层系统。

### 展开

建议用下面这套抽象：

```text
Spatiotemporal Data OS
├── World Interface
├── World Model
├── Data Planes
│   ├── Raw Plane
│   ├── Model Plane
│   ├── Feature Plane
│   ├── Experiment Plane
│   └── Knowledge Plane
├── Method Fabric
├── Agent Control Plane
└── Decision Plane
```

这里最关键的三件事：

- `World Model`：把数据组织成世界对象
- `Method Fabric`：把方法组织成可搜索能力
- `Decision Plane`：把分析结果转成行动

### 一句话结论

**Data OS 的核心不是“有个大数据库”，而是“世界模型 + 方法织布层 + 决策闭环”。**

## 第 9 页：和 Butler 现有架构怎么对齐

### 核心 message

这个方向不是另起炉灶，而是可以直接长在 Butler 现有骨架上。

### 展开

Butler 当前已有：

- `frontdoor / domain plane`
- `orchestrator / control plane`
- `process runtime`
- `agent runtime`

可以这样映射：

- `frontdoor`：接问题、任务和观察入口
- `orchestrator`：做方法路由、任务编排、agent 协作
- `process runtime`：承接实验账本、artifact、handoff、verification
- `agent runtime`：执行单次分析、单次试验、单次计算

建议新增的真源对象：

- `WorldObject`
- `Dataset`
- `MethodPackage`
- `Evaluator`
- `ExperimentRun`
- `Artifact`
- `Insight`
- `Decision`

### 一句话结论

**Butler 现在缺的不是新大脑，而是让现有 runtime 和 orchestrator 理解“方法对象”和“实验对象”。**

## 第 10 页：最后要带走的 5 个判断

### 1. 方法问题已经从“有没有方法”变成“怎么组织方法”

### 2. 多数真实系统里不存在单一永久最优解，只有条件最优与 Pareto 前沿

### 3. agent 自主实验很重要，但它只是方法织布层中的实验竞技场

### 4. 巨量时空数据系统的核心不是数据库，而是世界模型、方法织布层和决策闭环

### 5. 最终目标不是做一个更大的分析平台，而是做一个能持续理解世界、持续搜索更优解、并持续支持行动的操作系统

---

## 结尾一句

**我们真正要构建的，不是“数据库 + 多 agent + 方法库”的拼装系统，而是一个以世界模型为核心、以方法织布层为中枢、以实验竞技场为进化引擎的时空数据操作系统。**
