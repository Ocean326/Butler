## 1. 原文关键信号（技术与趋势）

- **Sub Agent 特征**
  - 进程内、毫秒级启动、生命周期用 `Map` 管、结果用 `EventEmitter` 投递。
  - 无法互相通信，本质是「同一身体里的轻量子程序」，适合后台型小任务。

- **Agent Team 特征**
  - 基于 SDK、每个成员是独立进程，通过 mailbox 通信。
  - AI 有能力自主设计团队结构、分工和汇总策略，更偏向「自治团队」。

- **双引擎 + 统一记忆**
  - 用命令（例如 `/team`）在两套执行引擎之间切换。
  - 通过适配层共享 Provider 配置和记忆系统，避免「两张皮」。

- **行业趋势判断**
  - MCP 成为标准，CrewAI / AutoGen / LangGraph 等多智能体框架并存。
  - 预期将出现可以自组织的 Agent 团队，以及跨平台、多工具协作。

## 2. Butler 架构视角：SubAgent vs AgentTeam 分工草案

- **SubAgent（Butler 内部子代理）**
  - 对应：当前 `skills` + 轻量 SubAgent / 工具型任务。
  - 适合：
    - 单轮完成的小任务：单篇网页抓取、单文件解析、一次性转换；
    - 心跳里的「探针任务」：检测某个状态、更新缓存。
  - 设计关注点：
    - 生命周期 = 单任务 / 单心跳片段；
    - 不要求跨任务长期记忆，由 Butler 主体负责长记忆。

- **AgentTeam（外部或重型团队）**
  - 对应：IDE 协作 / 长时研究 / 多阶段规划等重任务。
  - 适合：
    - 「研究一整个主题」；
    - 「帮助用户做系统级架构设计或长篇内容生产」；
    - 「跨多技能、多数据源的综合分析」。
  - 设计关注点：
    - 与 `task_ledger` 的任务边界与回执格式；
    - 如何让 Team 的中间过程可观测、可插手、可中止。

## 3. 与当前 Butler 的差距与提问

- **差距感**
  - 当前 Butler 内部的「子流程 / 子技能」更多是函数级封装，还没有一套显式的 SubAgent 生命周期与状态机。
  - AgentTeam 更多停留在概念层（或工具链实验），尚未形成统一抽象：什么时候派 Team、怎么回写结果。

- **需要回答的问题**
  - Q1：在 Butler 的心跳 / 日常对话中，哪些触发条件适合自动起一个 SubAgent 而不是直接 inline 处理？
  - Q2：哪些情况必须升级为 AgentTeam？（例如「跨度 > N 小时」「涉及 ≥3 种外部资源」「需要并行多路径探索」）
  - Q3：双引擎切换时，如何保证：
    - 对用户来说只有一个「任务 ID / 会话主线」；
    - 对内部来说可以清楚地区分「本体处理」vs「外包给 Team」？

## 4. 下一步可行的小实验（候选）

- **实验 A：定义一个最小 SubAgent 类型**
  - 从现有一个 skill（如 web 抓取或 OCR）里抽象出 SubAgent 概念：
    - 明确：输入结构 / 状态记录 / 超时策略 / 回执格式。
    - 在 `heartbeat_orchestration` 或某个 skill pipeline 里试点。

- **实验 B：为 AgentTeam 设计统一「任务包装器」**
  - 定义：从 Butler → AgentTeam 的任务描述模板 + 回写模板。
  - 选一个典型长任务（例如「对比 3 篇小红书 agent 文章，给出 Butler 架构建议」）作为试点。

- **实验 C：把「双引擎」写进架构图 / 状态机**
  - 在 Butler 内部架构图里加一层：`Execution Engine` = { Butler 本体、SubAgent 引擎、AgentTeam 引擎 }。
  - 标记清楚：谁负责记忆、谁负责执行、谁负责治理（MAS harness 视角）。
