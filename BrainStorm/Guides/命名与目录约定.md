# BrainStorm 命名与目录约定

这份文档现在的目标很明确：定义 **四个前台区怎么分工，后台加工层怎么不挡路**。

---

## 一、总结构

BrainStorm 现在按 **前台四区 + 后台一层** 组织：

1. `Ideas/`：你的想法入口
2. `Raw/`：碎资料区
3. `Insights/standalone_archive/`：Insight 区
4. `Insights/mainline/`：长主线知识线区
5. `Working/`：agent 后台加工层

额外入口：

- `BrainStorm/Insights/README.md`：默认阅读入口
- `BrainStorm/STATE.md`：当前状态与行动项
- `BrainStorm/MEMORY.md`：长期记忆

---

## 二、先判断你手里的东西是什么

### 属于 `Ideas/`

- 它主要是你的想法，不是资料
- 还没有形成稳定结论
- 你只是怕以后忘掉
- 它可能很粗、很跳、甚至互相矛盾

### 属于 `Raw/`

- 它主要是外部材料，不是你的判断
- 比如网页、摘录、截图、OCR、抓取结果、数据样本
- 重点是“保留证据”，不是“形成结论”

### 属于 `Insight`

- 它已经不是单纯资料
- 你或 agent 已经做过一轮提炼
- 它能明确说明“这批东西说明了什么”

### 属于 `mainline`

- 它不是单篇结论，而是一类稳定主题
- 后续新增内容更适合并进去，而不是继续平铺新文件

### 属于 `Working/`

- 它只是中间加工稿
- 主要给 agent 整理、合并、试写结构
- 默认不作为人的第一落点

---

## 三、命名规则

### 1. `Ideas/`

#### `Ideas/inbox/`

- 用于随手抛想法
- 推荐命名：
  - `YYYYMMDD_一句话主题.md`

#### `Ideas/threads/`

- 用于跟踪一条正在长大的想法线
- 推荐命名：
  - `YYYYMMDD_主题_idea_thread.md`

### 2. `Raw/`

- 按日期整理的人工 Raw 笔记统一进入：
  - `Raw/daily/YYYYMMDD/YYYYMMDD_<source>_<topic>.md`
- 抓取/OCR 原产物保留脚本友好命名：
  - `{platform}_{id}.md`
  - `{platform}_{id}.json`
  - `{platform}_{id}_ocr.md`
  - `{platform}_{id}_ocr.json`
- 无明确日期或以平台 id 为主键的抓取原件，可继续保留在 `Raw/` 根下
- 图片资产：
  - `Raw/images/`
  - 或专题子目录自己的 `images/`
- 专题系列资料：
  - 进入各自子目录，例如 `Raw/Simon_agent_xhs_series/`

### 3. `Working/`

- 统一使用：
  - `YYYYMMDD_<topic>_to_brainstorm.md`
- 这是过渡层，不是目标层。

### 4. `Insights/standalone_archive/`

- 单主题 insight 统一使用：
  - `YYYYMMDD_<topic>_Insight.md`
  - 或兼容历史的 `..._insight.md`

### 5. `Insights/mainline/`

- 文件名必须表达稳定主题，不按单篇文章命名
- 一条主线建立后，后续优先并入，不再横向散开

---

## 四、两条标准流

### 流 A：从想法开始

1. 先记到 `Ideas/inbox/`
2. 值得持续跟时升格到 `Ideas/threads/`
3. 需要证据时把外部材料收进 `Raw/`
4. agent 在 `Working/` 加工
5. 形成 `Insight`
6. 多个 Insight 收敛后并回 `mainline/`

### 流 B：从资料开始

1. 外部材料先进入 `Raw/`
2. agent 在 `Working/` 加工
3. 形成 `Insight`
4. 成熟后并回 `mainline/`

---

## 五、知识树治理规则

### 默认入口规则

- `BrainStorm/Insights/README.md` 是默认阅读入口
- `BrainStorm/Insights/index.md` 只保留兼容索引
- `Ideas/` 不纳入知识树，它是入口池，不是树干

### 自动刷新

执行：

`python BrainStorm/tools/refresh_brainstorm.py`

会更新：

- `Insights/README.md`
- `Insights/index.md`
- `Insights/knowledge_tree.json`

### 挂靠规则

- 新的 standalone Insight 必须能挂到某个知识分支
- 若脚本无法自动分类，优先补 `knowledge_tree_config.json` 的关键词
- 当同一分支积累 3+ 篇同主题洞察时，优先考虑合并回对应 `mainline/`

---

## 六、最小闭环

任何新内容，尽量满足下面闭环之一：

### 想法闭环

1. 先进入 `Ideas/`
2. 有证据后进入 `Raw/`
3. 加工后进入 `Insight`
4. 成熟后进入 `mainline/`

### 资料闭环

1. 先进入 `Raw/`
2. 加工后进入 `Insight`
3. 成熟后进入 `mainline/`

---

## 七、治理目标

- 你以后有新脑暴点时，不需要先判断它属于哪条主线，直接去 `Ideas/`
- `Raw/` 承担资料留存，不承担结论表达
- `Insights/` 承担知识沉淀，不再承担随手记
- `Working/` 继续存在，但降级为后台厨房
- 以后读知识时，优先看树；以后记想法时，优先去入口池
