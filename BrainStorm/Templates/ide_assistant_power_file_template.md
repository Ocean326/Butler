## 核心结构：一个文件里要放什么

> 把原笔记压缩成 3–5 个模块，每个模块下面可以继续用要点列出内容。  
> 参考「一个文件让 Claude Code 战斗力翻倍」一类笔记，把“抽象出来的结构”写在这里。

### 1. 环境与身份设定

- **开发环境说明**：IDE / 编辑器版本、语言栈、操作系统、常用插件（例：Claude Code / Cursor / VS Code + 扩展列表）。
- **项目类型与规模**：单体脚本 / Web 服务 / 多仓协同 / Mono-repo 等。
- **当前角色定位**：`个人开发者 / 团队内主程 / 学习项目`，这会影响助手应该怎么协作。

### 2. 高频工作流（Workflows）

- **日常循环任务**：例如「新需求从 0 到上线」「调试线上 bug」「重构一段遗留代码」等，每个用 5–8 步的 checklist 表达。
- **关键工作流拆解**：为每个高频任务写清「触发条件 → 输入 → 步骤 → 期望输出」，方便让助手接手局部流程。
- **工具组合**：哪些任务会同时用到 git / 测试框架 / 构建工具 / AI 助手（注明调用顺序和注意事项）。

### 3. 常用命令与操作清单

- **项目级常用命令**：启动 / 测试 / 构建 / 格式化 / Lint / 部署命令，一行一个，附上简短说明。
- **调试与排错命令**：日志查看、端口占用排查、数据库连接测试等。
- **危险命令标记**：需要谨慎执行或必须二次确认的命令（加粗并写清“什么时候绝对不要用”）。

### 4. 常见坑与解决方案

- **环境 / 依赖相关坑**：版本不兼容、依赖安装失败、平台差异（Windows / macOS / Linux）导致的问题及解决方案。
- **工程结构 / 模块化坑**：容易误解的目录、需要遵守的约定（例如：配置放在哪里、哪些文件不能直接改）。
- **AI 助手易犯错点**：例如「喜欢改错文件」「误删配置」「不会跑集成测试」，以及对应的防御性提示。

### 5. 示例对话与提示语（Prompts）

- **典型高质量请求示例**：为 2–3 个高频任务各写一段「对 IDE/助手说的话」，包括上下文给法与输出要求。
- **负面示例 / 反例**：列出几条模糊、容易导致跑偏的请求，并写明如何改写。
- **结构化输出约定**：例如「一律用 checklist / diff patch / code fence 返回」，方便复制进 git / 终端。

## Butler / Claude / Cursor 适配区

> 把上面的内容映射到不同助手的实际使用方式，便于以后快速接入或迁移。

### 0. Claude Code / Agent 工程化示例对齐

- 参考小红书笔记「把 Claude Code 拆开看，Agent 就不神秘了」：
  - **核心五问**：状态怎么存｜上下文怎么控｜任务怎么追｜失败怎么回滚｜协作怎么对齐。
  - 本模板可以在上述五问下各留出 3–5 行，用来写清当前项目里：
    - 状态/记忆存放位置（如 BrainStorm/Raw / Insights / MEMORY、自定义 STATE）；
    - 上下文注入与裁剪方式（例如「总是先读本文件」的约定）；
    - 任务跟踪与验收路径（任务看板 / task_ledger / 心跳任务 id）；
    - 回滚与降级策略（例如「只读模式」「不改 butler_main/butler_bot_code」等）；
    - 与人类/其他 agent 的协作边界。
  - 「一个文件让 IDE/助理战力翻倍」的最小可用版本，只要把上述五问填到能让新助手 10 分钟内读懂并接上手即可，不必一次写满所有细节。

### 1. Butler 适配

- **放置目录建议**：`./工作区/IDE-Assistant/` 或当前项目下的 `docs/assistant/`（根据具体仓库习惯选择其一）。
- **与 Butler 的关联方式**：
  - 作为「工作记忆」：在对话中引用该文件路径，请 Butler 先阅读再执行任务。
  - 作为「技能入口说明」：若已有相关 skill（如 `web-note-capture-cn` / `web-image-ocr-cn`），在技能说明里链接本模板，作为 ingest 下一步。
- **可抽成配置 / prompt 的字段**：
  - 高频工作流 → 可变成 Butler 分支模板或心跳任务 checklist。
  - 常用命令 / 常见坑 → 可做成运行前提示或 pre-check 列表。
  - 示例对话 → 可沉淀为通用 prompt 片段库。

### 2. Claude 系（Claude Code / Claude Desktop）

- **推荐放置位置**：项目仓库内 `CLAUDE_HELP.md` / `assistant_guide.md`，或本地常驻知识库目录。
- **注入方式**：
  - 打开项目时，优先让 Claude 阅读本文件，再开始编写或重构。
  - 对复杂任务，在对话第一轮直接粘贴「当前工作流片段 + 常见坑小节」。
- **特别适配点**：
  - 强调「状态怎么存 / 上下文怎么控 / 任务怎么追」这三块，让 Claude 帮你维护执行进度。

### 3. Cursor / 其他 IDE 助手

- **推荐放置位置**：`docs/ide_assistant_guide.md` 或 `.cursor/` 相关说明文件中添加链接。
- **注入方式**：
  - 作为「项目导读页」，在每次开新分支 / 重构前阅读。
  - 把例子 prompt 整理进 Snippets 或快捷指令中。
- **适配注意**：
  - 标记哪些部分适合做成「自动建议」或「命令面板动作」，哪些只适合作为人工参考文档。

## 使用说明（给未来的我）

1. **先填再用**：新项目落地时，先花 15–30 分钟按本模板补齐关键空白，再把文件交给 IDE/助手阅读。
2. **任务前复读一遍高频工作流**：每次要开新需求 / 大改动前，先快速浏览「高频工作流 + 常见坑」，防止 AI 和自己一起踩坑。
3. **对话里引用具体小节**：和 Butler / Claude / Cursor 说话时，尽量引用「本文件中的某一节」而不是整篇贴过去，减少噪音。
4. **按迭代补全**：每次踩到新坑或形成新 workflow，就回到这里补几行，而不是另开零散文档。
5. **定期回收成标准化资产**：当本模板在一个项目里跑顺后，可以复制到其他项目，仅修改环境与工作流部分，形成自己的「IDE/助手战力基线包」。
