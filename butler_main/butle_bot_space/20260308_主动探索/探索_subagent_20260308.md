## 1. 概念与文献要点

### 1.1 软件 Agent 与层级（Wikipedia / 经典 MAS）

- **Agent 基本属性**：反应环境、自主性、目标导向、持久运行；可**调用其他任务**（含通信），具备**社会能力**（与其他组件协作）。
- **设计议题**（与 subagent 直接相关）：
  - **Agent 层级**：何种层级有用（如任务执行 Agent、调度 Agent、资源提供者等）。
  - **消息与通信**：如何实现消息传递与协议。
  - **任务优先级、调度与同步**：子任务如何分配与汇总。
- **多 Agent 系统（MAS）**：分布式、协作达成单 Agent 无法完成的目标；与「主 Agent 调用子 Agent」的层级模式互补。

### 1.2 Subagent 在本工作区的对应

- **主控**：`feishu-workstation-agent`（飞书管家）负责路由、分派、工作区与角色设定。
- **子 Agent**：orchestrator、secretary、literature、file-manager、research-ops、engineering-tracker、discussion、governance 等；定义在 `.cursor/agents/`，经 Cursor CLI（如 `agent -p --force --trust ...`）或 MCP `mcp_task` 调用。
- **任务分解与汇总**：orchestrator 拆解任务、排优先级、分发给各 specialist；各 specialist 产出汇总到「今日闭环清单 + 明日优先级」；飞书工作站做每周巡检与治理。

## 2. 边界与权限（提炼）

| 维度 | 本工作区约定 | 可复用要点 |
|------|--------------|------------|
| **职责边界** | secretary 只做记录与闭环不替代学术判断；literature 只做文献与证据不给未验证结论；file-manager 不修改研究结论只管理文件；治理仅飞书工作站 | 子 Agent 按「能力边界」划分，避免越界与重复 |
| **产出路径** | 统一 `./研究管理工作区`，各 Agent 对应子目录；调用时**必须指定工作区**并在 prompt 中写明「产出写入 xxx」 | 调用子 Agent 时显式约定输出路径，便于汇总与回退 |
| **结果汇总** | 子 Agent 结果返回主控（如 mcp_task 的 return）；主控负责整合、写回 chat / 记忆 | 主控拥有「结果解释权」与写入权限，子 Agent 不直接改主控记忆 |
| **权限/回退** | 所有「丰富自己」的修改在 work-tree/备份 之后进行 | 子 Agent 调用链中的写操作应有明确范围，关键状态可回退 |

## 3. 与首轮探索（agent teams）的衔接

- **agent teams** 侧重多 Agent 协作、编排、通信与决策协议；**subagent** 侧重「主—子」调用、任务分解、结果汇总与边界。
- 二者结合：当前架构 = 团队编排（orchestrator + 各 specialist）+ 主控调用子 Agent（飞书工作站 + Cursor CLI / mcp_task）；下一步可细化「子 Agent 边界」与「自我手术」的回滚策略（见任务主文件执行记录）。

## 4. 简要提炼（供下次心跳或丰富自己）

- **Subagent 三要素**：**调用约定**（谁在什么时机调谁）、**任务/结果契约**（输入 prompt、输出路径与格式）、**边界与权限**（职责不重叠、写操作范围与回退）。
- **本工作区已具备**：角色一览与产出路径（AGENTS_ARCHITECTURE）、调用时指定工作区与产出路径（feishu-workstation-agent、SKILL.md）、治理与记忆约定（local_memory）。
- **可后续丰富**：在 local_memory 或 governance 中增加一条「子 Agent 调用与边界」备忘；若引入新子 Agent，在 AGENTS_ARCHITECTURE 与备份中同步更新。
