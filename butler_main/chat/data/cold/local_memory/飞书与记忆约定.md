- **约定**：不要什么都往人设文档（Agent 角色说明）里放；与人设无关的约定、偏好、技术备忘等写入 `local_memory`，便于维护且避免人设文档膨胀。

- **约定**：长期记忆整理与加载按**优先级**执行——越重要、越与运行/底层长期记忆相关，保留与加载必要性越高。

## 使用规则（recent_memory）

- **约定**：
  - 用户**未**明确说明「全新任务 / 全新情景」时，**优先沿用 recent_memory** 续接上下文。
  - 用户**明确开启新任务**时，**忽略 recent_memory**，不沿用上一轮上下文。
- **适用对象**：飞书工作站在回复前按此规则决定是否加载 recent_memory；**负责记忆整理的 agent（file-manager-agent）执行长期记忆整理时也需知晓此规则**，以便在整理/压缩 recent 时保留「是否为新任务开启」的语义（例如在摘要或 long_term_candidate 中标注），便于后续会话正确续接或重置上下文。

## 飞书 chat_id / 凭证查证原则（2026-03-25 合并）

- **先查证再下结论**：只要机器人已经能实际发送飞书消息，就默认 `chat_id`、凭证与对应链路必然存在；遇到相关问题时先顺着代码、配置、环境变量与运行链路查证，不用“拿不到/可能没有”作偷懒结论。
- **明确边界层级**：要区分代码中的概念槽位、环境变量名与当前运行态的具体值；即使拿不到具体值，也要明确说明卡在哪一层、为什么受限、下一步该去哪里查。

- **能力盲区要补齐**：若现有 skill / 脚本在飞书凭证、`chat_id` 来源、聊天历史读取上存在盲区，优先升级 skill 或补文档，不让同类问题长期停留在口头解释层。

## 后台维护 Cursor CLI 调用与对话路径对齐（2026-03-08）

**问题**：后台维护每次超时且出现「API key is invalid」等；对话（飞书发消息）可正常请求与返回。

**根因**：后台维护子进程内调用 Cursor CLI 的方式与对话路径不一致——此前后台维护用 `Popen` + 二进制流 + 自解 GBK，且未显式传入 `env`；对话路径使用 `subprocess.run` + `text=True` + `encoding="utf-8"` + `errors="replace"`，且在同一进程/环境中运行。

**修复**（已落代码）：
- **memory_manager._run_model_subprocess**：改为与对话一致的 `subprocess.run(..., text=True, encoding="utf-8", errors="replace", env=os.environ.copy())`，使后台维护与对话使用同一套 CLI 调用与解码方式，并继承相同环境（含 Cursor 相关 env）。
- **butler_bot**：`_run_agent_via_cli`、`_list_available_models` 的 `subprocess.run` 增加 `errors="replace"`，避免 Windows 下 CLI 输出 GBK 时产生 `UnicodeDecodeError`。

**生效**：修改了 `butler_bot_code` 下代码，需重启主进程/后台维护后新逻辑才会生效；已创建 `工作区/restart_request.json`，后台维护本轮结束后会自动重启。

**关键词**：后台维护、Cursor CLI、对话、subprocess、encoding、env、restart

## 「抓-读-脑」统一约定（2026-03-17）

- **约定**：从现在起，飞书侧所有「抓-读-脑」类流程中，`读` 这一步（包括**读网页正文、读图片/截图内容**）默认由当前模型直接完成，不再依赖外部 OCR 作为前置必需条件。
- **行为**：
  - 当用户甩链接或发图，只要当前模型具备读图/读文能力，**优先直接在对话里理解内容并给出结论**；
  - 仅当需要**长期沉淀为可复用文件产物**时，才调用 `web-image-ocr-cn` 等 OCR skill，把图片文字结构化落盘到 `BrainStorm/Raw`。
- **边界**：
  - 不要求用户为了「读图」额外跑本机 OCR 命令或配置 PaddleOCR；环境不足时，优先用模型直读，其次再考虑补环境做结构化沉淀；
  - 任何流程文档/skill 中若出现「先 OCR 才能读懂内容」的说法，以本条约定为准，视为过时。
