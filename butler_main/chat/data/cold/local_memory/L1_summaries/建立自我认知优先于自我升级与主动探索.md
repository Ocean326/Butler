# 建立自我认知优先于自我升级与主动探索（L1 摘要）

## 用途与范围
- 这是 Butler 的“自我认知优先”入口，用于在后台维护/调度/执行链路里提醒：先把边界、真源、失败模式与可验证回执补齐，再谈自我升级或主动探索扩张。
- 该入口面向的是“怎么保证下一步可控且可验证”，而不是记录过程日志。

---

## 当前有效结论（摘要）

1. 先立真源与控制面边界，再执行/再扩张
- 后台维护真源以 `task_ledger.json` 与结构化 run trace 同步视图为准；planner 输入只来自活跃看板与受控的 recent/local memory 命中/运行态上下文，`archive/quarantine` 不回灌。

2. fallback/degrade 默认收敛：无白名单任务 `status-only`，缺 `selected_task_ids` 直接降级
- 不在“缺证据/缺选择集”的状态下继续主动探索；避免脏任务池污染扩张。

3. 并行预算内优先“可验证/可快速形成回执”的任务；探索要显式允许
- 并行只能用于更快完成已选中的明确任务；探索与探索型起念必须受开关与白名单约束。

4. talk 的硬约束与 skill 稳定性：把“当前对话硬约束”写进 profile section；skill 不要只靠关键词运气
- 确实要用 skill 时，必须在回复里明确“使用了哪个 skill + SKILL.md 路径”；否则要明确说没命中。

5. 失败处理对齐 Wave4/5：结构化 acceptance receipt + failure_class；用“诊断 -> 换路/修正 -> 复试”闭环
- 不把失败当文本噪音吞掉；失败需进入可分类体系，并能用于下一轮替代策略选择。

---

## 2026-03-19 本轮可复用条目（供后续调度/开口直接套用）
- 选任务时：只承接来自 `background_tasks/active/` 且最终会写入 ledger 的明确任务；无 `selected_task_ids` -> 直接 status-only 并在回执中写清 degrade 原因。
- 并行时：在预算内尽量选短/高回执的明确任务；回执用 run trace 的 selected/rejected 来证明选择逻辑有效。
- 开口时：规则优先落 `Current_User_Profile.private.md` 的独立 section；skills 以“明确命中 + 明确声明使用路径”为唯一可信执行证据。
- 失败时：回执必须包含 `evidence/artifacts/uncertainties/next_action/failure_class`（或等价结构字段），并明确下一轮采用的“换路/修正”方向。
