# idea_loop 设计方案（v0）

> 日期：2026-03-20

> 来源参考：`BrainStorm/Raw/daily/20260318/20260318_github_autoresearchclaw_note.md` 与 BrainStorm 中关于多轮迭代、独立验收、Sentinel、autoresearch/AutoResearchClaw 对照的相关材料。

---

## 1. 目标

在 Butler 中先落一个科研场景的最小 harness 单元，不追求一开始就做成 `AutoResearchClaw` 那样的全管线，而是先实现一个足够小、足够稳、可反复复用的最小 loop：

```text
idea -> plan -> (改代码 -> 测试 -> 验收)^N -> 最终验收
```

这个 loop 的目的不是“帮模型多试几次”，而是把一次方法改进过程结构化为：

1. 可约束
2. 可观察
3. 可验收
4. 可复盘
5. 可积累经验

---

## 2. 设计原则

### 2.1 默认多轮，不默认一次成功

`idea_loop` 默认任务不会在一次 `plan -> execute` 中完成，而是内建多轮迭代状态：

- `PROCEED`：当前方向有效，继续推进
- `REFINE`：方向不变，但方案、参数、代码实现需要微调
- `PIVOT`：当前 hypothesis 价值不足，切换到新的方法方向
- `STOP`：达到完成条件或预算耗尽

### 2.2 执行与验收分离

执行者负责“改什么、怎么试”，但不负责最终裁决“是否算通过”。

因此必须有独立于执行者的验收层：

- **轮内验收**：判断这一轮是否有效
- **最终验收**：判断是否值得固化为本次研究结论

### 2.3 不可变信任边界

首版必须显式区分：

- **可编辑区**：方法实现、候选配置、实验脚本中的局部参数
- **不可编辑区**：评测脚本、数据边界、预算限制、日志协议、验收规则

也就是说，执行 loop 的 agent 不能通过修改评测器来“让自己显得成功”。

### 2.4 失败即数据

每一轮失败都不是废物，而是经验样本。最小版本先做到：

- 留下失败原因
- 留下本轮尝试与结果
- 留下下一轮不该重复什么

后续再演进成 lessons/skills 注入。

### 2.5 评判产出，不苛求固定路径

验收层主要看：

- 代码是否真的跑过
- 结果是否满足指标
- 是否违反边界条件
- 结论是否与证据一致

而不是强行要求执行者必须沿某个固定步骤操作。

## 3. 最小状态机

建议首版状态机如下：

```text
IDEA_LOCK
  -> PLAN_LOCK
  -> ITERATE[k]
       -> PATCH
       -> TEST
       -> VERIFY
       -> DECIDE(PROCEED | REFINE | PIVOT | STOP)
  -> FINAL_VERIFY
  -> ARCHIVE
```

### 3.1 IDEA_LOCK

将用户的“方法改进 idea”压缩成一个可执行、可证伪的对象：

- 研究问题是什么
- 这次改进的 hypothesis 是什么
- 预期改善哪个指标
- 不打算动哪些边界

产物建议：

- `idea.md`
- `hypothesis.json`

### 3.2 PLAN_LOCK

生成当前 run 的计划，但计划需要满足“收敛性约束”：

- 一轮只允许 1 个主改动
- 必须写清 expected signal
- 必须写清失败判据
- 必须写清最大尝试预算

产物建议：

- `plan.md`

### 3.3 ITERATE[k]

每一轮最小包含四步：

1. **PATCH**：改代码或配置
2. **TEST**：跑最小测试 / 冒烟测试 / 固定评测
3. **VERIFY**：独立验收层判定
4. **DECIDE**：决定下一步是 `PROCEED` / `REFINE` / `PIVOT` / `STOP`

### 3.4 FINAL_VERIFY

对“当前最佳版本”做一次更严格的最终验收：

- 在干净环境复跑
- 与 baseline 对比
- 检查是否有作弊或边界污染
- 输出最终结论与风险说明

### 3.5 ARCHIVE

把本次 run 的关键产物归档，至少包括：

- 最终 patch
- 指标结果
- 验收结论
- lessons

## 4. 每轮的输入 / 输出契约

### 4.1 输入契约

每一轮都应读取统一结构：

```json
{
  "run_id": "idea_loop_20260320_xxx",
  "baseline_ref": "git commit or tag",
  "hypothesis": "一句话假设",
  "editable_scope": ["允许改动的文件/目录"],
  "frozen_scope": ["禁止改动的文件/目录"],
  "primary_metric": "主指标",
  "guardrails": {
    "time_budget_sec": 1800,
    "max_iterations": 3
  },
  "done_criteria": [
    "达到主指标增益阈值",
    "通过独立验收"
  ]
}
```

### 4.2 输出契约

每一轮建议统一输出：

- `patch.diff`
- `test_result.json`
- `metrics.json`
- `acceptance.json`
- `lesson.md`
- `decision.json`

这一步很关键：没有结构化输出，后面就无法做可靠复盘和经验回灌。

## 5. 验收机制

### 5.1 轮内验收

轮内验收只回答一个问题：**这一轮是否值得继续沿当前方向推进？**

最小检查项：

- 代码是否成功运行
- 是否触发硬错误或明显回归
- 主指标是否出现正向信号
- 结果是否与 hypothesis 至少部分一致
- 是否违反 frozen boundary

输出建议：

```json
{
  "accepted": true,
  "status": "weak_positive",
  "reason": "metric improved but variance still high",
  "decision": "REFINE"
}
```

### 5.2 最终验收

最终验收比轮内验收更严格，回答的是：**这个结果能不能作为本次方法改进的有效结论？**

最小检查项：

- 是否在干净环境复跑成功
- 是否相对 baseline 有稳定增益
- 是否存在硬编码指标、数据泄漏、跳过评测等可疑行为
- 结论描述是否与结果一致
- 交付物是否完整

### 5.3 独立验收层的角色

首版不一定要做成单独 agent，但在职责上必须独立：

- 执行者产出结果
- 验收者只读产物和规则做判断
- 验收者不能“帮忙补代码后再说通过”

否则执行和裁决混在一起，loop 很容易退化成自说自话。

## 6. 首版建议的决策规则

建议先用简单明确的规则，避免一开始搞复杂策略：

### 6.1 进入 `REFINE`

出现以下情况之一：

- 代码没有跑通，但修复路径明确
- 指标没有提升，但发现了明显实现问题
- 指标有改善，但方差较大或稳定性不足

### 6.2 进入 `PIVOT`

出现以下情况之一：

- 连续 2~3 轮无实质进展
- 当前 hypothesis 被实验结果否证
- 继续微调的收益明显低于切方向

### 6.3 进入 `PROCEED`

出现以下情况之一：

- 主指标出现稳定增益
- 当前方向已具备进入下一子问题的条件

### 6.4 进入 `STOP`

出现以下情况之一：

- 已达到 done criteria
- 已耗尽时间/轮数预算
- 当前阶段无法继续，需人工接管

## 7. 建议目录结构

首版建议在 `butler_main/incubation/research/units/research_idea/idea_loop` 下按下面结构推进：

```text
idea_loop/
  README.md
  specs/
    loop_spec.template.md
    acceptance_schema.json
  prompts/
    planner.prompt.md
    verifier.prompt.md
  scripts/
    run_loop.ps1
    run_eval.ps1
    final_verify.ps1
  runs/
    .gitkeep
```

其中：

- `specs/`：定义统一输入/输出协议
- `prompts/`：定义 planner/verifier 的最小职责
- `scripts/`：封装执行入口
- `runs/`：沉淀每次实验记录

## 8. 与 AutoResearchClaw / autoresearch 的关系

这个 `idea_loop` 不直接复制大管线，而是取两者的最小共识：

### 来自 autoresearch

- 极简主循环
- 明确不可变信任边界
- 单轮预算约束
- 失败保留为数据

### 来自 AutoResearchClaw

- 内建 `PROCEED / REFINE / PIVOT`
- 执行与验收分层
- 横切式质量检查思路
- 经验沉淀为后续可复用资产

因此它的定位不是论文工厂，而是：

> **一个面向“方法改进”任务的最小科研实验壳。**

## 11. 一句话定义

`idea_loop` 是 Butler 中面向科研方法改进任务的最小 harness：  
**先锁定 hypothesis 和边界，再在受控预算内做多轮 patch-test-verify，最后由独立验收层决定结果是否成立。**
