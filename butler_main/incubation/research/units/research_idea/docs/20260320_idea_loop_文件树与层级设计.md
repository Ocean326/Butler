## 1. 设计思路

这次 `idea_loop` 的文件树不直接照搬单一项目，而是做三种模式的融合：

1. **BrainStorm 的四层沉淀**：`Raw -> Working -> Insights/归档 -> STATE`
2. **任务型项目的三层执行结构**：`artifacts -> runtime -> deliveries`
3. **成熟工程仓库的静态实现层**：`docs/config/scripts/tests/...`

因此 `idea_loop` 建议拆成三大块：

- **静态协议层**：定义规则、schema、prompt、脚本、模板
- **运行态层**：放队列、锁、检查点、当前状态
- **沉淀层**：按 `raw -> working -> deliveries -> archive` 沉淀每次 run 的产物

## 3. 各层级职责

### 3.1 `README.md`

默认入口，作用类似 BrainStorm 的总入口：

- 说明 `idea_loop` 是干什么的
- 指向最新设计文档
- 指向协议层和运行层

### 3.2 `specs/`

这是 **不可轻易漂移的协议层**，放稳定契约：

- run 输入结构
- acceptance 结构
- decision 结构
- final verify 结构

它对应成熟项目里的 `config/` + schema，也对应 BrainStorm 的“规则层”。

建议后续典型文件：

- `loop_spec.template.md`
- `acceptance.schema.json`
- `decision.schema.json`
- `run_manifest.schema.json`

### 3.3 `prompts/`

这是 **角色行为层**，放 planner / verifier 等角色的提示词，而不是 run 结果。

建议拆成：

- `planner.prompt.md`
- `executor.prompt.md`
- `verifier.prompt.md`
- `summarizer.prompt.md`

这样 prompt 与协议分离，后面更容易调优。

### 3.4 `scripts/`

这是 **执行入口层**，只放可运行脚本。

建议脚本职责保持单一：

- `create_run.ps1`：初始化 run
- `run_iter.ps1`：执行单轮 loop
- `run_eval.ps1`：跑固定评测
- `final_verify.ps1`：最终验收
- `archive_run.ps1`：归档 run

对应 `research-claw` 的 `scripts/` 风格。

### 3.5 `adapters/`

这是 **场景适配层**，专门吸收业务差异。

原因是你的 BrainStorm 和主线里已经反复强调：

- 通用 loop 不要硬编码具体业务逻辑
- 场景差异应尽量进入 adapter

所以这里放：

- repo 级适配
- 数据集级适配
- 任务类型级适配

例如：

- `repo_adapter.example.yaml`
- `vision_benchmark_adapter.yaml`
- `llm_finetune_adapter.yaml`

### 3.6 `templates/`

这是 **文档模板层**，帮助 run 产物保持统一结构。

建议放：

- hypothesis 模板
- plan 模板
- lesson 模板
- final report 模板

它相当于 BrainStorm 的“Working 模板化能力”。

### 3.7 `examples/`

这是 **教学与对照层**，放一个最小成功样例，帮助新 agent 或新开发者快速上手。

建议至少保留：

- 一个最小 `loop_spec`
- 一个最小 run 目录快照
- 一个成功验收样例

### 3.8 `tests/`

这是 **harness 自测层**，测试的是 loop 本身，而不是研究方法本身。

建议覆盖：

- schema 校验
- 路径冻结规则
- 决策流转规则
- 归档流程

对应成熟项目里的 `tests/` 与 `test/` 分工思路。

### 3.9 `runtime/`

这是 **运行态层**，只放“活状态”，不放长期知识。

子层建议：

- `queue/`：待执行 run / 待处理任务
- `locks/`：并发锁、防重复执行
- `checkpoints/`：当前 run 的阶段快照
- `state/`：索引、当前活动 run、最近一次决策

这个层对应：

- BrainStorm 的 `STATE.md`
- 任务型项目里的 `runtime/`

建议这里主要放 `.json/.md` 状态文件，而不是大体量结果。

### 3.10 `runs/`

这是 **沉淀层**，是整个 `idea_loop` 最重要的一层。

这里不建议把所有文件扔在一个目录里，而是参考 BrainStorm 的分层治理：

#### `runs/raw/`

放原始实验产物，类似：

- stdout/stderr
- 原始 patch
- 原始 metrics
- 原始测试输出
- 原始 verifier 输入

对应 BrainStorm 的 `Raw/`，也对应任务项目里的 `artifacts/`。

#### `runs/working/`

放中间加工层，类似：

- 结构化 plan
- 中间分析
- 对比表
- hypothesis 修订稿
- run 小结草稿

对应 BrainStorm 的 `Working/`。

#### `runs/deliveries/`

放通过验收、适合给人直接消费的交付物，类似：

- 本轮结论
- 最终验收报告
- 可提交 PR 的摘要
- 下一步建议

这里吸收的是目标项目中的 `deliveries/` 设计，而不是继续混在 raw 里面。

#### `runs/archive/`

放已经结束的 run 冷归档：

- 成功 run 的完整快照
- 失败 run 的保留版
- 已被替代的旧版本记录

这样可以避免 `raw/working/deliveries` 越积越乱。

## 4. 为什么不用把 `docs/` 放进 `idea_loop/`

因为你已经在 `butler_main/incubation/research/units/research_idea/docs/` 采用了按日期沉淀方案文档的方式，这和 BrainStorm 很像：

- **目录内实现区**：放可执行资产
- **目录外 docs 区**：放按日期演化的设计与复盘

所以建议继续保持：

- `idea_loop/`：实现与运行资产
- `research_idea/docs/`：日期化设计、复盘、决策记录

不要在 `idea_loop/` 内再平行造一套 `docs/`，否则会出现双入口。

## 5. 命名建议

### 5.1 运行目录命名

建议每次 run 用：

- `YYYYMMDD_<task_slug>_<run_id>`

例如：

- `20260320_resnet_dropout_ablation_r001`
- `20260320_reward_model_prompt_tune_r002`

### 5.2 文件命名

建议按“类型优先 + 机器友好”命名：

- `plan.md`
- `metrics.json`
- `acceptance.json`
- `decision.json`
- `lesson.md`
- `final_report.md`

不要在同一 run 里混太多自由命名文件，否则后面不利于自动处理。

## 6. 推荐的数据流

```text
specs/prompts/templates
        ↓
    scripts/adapters
        ↓
      runtime
        ↓
runs/raw -> runs/working -> runs/deliveries -> runs/archive
```

其中：

- 上层定义规则
- 中层负责执行
- 下层沉淀结果

这是把 BrainStorm 的知识分层，和任务型项目的执行分层，叠成一个最小研究 harness。

---

## 7. 当前结论

`idea_loop` 最适合采用：

- **顶层少而稳**：入口、协议、prompt、脚本、适配器、测试
- **运行态单独隔离**：runtime 不和长期资产混放
- **结果分层沉淀**：raw / working / deliveries / archive

一句话总结：

> `idea_loop` 不是单纯代码目录，而是一个带协议层、运行态层、沉淀层的最小研究操作壳。
