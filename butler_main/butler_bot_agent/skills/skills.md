# Butler Skills 使用总则

> 用途：作为所有角色在“什么时候必须/优先使用 skill、如何调用 skill”时的统一约定入口；具体到某个 skill 时，再读取该目录下对应的 `SKILL.md`。

## 1. 什么时候必须 / 优先使用 skill

- **有匹配 skill 的任务场景时，要优先用 skill**
  - 只要用户的需求，和某个已登记的 skill 能力明显对应，就应优先通过 skill 完成任务，而不是直接改核心代码或临时造一次性脚本。
  - 在考虑新增 skill 前，先搜索本地 `./butler_bot_agent/skills` 与外部资源，尽量复用现有轮子；仅在无合适轮子或严重不适配时再新建 skill，并在说明中记录已检索过哪些方案、为何需要新轮子。

- **典型触发场景（在使用过程中持续补充）**
  - 飞书侧能力：历史消息、文档检索、群/会话工具等 → 优先查 `feishu_*` 相关 skills；发云文档链接、心跳维护任务云文档 → `feishu-doc-sync`。
  - 外部平台抓取与学习：小红书、论坛、网页资料等 → 优先查 `web-note-capture-cn` 等相关 skills。
  - 中文内容平台单篇抓取：知乎专栏、小红书分享页、`xhslink` 等 → 优先查 `web-note-capture-cn`，默认走本地脚本与可选 cookie，不直接信任外部 skill 市场脚本。
  - **从 OpenClaw/Skills.sh 查找技能**：用户说「找个技能」「有没有能做 X 的技能」「从 OpenClaw 下好 find skills」等，或 Butler 缺能力需从公开技能库补充时 → 使用 `openclaw-find-skills`（按 `npx skills find` / `npx clawhub search` 流程查技能并给出安装命令）。
  - 日常巡检与治理：启动巡检、健康检查、配置/目录巡检等 → 优先查 `daily-inspection` 及后续运维类 skills。
  - 其他可复用流程：批处理、导出、整理套路等 → 优先查 `operations` / `general` 分类下的 skills。
  - 若在心跳或自我升级分支中涉及“skills 探索 / 外部学习”，应显式说明本轮会参考哪些既有 skills，并在产出中完善本节的典型场景列表。

## 2. 调用 skill 的标准流程

1. **匹配 skill**
   - 当用户提到“skill / 技能 / 调用 skill / 调用某某 skill”或任务明显属于某类外部能力时：
     - 先在 `./butler_bot_agent/skills` 中按名称、描述与 `SKILL.md` 内容匹配出最相关的一个或少数几个 skill。
     - 若用户没有点名具体 skill，由当前角色根据任务意图自行选择最合适的 skill，并在回复中说明选择依据。

2. **读取 `SKILL.md`**
   - 在真正执行前，**必须**用 Read 读取该 skill 目录下的 `SKILL.md`：
     - 按 `SKILL.md` 中的“适用场景 / 前置条件 / 步骤 / 配置来源 / 风险点”来设计本轮操作。
     - 如有 `reference.md`，在遇到细节不清晰或需要扩展用法时再按需阅读。

3. **按说明执行**
   - 执行即**按 SKILL 说明书构建脚本/命令，在本地环境实际运行**；不是“名义上调用”，而是真的跑起来（例如 Python 类 skill 用 Python 在终端执行、需 CLI 的按 SKILL 约定调 CLI）。若 SKILL.md 中写明需在终端执行 `python`/`python3` 或其它运行时，就按该方式执行。
   - 执行时遵守 `SKILL.md` 的约定，不随意突破其边界：
     - 不把 DNA 级能力（核心运行机制、生命周期、日志/安全闸门等）偷偷塞回 skill。
     - 如发现 `SKILL.md` 本身过时或有问题，应在本轮产出中记下“需更新该 skill 说明”的结论，而不是临时擅自改写行为。

4. **在回复中透明说明**
   - 在对用户的自然语言回复中，**明确说明**：
     - 本轮使用了哪个 skill（名称 + 相对路径）。
     - 该 skill 在本轮大致做了什么（1–2 句话）。
     - 若未找到合适 skill，也要如实说明“当前未找到匹配的 skill”，而不是装作已经使用。

5. **缺能力时主动补能力，再回到原任务**
  - 如果当前任务明确卡在“本地没有现成 skill / MCP / capability”，默认不要只停在“做不到”或“缺少能力”的汇报。
  - 先检查本地已登记 skills、已有 capability 字段与既有 agent 角色是否已经能覆盖；若仍不足，再进入外部补能力流程。
  - 外部补能力流程默认顺序：
    1. 检索公开方案：优先官方文档、可信 GitHub 仓库、公开技能库与项目内已有引用来源；
    2. 做安全审阅：检查来源可信度、许可证/用途、权限要求、是否会直接改身体 DNA、是否包含不明可执行脚本、是否与现有架构冲突；
    3. 仅在审阅通过后，才下载/适配为本地 skill 或 MCP 接入方案；
    4. 接入后立刻回到原任务重试，而不是把“学技能”当成本轮终点；
    5. 在产出中留下来源、审阅结论、接入位置、未解决风险与重试结果。
  - 若审阅不通过，要明确写出拒绝引入的原因，而不是为了完成任务硬接高风险能力。

6. **skill 与 MCP 的落地边界**
  - 复用型执行套路、公开工具封装、稳定工作流说明，优先沉淀为 skill。
  - 需要标准化工具调用接口、跨任务复用能力入口、或外部服务协议接入时，可设计/接入 MCP。
  - 无论是 skill 还是 MCP，都不能偷带 Butler 身体 DNA：运行主循环、守护、核心记忆机制、关键安全闸门、核心 schema 与进程生命周期仍属于主代码/配置层。
  - 新能力落地后，至少要补最小说明：用途、来源、前置条件、风险边界、失败回退方式。

## 3. 与各角色的协作约定

- **Soul 层（Butler_SOUL）**
  - 把“优先善用 skills、优先复用现有轮子”视为人格与价值排序的一部分，而不是可有可无的工程细节。

- **对话入口角色**
  - `feishu-workstation-agent` 与 `butler-continuation-agent`：
    - 在各自 role 文件中引用本 `skills.md` 作为“skills 使用总则”。
    - 当用户说“调用 skill”时，默认按本文件的匹配与调用流程执行。

- **心跳与子 Agent**
  - 心跳 planner 在规划涉及外部能力或自我升级/skills 学习的分支时：
    - 优先安排“读取对应 `SKILL.md` + 按说明执行”的小步动作。
    - 在 branch 的 `prompt` 头部明确写出将要使用的 skill 名称及主要目标。
    - 若判断现有能力不够，应显式安排“检索公开方案 -> 安全审阅 -> 落 skill/MCP -> 回到原任务重试”的闭环，而不是只把缺口上报就结束。

## 4. 后续维护方式

- 本文件只描述“通用使用规则与触发场景示例”，不承载具体某个 skill 的细节。
- 每当新增或大幅调整一个 skill 时：
  - 在对应 `SKILL.md` 中写清楚适用场景与调用示例。
  - 视情况在本文件的“典型触发场景”中补充一行，让其它角色更容易发现并复用该 skill。
