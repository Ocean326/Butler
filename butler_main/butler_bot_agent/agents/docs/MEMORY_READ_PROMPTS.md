# 记忆读取提示词（短期 + 长期）

> 本文件为**可复用提示词**：任何需要在回复前「先判断是否需要近期记忆、再按需读长期记忆」的 Agent（如飞书管家、新建的管家类 Agent）均可引用此段内容，放入自己的角色说明中。

---

## 何时使用

在**做决定或回复用户之前**，若该 Agent 启用了「短期 + 长期记忆」机制，应结合 `ANTI_OVERCONSTRAINT.md` 先判断是否值得加载记忆，再按下列步骤完成读取与思考。

---

## 读取顺序与要点

### 1. 先判断是否需要短期记忆（recent_memory）

- **路径**：由记忆机制约定给出，默认 `./butler_bot_agent/agents/recent_memory/recent_memory.json`（若 Agent 使用独立记忆空间，则见该 Agent 的 memory 配置）。
- **用途**：续接最近几轮对话的上下文（话题、结论、待办、long_term_candidate 等）。
- **默认口径**：优先做一次快速判断，而不是机械读取。
- **建议读取的场景**：
  - 当前请求明显是上一轮的继续、追问、收尾或恢复现场；
  - 需要接上既有待办、约定、未完成动作；
  - 用户在问“上次做到哪”“继续”“按之前那个来”；
  - 当前任务若不看 recent_memory 很容易重复劳动或接错上下文。
- **可以先不读的场景**：
  - 当前请求本身独立、边界清晰、可直接执行；
  - 读取 recent_memory 的成本高于它可能带来的收益；
  - 用户明确开启新任务，本轮视为全新起点。
- **使用方式**：若决定读取，再将最近若干条的 `topic`、`summary`、`next_actions`、`long_term_candidate` 纳入思考，避免重复问、漏掉待办或遗忘刚达成的约定。

### 2. 按需读取长期记忆（local_memory）

- **路径**：由记忆机制约定给出，默认 `./butler_bot_agent/agents/local_memory`（若 Agent 使用独立记忆空间，则见该 Agent 的 memory 配置）。
- **用途**：与当前请求相关的长期约定、偏好、技术备忘、人事/生活等。
- **触发**：当用户问题涉及多轮前的约定、项目/人物/主题、或需要「回忆」某类信息时，按**主题/关键词**或**优先级**读取对应文件。
- **优先补读当前用户画像**：若当前请求明显依赖“这位用户平时喜欢怎样的语气、输出密度、emoji、主动 talk、协作方式、关系距离”，优先查看 `./butler_bot_agent/agents/local_memory/Current_User_Profile.private.md`；若该文件不存在，则参考 `Current_User_Profile.template.md`。这类信息不应继续硬编码在项目公共 soul 或 role 里。
- **加载优先级**（若机制约定中有）：越重要、越与运行/底层长期记忆相关的内容，优先读取（如研究管理输出路径、飞书与记忆约定、关键流程等）。
- **使用方式**：只读与当前请求相关的文件或段落，在回复中自然引用，勿无意义罗列全文。
- **禁止事项**：不要为了显得稳妥，把 `local_memory` 当作默认全量背景材料机械扫读。

### 3. 综合与回复

- 将短期记忆（近期上下文）与长期记忆（相关约定/背景）综合后，再组织回复。
- 回复中避免无意义的引导词（如「正在读取…」），直接给出有用信息或下一步动作。
- 若不读取记忆也能高质量完成当前请求，就直接完成，不为流程感增加额外步骤。

---

## 引用方式

- **新建管家类 Agent**：在角色说明的「做决定/回复之前」部分加入一句：「请遵循 `./butler_bot_agent/agents/docs/MEMORY_READ_PROMPTS.md` 与 `./butler_bot_agent/agents/docs/ANTI_OVERCONSTRAINT.md`，先判断是否需要加载记忆，再完成按需读取后回复。」
- **或**：将上述「读取顺序与要点」整段复制到该 Agent 的说明中，并把路径替换为该 Agent 的 memory 路径（若使用独立记忆空间）。
