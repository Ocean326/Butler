# 心跳机制与外部任务组织说明

更新时间：2026-03-10

本文档用于描述**当前已上线机制**。重点说明：heartbeat planner 的 role / template / context 真源、任务真源、recent/local memory 的角色分工，以及 `工作区/local_memory` 中各类说明文档与运行真源之间的关系。

---

## 一、心跳决策器（规划器）的 prompt 从哪来

每次心跳执行时，**规划器**会组一个 prompt，调用模型输出 JSON 计划。该 prompt 的组成如下。

### 1. role 文件

- **路径**：`butler_main/butler_bot_agent/agents/heartbeat-planner-agent.md`
- **作用**：规划器的角色、职责、人格基线、边界与长期判断原则。

### 2. 模板文件

- **路径（主入口）**：`butler_main/butler_bot_agent/agents/heartbeat-planner-prompt.md`
- **作用**：规划器的 JSON 提示词主入口，负责 JSON 计划的字段约定、决策原则与执行约束。
- **路径（JSON 契约镜像）**：`butler_main/butler_bot_code/prompts/heart_beat.md`
- **作用**：仅保留为 JSON 契约与兼容模板，不再承载 role 或决策规则语义。

### 3. 额外上下文文件

- **路径**：`butler_main/butler_bot_agent/agents/heartbeat-planner-context.md`
- **作用**：当前阶段的附加上下文或临时优先级，不再放在 `configs/butler_bot.json` 的 `heartbeat.agent_prompt` 中。

### 4. 填入的上下文（HeartbeatPlanningContext）

| 占位符 / 内容 | 来源 | 说明 |
|---------------|------|------|
| **role_text** | `_load_heartbeat_role_excerpt(workspace)` | **heartbeat-planner-agent.md** 的摘录。心跳规划器的角色真源已经独立，不再复用 `feishu-workstation-agent.md` 作为 planner 角色说明。 |
| soul_text | `_load_butler_soul_excerpt(workspace)` | Butler_SOUL 摘录。 |
| **tasks_md_text** → tasks_context | `_load_heartbeat_tasks_md(workspace)`，空则 `_render_legacy_heartbeat_tasks_md(...)` | 规划器看到的**任务文本视图**。主来源是 `heartbeat_tasks.md`；若该文件为空，才从 legacy JSON store 渲染兜底。 |
| recent_text | `_render_unified_heartbeat_recent_context(workspace)` | **统一近期流**，同时包含 talk recent 与 beat recent。 |
| local_memory_text | `_render_heartbeat_local_memory_query_hits(workspace, ...)` + `_render_heartbeat_local_memory_snippet(workspace)` | 基于 `L0_index.json` 的长期记忆索引：前者按「任务 + 上下文 + 近期流」动态检索命中片段，后者提供默认基线摘录，二者合并后作为 planner 看到的长期语义视图。 |
| skills_text | `_render_available_skills_prompt(workspace)` | 可复用 skills。 |
| context_text | `_load_heartbeat_context_excerpt(workspace, heartbeat_cfg)` | 优先读取 `heartbeat-planner-context.md`；旧 `agent_prompt` 仅作兼容 fallback。 |

结论：  
- **「心跳决策器的 role」** = `heartbeat-planner-agent.md` 的摘录。  
- **「心跳决策器的 context」** = `heartbeat-planner-context.md` 的摘录。  
- **「心跳看到的任务」** = `heartbeat_tasks.md` 的全文或 legacy 渲染结果，在 prompt 里是 `{tasks_context}`，和 role 是**分开的两块**。  
- **「心跳看到的近期上下文」** = talk + beat 的统一 recent，而不是只看旧的 `heart_beat_memory.json`。  
- **「工作区/local_memory 文档」** = 主要是镜像、说明、草稿或人工维护索引，不是 heartbeat 运行时直接加载的唯一真源。

## 二、心跳决策器「角色」真源：别把任务写进去

- **planner 角色文件**：`butler_main/butler_bot_agent/agents/heartbeat-planner-agent.md`
- **对外表达角色文件**：`butler_main/butler_bot_agent/agents/feishu-workstation-agent.md`
- **设计上**：这两类文件只应放**角色设定、规则、职责、边界**，不应放「待办列表」「长期任务补充」等任务类内容。

若主进程（对话时）把用户说的「长期任务补充」写进角色文档，就会发生：

- 下次加载 role excerpt 时，会把这段也当「角色」读进去；
- 表现就是：**长期任务被当成角色的一部分**，看起来像「直接追加到决策器 role 的末尾」。

所以：**长期任务/任务补充不应写入任何角色文档**，应写入下面说的「任务真源」。

---

## 三、心跳相关「任务」真源：应放哪里、怎么组织

### 1. 给规划器看的「任务列表」文本（planner 读口）

- **路径**：`butler_main/butler_bot_agent/agents/local_memory/heartbeat_tasks.md`
- **作用**：规划器每轮读的**任务文本视图**来自这里（`_load_heartbeat_tasks_md` → prompt 里的 `{tasks_context}`）。
- **注意**：当前实现仍可能把「来自对话」追加块写进这个文件，所以它是 planner 的读口，但内容有过载风险；正式新增任务仍应优先走结构化账本或现有合并链路，而不是长期把各种规则和历史聊天都堆进这里。
- **代码**：有 `_append_to_heartbeat_tasks_md(workspace, text)`，用于往该文件追加内容（例如「## 来自对话」+ 若干条）。

### 2. 结构化任务账本（运行真源）

- **路径**：`butler_main/butler_bot_agent/agents/state/task_ledger.json`
- **作用**：短期任务、长期任务、提醒、执行状态等的**统一结构化账本**。
- **设计**：当前更接近运行真源的是 **task_ledger.json**；`heartbeat_tasks.md` 是给 planner 看的文本读口。两者现在仍会互相影响，但语义上不再处于同一层。

### 3. 兼容期仍被读取的旧 store（不作为新增任务主落点）

- **heart_beat_memory.json**：`butler_bot_agent/agents/recent_memory/heart_beat_memory.json`，兼容期短期任务镜像。
- **heartbeat_long_tasks.json**：`butler_bot_agent/agents/local_memory/heartbeat_long_tasks.json`，兼容期长期/定时任务镜像。
- 这两个 JSON 仍会参与 legacy fallback，但**不应继续被文档描述为唯一主真源**。

## 四、主进程（对话）侧：任务应如何落笔

- **预期行为**：用户说「加一个长期任务 / 长期任务补充」时，主进程应优先走两条路：
  - 在**记忆沉淀**里产出带 `heartbeat_tasks` / `heartbeat_long_term_tasks` 的 entry，由现有逻辑在 `on_reply_sent` 后合并进 task ledger，并视需要刷新 planner 读口；
  - 或显式维护 `heartbeat_tasks.md` 中面向 planner 的任务段落。
- **错误行为**：把长期任务/补充**写进 feishu-workstation-agent.md**。这样会污染「角色」，并被心跳当成 role 末尾内容读进去。

## 五、工作区 `local_memory` 的位置

`./工作区/local_memory` 这批文档很重要，但它们的角色应统一理解为：

- **工作区镜像 / 说明层**：给人看、给治理看、给后续任务整理看。
- **方案与草稿层**：例如四层模型草稿、role/output_dir 设计笔记、persona sandbox。
- **不是单一运行真源**：heartbeat 运行时真正读取的是 `prompts/heart_beat.md`、`heartbeat-planner-agent.md`、`agents/local_memory/*`、`recent_memory/*`、`state/task_ledger.json` 等实现层文件。

因此，工作区文档要尽量避免自称“唯一真源”；更准确的表述是：它们是**工作区侧的镜像、说明、治理索引或待落地设计**。

当前实现里，主进程**没有**在代码里强制「用户说加长期任务时必须写 heartbeat_tasks.md 或 task_ledger」；模型可以自由选文件。若模型选成改 feishu-workstation-agent.md，就会出现你看到的现象。

---

## 六、小结表：心跳机制与外部任务放哪

| 内容类型 | 应放置位置 | 不应放置位置 |
|----------|------------|--------------|
| 心跳规划器角色 | heartbeat-planner-agent.md | 在角色文档里堆任务清单 |
| 对外表达角色 | feishu-workstation-agent.md / butler-continuation-agent.md | 在角色文档里堆任务清单 |
| 长期任务 / 任务补充 | task_ledger.json + heartbeat_tasks.md | 任意角色文档 |
| 短期待办、提醒 | task_ledger.json + heartbeat_tasks.md | 任意角色文档 |
| 规划器角色、人格约束、职责边界 | heartbeat-planner-agent.md、Butler_SOUL | 不要混写进任务清单 |
| 规划器附加上下文 | heartbeat-planner-context.md | 不要继续塞回 heartbeat 配置 |
| 规划器 JSON 契约与模板 | heart_beat.md | 不要混写成长角色文档 |
| 工作区说明、镜像、设计草稿 | `./工作区/local_memory/*` | 冒充运行真源 |

- **planner role 的来源**：`heartbeat-planner-agent.md`。  
- **heartbeat 任务的来源**：`task_ledger.json`（结构化真源）+ `heartbeat_tasks.md`（planner 文本读口）。  
- **工作区 local_memory 的来源性质**：治理镜像、辅助说明、草稿和人工整理层，不是单一运行真源。

若后续要继续降噪，下一步应把 `heartbeat_tasks.md` 从“任务池 + 历史追加区 + 杂项说明”进一步拆层。本次先统一文档口径，不改运行行为。
