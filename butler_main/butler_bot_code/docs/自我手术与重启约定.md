# 自我手术与重启约定

> 管家 bot 在心跳中可修改自身代码/配置（自我手术）；若需重启主进程使改动生效，通过「重启标记」触发自动重启，实现不断心跳下的自我升级。

## 1. 何时需要重启

- 修改了 `butler_bot_code` 下 **Python 代码**（如 `memory_manager.py`、`butler_bot.py`、`agent.py`）后，当前主进程仍运行旧代码，需重启才能加载新逻辑。
- 修改了 `butler_bot_code/configs/butler_bot.json` 等**配置**后，若希望立即生效（如心跳间隔、主动说话窗口），也需重启主进程重新读配置。
- 修改了 `butler_bot_agent/agents/heartbeat-planner-context.md` 这类 agent 侧上下文文件时，通常**不需要重启**；下次心跳自然会读到。
- 仅修改 **工作区** 内文档、或 `./butler_bot_agent/agents` 下角色/记忆时，**不需要**重启（下次心跳或下次对话会自然读到新内容）。

## 2. 心跳任务如何请求重启

在心跳执行任务时，若本次**自我手术**涉及上述代码或配置且需重启生效，请在本轮产出中**任选一种**方式写入重启标记：

- **方式 A**：在工作区根下创建文件 `工作区/restart_request.json`，内容示例：
  ```json
  { "reason": "修改了 butler_bot 心跳间隔配置，需重启生效" }
  ```
- **方式 B**：在 `butler_bot_code/run/` 下创建空文件 `restart_requested.flag`（或写入一行说明）。

写入后**无需再执行任何命令**；本次心跳结束后，心跳子进程会检测到标记并自动执行重启流程。

## 3. 自动重启流程（实现侧）

1. 每次 `_run_heartbeat_once` 结束（发送心跳消息并回写任务状态）后，调用 `_check_and_perform_restart(workspace)`。
2. 若存在 `工作区/restart_request.json` 或 `butler_bot_code/run/restart_requested.flag`：
  - 读取重启原因（若 JSON 里有 `reason`）；
  - 删除这两个标记（避免重复重启）；
  - 启动独立守护进程 `butler_bot_code/butler_bot/restart_guardian_agent.py`；
  - 当前心跳子进程 `sys.exit(0)` 退出。
3. `restart_guardian_agent` 会执行：
  - 先调用 `manager.ps1 restart butler_bot`；
  - 若未拉起成功，自动执行兜底：`manager.ps1 stop --all` + `manager.ps1 start butler_bot`；
  - 作为常驻守护进程每 30 秒巡检一次：
    - **对话存活**：优先读取 `butler_bot_code/run/butler_bot_main_state.json` 中的主进程运行态（PID + `updated_at` 心跳）；仅在状态文件缺失时再回退 `manager.ps1 status`；
    - **心跳存活**：`heartbeat_last_sent.json` 是否在阈值内更新；
  - 若检测到主进程死亡/心跳超时，则自动触发「抢救」与「心脏起搏器（stop-all + start）」；
  - 解析最新 `*.err.log` 提取常见故障模式（配置缺失、Cursor CLI 缺失、权限/执行策略、Traceback 等）；
  - 输出复盘报告到 `工作区/governance/self_upgrade_reports/`，并写入 `restart_guardian_latest.{json,md}`；
  - 同步写一条「自我升级守护报告」到 `./butler_bot_agent/agents/recent_memory/recent_memory.json`，供后续对话与升级规避参考。

## 3.1 心跳异常分级处理（新增）

- **Level 1 软恢复（仅心跳异常）**：
  - 条件：主对话进程存活，但心跳超时/异常；
  - 动作：守护进程终止 heartbeat 子进程，交给主进程内置 heartbeat watchdog 自动拉起；
  - 目标：最小扰动，不重启整个 butler。
- **Level 2 硬恢复（主流程重启）**：
  - 条件：软恢复后仍异常，或收到显式重启请求；
  - 动作：`manager.ps1 restart butler_bot`。
- **Level 3 心脏起搏器（兜底）**：
  - 条件：Level 2 后仍未恢复；
  - 动作：`manager.ps1 stop --all` + `manager.ps1 start butler_bot`。

以上每一级动作都会写入守护报告，便于后续复盘与升级规避。

## 4. 约定与注意

- **谁写标记**：由心跳执行任务时的 Agent（飞书工作站管家）在完成代码/配置修改后，在同一轮产出中写入上述文件之一。
- **何时写**：仅当确实修改了 `butler_bot_code` 下代码或配置、且需要重启才能生效时写入；避免无谓重启。
- **提示词注入**：`HEARTBEAT_WORKSPACE_HINT` 中已包含「自我手术·重启」一条，心跳执行器每次都会看到该约定。
- **安全**：重启会短暂中断当前主进程与心跳子进程；新进程启动后心跳会按配置继续运行（如每 5 秒一次）。
- **守护生命周期**：`manager.ps1 start butler_bot` 会自动拉起守护进程；`manager.ps1 stop butler_bot` / `stop --all` 会停止守护进程（守护自身触发抢救时不会误杀自己）。
- **职责边界**：主对话进程负责写入自身运行态与管理 heartbeat 子进程；guardian 只负责巡检主进程与升级/抢救，不直接接管 heartbeat 的常规重启节奏；heartbeat 只负责心跳任务与重启请求上报。

## 5. 相关路径与配置

| 用途           | 路径 |
|----------------|------|
| 重启标记（JSON）| `{workspace}/工作区/restart_request.json` |
| 重启标记（flag）| `{workspace}/butler_bot_code/run/restart_requested.flag` |
| 主进程运行态 | `{workspace}/butler_bot_code/run/butler_bot_main_state.json` |
| 重启脚本       | `{workspace}/butler_bot_code/manager.ps1`（参数：restart butler_bot）|
| 守护 agent     | `{workspace}/butler_bot_code/butler_bot/restart_guardian_agent.py` |
| 守护报告       | `{workspace}/工作区/governance/self_upgrade_reports/restart_guardian_latest.json` |

---

*本约定使管家能在不断心跳中完成「改代码 → 写标记 → 自动重启 → 新进程继续心跳」的自我升级循环。*
