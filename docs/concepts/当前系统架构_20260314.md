# 当前系统架构（2026-03-14）

## 1. 架构定位

相对 0313 方案，当前 Butler 已经从“飞书入口 + 大一统 memory_manager + 一轮式 heartbeat”推进到一个更接近 **前台入口 / 经理调度 / 执行角色 / 记忆底座 / 运行监视** 分层的结构，但还没有完全走到独立 `planner_service / supervisor_service / self_mind_app` 的终态。

当前更准确的定位是：

- 飞书 Bot 作为前台入口与同步交互面
- 心跳侧承担经理化调度、兜底规划、阶段推进
- branch 层开始具备 `process_role`、`runtime_profile`、验收回执
- memory 层开始从单文件重逻辑走向 backend/index/service 分层
- guardian 仍是守护工具，但当前在线恢复路径仍以主进程 + sidecar 手工可控为主

---

## 2. 当前分层

## 2.1 前台入口层

核心文件：

- `butler_bot/agent.py`
- `butler_bot/butler_bot.py`
- `butler_bot/request_intake_service.py`

职责：

- 接收飞书消息
- 做同步回复与流式反馈
- 将用户请求分诊为 `discussion_only / sync_quick_task / sync_then_async / async_program`
- 将复杂任务转交给后台 task/heartbeat 链路

与 0313 对照：

- 已落地“前台分诊骨架”
- 仍未完全把前台状态卡片、对话状态服务独立成单独模块

---

## 2.2 调度与经理层

核心文件：

- `butler_bot/heartbeat_orchestration.py`
- `butler_bot/runtime_router.py`
- `butler_bot/task_ledger_service.py`
- `butler_bot/acceptance_service.py`

当前形态：

- `HeartbeatOrchestrator` 仍是编排主入口，但已经不再纯粹是“一轮调度器”
- branch 已开始带有 `process_role`、`runtime_profile`
- runtime 决策从 `cli_runtime.py` 上移到 `runtime_router.py`
- 验收回执由 `acceptance_service.py` 统一构造并落回 task ledger / report

当前更接近的角色分工：

- `manager/planner`
  - 在心跳中做计划、回执、阶段推进
- `executor`
  - 执行 branch prompt
- `test/evaluator`
  - 以 `process_role=test` 或 review 型 branch 承担验证
- `acceptance`
  - 负责任务收口、产出回执、验收状态

与 0313 对照：

- `request_intake_service.py`、`runtime_router.py`、`acceptance_service.py` 已落地
- “经理制 + 多阶段 agent” 的 branch 骨架已落地
- 仍未完全拆成独立 `planner_service.py / supervisor_service.py`

---

## 2.3 执行层

核心文件：

- `butler_bot/agent_team_executor.py`
- `butler_bot/agent_capability_registry.py`
- `butler_bot/prompt_assembly_service.py`
- `butler_bot/cli_runtime.py`

职责：

- 团队 agent / capability catalog 管理
- planner / dialogue prompt 组装
- branch 运行时模型与 CLI 适配
- Cursor/Codex provider 配置与调用

当前状态：

- 团队执行、能力注册、prompt 装配已从主流程中抽出
- `runtime_router.py` 已将“用哪个 CLI/模型”的决策外提
- 当前线上默认更偏向 Cursor 稳定路径，Codex 保留但不再作为自动分支的强依赖

---

## 2.4 记忆与索引层

核心文件：

- `butler_bot/memory_manager.py`
- `butler_bot/memory_backend.py`
- `butler_bot/local_memory_index_service.py`

职责：

- recent memory / local memory / semantic event 的统一写口
- 本地长期记忆索引和关系维护
- heartbeat / self_mind / recent 共享的记忆底座

与 0313 对照：

- `memory_backend.py`、`local_memory_index_service.py` 已补出，memory 开始退化为 facade + orchestration
- 但 `memory_manager.py` 仍然偏重，`self_mind` 尚未真正拆成独立 app

---

## 2.5 运行与监视层

核心文件：

- `manager.ps1`
- `guardian/manager.ps1`
- `watch_stack.ps1`

职责：

- 主进程启动、停止、重启
- heartbeat sidecar 启停与守护
- 主进程 / 心跳 / watchdog / 日志的一屏观察

本轮新增：

- `watch_stack.ps1`
  - 作为快速后台监视通路
  - 一屏看主进程、heartbeat、watchdog 与最新日志尾部

---

## 3. 当前任务流

用户请求进入系统后，大体流程是：

1. 飞书入口接收消息
2. `request_intake_service.py` 判断是讨论、同步快任务还是异步 program
3. 若转后台，则写入 task ledger / heartbeat task source
4. `heartbeat_orchestration.py` 以经理视角规划 branch
5. `runtime_router.py` 为 branch 决定 runtime profile
6. `cli_runtime.py` / team executor 执行 branch
7. `acceptance_service.py` / `task_ledger_service.py` 回写验收信息与最终回执

这比 0313 之前的路径更清楚：前台、调度、执行、验收已经开始分层，不再全压在一个 prompt 和一个大文件里。

---

## 4. 当前仍未完成的 0313 项

仍未完全完成的关键项有四类：

- `HeartbeatOrchestrator` 还没有完全退化成 facade
- `planner_service.py / supervisor_service.py` 还没有独立成正式模块
- `self_mind` 还没有外挂化为独立 app
- guardian 的自动恢复链还没有完全稳定到可以替代人工清理 sidecar

---

## 5. 当前推荐理解方式

不要再把当前 Butler 理解为“一个会回复飞书的脚本”，而要理解为：

一个以飞书为前台入口、以 task ledger 为任务真源、以 heartbeat 为经理调度器、以 Cursor/Codex 为执行器、以 memory backend 为共享认知底座的多模块系统。

相对于 0313 方案，当前系统已经有骨架；相对于最终目标，当前仍处于“骨架已立、职责还在继续解耦”的阶段。
