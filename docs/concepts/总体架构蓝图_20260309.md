# Butler 重构总体架构蓝图（2026-03-09）

## 目标

把 Butler 从“运行时大总控 + 多套真源 + 文档补丁式治理”重构为：

- 更智能：模型只负责理解、规划、提议，不再被多套冲突真源干扰。
- 更易维护：状态、策略、运维、升级分层。
- 可受控自升级：低风险可自改自测，高风险必须批准。

## 目标分层

### 1. Transport

职责：飞书长连接、消息发送、附件下载、去重、回调入口。

当前主要文件：`butler_bot_code/butler_bot/agent.py`

### 2. Conversation Runtime

职责：

- 单轮对话编排
- recent context 注入
- 调用模型
- 将候选记忆/候选任务交给下游服务

### 3. Memory Service

职责：

- `talk_recent_memory`：短期对话续接
- `beat_recent_memory`：heartbeat / 连续思考续接
- `self_mind`：主意识回忆层与时间连续层，承接被动思考线索与主动思考主线
- `semantic_memory`：长期事实、偏好、约定，采用 L0/L1/L2 分层
- 候选提取与正式落账分离

当前约定：talk 与 beat 使用两个独立 recent 池，允许互相只读摘要，但不混写活跃 recent。

补充：Memory Service 不应只理解为“recent + semantic memory”。按当前 Butler 演化方向，还应显式包含两段思考链路：

- **被动思考链路**：`recent -> subconscious 整理 -> self_mind / semantic_memory 分流`
- **主动思考链路**：`butler-agent <- self_mind + recent + semantic_memory`，在此基础上形成主线、主动开口与自由活动方向

也就是说，Memory Service 不只是存储服务，也是“事件痕迹 -> 回忆层 -> 长期知识层”的连续组织服务。

### 4. Task Ledger

职责：

- 统一记录短期任务、长期任务、提醒、执行状态
- 成为统一的 machine-readable 任务结构真源（工作区文档与 local_memory 侧为说明/镜像层，不与之竞争写入真源）
- 对外输出只读镜像，不允许多处并写

### 5. Governor

职责：

- 评估动作风险
- 决定是否允许自动执行
- 控制自升级、自修改、自重启边界

### 6. Observer / Telemetry

职责：

- 心跳运行快照
- planner backoff
- 守护状态
- 主进程状态
- 仅负责运行观测，不写认知记忆

### 7. Upgrade Engine

职责：

- 接收升级提议
- 生成补丁/执行计划
- 调用 governor 审批
- 执行测试、上线、回滚

## 当前第一步实现

本轮已开始落地两项基础设施：

1. `task_ledger_service.py`
2. `governor.py`

并将 heartbeat `apply_plan()` 接入 `task_ledger`，形成新的正式落账入口。

## 分阶段迁移原则

- 先建新内核，不立刻拆旧逻辑。
- 先双写和镜像验证，再单写切流。
- 活跃目录只保留当前真源；历史日志与旧说明迁入 `过时/` 或 `99_归档/`。
