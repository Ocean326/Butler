## 一、先给出结论：Butler 应该成为什么

Butler 不应继续朝“会聊天的脚本集合”演化，而应明确朝下面这个方向演化：

**一个有自我边界、有分层记忆、有离线巩固、有任务账本、有安全治理、能持续积累经验并修正自身的长期陪伴型认知系统。**

这句话拆开后有六个工程含义：

1. **记忆不是附件，而是操作系统的一部分。** 记忆应参与规划、执行、反思、升级与风险控制，而不只是“为了续聊而存摘要”。
2. **近期记忆、长期语义、程序性经验、任务状态、自我模型必须分层。** 混成一锅会让检索、更新、冲突处理和安全治理全部失真。
3. **后台自动化不能只做提醒与执行，它还应承担离线巩固与再组织。** 这相当于 Butler 的“睡眠期”与“后台认知期”。
4. **长期记忆的核心不是多，而是可更新、可追溯、可冲突管理。** 否则系统会越来越像会遗忘来源、却坚持错误结论的人。
5. **Guardian 的价值不只是运维，而是系统的执行闸门。** 认知系统越复杂，自修改越要被约束在审阅、测试、回滚框架内。
6. **项目目标不是机械模仿人脑，而是借用认知科学的“功能结构”，构建更稳、更长期、更可维护的 AI 系统。**

因此，Butler 的长期路线不应是“继续补 prompt 和记忆字段”，而应是从以下五条主线持续演进：

1. 记忆分层化
2. 巩固流程化
3. 自我结构化
4. 任务账本化
5. 升级治理化

## 二、证据基础：认知科学给了什么，不能误读什么

本文采用的是“功能映射”而非“脑区拟真”。认知科学提供的不是一套可以直接照搬的代码结构，而是一组稳定的系统设计启发。

### 2.1 工作记忆告诉我们：上下文窗口必须被当作稀缺资源管理

Baddeley 将工作记忆描述为支持临时存储与操作的系统，用于语言理解、学习与推理，而不是无限容量的缓存池。后续综述也持续强调，工作记忆与注意控制、执行控制是绑定的。[1][2]

对 Butler 的工程含义是：

1. 每轮 prompt 上下文必须被视为稀缺资源，而不是“能塞多少塞多少”。
2. `Soul + role + recent + local memory` 的组装过程，本质上就是工作记忆的装配过程。
3. recent 的价值不是长期保存全部细节，而是在有限上下文内维持任务连续性。

因此，Butler 不该把“增加上下文长度”当成长期解法。长期解法应是：**更好的筛选、压缩、检索与分层记忆。**

### 2.2 记忆巩固研究告诉我们：近期经历需要离线整理，才能变成稳定知识

睡眠与系统巩固研究表明，新近经历不会自动成为稳定长期记忆；海马依赖的近期表征会在离线期被重放、重组，并逐渐转向更抽象、图式化的新皮层表示。Rasch 与 Born、Klinzing 等、Brodt 等的综述都指向这一点。[3][4][5]

对 Butler 的工程含义是：

1. recent 不应直接等同于 long-term。
2. `recent -> local memory (L0/L1/L2)` 的过程应被看作显式巩固流程，而不是普通文件写入。
3. 后台自动化最重要的长期价值之一，不是“定时帮忙做点事”，而是“在无人打扰时完成后台巩固”。

因此，Butler 后续要把后台 round 从“定时任务轮次”升级成“任务 + 认知巩固双引擎”。

### 2.3 再巩固研究告诉我们：被提取的记忆应允许变得可塑，但必须受控

Lee、Nader、Schiller 等综述指出，记忆在被提取后可能进入短暂的可塑窗口，并在再稳定前被更新、增强或削弱；Elsey 等对人类研究也强调，再巩固不是“随便覆盖旧记忆”，而是有条件、有限制的更新过程。[6][7]

对 Butler 的工程含义是：

1. 长期记忆不能只有“追加新块”与“当前结论覆盖”。
2. 当系统命中某条长期记忆并决定依据它行动时，这条记忆应进入“可更新状态”。
3. 更新必须记录来源、原因、时间、冲突对象和旧版本。

这意味着 Butler 的长期记忆应该从“Markdown 块更新”升级到“事件流 + 当前视图”的模式。

### 2.4 图式研究告诉我们：长期记忆的目标不是堆细节，而是形成可激活的稳定结构

Ghosh 与 Gilboa、Gilboa 与 Marlatte 的综述指出，schema 并不是任意摘要，而是能指导理解、预测与整合新信息的知识结构。schema-congruent 信息更容易整合，但 schema 过强也会造成偏差。[8][9]

对 Butler 的工程含义是：

1. L0 的 `current_conclusion + applicable_scenarios + keywords` 方向是对的，它比堆满长文本更接近“图式”。
2. 但系统必须区分“与既有图式一致”和“与既有图式冲突”。
3. 任何长期结论都应该绑定适用情境与反例边界。

所以，Butler 的长期记忆不是知识仓库，而应是**带适用条件的行动图式库**。

### 2.5 自传体记忆研究告诉我们：稳定自我不是一句 system prompt，而是结构化的自我-记忆系统

Conway 与 Pleydell-Pearce 的 Self-Memory System 模型提出，自传体记忆是在 working self 约束下构建出来的，而不是简单读取的静态库。[10]

对 Butler 的工程含义是：

1. `Soul` 不是装饰性人格文案，而是高优先级的稳定自我约束。
2. `role` 不是普通 prompt，而是当前 working self 的任务面。
3. recent 与 local memory 共同构成 Butler 的自传体材料，但其激活必须被当前目标约束。

这意味着 Butler 不能只有“记忆系统”，还必须有明确的“自我系统”。

## 三、证据基础：计算机科学与 AI 已经验证了什么

### 3.1 仅靠参数记忆不够，外部记忆是长期系统的必要部件

Roberts、Raffel、Shazeer 指出，大模型参数中确实可以存储并调用相当多的知识，但这并不消除外部知识与检索式系统的价值；在开放问答等任务上，显式检索仍然是重要路线。[11]

对 Butler 的含义非常直接：

1. 不应把“模型更大了”当作 Butler 记忆问题的最终解法。
2. 外部可检索记忆、任务账本、长期索引，仍然必须存在。
3. Soul、规则、任务与长期经验都不应只活在 prompt 里。

### 3.2 生成式 agent 研究证明：观察、反思、规划三段式架构是有效的

Park 等的 Generative Agents 论文中，agent 能形成较可信的长期行为，不是因为上下文更长，而是因为它们把经历记录、反思抽象和规划检索组合成了一个循环；消融实验也显示 observation、planning、reflection 都是关键组件。[12]

这与 Butler 当前方向高度一致，但也暴露出下一步要求：

1. 观察层要持续积累高质量事件，而不是只保留一句摘要。
2. 反思层要生成更高阶的结论，而不是只拼接文本。
3. 规划层检索的应是“当前有效结论”，而不是所有历史碎片。

### 3.3 长期对话记忆研究证明：递归摘要有效，但只能解决一部分问题

Wang 等关于递归摘要的研究显示，把局部对话先压缩成小记忆，再递归整合为新记忆，可以显著改善长对话一致性，而且能与长上下文和 retrieval-enhanced 系统互补。[13]

对 Butler 的启发是：

1. 递归摘要适合 recent 到 semantic 的过渡。
2. 但摘要并不天然等于真理，也不天然等于程序性经验。
3. 摘要链路必须与冲突检测、来源记录、长期更新流程联动。

换句话说，摘要是必要能力，但不是完整记忆架构。

### 3.4 MemGPT 说明：多层内存管理比单一上下文堆叠更接近可扩展系统

MemGPT 把 LLM 视作拥有有限主存的执行器，通过快慢不同的内存层与显式数据迁移，让系统在有限上下文里表现得像有更大内存。[14]

这对 Butler 的指导价值很高：

1. `当前 prompt`、`recent`、`L0 索引`、`L1/L2`、`工作区`、`日志/ledger`，本来就该被视为不同速度、不同成本、不同可信度的存储层。
2. 关键不在于“所有层都给模型看”，而在于“什么时候把哪一层搬进当前工作记忆”。
3. 未来 Butler 应把 memory orchestration 做成清晰的控制器，而不是散在各函数里的 if/else。

### 3.5 AI agent 工程界已经给出一个结论：经验库必须和技能库分离

近年的 agent 系统普遍把“发生过什么”与“怎么做某件事”分开管理。虽然 Butler 目前已经在 recent 与 local semantic 上走得很远，但“程序性记忆/技能记忆”仍然空缺。

这会直接带来两个问题：

1. 系统知道用户喜欢什么，却不知道“遇到某情境时该按什么稳定流程做”。
2. 系统知道上次做成了，但难以把成功做法抽象成可复用策略。

因此，Butler 的下一阶段必须显式引入 procedural memory，而不能继续把它隐含在 prompt 经验里。

## 四、对 Butler 的核心判断：项目现在最接近什么，还缺什么

结合现有文档与实现方向，Butler 已经有了下面这些非常正确的雏形：

1. **统一 recent schema**：这是从“聊天摘要”迈向“多通道近期记忆”的关键一步。
2. **SubconsciousConsolidationService**：这是把“写记忆”从单纯摘要，升级成预整理、预判断、预筛选的关键一步。
3. **L0/L1/L2 长期记忆结构**：这是从“文件堆”迈向“语义层 + 人类可读层 + 细节层”的关键一步。
4. **后台自动化读取 soul / role / recent / local index**：这使得后台规划不再只是机械定时器。
5. **Guardian 独立化**：这为自修改、自升级、自恢复提供了必要的边界系统。

但是，如果从长期系统视角看，当前仍有五个核心缺口：

### 4.1 缺少程序性记忆层

目前系统能记住偏好、事实、关系和任务，但还不能稳定表示：

1. 在什么情境下应启动什么流程
2. 哪种操作顺序更可靠
3. 某项任务的惯用策略是什么
4. 某个用户的偏好对应怎样的执行风格

### 4.2 缺少显式再巩固协议

现在有“历史演化”文本，但还没有一个完整的“检索命中 -> 进入可塑窗口 -> 对照新证据 -> 产生命题更新 -> 写回版本链”的状态机。

### 4.3 后台自动化仍偏执行器，尚未充分承担离线认知职责

当前后台自动化链路已经能规划和执行任务，但还没真正承担：

1. 高显著性 recent 重放
2. 跨轮主题聚类
3. 长期记忆修订
4. 经验抽象
5. 冲突检测与待核实队列维护

### 4.4 缺少来源可信度与冲突账本

长期系统迟早会遇到：

1. 用户后来改变偏好
2. Butler 的旧推断被新信息推翻
3. 后台轮执行结果与先前假设冲突
4. 多条记忆都像真的，但不能同时成立

如果没有显式 source/conflict ledger，系统会越来越“会说但不可信”。

### 4.5 缺少长期评测框架

当前项目已经有测试与治理意识，但还缺“认知系统级”的评测，例如：

1. 记忆命中是否帮助回答而不是污染回答
2. 长期更新后是否减少自相矛盾
3. 高 salience 条目是否更容易被正确回忆
4. 程序性经验是否真的提升任务成功率

没有这类评测，记忆系统很容易变成“感觉更聪明”，但没有证据证明真的更好。

## 五、长期目标架构：建议把 Butler 演化为七层认知系统

建议 Butler 的长期目标不是继续在单个 `MemoryManager` 上叠功能，而是逐步演化为以下七层。

### 5.1 第 1 层：Working Set

职责：为当前轮对话或当前次后台 round 组装最小必要上下文。

输入来源：

1. Soul
2. 当前 role
3. 近期高相关 recent
4. L0 当前有效结论
5. 当前任务账本状态

要求：

1. 有明确 token 预算
2. 有显式选择理由
3. 有降级策略

### 5.2 第 2 层：Episodic Stream

职责：记录“发生了什么”。

这层就是 unified recent，但后续建议进一步分出标准事件模型：

1. talk event
2. background observation
3. task event
4. relationship event
5. maintenance / guardian event
6. self-modification event

建议：这一层以**事件流真源**为主，不以摘要块为主。

### 5.3 第 3 层：Semantic Schema Store

职责：记录“现在认为稳定成立的结论是什么，以及何时适用”。

这层大致对应当前 L0/L1/L2，但后续需要增强：

1. 每条结论有唯一 schema_id
2. 每条结论有 version
3. 每条结论有 source provenance
4. 每条结论有 confidence 与 evidence count
5. 每条结论有 applicable_scenarios 与 anti_scenarios
6. 每条结论有 superseded_by / conflicts_with

### 5.4 第 4 层：Procedural Memory

职责：记录“怎么做”。

建议单独建层，不要继续混在语义记忆里。它应包括：

1. 用户偏好的执行风格模板
2. 常见任务 SOP
3. 经验证有效的 prompt / tool / test 组合
4. 特定情境下的策略路由规则

示例：

1. “涉及代码修改和重启时，先出 patch plan，再走 Guardian。”
2. “涉及工作区正式产出时，优先写入公司目录，而不是家目录。”
3. “用户要求长期提醒时，优先落 task ledger，不直接写普通记忆。”

### 5.5 第 5 层：Self Model

职责：回答“我是谁，我的边界是什么，我当前处于什么角色，我不能做什么”。

建议把 `Soul`、`role`、长期身份约束、与 Guardian/用户的关系边界整合成独立的自我模型层。它至少要包含：

1. 稳定原则
2. 当前职责
3. 权限边界
4. 失败恢复原则
5. 升级与审批边界

### 5.6 第 6 层：Offline Consolidation Engine

职责：在后台 round 或低负载窗口完成：

1. high-salience event replay
2. topic clustering
3. recent -> semantic 提升
4. semantic reconsolidation
5. episodic -> procedural 抽象
6. 待核实/冲突条目整理

这层是 Butler 后续“越来越像长期主体”的关键，不可缺。

### 5.7 第 7 层：Governance and Safety Gate

职责：对高风险动作施加边界。

这层应由 Guardian、Governor、测试、回滚和 patch plan 一起构成，确保：

1. 认知系统可以成长，但不能无闸门自改
2. 长期记忆可以更新，但不能静默污染
3. 任务可以自动推进，但不能突破风险边界

## 六、具体工程建议：未来两步最值得做什么

### 6.1 第一优先级：把记忆写入机制改造成“事件流 + 物化视图”

这是本文最重要的工程建议。

当前 L0/L1/L2 已经比普通项目成熟很多，但未来如果要支持再巩固、冲突管理、来源追踪和程序性抽象，仅靠直接改 Markdown 会越来越吃力。

建议演化为：

1. **真源**：append-only memory events
2. **视图 A**：recent working view
3. **视图 B**：semantic current view
4. **视图 C**：procedural profile view
5. **视图 D**：human-readable markdown mirrors

事件类型可包含：

1. `episode.recorded`
2. `memory.promoted`
3. `memory.retrieved`
4. `memory.reconsolidated`
5. `memory.conflict_detected`
6. `procedure.learned`
7. `procedure.deprecated`
8. `task.outcome_observed`

这样做的收益：

1. 再巩固终于有落点
2. 冲突管理终于能被追踪
3. 视图可以重建
4. 机器真源与人类阅读层可以解耦

### 6.2 第二优先级：把后台自动化扩展为“后台巩固器”

建议后台 round 每轮不只做任务规划，还评估是否进入以下模式之一：

1. `task_execution`
2. `memory_consolidation`
3. `reconsolidation_review`
4. `procedural_extraction`
5. `conflict_audit`
6. `idle_status`

最低可行版本建议：

1. 当没有高优先级任务时，扫描高 salience recent
2. 对过去 N 条 recent 做主题聚类
3. 对每个主题判断：保持 recent、提升为 semantic、更新既有 schema、抽象 procedural、还是送入待核实
4. 把结果写入相应视图

如果这一步做成，Butler 的后台能力会从“定时执行器”变成“持续整脑的人”。

## 七、关于程序性记忆：这是 Butler 下一阶段最该补的缺口

程序性记忆在本项目里不是锦上添花，而是从“会记事”走向“会长期做事”的分水岭。

建议单独设计为以下结构：

```json
{
  "procedure_id": "uuid",
  "title": "涉及代码改动或重启时走 Guardian 审阅流程",
  "trigger_conditions": ["code_change", "restart", "risk>=medium"],
  "steps": [
    "生成 patch plan",
    "选择测试范围",
    "执行测试",
    "请求 Guardian 审阅或备案",
    "执行变更",
    "记录结果与回滚点"
  ],
  "success_signals": ["tests_passed", "service_healthy"],
  "failure_signals": ["health_check_failed", "rollback_invoked"],
  "source": "learned_from_history|explicit_rule|user_teach",
```

程序性记忆的来源可以有三类：

1. 用户明确教的流程
2. 项目规则文档中的硬约束
3. 多次成功执行后抽象出来的经验流程

最重要的原则：**程序性记忆不是让模型更自由，而是让系统在已知情境下更稳定。**

---

## 八、关于自我系统：Soul、role、记忆、任务之间应该怎样分工

长期系统最怕四类东西混在一起：

1. 原则
2. 偏好
3. 事实
4. 任务

建议 Butler 后续把这四类严格分开：

### 8.1 Soul

放：稳定原则、身份、价值观、底线、长期人格。

不放：具体任务、短期偏好、临时策略。

### 8.2 Role

放：当前阶段职责、当前工作模式、当前目标函数。

不放：长期历史细节。

### 8.3 Memory

放：事实、偏好、关系、经验、程序、过往事件。

不放：尚未完成的待办真源。

### 8.4 Task Ledger

放：待办、计划、状态、截止时间、执行结果。

不放：用户人格偏好和长期世界知识。

这是 Butler 继续进化时必须守住的边界。边界一旦乱掉，检索质量和治理成本都会迅速恶化。

---

## 九、关于长期记忆更新：必须从“覆盖文本”转向“证据驱动的版本演化”

建议以后每条长期结论都遵循以下更新协议：

1. **检索命中**：某条 schema 在当前轮被命中。
2. **证据比对**：新 episode 是否支持、补充、削弱或推翻旧结论。
3. **更新决策**：
   - support：增加证据数，提高置信度
   - refine：细化适用情景
   - contradict：标记冲突并生成候选新版本
   - obsolete：旧结论失效但保留历史
4. **写回事件**：记录谁改的、何时改的、依据哪些 recent/task/result。
5. **更新视图**：刷新 L0 current view、L1 镜像、L2 细节层。

这套协议的关键不是复杂，而是让系统未来能回答：

1. 你为什么会这样认为？
2. 你什么时候改了看法？
3. 这条结论目前还有效吗？

如果 Butler 能稳定回答这三问，它的长期可信度会大幅提升。

## 十、关于评测：没有长期评测，记忆系统就只有幻觉性的“进步感”

建议建立 Butler 专用的认知评测集，至少覆盖六类指标。

### 10.1 Recall Precision

在应该记起时，是否能记起正确内容；在不该记起时，是否减少错误召回。

### 10.2 Consistency Over Time

同一主题跨 1 天、1 周、1 月的回答是否更稳定，且能正确反映更新。

### 10.3 Memory Pollution Rate

被写入长期记忆的条目中，有多少后来证明是噪音、误判或过度推断。

### 10.4 Reconsolidation Quality

当用户偏好、任务状态或项目规则发生变化时，系统是否能更新旧结论而非产生双重人格式矛盾。

### 10.5 Procedural Win Rate

引入程序性记忆后，某类任务的成功率、重试次数、平均修复时长是否改善。

### 10.6 Safety Compliance

涉及自修改、重启、升级时，是否遵守 Guardian/Governor 边界，是否保留 patch plan、测试记录和回滚点。

建议把这些指标分成两类：

1. 离线回放评测
2. 在线运行观测

这样 Butler 的“成长”才不只是主观感觉，而是可测量事实。

## 十一、风险清单：项目后续最容易走偏的地方

### 11.1 风险一：把所有问题都归结为 prompt 不够强

长期主体的问题，大多数不是 prompt 问题，而是记忆结构、更新协议和执行治理问题。

### 11.2 风险二：把所有记忆都塞进一个仓库

如果 episodic、semantic、procedural、task、self 全部混存，后续几乎必然出现污染、误召回和维护失控。

### 11.3 风险三：为了“像人”而过度神经科学拟真

Butler 借鉴认知科学，是为了得到稳定的功能结构，不是为了把脑科学名词机械映射成类名。

### 11.4 风险四：长期记忆只能增长，不能退役

没有失效机制、冲突机制和版本机制，长期记忆最终会把系统拖入僵化和自相矛盾。

### 11.5 风险五：让自修改能力跑在治理能力前面

越智能的自我升级，越需要 Guardian、测试和回滚先到位。

## 十二、推荐的阶段路线图

### 阶段 A：把现有统一记忆做稳

目标：让 unified recent、L0/L1/L2、background context 装配更加稳定、可测、可解释。

建议动作：

1. 统一 recent event schema
2. 强化 salience / confidence 的使用位置
3. 建立 source_type / source_reason 规范
4. 增加长期记忆命中与写回日志

### 阶段 B：引入再巩固与冲突账本

目标：让系统能“改观点”而不是“叠观点”。

建议动作：

1. 引入 memory event log
2. 为 schema 增加 version / superseded_by / conflicts_with
3. 后台 round 支持 reconsolidation_review 模式
4. 引入待核实队列

### 阶段 C：引入程序性记忆

目标：让系统沉淀“怎么做”的稳定经验。

建议动作：

1. 建 procedure store
2. 从规则文档和历史成功案例抽取 procedure
3. 在 planner 中引入 procedure retrieval
4. 验证 procedure 是否真的提升成功率

### 阶段 D：形成完整自我系统与升级闭环

目标：让 Butler 真正成为可长期演化的主体系统。

建议动作：

1. 明确 self model 契约
2. 明确 Guardian/Governor 参与边界
3. 建立长期认知评测集
4. 把升级行为纳入统一 ledger

---

## 十三、最终原则：项目后续的总方向应该怎么把握

如果把本文档压缩成最少的话，只有下面九条：

1. **把 Butler 当长期主体系统来设计，而不是当会话机器人来修补。**
2. **把记忆当控制平面来设计，而不是当摘要附件来堆积。**
3. **优先建设分层与边界，再追求拟人化与复杂性。**
4. **让后台自动化负责后台巩固，而不只是后台执行。**
5. **让长期记忆可更新、可追溯、可退役。**
6. **显式补齐程序性记忆。**
7. **让 Soul、role、memory、task ledger 各守其边界。**
8. **让 Guardian 成为自修改和高风险动作的必要闸门。**
9. **任何“变聪明”的主张，都尽量拿长期评测来证明。**

只要这九条不偏，Butler 后续无论往更强 agent、更强长期陪伴、更强自治心跳还是更强自我维护方向发展，都不会偏离主航道。
