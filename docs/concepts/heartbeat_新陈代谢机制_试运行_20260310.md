# Heartbeat 新陈代谢机制（试运行版）

更新时间：2026-03-10

## 1. 这套机制解决什么问题

Butler 是持续演化的系统。代码、prompt、agent 规则、README、设计文档会不断变化，而旧说明并不会自动失效。

如果系统仍把旧文档当作高可信真源，就会出现：

1. 按过时路径执行
2. 沿用旧认知做错误决策
3. 文档与运行事实逐步分叉
4. 心跳在错误说明的引导下反复加深偏差

所以需要一个轻量“新陈代谢机制”：不做强制大清洗，但持续识别“系统近期变了什么”“哪些说明可能老了”“本轮决策该优先信什么”。

## 2. 当前试运行版的核心原则

### 2.1 信息源可信度层级

heartbeat 规划时默认采用以下优先级：

1. 当前轮输入、当前运行状态、正在执行的动作、最新文件变化
2. unified recent / beat recent
3. local memory 当前有效结论
4. README / docs / 说明文档

同一来源内部，越新的权重越高；旧文档若落后于最新运行信号，应视为“待复核参考”，而不是绝对真源。

### 2.2 不是自动判错，而是轻量提示复核

本机制当前不自动删除、不自动大规模改写文档，也不直接判定某文件“错误”。

它只做：

1. 识别近期运行信号
2. 标记疑似过时说明
3. 把这些结果注入 heartbeat planner
4. 让 planner 在没有更高优先级任务时，选择轻量核对、更新索引或归档

## 3. 已接入 heartbeat 的位置

当前改为 **prompt / rules first** 的轻量接入方式。

接入点不再是额外的扫描器和快照生成逻辑，而是两层约束：

1. `heartbeat-planner-agent` 的角色规则
2. `butler_main/butler_bot_agent/agents/heartbeat-planner-prompt.md` 里的规划原则（`butler_bot_code/prompts/heart_beat.md` 仅保留为 JSON 契约兼容模板）

因此 planner 会在每轮规划时主动遵守：

1. 当前运行事实优先于旧文档
2. 越新的信息权重越高
3. 旧说明应被复核，而不是被机械执行
4. 如无更高优先级任务，可顺手做轻量“代谢型维护”

## 4. 当前如何工作

当前版本不再依赖硬编码扫描器，而是让 planner 在已有上下文里主动做这三步判断：

1. 先看当前轮与最近上下文里是否已经出现“项目刚变过”的信号
2. 再看长期记忆和文档是否与这些信号冲突或滞后
3. 如果没有更高优先级任务，把“复核说明 / 更新索引 / 标记旧路径”作为低风险治理动作

也就是说，当前版本的“新陈代谢”更像行为原则，而不是额外子系统。

## 5. 当前落点

当前主要落点在规则层，而不是新增真源文件：

1. `butler_bot_agent/agents/heartbeat-planner-agent.md`
2. `butler_bot_agent/agents/heartbeat-planner-prompt.md`

这样做的目的，是先验证“规则驱动的新陈代谢意识”是否已经足够有用，避免过早把机制写死在代码里。

## 6. 当前行为边界

当前试运行版明确不做：

1. 自动删除文档
2. 自动归档大批文件
3. 自动覆盖长期记忆
4. 自动否决旧文档
5. 自动发起高风险自修改

当前只允许 heartbeat 在无更高优先级任务时做这些低风险动作：

1. 更新 README 索引
2. 补写“当前有效说明”
3. 标注某文档可能过时
4. 轻量归档已明显失效的说明

## 7. 为什么这叫“新陈代谢”

类比生命体，新陈代谢不是“只长新东西”，而是：

1. 持续摄入新信号
2. 让旧成分退役
3. 维持系统当前结构与现实一致

对应到 Butler：

1. 新代码、新 prompt、新规则是“新输入”
2. 旧说明、旧路径、旧认知是“待代谢对象”
3. heartbeat 是最合适的轻量后台代谢器

## 8. 下一步怎么继续增强

试运行版之后，建议按这个方向增强：

1. 先观察 prompt / rules 版是否已经能稳定触发轻量维护，而不是一上来写死扫描规则。
2. 若后续确实反复出现“旧说明误导执行”，再补最小代码支撑，例如仅记录“最近一次被验证的文档”或“待复核清单”。
3. 把 README / docs / Soul / role / long memory 的关系纳入统一版本图，而不是各自漂着。
4. 让 heartbeat 在完成轻量维护后，把结论回写到 recent 和 long memory，形成真正的代谢闭环。
5. 只有在 prompt / rules 明显不够时，再考虑引入更强的 Guardian 级治理器。

## 9. 现阶段一句话总结

这版新陈代谢机制的本质不是“自动清理项目”，而是：

**先把“当前运行事实优先于旧文档、越新的信息权重越高、旧说明需要复核”固定成 heartbeat 的系统级判断习惯；只有当这种习惯不够用时，才补最小代码。**

## 10. 2026-03-16 心跳治理小记

本轮 heartbeat 实际在推进的长期线主要集中在 Web→BrainStorm 流转标准化、科研“龙虾工程方法”在 Butler 侧的工程化迁移，以及“说明规范（实际执行 vs 只读思路）”的落地对齐。  
当前心跳在并行/串行上采用“小并行 + 串行收束”的节奏：本轮规划约 3–4 条分支，其中一条聚焦 self_mind、一道偏治理、一到两条绑定长期工程线。  
就“主动意识槽 + 自我升级槽 + 工作任务槽”的平衡来看，本轮略偏向工作任务与治理协同，主动意识保持在可感知但不过载的水平，自我升级以记录型、小步验证为主。
