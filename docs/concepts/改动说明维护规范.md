# 改动说明维护规范（butler_bot）

- 适用范围：`butler_bot_code/` 下所有代码、配置、流程变更。
- 目标：让每次改动都可追溯、可复盘、可回滚。

## 一、何时必须写改动说明

满足任一条件，必须新增一份说明文档：

1. 涉及行为变化：用户可见行为、消息内容、任务状态语义变化。
2. 涉及执行链变化：心跳、记忆、消息处理、重启机制。
3. 涉及数据结构变化：JSON schema、memory 字段、配置项。
4. 涉及并发或可靠性变化：并行策略、超时、重试、去重。
5. 涉及测试补充或回归修复。

## 二、文档命名规则

放在 `butler_bot_code/docs/`，命名采用：

`变更说明_YYYYMMDD_主题.md`

示例：`变更说明_20260308_心跳并行与可见性.md`

## 三、每份改动说明必须包含的章节

1. 基本信息：日期、范围、目标。
2. 背景：改什么、为什么改。
3. 功能变更：逐条列行为变化。
4. 关键实现点：核心方法/配置项。
5. 验证结果：执行了哪些测试、结果如何。
6. 已知边界：当前限制与风险。
7. 回滚点：最小可回退单元。

## 四、README 索引维护规则

每新增一份改动说明，必须同步更新 `butler_bot_code/README.md`：

1. 在 README 增加或更新“版本与改动说明”小节。
2. 链接最新说明文档。
3. 保留最多最近 3 次重要改动链接；更早内容可在该小节留“历史见 docs”。

## 五、代码与说明的一致性规则

1. 改动说明中提到的关键方法名必须可在代码中检索到。
2. 改动说明中提到的配置项必须在代码中有实际读取。
3. 改动说明中提到的测试命令必须可运行（至少在当前环境可用的测试命令）。
4. 若未执行测试，必须明确写“未执行”及原因。

## 六、后续改动执行清单（PR/提交前）

1. 写/更新改动说明（docs）。
2. 更新 README 入口链接。
3. 运行相关测试并记录结论。
4. 检查说明与代码是否一致。
5. 若涉及线上行为，补充“观察点”（日志关键字或快照字段）。

- 范围：<文件路径列表>
- 目标：<一句话>

## 背景
...

## 功能变更
1. ...

## 关键实现点
- ...

## 验证结果
```powershell
<测试命令>
```
结果：...

## 已知边界
- ...

## 回滚点
1. ...
```

## 八、执行责任

- 实施改动的人负责起草说明。
- 合并前最后修改代码的人负责核对 README 索引。
- 若由心跳/自动化任务改动代码，也必须按本规范补文档。
