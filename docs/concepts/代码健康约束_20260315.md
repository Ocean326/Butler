## 一、原则

核心目标不是把文件拆碎，而是让功能尽量独立、依赖方向稳定、修改影响面可预测。

所以 Butler 后续所有“拆分”都必须服从这三件事：

1. 功能边界更独立
2. 调用链更清楚
3. 代码健康可自动检查

## 二、当前健康约束

### 1. 依赖方向

- `services/` 不反向依赖 `memory_manager.py`
- `runtime/` 不反向依赖 `memory_manager.py`
- `execution/` 不反向依赖 `memory_manager.py`

原因：

`memory_manager.py` 是总编排类，不应该重新变成所有模块都反向依赖的“黑洞”。

### 2. 文件体量预算

当前已对部分关键文件加预算检查：

- `memory_manager.py`
- `services/self_mind_runtime_service.py`
- `services/self_mind_state_service.py`
- `services/self_mind_prompt_service.py`
- `services/message_delivery_service.py`
- `services/upgrade_governance_service.py`
- `runtime/cursor_runtime_support.py`

原因：

预算不是为了追求数字好看，而是防止新模块刚拆出去又长成第二个大文件。

### 3. 健康检查自动化

当前代码健康检查已进入默认测试约束：

- `butler_main/butler_bot_code/tests/test_code_health.py`

这意味着后续如果有人把运行时层、服务层再次反向绑回 `memory_manager.py`，或者让新拆模块失控膨胀，默认回归就会直接报警。

## 三、当前已经落实的独立性改善

### 1. self_mind

已拆出：

- `self_mind_runtime_service`
- `self_mind_state_service`
- `self_mind_prompt_service`

### 2. 升级治理

已拆出：

- `upgrade_governance_service`

### 3. 发送链路

已拆出：

- `message_delivery_service`

### 4. Cursor 运行时支持

已拆出：

- `runtime/cursor_runtime_support.py`

它的目的就是切断 `runtime/cli_runtime.py -> memory_manager.py` 的反向依赖。

## 四、后续维护要求

1. 新功能优先落独立服务，不先塞进 `memory_manager.py`。
2. 新服务默认只处理一类职责，不承担额外编排。
3. 如果发现反向依赖，优先抽支持模块。
4. 每次重构后必须同步更新健康测试，不允许“代码变了，健康检查还停在旧结构”。
