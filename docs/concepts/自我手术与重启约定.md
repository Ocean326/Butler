> 状态说明：旧“后台 sidecar -> guardian -> 自动重启标记”协议已经退役。当前现行约定围绕 `chat core + orchestrator` 主链，通过 `manager.ps1` 与 `watch_stack.ps1` 做重启和验证。

## 1. 何时需要重启

- 修改了 `butler_main/chat/`、`butler_main/orchestrator/`、`butler_main/butler_bot_code/` 下会在进程启动时加载的 Python 代码。
- 修改了 `butler_main/butler_bot_code/configs/`、`registry.json`、启动参数或渠道 binding 等启动期配置。
- 修改了只在启动时装载的 prompt/bootstrap 资产。

通常不需要重启的情况：

- 只改文档、daily-upgrade、research 说明材料。
- 只改 recent memory / run data / 工作区产物。
- 只改可在下轮读取时自然生效的非启动期文本资产。

## 2. 当前重启方式

1. 先看状态：
   - `butler_main/butler_bot_code/manager.ps1 status`
2. 再执行重启：
   - `butler_main/butler_bot_code/manager.ps1 restart butler_bot`
3. 重启后观察：
   - `butler_main/butler_bot_code/watch_stack.ps1`
   - `orchestrator` 的 query / observe 输出
   - `chat` 最近一轮 recent memory 与渠道健康状态

## 3. 当前验证顺序

1. 确认 `chat core` 已重新拉起，主前台入口可响应。
2. 确认 `orchestrator` 查询口可返回 mission / session / 运行态。
3. 确认 recent memory 仍按 `session_scope_id` 正常隔离。
4. 如改动涉及后台流程，再额外确认 background automation 仍能被 `orchestrator` 正常推进。

## 4. 已废弃的旧协议

以下内容保留为历史概念，不再作为现行操作约定：

- `工作区/restart_request.json`
- `run/restart_requested.flag`
- 旧 sidecar runner 入口
- `restart_guardian_agent.py`
- 旧后台 watchdog
- “心跳结束后自触发重启”的旧链路

如果需要追溯这些旧协议，请把它们视为历史后台自动化时代的运维约定，而不是当前 Butler 的现行入口。

## 5. 当前职责边界

- `chat` 负责前台对话与会话级 recent memory。
- `orchestrator` 负责后台推进、控制面与观测面。
- `runtime_os / agents_os` 负责执行 substrate。
- `manager.ps1` 与 `watch_stack.ps1` 负责进程操作与人工观察，不再依赖独立 guardian/后台 sidecar 协议。
