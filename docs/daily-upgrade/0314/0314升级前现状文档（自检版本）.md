## 1. Talk 放入 / 取消心跳任务的协议与真源

### 1.1 任务看板真源

- **统一入口说明**：`local_memory/heartbeat_tasks.md`  
  - 用途：heartbeat planner 的**统一任务看板读口**。  
  - 原则：**任务意图先汇入分类看板，再同步到执行账本**；`task_ledger.json` 负责执行状态与回执，不承担所有任务入口。  
- **分类看板**（均为 Markdown，条目带 `task_id/status/source/priority/category`）：  
  - 工作任务：`heartbeat_tasks/01_工作任务.md`（来自**对话**明确指派、外部工作、需稳定收口）  
  - 主动想干：`heartbeat_tasks/02_主动想干.md`  
  - 定时 / 长期 / 整理清洁 / 闲时无聊：`03_定时.md`～`06_闲时无聊.md`  
- **变更日志**：`heartbeat_tasks/task_change_log.jsonl`  

### 1.2 「放入」心跳任务（Talk → 看板）

- **触发**：用户在 Talk（飞书对话）中说「放进心跳」「交给心跳」「排期吧」或明确指定「交给心跳任务，紧急程度：xx」等。  
- **执行方**：**feishu-workstation-agent**（当前对话侧）。  
- **动作**：  
  1. 判断任务类型，选择对应分类看板（多为 `01_工作任务.md`）。  
  2. 在看板**末尾**追加一条任务条目，格式示例：  
     `- [task_id=xxx][status=pending][source=conversation][priority=medium][category=work] 描述`  
  3. 若为「排期汇总」类，可写一条汇总型任务（如 `heartbeat-schedule-YYYYMMDD`），把多条待办收口为显式排期入口，供 planner 后续选取。  
- **真源/约定**：  
  - 调度与分类约定见 `heartbeat_tasks.md`「调度协议」「分类目录」；  
  - 飞书侧「用户说放进心跳就写入看板」为**约定行为**，未单独成篇 prompt，依赖 feishu-workstation-agent 的通用任务型协议与 `docs/TASK_ROLE_PROTOCOL.md`（若存在）。  

### 1.3 「取消」心跳任务

- **触发**：用户明确说「取消 / 删除 / 不要了」某任务（如指定任务内容或说「把 xx 任务删掉」）。  
- **执行方**：feishu-workstation-agent 或心跳侧在维护时。  
- **动作**：  
  - 在看板中找到对应条目，将 `status` 改为 `cancelled`（或从条目中移除）；  
  - 在 `task_change_log.jsonl` 中记一条变更（可选但建议），便于审计。  
- **说明**：当前「取消」的显式协议分散在对话习惯与任务看板格式中，未单独集中成「取消协议」文档。  

### 1.4 小结（放入/取消）

| 项       | 真源 / 位置 |
|----------|-------------|
| 看板定义 | `heartbeat_tasks.md` + `heartbeat_tasks/01_工作任务.md` 等 |
| 放入     | 对话中说「放进心跳」→ feishu-workstation-agent 在对应看板末尾追加条目 |
| 取消     | 用户说取消 → 改 status 为 cancelled 或移除 + 可选记入 task_change_log |
| 集中协议 | 未单独「Talk 放入/取消心跳任务」prompt 文档，依赖角色与看板约定 |

## 2. Planner 获取任务 → 分发 → 监督 → 验证 → 汇报 → 交付

### 2.1 整体链路（现状）

1. **获取任务**  
   - Planner 读取：`heartbeat_tasks/01_工作任务.md`～`06_闲时无聊.md` + `task_ledger.json`（及生命周期目录 `agents/state/task_workspaces/未进行|进行中|已完成`）。  
   - 综合：时间、DDL/定时、用户是否活跃、工作/成长权重等做动态调度（见 `heartbeat_tasks.md` 调度协议）。  

2. **分发**  
   - Planner 输出本轮的 `parallel_ready` / `serial_queue`（branch 列表）。  
   - 每个 branch 交给 **heartbeat-executor-agent**（或专长 sub-agent）执行；executor 的职责定义见 `sub-agents/heartbeat-executor-agent.md`。  

3. **监督与验证**  
   - **执行侧**：executor 按「目标、输出路径、完成标准」执行，交付前自检：`result / evidence / unresolved / next_step`。  
   - **任务型协议**：遵循 `docs/TASK_ROLE_PROTOCOL.md`（诊断→换路/修正→复试→仍失败再上抛）。  
   - 监督与验证目前主要依赖 executor 的自我验收与回执格式，未单独成「监督 agent」文档。  

4. **向 User 汇报进度**  
   - **机制**：`plan.tell_user` → 通过 **proactive-talk** skill 发到**对话窗（Talk）**，不是心跳窗。  
   - **约定**：用户期望「过程当中主动向 user 汇报进度」；skill 说明见 `skills/proactive-talk/SKILL.md`（阶段完成、阻塞提醒、下一步建议、小结回执）。  
   - **频控**：`heartbeat.proactive_talk` 在 `butler_bot_code/configs/butler_bot.json` 中配置：`min_interval_seconds`（当前 5400）、`min_heartbeat_runs_since_last`（3）、`min_completed_branches`（2）、`max_chars`（220），避免刷屏；以实际 config 为准。  

5. **交付任务完成情况**  
   - 执行账本（task_ledger）更新任务状态；  
   - 任务同步到生命周期目录（未进行 / 进行中 / 已完成），**完成时**生成最终汇总文档；  
   - 上述约定见 `heartbeat_tasks.md`「原则」「生命周期目录」。  

### 2.2 真源汇总

| 环节     | 真源 / 位置 |
|----------|-------------|
| 获取任务 | `heartbeat_tasks.md` + 各分类看板 + `task_ledger.json` + task_workspaces |
| 分发     | Planner 输出 branch → heartbeat-executor-agent / sub-agents |
| 监督验证 | executor 自验收 + `TASK_ROLE_PROTOCOL.md`，无独立监督 agent 文档 |
| 汇报     | `plan.tell_user` + proactive-talk skill → Talk 对话窗 |
| 交付     | task_ledger 更新 + 生命周期目录 + 完成时汇总文档 |

## 3. Recent 过时 + Summary + 归入 Local Memory 的机制

### 3.1 Recent 的「过时」与压缩

- **触发**：`memory_manager._compact_recent_entries_if_needed`  
  - 条件：recent 条数 **> 15** 或 总字符 **> 15k**（与 `recent_memory_compact_policy.md` 一致）。  
- **行为**：旧条被挤出 recent；被挤出的内容可进入摘要/归档（如 `recent_archive.md`），部分条目不在此步写入 local_memory（见下）。  

### 3.2 Summary

- **每轮**：`_summarize_turn_to_recent` 得到本条摘要与 `long_term_candidate`（含 `should_write`、`summary`）。  
- **压缩时**：被挤出的旧条可保留摘要快照；`recent_archive.md` 只存摘要快照，**不**记录「哪条写入了 local_memory」。  

### 3.3 归入 Local Memory 的三条路径（真源：L2_detail）

1. **每轮落盘时（即时沉淀）**  
   - 条件：`long_term_candidate.should_write === true` + `summary` 非空 + governor 放行。  
   - `should_write` 来源：模型提炼 或 启发式（对话含 LONG_TERM_HINTS 关键词）。  
   - LONG_TERM_HINTS：`"记住","以后","默认","偏好","必须","统一","固定","长期","沿用","always","default","remember","preference","must"` 等。  

2. **should_write 补偿迁移（防漏）**  
   - 每轮收尾 + recent 维护时，扫描 recent 中已标 `should_write` 但未提升的条目，按额度（每轮 2 条，维护 3 条）写入 local_memory。  

3. **压缩时（反思沉淀）**  
   - 仅针对「被挤出且时间 > 1 天」的条目，用关键词筛（反思、教训、下次、避免、必须、默认、偏好、规则、约束），最多 2 条写入 local_memory。  

### 3.4 写入流水（可追溯）

- **位置**：`local_memory/local_memory_write_journal.jsonl`  
- **来源类型**：per-turn / recent-sweep / recent-compact / manual  
- **用途**：回答「哪次写入了 local、写到了哪、来源是哪条 recent」。  

### 3.5 摘要入口

- **L1**：`local_memory/L1_summaries/recent到local_memory迁移与写入机制.md`  
- **L2 详情**：`local_memory/L2_details/recent到local_memory迁移与写入机制_detail.md`

## 4. 其他可能遗漏的链路 / 机制

- **recent_memory 使用规则**（飞书/记忆）：  
  - 用户**未**说「全新任务/全新情景」→ **优先沿用 recent_memory**；  
  - 用户**明确新任务**→ **忽略 recent_memory**。  
  - 真源：`local_memory/飞书与记忆约定.md`「使用规则（recent_memory）」。  

- **路径回退**：相对路径在当前仓库找不到时，再回退到用户明确提供的其他工作目录搜索；优先相对路径，不默认写死桌面绝对路径。  

- **任务型 Role 通用协议**：执行型行为遵循 `docs/TASK_ROLE_PROTOCOL.md`；feishu-workstation-agent、heartbeat-executor 均引用，诊断→换路→复试→上抛。  

- **Guardian 重启与断片**：对话中触发 guardian 重启可能导致当轮写回/记忆未完成；重启安全点与断片恢复机制有讨论与任务条目，是否已完全落代码需查 body 侧。  

- **心跳与 self_mind**：近期已从心跳体系中去掉「self_mind 作为一类人物/并行来源」的配置（heartbeat-planner、heartbeat_tasks、02_主动想干 等）；任务条目的 `source=self_mind` 仍可存在于看板，仅不再作为调度上的独立「一路」。  

- **tell_user 与对话窗**：心跳窗与 Talk 对话窗为两个不同 chat；主动找用户说话应走 tell_user → 对话窗，见「飞书与记忆约定」及 proactive-talk skill。  

- **每日聊天备份**：飞书聊天记录每日备份到 `工作区/with_user/feishu_chat_history`，约定见 local_memory 相关摘要。  

- **照镜子（行为记录复盘）**：拉取 Talk/心跳会话 → 自检表达质量 → 结论写 self_mind 或 local_memory；见长期记忆命中「照镜子」摘要。
