# 0316 现状分析与升级计划

更新时间：2026-03-16 01:15

## 一、本文档用途

本文档聚焦四个当前最值得继续收口的问题：

1. `memory_manager.py` 仍然过重
2. `self_mind` 运行态噪音过大
3. self_mind / heartbeat 相关飞书卡片发送链仍存在历史异常风险
4. planner 主动汇报链路虽然已接上，但稳定性和可观测性仍不够

本文档不是泛泛讨论，而是回答三件事：

1. 这四块当前已经做到什么程度
2. 还卡在哪些结构性问题上
3. 0316 之后应该怎么收口成稳定系统

## 二、当前现状判断

### 1. `memory_manager.py` 已开始拆分，但仍是事实上的“总黑洞”

0315 已经把第一层职责外提到服务层：

- `self_mind_runtime_service`
- `self_mind_state_service`
- `self_mind_prompt_service`
- `self_mind_cycle_service`
- `message_delivery_service`
- `delivery_flow_service`
- `approval_request_service`
- `upgrade_governance_service`

这说明拆分方向是对的，已经不是最早那种所有实现都平铺在一个大文件里。

但当前真实问题是：

1. `memory_manager.py` 仍然同时掌握运行时生命周期、heartbeat 规划、recent/local memory 维护、self_mind bridge、审批读写、发送兜底、状态文件、日志与 prompt 拼装入口。
2. 服务层虽然存在，但很多服务还是“由 `memory_manager.py` 驱动 + 反向调用大量私有方法”，还没有完全形成稳定、独立的服务边界。
3. planner prompt 模板、心跳调度默认规则、tell_user 续接协议、self_mind 周期协作规则，仍有相当一部分以大字符串或私有 helper 的形式挂在主类上。

所以当前准确判断不是“已经拆完”，而是：

`memory_manager.py` 已经从单体实现转向“总编排 + 半委托”状态，但还没有真正收口成轻量 orchestrator。

### 2. `self_mind` 已有真实执行能力，但当前噪音仍然偏大

当前 `self_mind` 不再只是一个空壳：

- 已有 cycle prompt
- 已有 `talk / heartbeat / hold` 决策
- 已有 direct talk
- 已有 bridge 更新
- 已有 handoff receipt
- 已有同焦点连续多轮后的 breakout 机制

但当前噪音主要来自四类问题：

1. **状态、原始流、可读视图混住在同一目录**
   - `current_context.md`
   - `raw_thoughts.json`
   - `thought_reviews.json`
   - `perception_snapshot.md`
   - `behavior_mirror.md`
   - `daily/`
   - `cognition/`

   它们同时承担“运行态真源、调试中间产物、用户可读摘要、长期认知沉淀”几种角色，导致信号混杂。

2. **self_mind 仍然容易把观察、续思、执行候选混成一股流**
   - 有些内容应留在认知层
   - 有些内容应转成 heartbeat/task ledger
   - 有些内容只该进入短期运行态日志

   但当前实际落盘仍偏“先都记下来，再局部回收”，噪音量会持续上升。

3. **`current_context.md` 仍承担过多汇总责任**
   - 它既像 prompt 输入
   - 又像运行态摘要
   - 又像近期自我叙述

   结果就是任何历史噪音、旧路径、旧焦点，都容易继续留在当前上下文里。

4. **debug receipt / direct talk / heartbeat handoff 同时存在时，外显信号可能过密**
   - self_mind 可直接 talk
   - 也可交给 heartbeat
   - 还能发 handoff receipt 到 heartbeat 窗口

   若没有统一的频控和通道优先级，用户看到的信息就容易碎。

所以当前对 self_mind 的判断应是：

它已经具备真实协作能力，但仍未完成“认知层 / 执行层 / 观测层”的降噪分流。

### 3. 飞书卡片异常风险没有完全消失，只是已具备基础兜底

这块不能简单说“已修好”或“还没做”，真实状态是：

#### 已有的稳定基础

1. `MessageDeliveryService` 已经建立统一的私聊发送链：
   - 优先 `interactive`
   - 失败后回退 `post`
   - 再失败回退 `text`
2. 当前测试已经覆盖：
   - schema `2.0`
   - body markdown 结构
   - `interactive -> post` 的失败回退
3. self_mind / heartbeat 的私聊发送已经开始走服务层，而不是散落在各处重复实现。

#### 仍然存在的风险点

1. `agent.py` 内仍保留了一套单独的 interactive card / card action 逻辑。
2. 当前“对话回复卡片链路”和“私聊发送服务链路”并没有完全统一到同一个发送真源。
3. 这意味着：
   - 卡片 schema 调整可能两边不同步
   - card action 回调与普通发送链的异常口径不一致
   - 某一边修复后，另一边仍可能保留旧问题

所以当前最准确的判断是：

飞书卡片异常已经从“没有兜底”推进到“有 fallback、有测试”，但还没有完成单一发送真源统一，因此仍有历史漂移风险。

### 4. planner 主动汇报链路已经接上，但仍缺闭环验证

当前 planner 主动汇报并不是空想法，已经有明确协议：

- `user_message`
  - 发往 heartbeat 窗口，说明本轮判断和执行意图
- `tell_user_candidate`
  - 只是候选，不等于最终直接发给用户的话
- `tell_user_reason / tell_user_type / tell_user_priority`
  - 作为下一轮继续心理活动与选择是否主动开口的依据

并且已有测试覆盖：

1. `user_message` 在 planner / fallback / status 场景可用
2. `tell_user_candidate` 不再走旧 talk_intent 主路径
3. branch 的 `heartbeat_tell_user_markdown` 能汇总为可同步内容
4. 并行分支、defer 信息、阶段性 receipt 已能反映到心跳消息

但当前还缺三类稳定性：

1. **闭环可观测性不足**
   - 规划器写了 `tell_user_candidate`
   - subconscious 生成了 tell_user intention
   - 下一轮 reflective continue 决定是否开口
   - 最终通过发送链发到用户

   这条链虽然存在，但当前缺少单一 run_id / audit trace 来把整条链串起来。

2. **协议还分散在多个层**
   - planner schema 在 `heartbeat_orchestration.py`
   - tell_user 意图整理在 `subconscious_service.py`
   - 继续心理活动与发送逻辑还在 `memory_manager.py`

   这会让“主动汇报到底由谁决定”继续变得模糊。

3. **缺少更接近真实链路的回归测试**
   - 当前多是单层或半集成测试
   - 还缺“planner -> branch results -> intention -> next-round send”这种完整链路测试
   - 也缺 heartbeat/self_mind/card 三条发送通道的统一行为测试

所以这块当前判断应是：

planner 主动汇报流程已经从协议层进入运行层，但还没有达到“可追踪、可调试、可稳定复现”的程度。

## 三、0316 核心问题归因

归根到底，这四个问题并不是四个孤立 bug，而是一个共同症状：

**运行编排、认知编排、发送编排还没有完全收口到单一真源。**

具体表现为：

1. `memory_manager.py` 仍然知道太多细节
2. self_mind 的运行态和认知态没有彻底分层
3. 飞书发送链仍然存在“双实现”
4. planner 主动汇报虽然有协议，但跨层链路还太散

所以 0316 的目标不应该是“继续补几个点状修复”，而应该是：

把这四个问题收敛成一套统一的编排与发送真源治理。

## 四、0316 升级目标

### 目标 A：把 `memory_manager.py` 压缩为真正的总编排层

要求不是把大文件拆碎，而是把以下职责继续外提成稳定服务：

1. heartbeat 规划与结果应用编排
2. self_mind 认知流与 bridge 流
3. 主动汇报 / tell_user 续接流
4. 发送链与发送审计

最终目标：

- `memory_manager.py` 只保留生命周期编排、服务装配、跨层协调
- 不再继续承担大段具体实现

### 目标 B：把 self_mind 从“高噪运行流”压成“低噪认知层”

要求：

1. 认知真源、运行态日志、用户可读视图分目录
2. `current_context.md` 不再承担所有原始噪音汇总
3. self_mind 只保留真正值得继续的焦点与认知信号
4. 可执行事项优先交给 heartbeat / task ledger， self_mind 可读最近事项，但不再有任务分给self-mind

### 目标 C：统一飞书发送真源

要求：

1. 心跳 / self_mind / 对话回复尽量走同一发送服务协议
2. interactive/post/text fallback 只维护一套
3. card action 与普通发送至少共享同一份 card schema 构造逻辑
4. 发送结果要带结构化审计，而不只是 print

### 目标 D：把 planner 主动汇报链路做成完整闭环

要求：

1. planner 只负责产出候选意图与心跳窗口说明
2. 意图沉淀与继续心理活动有单独真源服务
3. 最终对用户开口的动作有单独审计链
4. 回归测试能覆盖整条链，而不是只覆盖其中一个片段

## 五、执行计划

### P0. 先统一“发送与主动汇报”真源

这是最优先的，因为它同时影响飞书卡片异常、planner 主动汇报和 self_mind 外显噪音。

#### 本轮要做

1. 新建单一的 `tell_user / message delivery` 编排层
   - planner 只产出候选
   - reflective continue 只负责二次判断
   - 最终发送统一走 message delivery service

2. 收口 card schema 真源
   - `agent.py` 内的卡片构造与 `MessageDeliveryService` 对齐
   - 不再允许两个发送入口各自维护不同 interactive body

3. 增加发送审计
   - 至少记录：来源、channel、msg_type、fallback 路径、最终状态、失败原因

#### 验收口径

1. heartbeat / self_mind / 对话回复的 interactive schema 一致
2. 同类异常不会出现“一条链能回退，另一条链直接炸掉”
3. planner 主动汇报链每一步都能追到

### P1. 继续拆 `memory_manager.py` 的四块重职责

优先拆下面四块，而不是随机继续挪函数：

1. `HeartbeatPlanningFacade`
   - planner prompt
   - plan normalize
   - fallback
   - branch group contract

2. `TellUserFlowService`
   - candidate/intention 读写
   - reflective continue
   - cooldown / dedupe / send trigger

3. `SelfMindCognitionService`
   - context refresh
   - cognition routing
   - bridge signal normalize
   - pending item compaction

4. `RuntimeStateAuditService`
   - heartbeat/self_mind/main state 文件
   - send audit
   - lifecycle audit

#### 验收口径

1. `memory_manager.py` 不再继续增长
2. 新服务不反向依赖 `memory_manager.py`
3. 新的服务职责名称和文件名能与实际边界对应

### P2. 对 self_mind 做一次结构化降噪

建议把 self_mind 目录的角色分成三层：

1. `state/`
   - 当前运行真源
   - bridge/pending/streak 等

2. `streams/`
   - 原始 thought/review/event 日志
   - 只给系统调试与回溯用

3. `views/`
   - `current_context.md`
   - `perception_snapshot.md`
   - 对 prompt 可读、对用户可读的轻量视图

同时加三条规则：

1. 原始流不直接进 `current_context.md`
2. 同焦点重复内容要做去重或压缩
3. 超时未续接的 pending item 自动降级或归档

#### 验收口径

1. `current_context.md` 长度和主题漂移可控
2. self_mind 对 heartbeat 的委托不再制造大量重复噪音
3. 用户可见的主动发话频率明显更稳定

### P3. 给 planner 主动汇报做完整链路测试

新增测试重点：

1. `planner -> tell_user_candidate -> subconscious intention -> next-round continue -> send`
2. `self_mind direct talk / heartbeat tell_user / heartbeat user_message` 三条链互不混线
3. interactive 失败后的 fallback 在三条链上都一致
4. debug receipt 开关不会污染正常用户通道

#### 验收口径

1. 主动汇报链可重复回归
2. 能明确知道“这句话是谁决定的、为什么发、从哪条链发出的”
3. 不再依赖人工盯日志判断是否正常

## 六、建议的实施顺序

1. 先做 `P0`，统一发送与 tell_user 真源
2. 再做 `P1`，继续拆 `memory_manager.py`
3. 再做 `P2`，对 self_mind 做目录与数据流降噪
4. 最后做 `P3`，用回归测试把链路钉住

原因很简单：

如果先拆 self_mind、但发送链还没统一，噪音只会换一种形式继续出现；  
如果先拆 `memory_manager.py`、但 tell_user 真源还散着，后面还得返工一次。  

所以 0316 的正确顺序是：

**先统一外显链，再拆内部链，最后收口认知噪音。**

## 七、0316 一句话结论

Butler 当前已经从“完全混在一起”推进到“开始分层”，但还没有到稳定工程系统的程度。

0316 最应该做的不是继续加新能力，而是把：

- `memory_manager` 的重职责
- self_mind 的高噪运行流
- 飞书卡片的双实现风险
- planner 主动汇报的分散链路

统一收口成一套可追踪、可验证、可长期维护的编排真源。
