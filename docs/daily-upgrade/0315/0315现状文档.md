## 三、当前后台真实结构

### 1. 主对话进程

- 入口：`butler_main/butler_bot_code/butler_bot/butler_bot.py`
- 职责：接收用户消息、处理普通对话、处理显式后台控制命令、在回复后回写记忆

### 2. heartbeat

- 由 `MemoryManager.start_background_services()` 在主进程启动时直接拉起
- 独立 sidecar 进程运行
- 主进程内置 watchdog 看护

### 3. self_mind

- 由主进程内线程运行
- 当前支持 `talk / heartbeat / hold` 这一轮决策结构
- 是否进入 body loop 由配置控制

### 4. 当前后台控制方式

当前 talk 已支持直接控制：

- `重启后台`
- `重启心跳`
- `重启意识循环`
- `重启 self_mind`
- `停止心跳`

所以当前的真实判断是：

Butler 的后台控制真入口已经是主对话进程与其生命周期控制，而不是外部 guardian 守护链。

## 五、这轮测试中暴露并已处理的问题

### 1. 测试区本身曾经混入手工脚本

最开始 `pytest` 收集阶段直接被 `run_decide_e2e_test.py` 这类脚本打断，因为它依赖历史目录结构，不属于标准自动回归测试。

当前已处理为：

- 手工脚本移入 `tests/manual/`
- 默认回归只收集 `test_*.py`

### 2. 当前环境的系统临时目录不可继续写入

本轮确认了一个环境级问题：

- Python 的 `tempfile.TemporaryDirectory()` 在系统 Temp 下创建出的目录，在当前机器上会出现后续继续写入被拒绝的情况

当前已通过 `tests/conftest.py` 统一替换到工作区内的稳定临时目录，避免这类环境噪音持续污染测试结果。

### 3. 一批测试断言已经过时

这轮还修正了一批“测试本身还停留在旧结构假设”的问题，例如：

- 仍假定 `butler_main` 是项目根
- 仍假定 guardian 是现役组件
- 仍假定旧私有 API 存在
- 仍假定 `_run_model_subprocess` 走旧的 `subprocess.run`

这些已通过“更新断言 / 移出默认套件 / 改成当前真源口径”处理掉。
