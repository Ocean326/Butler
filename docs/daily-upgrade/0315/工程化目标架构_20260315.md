# Butler 工程化目标架构

更新时间：2026-03-15

## 一、目标

从 0315 起，Butler 不再按“能跑就继续堆”推进，而按长期维护系统推进。

目标不是一次性重写，而是把现有真实主链路收口成：

1. 架构边界清晰
2. 协议真源单一
3. 状态真源稳定
4. 重构可回归
5. 长期升级不再反复打散系统

## 二、目标架构分层

### 1. 主机制层

位置：`butler_main/butler_bot_code/butler_bot/` 根目录

只保留主链路入口与总编排：

- `agent.py`
- `butler_bot.py`
- `memory_manager.py`
- `heartbeat_orchestration.py`
- `heartbeat_service_runner.py`
- `governor.py`
- `butler_paths.py`
- `memory_cli.py`

要求：

- 根目录不继续堆业务碎模块
- 这里只负责入口、总编排、生命周期和跨层协调
- 复杂逻辑必须继续外提到职责子目录

### 2. 服务层

位置：`services/`

职责：

- 任务账本
- 验收回执
- 记忆读写
- 提示词装配
- 请求分析
- 潜意识/总结等可独立服务

要求：

- 以纯服务能力为主
- 少依赖入口层全局状态
- 尽量具备独立测试入口

### 3. 运行时层

位置：`runtime/`

职责：

- CLI 执行
- Runtime 路由
- 运行日志

要求：

- 把“选哪个执行器”和“怎么调用执行器”收口
- 高权限执行参数必须可治理、可审计

### 4. 协作执行层

位置：`execution/`

职责：

- sub-agent / team 执行
- 团队协作的串行/并行控制

要求：

- 协作失败不能拖垮全局主链路
- 并行异常必须结构化回执

### 5. 注册与资产层

位置：`registry/`

职责：

- skills
- sub-agents
- teams
- 公共能力目录

要求：

- 只做注册、发现、渲染
- 不承担运行时执行副作用

### 6. 工程规范层

位置：`standards/`

职责：

- 架构清单
- 长期有效协议
- 规范性 prompt 真源

这是 0315 之后必须新增的一层。它的意义是：

以后 Butler 的任务协作、自我更新、自我认识协议，不再散落在 `memory_manager.py`、`agent.py`、`heartbeat_orchestration.py` 的硬编码字符串里，而是有单一真源。

### 7. 过时归档层

位置：`obsolete/`、仓库根 `过时/`

职责：

- 隔离已经退出主链路的旧代码
- 保留参考价值但不再参与当前系统认知

要求：

- 旧链路不再被 README、测试默认套件、观察脚本视为现役组件

## 三、长期稳定的三套协议

### 1. 任务协作协议

目标：

- 对话、heartbeat、self_mind 不再维护三套任务池
- `task_ledger.json` 成为执行状态真源
- 任务工作区承担可读投影、过程记录和最终报告

最低要求：

- 新任务必须有目标、边界、完成判据、阻塞
- 执行结果必须能回写验收摘要和过程角色

### 2. 自我更新协作协议

目标：

- 让 Butler 可以持续升级，但不以牺牲稳定性为代价

最低要求：

- 升级先出方案，再审批，再改动，再验证
- 身体层高风险改动必须经过统一 upgrade / governor 闸门
- 升级必须留下回执、风险与回滚点

### 3. 自我认识协作协议

目标：

- 让 self_mind 成为长期稳定的认知层，而不是漂移的第二调度中心

最低要求：

- self_mind 负责认知和续思，不直接替代执行系统
- 认知线索、执行任务、观察性线索必须分流
- 可执行事项交接到 heartbeat / task ledger

## 四、长期架构约束

1. 根目录仅保留主机制代码。
2. 协议文本必须有单一真源。
3. 高风险升级不得绕过 governor。
4. `task_ledger` 是执行状态真源。
5. self_mind 不维护平行执行系统。
6. 每次重构必须同步更新测试与文档。

## 五、0315 之后的主改造顺序

1. 先建立标准层与协议真源。
2. 再把散落在 prompt 里的协议逐步收口到标准层。
3. 再拆 `memory_manager.py` 的重职责区块。
4. 再收观测、CI、依赖锁、跨平台启动等工程化外围。

## 六、当前结论

Butler 现阶段不该再继续靠“多加几段 prompt”和“在大文件里补几百行函数”推进。

0315 后的正确方向是：

先把架构与协议变成可维护真源，再在这个骨架上持续拆重、补稳、补测试。
