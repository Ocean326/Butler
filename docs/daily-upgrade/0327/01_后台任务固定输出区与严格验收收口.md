## 一句话裁决

本轮已经把“后台任务假完成”和“产物溯源困难”这两件事从排查结论落成代码事实：

`negotiation frontdoor -> campaign create -> bundle root -> codex implement evidence -> reviewer blockers -> query/feedback`

这条链现在能够明确回答三件事：

1. 任务产物应该去哪找
2. 任务为什么还没算 done
3. 哪些看起来像结果的 workspace 内容其实不是这次任务的 deliverable

## 故障复盘

### 1. 真故障不在“后台没启动”

本轮排查确认，之前看到的一个典型 `campaign` 样本会按既定 deterministic 流程稳定跑完；真正的问题是：

1. `implementation_report` 默认是 placeholder
2. reviewer 在 negotiation 创建的长期任务上没有把 placeholder / missing deliverable refs / pending correctness checks 当作硬阻塞
3. 用户又会在 workspace 里看到别的历史任务留下的现成产物，于是误判“后台任务已经产生了结果”

所以这轮不能只补前门，而必须把 `implement -> reviewer -> query -> feedback` 一起收口。

### 2. 前门弱化的正确边界

这轮没有把“前门弱化”解释成“前门不设门禁”。当前正式口径是：

1. frontdoor 可以允许 exploratory 启动，避免一开始就过度阻塞
2. 但 chat / 后台任务必须先形成边界澄清和初始计划
3. 最终 done 与否由 reviewer 和 acceptance evidence 决定，不由“任务已经启动”决定

## 当前实现

### 1. 固定输出区

后台任务 bundle 现在统一由 `background_task_bundle.py` 负责准备，输出区固定为：

- `工作区/Butler/deliveries/background_tasks/`

命名规则固定为：

- `YYYYMMDD_HHMMSS_<topic_slug>_<primary_id>`

当前 bundle 结构最小集合为：

- `briefs/`
- `artifacts/`
- `deliveries/`
- `manifest.json`
- `progress.md`

这条规则的作用不是美化目录，而是让所有后续排查都能直接按时间、主题和主 id 反查。

### 2. negotiation campaign 的严格验收

对 `created_from=campaign_negotiation` 的 `/delivery` 和 `/research`，当前口径是：

1. 默认 `campaign_runtime.mode=codex`
2. 默认 `strict_acceptance_required=true`
3. reviewer 在 acceptance 时必须检查：
   - `placeholder_implement_artifact`
   - `missing_deliverable_refs`
   - `pending_correctness_checks`

只要这三类 blocker 之一存在，就不会把任务误记为 done。

### 3. implement 证据与 deliverable 升格

`domains/campaign/codex_runtime.py` 现在不再只挂一个旁路 `codex_*_report`。当前实现会直接把主相位产物升级为可验收 evidence：

1. discover 写入 `briefs/discover_plan.md`
2. implement 写入 `deliveries/implement_iteration_XX.md`
3. iterate 写入 `artifacts/iterate_iteration_XX.md`
4. `implementation_report` 会在 codex 成功时补：
   - `placeholder=false`
   - `deliverable_refs=[...]`
   - `summary / execution_summary / next_action`

这样 reviewer 看的是“主 implement 证据是否真实”，而不是“旁边有没有一份 codex 附件”。

### 4. 状态与反馈投影

这轮新增的状态投影字段包括：

- `bundle_root`
- `bundle_manifest`
- `runtime_mode`
- `pending_checks`
- `resolved_checks`
- `waived_checks`
- `latest_acceptance_decision`
- `latest_acceptance_blockers`
- `not_done_reason`

它们已经统一进入：

1. `OrchestratorQueryService`
2. `chat/task_query.py`
3. `feedback_notifier.py`

因此现在查任务状态时，系统不仅会告诉你“任务在 implement / iterate”，还会明确告诉你“为什么没有 done”以及“真实输出区在哪”。

### 5. Chat 侧后台任务入口临时熔断

`2026-03-27 23:17:03` 的线上日志已经出现一条直接反证：用户明确要求“把现在 chat 启动后台 /任务的相关功能暂时都关闭”，旧前门仍把这条消息改写成“进入后台任务入口态”。

因此本轮补了一层产品面熔断，而不是继续调整 negotiation 细节：

1. 新增 `butler_main/chat/feature_switches.py`
2. 新开关固定为：
   - `features.chat_frontdoor_tasks_enabled`
3. 开关关闭时，chat 前门执行下面的统一回落：
   - `mission_ingress` 关键词与别名回落为普通 `chat`
   - `task_query / campaign_negotiation` 不再运行
   - `/status /plan /delivery /research` 不再作为前门命令处理
   - prompt 不再注入前门/后台/任务协作协议
4. 当前线上配置口径：
   - `butler_main/butler_bot_code/configs/butler_bot.json -> false`

## 当前残余边界

### 1. 已经收口的部分

1. 假完成 blocker 已落到 reviewer
2. 固定输出区与命名规则已落到 bundle helper
3. query / feedback 已能投影 closure state
4. 定向回归已覆盖 negotiation、frontdoor、campaign、observe、feedback

### 2. 本轮未做的部分

1. 没有做 live orchestrator 重启；本轮只做了 `butler_bot` 的 live restart 与在线 health/status 复验
2. 没有进一步收口 mission ingress 的“单 primary carrier”保护
3. 没有新增更重的治理协议或新模板体系

这三项不影响本轮故障闭环成立，但如果后续继续扩大后台任务入口面，仍应作为下一轮控制面治理专题。

## 最终结论

从 `2026-03-27` 起，后台任务的正式判断口径应改为：

1. 是否有固定 bundle root
2. implement 是否产出了真实 deliverable refs
3. reviewer 是否消除了 acceptance blockers
4. query / feedback 是否能复述同一份 closure state

只要这四项里有一项答不上来，就不能把任务记成真正完成。
