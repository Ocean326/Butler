## 今日裁决

1. `0327` 的主线不是再讨论“后台任务应该怎么设计”，而是把昨天已经确认的故障链真正改进代码、测试和文档。
2. 当前已确认的核心故障是：
   - negotiation 创建的 `campaign` 会在只有 placeholder implement artifact 的情况下继续走 reviewer，形成“任务看起来完成了”的假信号
   - workspace 里已有产物经常来自别的历史任务，因此不能把现成目录内容误判为这次后台任务的真实结果
3. 本轮正式收口为三条稳定口径：
   - 后台任务固定输出区统一落到 `工作区/Butler/deliveries/background_tasks/`
   - negotiation 创建的 `/delivery` 与 `/research` 默认进 `codex` runtime，但 reviewer 只对真实 deliverable + correctness checks 收敛
   - query / task_query / feedback_notifier 必须显式暴露 `bundle_root / runtime_mode / checks / latest_acceptance_decision / not_done_reason`
4. 本轮补收口一版系统抽象分层：
   - `Product Surface（产品表面层） -> Domain & Control Plane（领域与控制平面） -> L4 Multi-Agent Session Runtime（多 Agent 会话运行时） -> L3 Multi-Agent Protocol（多 Agent 协议层） -> L2 Durability Substrate（持久化基座） -> L1 Agent Execution Runtime（Agent 执行运行时）`
5. `Observe（观察）` 从今天起固定理解为 `Observability（可观测性） + Projection（投影读模型）`，并补入事件分层口径

## 本轮实现结论

### 1. Campaign / control plane

1. `orchestrator/interfaces/campaign_service.py`
   - 统一在创建时准备后台任务 bundle metadata
   - negotiation 创建的 `campaign` 默认补 `strict_acceptance_required=true`
   - `/delivery` 与 `/research` 若来自 negotiation，默认补 `campaign_runtime.mode=codex`
2. `domains/campaign/service.py`
   - implement artifact 会回写到 `instance.metadata["latest_implement_artifact"]`
   - reviewer 的 acceptance blockers 会回写 `latest_acceptance_decision / latest_acceptance_blockers / not_done_reason`
   - correctness checks 会在 implement 后同步回实例 metadata 与 `spec.metadata`
3. `domains/campaign/codex_runtime.py`
   - discover / implement / iterate 会把阶段产物写入固定 bundle 目录
   - codex implement 成功时会把主 `implementation_report` 从 placeholder 升格为真实 deliverable evidence

### 2. Query / feedback / frontdoor

1. `orchestrator/interfaces/query_service.py`
   - 新增 `bundle_root / bundle_manifest / runtime_mode / pending_checks / resolved_checks / waived_checks / latest_acceptance_decision / not_done_reason`
   - 不再把“缺 planning_contract 的 campaign”一律默认渲染为 `delivery`
2. `chat/task_query.py`
   - 前门状态查询现在会直接向用户回显输出区、runtime mode、closure blockers 和 not-done reason
3. `orchestrator/feedback_notifier.py`
   - Feishu task doc 和 push 文本会附带 bundle root、runtime mode 与 acceptance 状态

### 3. 固定输出区规则

1. 根目录固定为：
   - `工作区/Butler/deliveries/background_tasks/`
2. bundle 命名规则固定为：
   - `YYYYMMDD_HHMMSS_<topic_slug>_<primary_id>`
3. 每个 bundle 当前至少维护：
   - `briefs/`
   - `artifacts/`
   - `deliveries/`
   - `manifest.json`
   - `progress.md`

### 4. Skill exposure plane

1. skill 真源、collection 暴露、provider 注入方式正式分离
2. `chat/runtime.py` 与 `domains/campaign/codex_runtime.py` 统一改为消费 `SkillExposureContract`
3. `Codex CLI` 只消费 exposure contract，不再承担 skill 真源或选择逻辑
4. `campaign_runtime.mode=codex` 的任务现在默认补：
   - `skill_exposure.collection_id=codex_default`
   - `skill_exposure.injection_mode=shortlist`
   - `skill_exposure.provider_skill_source=butler`
5. 控制面补做了一轮 V1 优化：
   - collection registry 增加 owner / runtime / injection mode 等元数据
   - query / console 暴露 `skill_exposure_observation`
   - console 提供 skill collections / diagnostics API
   - console draft 支持任务级 `skill_selection`

### 5. 系统抽象与 runtime_os 公开面

1. `runtime_os` 新增三组公开面：
   - `runtime_os.multi_agent_protocols`
   - `runtime_os.multi_agent_runtime`
   - `runtime_os.durability_substrate`
2. `runtime_os.process_runtime` 保留兼容别名，但不再作为长期命名目标
3. `WorkflowSessionEvent` 已补最小事件封套字段：
   - `event_id / event_type / layer / subject_ref / causation_ref / created_at / payload`
4. `Workflow IR` 已补新的边界标签：
   - `protocol_owner`
   - `session_runtime_owner`
   - `durability_owner`
5. 当前仍保留旧的 `process_runtime_owner / collaboration_owner` 兼容字段，避免现有调用面一次性断裂

### 6. Chat 前门后台任务入口临时熔断

1. `butler_main/chat/feature_switches.py`
   - 新增 `chat_frontdoor_tasks_enabled`
   - 统一读取 `features.chat_frontdoor_tasks_enabled`，并允许 `BUTLER_CHAT_FRONTDOOR_TASKS_ENABLED` 环境变量临时覆盖
2. `butler_main/chat/routing.py`
   - 当开关关闭时，`mission_ingress` 相关别名与关键词一律回落为普通 `chat`
3. `butler_main/chat/mainline.py`
   - 当开关关闭时，跳过 `mission_ingress / task_query / campaign_negotiation`
   - `/status /plan /delivery /research` 不再触发前门能力，直接按普通聊天文本处理
4. `butler_main/chat/prompting.py`
   - 当开关关闭时，不再注入 `统一前门合同 / 前门协作协议 / 后台入口协作协议 / 状态查询协作协议 / 任务协作协议 / 前台分诊`
5. 线上活配置已改为：
   - `butler_main/butler_bot_code/configs/butler_bot.json -> features.chat_frontdoor_tasks_enabled=false`

## 最终裁决

1. 这轮“后台任务已经完成”的错误判断不再被单个 placeholder artifact 轻易触发。
2. 后台任务现在有固定输出区、可追溯命名规则和 manifest，可直接反查到时间、主题和主 id。
3. 前门弱化后的口径已经成立：
   - 可以允许 exploratory 启动
   - 但不能绕过边界澄清、初始计划和 reviewer 的严格 acceptance
4. 本轮已对 `butler_bot` 做 live `restart + status` 复验；`orchestrator / console` 本轮只做 `status` 确认，未额外重启，因为补丁只落在 chat 前门。
5. skill 注入从今天起应按“第 `4` 层真源 / 第 `3` 层选择 / 第 `1` 层消费 / 第 `2` 层记事实”理解，不再把 Codex local skill 目录误当 Butler 全局真源。
6. `multi-agent` 从今天起默认只指 `L3 + L4`；`Observe（观察）` 从今天起默认拆成 `Observability（可观测性） + Projection（投影读模型）`。
