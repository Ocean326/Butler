## 一句话裁决

当前 Butler 的主问题已经不是“runner 能不能跑”，而是长任务主线仍存在 4 类系统性偏差：

1. 前门与后台任务身份仍会漂移
2. 某些应该转后台的研究任务仍停留在 chat 前台
3. feedback harness 被文档写成“已成立”，但 live 任务默认并没有稳定挂上 Feishu 反馈合同
4. 系统级升级流程还没有被固定成“多路并行 -> 再规划 -> 再并行 -> 文档回写”的标准动作

## 本轮审计样本

本轮固定只用 4 个样本，不再靠印象判断：

1. `mission_ca4ffdcca385` / `campaign_1d10b299c5cb`
   - 当前唯一真实后台 campaign 长任务
2. `mission_7929ee3e7881`
   - 与上面同主题的 `talk_ingress` 任务，且持有长结果摘要
3. `mission_9502f7af440b`
   - SSH `pwd` 探测样本，用来区分“前门探测”和“后台长任务”
4. `aa50cd9b1e8acb57/recent_raw_turns.json`
   - Anthropic / Harness Engineer 研究样本，用来确认是否发生“在 chat 分析后终止，未转后台”

## 首轮问题清单

### P0-1 长任务身份漂移

Transfer_Recovery 同一用户意图同时留下了：

1. 一个真实后台 `campaign`
2. 一个 `talk_ingress` mission
3. 且 `talk_ingress` 的 `workflow_session` 中还写入了完整长任务结果摘要

这说明当前系统并没有做到“同一长任务只认一个后台承载实体”。  
这类问题的后果不是文档脏，而是：

1. 用户会误判后台任务数量
2. query / observe / status 可能命中错对象
3. 后续 follow-up 无法稳定 append 到唯一任务

直接证据：

1. `mission_ca4ffdcca385` 是 `campaign` 且已完成
2. `campaign_1d10b299c5cb` 已完成 3 次 iteration 并收敛
3. `mission_7929ee3e7881` 是 `talk_ingress`，但 `workflow_session_35b6183343d5` 也持有文献库、矩阵和写作输出摘要

### P0-2 研究任务可能停留在 chat 前台而不进入后台

Anthropic / Harness Engineer 样本显示：

1. chat 给出了长篇研究分析
2. recent memory 中没有对应的 mission / campaign 证据
3. 随后的 `/observe` 直接返回“已终止当前执行”

这和下一阶段主线里“长期任务应沉淀为 artifact / contract / resume”的方向冲突。  
如果产品预期这是一个需要持续推进的研究任务，那么当前行为就是 P0 路由失败。

### P1-1 Feishu feedback contract 没有形成默认稳定口径

代码口径显示：

1. `chat/negotiation.py` 只有在存在 `delivery_session` 时才挂 `feedback_contract`
2. `feedback_notifier.py` 只处理 `mission.metadata.feedback_contract.platform == "feishu"` 的 campaign

运行态口径显示：

1. 当前 live `orchestrator_run_state.json` 为 `feedback_docs=0 | feedback_pushes=0`
2. 当前 live campaign run-data 中没有可复读的 `feedback_doc` / `document_id` / `push_count`

所以现在不能把“feedback harness 已成立”直接等价成“Feishu 启动的后台任务默认会有 doc/push 交付面”。

### P1-2 query / observe 对“没挂反馈合同”的解释不够产品化

当前系统更像把内部计数暴露出来，而不是把用户真正关心的状态说清楚。  
对于一个已经后台完成但没有反馈合同的任务，产品需要明确回：

1. 任务是否真的启动了
2. 为什么没有 Feishu doc
3. 当前是否只有内部 run-data
4. 后续补消息会 append 还是新开任务

这一层现在在 live 证据上仍不够清晰。

### P2-1 稳定态文档口径存在过宣称

`0326/00` 与 `0326/03` 把 feedback harness 写成“已成立”，这个方向不算错，但当前 live 真相更接近：

1. feedback 能力可用
2. push-only 路径修过
3. 但 live 任务没有 active Feishu feedback contract
4. 因此 `doc/push` 指标仍为 `0`

也就是说，当前更准确的表述应该是“链路能力已具备，但默认挂接策略和 live 交付口径还未收口”。

### P2-2 任务统计口径仍会混淆轻探测和长任务

当前 3 个 mission 中既有：

1. 一个真实后台 campaign
2. 一个同主题 long result `talk_ingress`
3. 一个 SSH `pwd` 探测

如果不先做 mission taxonomy 清洗，系统外显“当前后台有几个任务”会长期混乱。

## 升级节奏

### 第一波并行

第一波固定拆 4 条 lane，并且允许并行推进：

1. `routing-lane`
   - 收口 frontdoor / negotiation / mission ingress
   - 目标：同一长任务只保留一个后台承载身份
2. `campaign-feedback-lane`
   - 收口 `feedback_contract` 挂接、doc/push 创建条件、query/status 提示语义
   - 目标：把“有能力”改成“有稳定默认口径”
3. `observe-lane`
   - 收口 `/status`、`/observe`、follow-up append 的外显行为
   - 目标：用户能一眼看懂任务当前处在哪个承载层
4. `docs-lane`
   - 同步收口 project-map、当日真源、acceptance 证据
   - 目标：不再让文档宣称跑在代码和 live 事实前面

### 第一波后的再规划

第一波结束后必须强制 replan，一律按下面方式做：

1. 重新跑 4 个固定样本
2. 把所有偏差重新归到：
   - 实现 bug
   - 设计缺口
   - 文档过宣称
   - 观测性缺失
   - 产品语义错位
3. 只保留 `P0/P1` 进入第二波
4. 若某问题已经被证实只是历史 run-data 噪音，就从主线阻塞项中降级，不继续误导实现

### 第二波并行

第二波不再按模块拆，而按问题束拆：

1. `identity-bundle`
   - 任务唯一身份、follow-up append、talk/campaign 边界
2. `delivery-bundle`
   - Feishu contract 默认策略、doc/push 创建、query/status 摘要
3. `artifact-bundle`
   - 研究任务转后台时的 planning/spec/progress/verdict 沉淀
4. `acceptance-bundle`
   - 更新 acceptance 样本、补充外显证据、回写文档真源

## 修复动作与验收口径

### 对 P0-1

修复动作：

1. 前门明确区分“轻量 talk 执行”和“需后台持续推进的长任务”
2. 同一 `source_invocation_id` 命中后台启动后，禁止再把完整长结果写回 `talk_ingress` 主承载
3. follow-up 只允许 append 到唯一 active campaign

验收：

1. Transfer_Recovery 样本只剩一个主承载实体
2. `/status` 命中同一任务时不再返回分叉对象

### 对 P0-2

修复动作：

1. 对研究型长任务增加显式后台升级门槛
2. chat 允许先做短答，但一旦进入“持续研究 / 再继续 / 后续迭代”语义，必须沉淀为后台任务
3. `/observe` 不能再把“未入后台的 chat 执行”伪装成可观察执行

验收：

1. Anthropic 类研究任务能形成 mission/campaign 或明确返回“仅前台问答，未启动后台”
2. recent memory、query、run-data 三处口径一致

### 对 P1-1 与 P1-2

修复动作：

1. 固定 Feishu 启动后台任务时的 `feedback_contract` 默认策略
2. 若未挂合同，query/status 必须明确说出原因
3. runner 周期摘要与用户可见 query 文案要能区分“未挂合同”和“挂了但发送失败”

验收：

1. 至少一条 Feishu 发起的 campaign 样本能看到稳定的 doc 或 push 证据
2. 未挂合同样本在 query/status 中能看到明确解释

### 对 P2-1 与 P2-2

修复动作：

1. 收紧稳定态文档口径
2. 统一 mission taxonomy，把 probe、talk、campaign 从运维统计和用户摘要里区分开

验收：

1. `docs/project-map/03_truth_matrix.md` 与 `0326/00`、`0326/03` 不再出现过度承诺
2. 运维摘要能区分 long-running campaign 和一次性 probe

## 当前裁决

从系统升级方法论上，后续不再接受“单路找代码 -> 单点修补 -> 结束”的推进方式。  
系统级问题固定按：

`并行首轮 -> 再规划 -> 并行二轮 -> acceptance -> 文档回写`

执行；否则 Butler 会继续在定位、身份、反馈与文档口径上反复漂移。
