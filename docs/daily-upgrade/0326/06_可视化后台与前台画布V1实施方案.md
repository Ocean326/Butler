## 一句话裁决

V1 可视化后台只做“观测 + 受控操作 + 结构化任务定制”，并与外部 chat 入口协作；自由画布建模进入 V2，不在 V1 主体落地。

## 1. 目标与范围

### 1.1 V1 目标

1. 建立 `Web 控制台` 作为长期任务的可视化观察与操控入口。
2. 建立 `Frontdoor Draft Board`，让高级用户可结构化定制任务而不直接改图。
3. 把 `WorkflowIR / campaign / session / artifacts / verdict / feedback` 映射成图面与面板对象。
4. 让外部 chat 仍作为主入口，但能被控制台复用和回显。

### 1.2 V1 范围

- `观测`：任务概览、图面、timeline、session/contract/feedback、运行状态。
- `控制`：campaign 级为主，少量节点级动作。
- `前门草案`：结构化任务定制，不做自由节点编辑。

### 1.3 V1 非目标

1. 不做桌面端。
2. 不引入 Node-first 后端。
3. 不改 `WorkflowIR` 真源或引入第二套 DSL。
4. 不在 V1 开放空白画布自由建模。
5. 不替换既有 `frontdoor -> negotiation/query -> campaign` 主链。

## 2. 所属层级与依赖方向

- 主层级：第 4 层 `frontdoor / domain / product plane`
- 次层级：第 3 层 `orchestrator / control plane`
- 依赖方向：`console -> orchestrator.interfaces -> runtime_os`，禁止倒依赖或绕过控制面。

## 3. 关键真源对象

1. `WorkflowIR`：图面结构的唯一编译真源。
2. `campaign`：任务生命周期、phase、verdict、contract。
3. `workflow_session`：运行态 evidence 与协作状态。
4. `artifacts`：产物 registry 与关联引用。
5. `planning/evaluation/governance` summaries：前门合同投影。
6. `CampaignNegotiationDraft`：前门草案与任务定制入口。

## 4. V1 产品形态

### 4.1 页面信息架构

1. `Overview Board`
2. `Campaign Graph`
3. `Run Timeline`
4. `Inspector Panel`
5. `Draft Board`
6. `Chat Context Panel`

### 4.2 图面能力

- 图面来自 `GraphSnapshot`，不直接读取内部 store 文件。
- 图面必须能标识：
  - phase path
  - active path
  - artifacts
  - handoffs
  - review/approval checkpoints

### 4.3 受控操作

campaign 级：
- `pause`
- `resume`
- `append_feedback`
- `request_approval`
- `resolve_approval`

节点级（只读为主，少量动作）：
- `inspect_node`
- `retry_current_node`
- `reroute_to_allowed_target`

所有动作必须通过 control plane 服务回写，不允许直接编辑 runtime 文件。

### 4.4 Draft Board

Draft Board 固定承接 `CampaignNegotiationDraft`，只支持结构化字段：

- `goal`
- `materials`
- `hard_constraints`
- `acceptance_criteria`
- `template_id / method_profile`
- `governance defaults`
- `feedback surface`

`Draft -> Launch` 仍走现有 `negotiation -> campaign create` 链路。

## 5. V2 自由画布方向

V2 引入 `CanvasDraftDocument`，只用于 draft，不直连运行时：

1. `CanvasDraftDocument -> compile -> workflow_template`
2. `workflow_template -> CampaignSpec -> WorkflowIR`
3. deployed snapshot 与 running campaign 均只读

## 6. 技术路线与栈

- 当前交付：Python-first，`stdlib WSGI + REST + SSE + static html/css/js`
- 可选后端升级：`FastAPI + REST + SSE`
- 前端升级目标：`React + TypeScript + Vite + @xyflow/react`
- 状态升级目标：`Zustand + TanStack Query`
- UI 升级目标：`Radix + Tailwind + design tokens`

## 7. 里程碑与并行节奏

### Round 0：方案与合同

1. 方案文档固化为真源
2. API/contract 固化为 runtime 文档
3. 复用与许可清单固化为 daily-upgrade 文档

### Round 1：控制台骨架 + Draft Board

1. 后端 console API 骨架
2. 前端 shell + graph + inspector
3. Draft Board 与 launch 链路

### Round 2：控制动作与事件流

1. campaign 控制动作
2. SSE 事件流
3. timeline + evidence 面板

## 8. 验收标准（V1）

1. Console 可列出任务并展示 `GraphSnapshot`。
2. Console 可显示 `planning / governance / evaluation` 摘要。
3. Draft Board 能改字段并能启动任务。
4. `pause/resume/feedback/approval` 操作能回写并被 query 观察。
5. runtime stale 时，所有 mutating action 被阻断。

## 9. 风险与约束

1. 双真源风险：图面不得成为第二套 runtime 真源。
2. 许可风险：禁止引入 EpicStaff 代码；Flowise enterprise 目录禁止复用。
3. 依赖方向风险：console 只能依赖 orchestrator interfaces。

## 10. 2026-03-27 收尾补丁

### 10.1 时间轴交互裁决

1. Run Timeline 继续保持“上层事件节点 + 下层时间标尺”两层结构。
2. 下层不再拖动整块事件区，而是通过可拖动 viewport thumb / scrubber 驱动共享的横向滚动。
3. 时间项固定为单排行内紧凑卡，不再承担大段摘要展示；详细信息进入 hover popover 或 agent detail。
4. 时间接近的事件允许相对真实时间点向右顺延，但必须保留真实时间锚点和引导线，禁止出现第二排和视觉重叠。
5. timeline 的 `scrollLeft / selected item / collapsed state` 需要按 scope 保持，自动刷新后不回到起点。

### 10.2 Agent Detail 交互裁决

1. 左侧 agent list 和中间 node card 统一增加显式详情图标，不再依赖隐式点击进入详情。
2. 点击卡体仍然只负责 `select + focus`；点击详情图标必须 `stopPropagation()`，单独打开 detail。
3. 旧右侧 inspector-panel 退出主链路，agent detail 改由全屏详情层统一承接。
4. 详情层第一版固定展示：
   - 概览与运行态
   - run records
   - planned input
   - artifacts
   - raw records

### 10.3 数据口径补充

1. agent detail 继续只消费现有 control plane / runtime 持久化真源，不新造前端私有日志。
2. 第一版 detail 数据优先来自：
   - `campaign_events`
   - `workflow_session.events`
   - `workflow_session.blackboard.entries`
   - 已有关联 artifacts

## 11. 2026-03-27 前端工程化升级

### 11.1 当前前端交付形态

1. `console/webapp/` 现在采用“双链并存”：
   - 根目录 `index.html / styles.css / app.js` 保留为 legacy fallback
   - 新前端源码进入 `console/webapp/spa/`
2. `spa/` 当前采用：
   - `React + TypeScript + Vite`
   - `TanStack Query`
   - `Zustand`
   - `@xyflow/react`
3. Python console server 当前默认托管规则改为：
   - 若 `console/webapp/dist/` 存在，优先托管构建产物
   - 若 `dist/` 不存在，自动回退到 legacy 静态页

### 11.2 本轮前端裁决

1. 这轮前端升级不是继续把 `app.js` 扩成更大的单体文件，而是正式切换到可维护的 SPA 结构。
2. legacy 页面只承担回退职责，不再承接新功能主线。
3. `Draft Board` 不再只存在于后台数据流里，前端必须为其保留正式入口与后续扩展空间。
4. graph / timeline / detail 仍然只消费 control plane 与 runtime 真源，不在前端新造运行时私有真相。

### 11.3 长期维护要求

1. 新增 console 前端能力默认进入 `spa/`，不要再往根目录追加大块 UI 逻辑。
2. 若后续继续做响应式、timeline、agent detail 或 Draft Board 升级，优先在 React 状态层和组件层收口，不回退到 DOM 拼接式实现。
3. `/console/api/*` 的合同仍保持稳定；前端升级优先通过 view-model 层消化差异，而不是让后端为页面组织反向漂移。
