# 08 第四层接口冻结 V1 简化版

日期：2026-03-25
时间标签：0325_0008
状态：Lane D / 第四层消费面冻结件

## 目标

1. 为 `0325-d-closure` 提供一份可执行的第四层冻结件。
2. 先冻结第四层的消费端口、统一证据集和依赖规则，不提前冻结下层实现细节。
3. 让 `chat / ingress / query / observe / smoke / domain pack` 在同一套口径下收口。
4. 让另一边继续推进 `3 -> 2 -> 1` 收缩时，第 4 层不需要跟着反复返工。

## 定位

1. 本文中的“第四层”指 `domain & product plane`，不是 `orchestrator control plane`，也不是 `runtime_os`。
2. 第四层是产品和领域的稳定消费面：
   - 向上接用户、渠道、前台产品和领域流程。
   - 向下只消费 control plane 与 runtime 的公开口。
3. 第四层先冻结的是 `contract`，不是 `implementation`。
4. `chat` 在本轮被视为第四层里的 `FrontDoorPort`，不是完整第四层本体。

## 本轮冻结原则

1. 冻结端口，不冻结大批细碎 API。
2. 冻结统一证据集，不冻结所有返回字段。
3. 冻结依赖方向，不冻结下层拆文件方式。
4. 第四层不拥有 runtime / control plane 真语义。
5. 第四层不直接读取 session 私有落盘，不直接解释 gate 执行过程。

## 四个端口

### 1. FrontDoorPort

定位：

- 前台入口
- 用户消息进入 Butler 的第一层消费口

当前主要落点：

- `butler_main/chat/routing.py`
- `butler_main/chat/mainline.py`
- `butler_main/chat/runtime.py`

只负责：

- 接收用户输入
- 判定 `chat` 还是 `mission_ingress`
- 组织 `channel / delivery / prompt / memory` 的前门事实
- 把结果交还给前台渠道

本轮只冻结的核心动作：

- `handle(invocation)`
- `route(invocation)`

明确不负责：

- `Mission / Node / Branch` 调度
- `workflow_session` 内部状态推进
- `approval / verification / recovery` 执行语义

### 2. MissionFacadePort

定位：

- 第四层对后台任务控制面的产品门面

当前主要落点：

- `butler_main/orchestrator/mission_orchestrator.py`
- `butler_main/orchestrator/ingress_service.py`
- `butler_main/orchestrator/query_service.py`

只负责：

- 提交任务
- 查询任务、branch、session 状态
- 控制任务
- 追加用户反馈

本轮只冻结的核心动作：

- `submit`
- `query`
- `control`
- `feedback`

明确不负责：

- `WorkflowIR` 编译细节
- `WorkflowVM` 内部执行逻辑
- session 文件结构与落盘路径

### 3. ObservationPort

定位：

- 第四层统一观测与验收口

当前主要落点：

- `butler_main/orchestrator/query_service.py`
- `butler_main/orchestrator/observe.py`
- `butler_main/orchestrator/smoke.py`
- `butler_main/orchestrator/runner.py`

只负责：

- 给人看系统是否真实运行
- 给 `Lane D` 提供统一验收证据
- 给 demo / smoke / dashboard / chat 回显提供同一套观测语义

本轮只冻结的核心动作：

- `snapshot`
- `mission`
- `session`

明确不负责：

- 发明新的 runtime 事件模型
- 读取私有 session 文件布局
- 重新定义 control plane 事件语义

### 4. DomainPackPort

定位：

- 领域流程包的接入位

本轮第一优先：

- `research`

后续候选：

- `background automation`
- `operator plane`

当前主要落点：

- `butler_main/orchestrator/research_projection.py`
- `butler_main/orchestrator/research_bridge.py`

只负责：

- 领域输入如何映射成 mission / template request
- 领域结果如何投影回产品视图
- 领域依赖哪些 package / policy / contract refs

本轮只冻结的核心动作：

- `build_request`
- `project_view`
- `resolve_refs`

明确不负责：

- 把领域规则塞回 orchestrator core
- 自己解释 runtime / control plane 真语义

## 统一证据集

本轮第四层只冻结下面这一组最小证据字段：

- `mission_id`
- `branch_id`
- `workflow_id`
- `workflow_session_id`
- `status`
- `active_step`
- `workflow_ir_compiled`
- `workflow_vm_executed`
- `workflow_session_count`

补充要求：

1. 还需要至少一类 `governance / package binding` 证据可见。
2. 但本轮不冻结其全量 payload 结构。
3. `observe / query / smoke` 三处必须共享同一套证据名，不再各自发明命名。

## 允许依赖

第四层新增代码允许直接依赖：

- `butler_main/chat/*` 自身前门与渠道层
- `butler_main/runtime_os/agent_runtime/*`
- `butler_main/runtime_os/process_runtime/*`
- `butler_main/orchestrator` 已公开的 façade

说明：

1. 兼容期内允许通过 `runtime_os` 去消费旧 `agents_os / multi_agents_os` 导出面。
2. 但第四层不应继续直接绑定旧目录私有实现。

## 禁止依赖

第四层新增代码不应新增对以下实现的直接依赖：

- `butler_main/orchestrator/service.py`
- `butler_main/orchestrator/workflow_vm.py`
- `butler_main/orchestrator/workflow_ir.py`
- `butler_main/orchestrator/runtime_bridge/workflow_session_bridge.py`
- `butler_main/agents_os/process_runtime/session/*`
- `butler_main/agents_os/process_runtime/factory/*`
- `butler_main/multi_agents_os/*`
- 任何 session 私有落盘路径或文件布局

补充裁决：

1. 旧代码中已存在的越层导入，本轮不要求一次清零。
2. 但本轮之后第四层新增改动不应继续扩大这些依赖面。

## 本轮不冻结的东西

1. `OrchestratorService` 的内部拆法。
2. `WorkflowIR` 的最终目标形态。
3. `runtime_os` 的物理目录最终落位。
4. `approval / verification / recovery` 的内部执行细节。
5. `workflow_session` 的全量 payload 结构。
6. 各渠道最终的全部展示细节。

## 后续扩展位

1. `08` 仍然是第四层消费面冻结件，冻结的是 `port / evidence / dependency boundary`，不是长期自治业务协议总稿。
2. `campaign supervisor`、`CampaignSpec`、`WorkingContract`、`EvaluationVerdict`、长生命周期外层循环等长期自治任务层讨论，转由 [09_长期自治Campaign任务层_讨论草稿.md](./09_长期自治Campaign任务层_讨论草稿.md) 承接。
3. 后续若 `09` 进入实现阶段，也默认只能建立在 `08` 已冻结的第四层边界之上，而不应反向改写本文的消费口、证据集与依赖裁决。
4. 因此，`08` 在后续阶段继续承担“冻结件”角色，`09` 则承担“长期自治任务层草案”角色，两者不合并、不混写。

## P0 / P1 / P2 执行计划

### P0 口径冻结

目标：

1. 冻结四个端口的定义。
2. 冻结统一证据集。
3. 冻结允许依赖与禁止依赖清单。
4. 为 `Lane D` 建立单一 closure 基线。

最小交付：

- 本文档
- 一份薄的第四层 contract 模块或等价稳定入口

验收：

- `chat / ingress / query / observe / smoke` 都能映射到本文四端口口径。
- A/B/C/D 不再围绕“第四层到底是什么”反复争论。

### P1 消费侧收口

目标：

1. 让第四层消费侧开始围绕统一 contract 工作。
2. 让 `query / observe / smoke` 明确使用统一证据集。
3. 让前门在命中 `mission_ingress` 时只走产品 façade，不越层直连私有实现。

建议落点：

- `butler_main/chat/mainline.py`
- `butler_main/orchestrator/mission_orchestrator.py`
- `butler_main/orchestrator/query_service.py`
- `butler_main/orchestrator/observe.py`
- `butler_main/orchestrator/smoke.py`

验收：

- 第四层主要消费点可以通过统一 contract 获得 mission / branch / session / observation 结果。
- 不新增对 control plane / runtime 私有实现的直接耦合。

### P2 验收闭环

目标：

1. 至少打通一条非 research 的 session-aware 真实证据链。
2. 让 `Lane D` 的总收口不只停留在文档和口头口径。

最小验收对象：

1. 一条普通 path mission。
2. `observe / query / smoke` 至少两处可见同一组证据字段。
3. 至少存在：
   - `workflow_ir_compiled`
   - `workflow_vm_executed`
   - `workflow_session_count > 0`
   - 一类 `governance / package binding` 证据

## 对 Lane D 的执行裁决

`Lane D` 本轮的第四层工作重点应是：

1. 吸收前三路的消费侧尾项。
2. 冻结第四层 contract 和统一证据集。
3. 让 `query / observe / smoke / tests / compat` 对齐。

`Lane D` 本轮不应扩成：

1. 新一轮 runtime 重构。
2. control plane 内核重写。
3. 大规模 domain pack 业务开发。

## Done 定义

满足下面条件时，可认为第四层接口冻结 V1 简化版成立：

1. 第四层被收成四个端口，而不是散乱接口堆。
2. `chat` 明确只做前门，不再继续长成第二个 orchestrator。
3. `MissionFacadePort` 成为第四层操作后台任务的唯一产品门面。
4. `observe / query / smoke` 使用同一套统一证据集。
5. 第四层新增代码不再直接绑定 `service.py`、session 私有实现或旧 `multi_agents_os` 私有目录。
6. 另一边继续推进 `3 -> 2 -> 1` 收缩时，第四层大部分消费代码不需要重写。

## 本文结论

1. 第四层本轮不追求“接口多”，而追求“消费口稳定”。
2. 第四层先稳定的是端口、证据和依赖边界。
3. 只要这三件事先稳住，`Lane D` 就能在不反向侵入第 2/3 层的前提下完成总收口与集成验收。
