## 这份文件解决什么

这份文件不是新的架构文，而是把 `0325` 总纲压成几个可以并行推进的轻任务包，并给出可直接复制的提示词。

原则：

- 不把任务拆得太死。
- 不预先规定过多实现细节。
- 每个任务包只冻结：目标、边界、参考文档、最小交付。
- 允许先做审计、脚手架、迁移方案，不强迫每个并行任务一步到位做完所有代码改造。

## 本轮主线思想与硬要求

本轮并行不是“各自做一小坨重构”，而是围绕同一条主线施工：

1. 先冻结 `3 -> 2 -> 1` 主轴：
   - `3 = orchestrator / control plane`
   - `2 = runtime_os / process runtime`
   - `1 = runtime_os / agent runtime`
2. 后续所有包都必须服从这条主线：
   - `3` 只回答“跑什么 / 何时跑 / 如何回写”
   - `2` 只回答“一个通用多步过程如何 dispatch / gate / resume / finalize”
   - `1` 只回答“单次 agent run 如何执行完”
3. 不允许逆向增长：
   - 不再往 `agents_os` 或 `multi_agents_os` 长新的真语义
   - 不再让 `orchestrator` 继续吸收 `approval / verification / recovery` 真执行语义
   - 不再让第 1 层继续对外导出第 2 层对象
4. `WorkflowIR` 当前只允许作为 `3 -> 2` 的过渡载体：
   - 当前可保留
   - 不允许继续混装扩张
   - 目标态是 `MissionExecutionPlan + ProcessStartRequest`
5. 并行包不以“完成所有代码搬迁”为目标，而以“冻结边界、建立目标目录、推进真源回收、减少错误依赖”为目标。

## 命名假设

当前默认把 `agents_os` 的候选新总名设为 `runtime_os`。

理由很直接：

- `agents_os` 已经不足以覆盖现在的真实职责，因为它不只承载单 agent runtime，还在承载 process runtime。
- `multi_agents_os` 又把第二层 process runtime 藏进了另一个名字里，进一步放大理解成本。
- `runtime_os` 至少比 `agents_os` 更接近“单次 run + 通用过程 runtime”的组合体。

本轮不要求所有并行任务都最终接受这个名字，但默认先按这个靶名工作。除非代码核查发现明显冲突，否则不要再发散出第三套命名体系。
