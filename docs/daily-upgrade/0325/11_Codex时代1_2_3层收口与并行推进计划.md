# 11 Codex 时代 `1 / 2 / 3` 层收口与并行推进计划

日期：2026-03-25
时间标签：0325_0011
状态：架构收口补充计划 / 并行推进补充件 / 不替代 `05`

## 一句话定位

1. 本文不是新的唯一活动计划，而是对 [05_Orchestrator完善设计与实施计划.md](./05_Orchestrator完善设计与实施计划.md) 的补充收口件。
2. 它回答的核心问题不是“今天的 MVP 先做什么”，而是：**在 `Codex` 已具备当前原生能力的前提下，Butler 的 `1 / 2 / 3` 层哪些应删除、哪些应变薄、哪些反而要强化。**
3. `Campaign MVP` 与 `research path` 在本文中统一降级为“架构验针”，用于验证长期边界是否正确，而不是反向决定总架构。

## 当前总判断

1. `Codex` 当前已经原生提供一批过去 Butler 容易重复建设的能力：
   - `subagents / multi_agent`
   - `review`
   - `resume / fork`
   - 项目级 `.codex/agents` role pack
   - `MCP`
2. 因此，Butler 现在不应再在 `1 / 2 / 3` 层重复发明这些能力的平行外壳。
3. 但这不意味着 `1 / 2 / 3` 层失效；更准确的裁决是：
   - 第 `1` 层保留，但显著变薄
   - 第 `2` 层保留，但只保留 Butler 自己必须掌握的过程真语义
   - 第 `3` 层不能删，反而要继续强化
4. 当前主流 CLI 的共性能力一律不再重复造轮子；`Codex` 当前是最强消费方，但 `1 / 2 / 3` 层仍保持 Butler 自己的真源定义。

## 第 `1` 层裁决：`agent runtime`

### 保留

1. 统一的外部执行适配入口：
   - 哪个 provider 被调用
   - 这次 run 的输入输出合同
   - 上下文、约束、role 注入
2. 单次 run 的事实记录：
   - `provider`
   - `model`
   - `role`
   - `external_session_id`
   - `resume_capable / fork_capable / review_capable`
3. CLI 能力探测与策略路由：
   - 哪些能力原生可用
   - 哪些只允许包裹复用
   - 哪些必须退回 Butler 自己语义

### 删除

1. Butler 自己维护的平行 `subagent runtime`
2. Butler 自己维护的平行 `review runtime`
3. Butler 自己维护的平行 `resume / fork` 外壳
4. 任何只是为了复刻主流 CLI 原生协作能力而存在的旧执行壳

### 变薄

1. `host / kernel` 只负责发起一次执行和收回结果，不继续解释多步 workflow。
2. `instance / session` 只保留单次 run 或 provider 实例级语义，不承接第 `2` 层 process/session 真语义。
3. `capability registry` 只做能力声明与路由，不扩成业务控制面。

### 强化

1. 统一 `ExecutionRequest / ExecutionResult / ExecutionFact` 合同。
2. 统一 provider 能力探测与事实记录。
3. 统一项目级 role pack 导出与 schema 守卫。

一句话结论：

第 `1` 层从“自造 agent 内核”收成“CLI-native execution adapter layer”。

## 第 `2` 层裁决：`process runtime`

### 保留

1. Butler 自己的 `workflow_session`
2. `checkpoint / resume substrate`
3. `artifact_registry / shared_state / workflow_blackboard`
4. 与业务无关的 `gate / receipt / recovery` 合同

### 删除

1. 任何只是为了模仿 `Codex` thread/session/subagent/review 形态而存在的中间对象
2. 不承载 Butler 独立过程语义、只服务旧 runtime 的重复协作壳

### 变薄

1. 第 `2` 层不再尝试做“完整通用 workflow engine”。
2. 第 `2` 层不再接管 CLI 已能原生完成的单轮内部协作。
3. 第 `2` 层只保留 Butler 必须跨轮掌握的过程连续性与可追溯性。

### 强化

1. 明确固定：`CLI session/thread != Butler workflow_session`
2. 明确固定：`CLI review/result != Butler verdict/writeback contract`
3. 把跨 phase、跨迭代、跨 provider 的状态保持继续压回 Butler 第 `2` 层

一句话结论：

第 `2` 层从“想包完所有流程执行”收成“跨轮过程真源与持久化 substrate”。

## 第 `3` 层裁决：`orchestrator / control plane`

### 保留

1. `Mission / Node / Branch / LedgerEvent` 真源
2. 调度、预算、激活、写回
3. `query / observe / run-data / evidence window`
4. 第 `4` 层 domain 对第 `2` 层 runtime 的编排与投影

### 删除

1. 第 `3` 层里残留的通用 runtime 执行语义
2. 直接操纵 session substrate 私有文件结构的逻辑
3. 任何把 control plane 写成执行器内核的历史包袱

### 变薄

1. `service.py` 继续退回 façade / use-case 层。
2. 编译桥只做 `mission/domain -> process request / run request` 转换，不承接执行真语义。

### 强化

1. 统一多 provider 结果的 writeback。
2. 统一 mission / branch / session / artifact 的观测面。
3. 固定第 `3` 层对第 `2` 层只走公开 API，不再摸私有实现。

一句话结论：

第 `3` 层不是“可以削掉的中间层”，而是 Butler 最不能丢的长期资产。

## 第 `4` 层与 `Codex` 的正确关系

1. 第 `4` 层最积极消费 `Codex` 原生能力。
2. 第 `4` 层推荐采用：
   - 固定 role pack
   - 局部覆盖注入
   - phase recipe
   - 独立 reviewer 终判
3. 当前推荐最小角色集：
   - `campaign-supervisor`
   - `campaign-implementer`
   - `campaign-reviewer`
4. 允许注入的局部覆盖只包括：
   - `top_level_goal`
   - `hard_constraints`
   - `working_contract`
   - `iteration_budget`
   - `workspace / repo root`
   - artifact refs
5. 不允许通过局部覆盖改写：
   - reviewer 最终独立性
   - `top_level_goal` 可变性
   - Butler 状态真源写入口

## 并行推进裁决

今天之后不再让 `Campaign MVP` 单线绑架总架构，而按四条并行线推进。

### Lane A：第 `1` 层收口

1. 建立统一 `ExecutionRequest / ExecutionResult / ExecutionFact`
2. 继续吸收 `Codex` 项目级 role pack
3. 删除明显重复的旧执行壳

### Lane B：第 `2` 层真语义压实

1. 固定 `workflow_session / checkpoint / artifact / shared_state` 真源
2. 切断 `CLI session` 与 `Butler session` 的概念混淆
3. 继续削减伪装成 process runtime 的重复协作壳

### Lane C：第 `3` 层强化

1. 继续把 `orchestrator` 收成纯 control plane
2. 强化 `query / observe / run-data`
3. 禁止第 `3` 层继续吸 runtime 私有实现

### Lane D：第 `4` 层验针

1. `Campaign` 用于验证：
   - 半自治 supervisor
   - 独立 reviewer
   - role pack + phase recipe
   - contract rewrite proposal
2. `research path` 用于验证：
   - 更长链路
   - 多角色
   - 更复杂证据面
3. 两条线都只是验针，不反向改写 `1 / 2 / 3` 层真源定义

## 近期实施顺序

1. 先做 `Lane A + Lane C`：
   - 因为第 `1` 层重复造轮子和第 `3` 层职责过满，是当前最容易继续扩散的风险点。
2. 再做 `Lane B`：
   - 把 Butler 自己必须保留的 process/session 语义钉死。
3. `Lane D` 同步以最小样例跟跑：
   - `Campaign`
   - `research`
4. 后续任何第 `4` 层新增功能，都必须先回答：
   - 这是不是主流 CLI 已有原生能力
   - 如果是，Butler 是否仍有必要重复实现

## 验收口径

1. 第 `1` 层不再继续新增平行 `subagent / review / resume` 外壳
2. 第 `2` 层能够清楚回答“哪些是 Butler 自己的跨轮语义”
3. 第 `3` 层能够稳定统一：
   - `mission`
   - `branch`
   - `session`
   - `artifact`
   - `runtime fact`
4. `Campaign` 与 `research` 两条验针都能跑，但都不再反向主导架构定义
5. 新增代码默认优先消费 provider-native 能力，而不是先在 Butler 里重复实现一版
