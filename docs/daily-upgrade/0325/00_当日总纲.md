## 四层总表：本轮判断尺子

这张表不是补充阅读材料，而是 `0325` 后续拆目录、拆 `service.py`、迁移语义真源时的统一判断尺子。后文第二张“核心文件迁移与动作类型”表，默认都以这里的四层边界为前提。

### 总图

| 层 | 名称 | 主要回答的问题 | 典型输出 |
| --- | --- | --- | --- |
| 1 | `agent_os / Agent Runtime` | 一个有边界的单次 agent run 怎么活完 | `RunResult / Artifact / Trace / Session` |
| 2 | `agent_os / Process Runtime` | 一个通用多步过程怎么推进、停机、恢复、过闸 | `WorkflowCursor / Checkpoint / DecisionReceipt` |
| 3 | `orchestrator / Control Plane` | 该跑什么、何时跑、结果怎么回写任务控制面 | `Mission / Node / Branch / LedgerEvent` |
| 4 | `domain & product plane` | 对用户和业务来说，到底要做成什么流程和产品体验 | `proposal/review/refine`、`background automation`、`chat/channel`、`operator plane` |

### 第 1 层：`agent_os / Agent Runtime`

| 子层 | 概念 | 功能 | 当前主要文件 | 不该负责 |
| --- | --- | --- | --- | --- |
| 1.1 Run Contract | `Run` `RunInput` `WorkerRequest` `RunResult` `Artifact` `TraceEvent` | 定义一次 bounded run 的输入输出和最小事实记录 | `butler_main/agents_os/runtime/contracts.py` | `Mission/Node/Branch`、`research/chat` 语义 |
| 1.2 Instance / Session Lifecycle | `Instance` `Session` `status` `retire/resume` | 单 agent 实例创建、恢复、退役、上下文快照 | `butler_main/agents_os/runtime/instance.py`、`butler_main/agents_os/runtime/host.py` | 多步 workflow 业务顺序 |
| 1.3 Execution Host | `host/kernel/runtime host` | 真正把一次 run 发出去执行，并收回结果 | `butler_main/agents_os/runtime/host.py` | `approval/verification` 的业务判定 |
| 1.4 Context / Capability Envelope | `context resolver`、`capability registry` | 决定这一轮 agent 看到什么、能调用什么 | `butler_main/agents_os/runtime/context_resolver.py`、`butler_main/agents_os/runtime/capability_registry.py` | `mission graph`、domain workflow |

这一层一句话边界：

`agent runtime` 只关心一个 agent run 怎么执行完，不关心这个 run 在整个任务图里处于第几步。

### 第 2 层：`agent_os / Process Runtime`

| --- | --- | --- | --- | --- |

这一层一句话边界：

`process runtime` 关心过程怎么跑，不关心为什么要跑这个过程。

当前最大的模糊点就在这里。第 2 层今天被拆散在三处：

- `agents_os/runtime/execution_runtime.py`
- `multi_agents_os/session/*`
- `orchestrator/service.py` 里的 gate/recovery 逻辑

这三处不重新命名和收口，后面会一直混。

## 五路分工原则

1. `Lane E` 只做 control plane 分层、迁移顺序、边界冻结，不继续吸收 runtime 语义。
2. `Lane C` 只做 process runtime substrate，不在这里扩写 orchestrator 业务语义。
3. `Lane A` 只做知识平面与映射平面，不顺手实现 compiler。

5. `Lane D` 做演示链路、fixture、smoke、observe、兼容与最终验收收口，消费前三路结论，但不回头重定义前四路的概念。

## Chat 升级补充与落地记录

### 本轮新增冻结点

1. `chat` 继续定位为第 4 层前台入口，不再自带一套并行执行内核。
2. 普通 `chat` 路线如果用户明确要求并行、分工、subagent，则只通过 prompt 明示“依赖 Codex 原生 parallel / subagent 能力”，不再维护 chat 自定义 `sub-agent / team execution / public agent library` 调度壳。
3. `chat` 的固定资产继续留在 `butler_main/chat/`：
   - prompt 资产
   - role / bootstrap / dialogue 资产
   - chat 自有 memory 目录布局
   - chat 到上层/下层的接口适配
4. 只有通用能力才下沉到第二层或通用 runtime 接口：
   - 对话 turn 级 memory provider 接口
   - `session_scope_id` 这类跨前台可复用的会话作用域参数
5. 微信阶段目标不是“真正多用户系统”，而是“多个微信绑定共享一个 chat 核心，仅隔离 recent memory 会话作用域”。

### 已落地实现

1. 普通 `chat` 路线已移除旧自定义执行壳接入：
   - 删除 chat 侧 `runtime_executor` 注入与旧 follow-up/extract runtime request 链路
   - 删除 `butler_main/chat/providers/butler_runtime_executor.py`
   - prompt / role / bootstrap 全部改为说明“Codex 原生并行协作”
2. 微信已支持多 binding：
   - `chat/core.py` 增加 `WeixinBindingConfig`
   - 每个 binding 独立 `state_dir / session_file / listener thread`
   - `/health` 输出聚合状态和逐 binding 状态
3. recent memory 已按会话作用域隔离：
   - `session_scope_id` 从 `chat/runtime.py` 注入到 `ConversationTurnInput.metadata`
   - 第二层 turn / memory provider 接口支持 `session_scope_id`
   - chat recent memory 改为按 scope 落盘到 `butler_main/chat/data/hot/recent_memory/scopes/<hash>/`
4. chat 自身 compat 做了第一轮清理：
   - 去掉 `pathing.py` 里的旧树迁移逻辑
   - `memory_policy.py` 去掉 `legacy_compatible`
   - chat 前台的旧后台 compat 壳与遗留 mission runtime 目录已删除
5. 全仓旧后台残留清理进入落地阶段：
   - `butler_bot_code` 的旧后台主链 / `memory_manager` / `team execution` 主链已删除
   - `manager.ps1` 与 `watch_stack.ps1` 改为以 `chat/core.py` + orchestrator 状态为主
   - `registry.json` 默认启动入口已切到 `chat/core.py`
   - `0325` 后续剩余工作以文档与历史资产收尾为主，不再保留旧后台 compat 壳

### 当前非目标

1. 本轮不推进真正的用户体系、账号体系或完整 multi-tenant。
2. 本轮不重写 orchestrator / mission 侧历史 team execution 语义。
3. 本轮只隔离微信 recent memory，会话之外的更深层用户画像归并暂不处理。

### 验收口径

1. 普通 `chat` prompt 中仍可表达“允许并行协作”，但不再暴露 chat 自己维护的 sub-agent / team executor 壳。
2. 微信多个 token / binding 可以挂到同一个 chat 核心上运行。
3. 不同微信会话的 recent memory 不再共享同一份 `recent_memory.json` / `recent_raw_turns.json`。
4. chat 侧代码树继续收缩到“前台资产 + 接口适配 + chat 私有实现”，通用 turn / memory 接口下沉到第二层。
