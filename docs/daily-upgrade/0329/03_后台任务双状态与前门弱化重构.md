## 一句话裁决

本轮把后台任务从“一个状态同时表示正在推进和是否最终完成”改成了双状态口径：

- `execution_state`
- `closure_state`

从现在起，后台任务可以明确表达：

1. 任务仍在推进
2. 当前阶段已经有产物
3. 为什么还没有最终闭环

而不会再把这三件事压缩成一个模糊的 `active / not_done_reason`。

## 本轮落地

### 1. 控制面真源改成双状态

`CampaignInstance.metadata` 当前新增并持续回写：

- `execution_state`
- `closure_state`
- `progress_reason`
- `closure_reason`
- `operational_checks_pending`
- `closure_checks_pending`

兼容字段 `not_done_reason` 继续保留，但语义降级为：

- `closure_reason` 的兼容镜像

### 2. pending checks 不再只是一桶

当前 `pending_correctness_checks` 仍保留，但对外口径已拆成两类：

1. `operational_checks_pending`
   - 影响“现在能不能继续安全推进”
2. `closure_checks_pending`
   - 影响“最终能不能判 accepted”

这意味着像 `ssh_reachable / target_path_exists / research_anchor_confirmed` 这类条件，不再只会被解释成“任务没完成”，而能被解释成：

- 当前等待确认或恢复
- 但阶段推进与最终闭环是两件事

### 3. query / chat / feedback 同步切口径

当前统一新增或标准化输出：

- `execution_state`
- `closure_state`
- `progress_reason`
- `closure_reason`
- `latest_stage_summary`
- `stage_artifact_refs`
- `acceptance_requirements_remaining`
- `operator_next_action`

覆盖面包括：

1. `OrchestratorQueryService`
2. `chat/task_query.py`
3. `feedback_notifier.py`
4. `console` detail 消费面

### 4. chat 前门继续弱化

本轮没有恢复 chat 自然语言建后台任务主入口。当前正式口径保持为：

1. `chat` 优先承担：
   - 查询已有后台任务
   - 给已有后台任务补充反馈
2. 新建后台任务的长期主入口，后续应转向更显式的控制面 / console 面

这条裁决的目的，是先把控制面语义和对外状态讲清楚，再决定是否重开前门。

## 当前业务流

当前后台任务业务流应理解为：

`显式创建 / 兼容入口 -> campaign control plane -> runner/session 推进 -> query/feedback/console 投影`

其中：

1. `execution_state`
   - 回答“它现在是在跑、在等、已暂停，还是已终止”
2. `closure_state`
   - 回答“它是 open、已有阶段产物、被 acceptance 卡住，还是已 accepted”

当前正式解释是：

- `accepted` 只表示最终闭环
- 不再等于“系统有推进动作”
