# 0317 心跳与任务循环现状问题报告

> 更新时间：2026-03-17
>
> 本文记录 0317 时点 Butler 的 heartbeat + task loop 真实状态、可见症状、代码入口和当前已确认的问题，不是新方案文档。

## 1. 目的

这份文档只回答四件事：

1. 现在心跳链路处于什么状态
2. 为什么会出现“经常停跳”
3. 为什么会执行一些“不知所谓”的任务
4. 后续修复时应先从哪几层下手

---

## 2. 当前运行状态

0317 为了先保住主对话链路，当前运行上已经临时收敛为：

1. `butler_bot.json`
   - `heartbeat.enabled = false`
   - `memory.self_mind.enabled = false`
2. 主 talk 进程单独运行
3. heartbeat / self_mind 暂时不参与在线链路

这不是长期方案，而是诊断态。

---

## 3. 已确认的核心现象

### 3.1 心跳状态文件已经撕裂

当前两个关键状态文件互相对不上：

1. `butler_main/butler_bot_code/run/heartbeat_run_state.json`
   - `updated_at = 2026-03-16 18:56:46`
   - `state = running`
   - `phase = execute`
   - `heartbeat_pid = 22440`
2. `butler_main/butler_bot_code/run/heartbeat_watchdog_state.json`
   - `updated_at = 2026-03-17 11:47:45`
   - `state = cooldown`
   - `heartbeat_pid = 0`
   - `cooldown_until = 2026-03-17 11:50:34`

这说明：

1. 运行态文件还认为心跳在跑
2. watchdog 文件已经进入 cooldown
3. 真实进程状态与两者并不一致

也就是说，当前不是“单个状态错了”，而是运行态、看门狗态、真实进程三者已经分叉。

### 3.2 日志显示心跳被旧 PID 长时间卡住

日志文件：

- `butler_main/butler_bot_code/logs/butler_bot_heartbeat_20260317_001.log`

其中从 `2026-03-17 08:15:56` 到 `2026-03-17 11:40:32` 持续重复同一类日志：

- 检测到已有心跳进程在运行 `(PID=22440)`，本次启动跳过

但 `22440` 已经是陈旧 PID。

这意味着当前心跳恢复逻辑容易被 stale PID / stale run state 卡死，直接表现为：

1. watchdog 认为已经有进程
2. 新启动被跳过
3. 结果用户看到的是“经常停跳”

### 3.3 状态展示会误导人

即使临时把 heartbeat 和 self_mind 都关闭，`manager.ps1 status` 的展示仍会给出：

1. 心跳与意识循环现由主进程统一管理

这句话没有区分：

1. 功能是否启用
2. 功能由谁托管
3. 当前是否处于 cooldown / stale / disabled

因此它更像“归属描述”，而不是“实时健康状态”。

## 4. 当前代码入口是怎样串起来的

### 4.1 `memory_manager.py` 仍掌握 heartbeat 的主接线

当前几个关键入口：

1. `start_background_services()`：`memory_manager.py:582`
2. `_use_external_heartbeat_process()`：`memory_manager.py:635`
3. `_heartbeat_process_watchdog_loop()`：`memory_manager.py:958`
4. `_heartbeat_loop()`：`memory_manager.py:1156`
5. `_run_heartbeat_once()`：`memory_manager.py:1239`
6. `_seconds_to_next_heartbeat()`：`memory_manager.py:6526`

当前 `_use_external_heartbeat_process()` 返回 `False`，意味着 heartbeat 默认由主进程侧的内嵌链路管理，而不是一个彻底独立、边界更硬的外部 sidecar。

这会带来两个问题：

1. 启动、watchdog、执行、状态写回都集中在 `memory_manager.py`
2. 一旦状态文件或运行条件脏掉，排查面会同时跨 runtime、watchdog、heartbeat 执行三个层面

### 4.2 `HeartbeatOrchestrator` 负责规划与执行上下文拼装

关键入口：

1. `build_planning_context()`：`heartbeat_orchestration.py:223`

它会从多个来源同时取上下文：

1. `heartbeat_tasks.md`
2. task workspace
3. unified recent / beat recent
4. local memory hits

因此 planner 的输入口是宽的，不只是一个“任务板”。

## 5. 为什么会“执行一些不知所谓的任务”

这个问题不是单个 planner prompt 的错，而是任务真源、账本状态和调度策略三层叠加。

### 5.1 任务真源已经明显污染

当前 heartbeat 真源目录：

- `butler_main/butler_bot_agent/agents/local_memory/heartbeat_tasks/`

其中几个文件的体量和内容已经说明问题：

1. `01_工作任务.md`
2. `02_主动想干.md`
3. `03_定时.md`
4. `05_整理清洁.md`
5. `06_闲时无聊.md`
6. `task_change_log.jsonl`

已观察到的污染特征：

1. `02_主动想干.md` 中存在大量重复或近重复任务
2. 同一主题被反复挂成 `pending / in_progress / done`
3. 特别明显的一条线是网易云歌词脚本，被多次重复挂任务
4. `05_整理清洁.md` 中混入大量“对话原文式任务”
5. `06_闲时无聊.md` 中包含较多探索、灵感、闲聊类任务
6. `task_change_log.jsonl` 已经非常大，说明任务流转历史长期堆积
7. `03_定时.md.38800.32860.tmp` 说明任务目录中还混入了运行期临时产物

这类任务池会直接把 planner 往“看起来能做，但不一定值得做”的方向拉。

### 5.2 `task_ledger.json` 的运行理由已经形成固定偏置

文件：

- `butler_main/butler_bot_agent/agents/state/task_ledger.json`

当前大量任务条目都带着近似的 `manager_state.reason`：

1. 用户在线无硬 DDL
2. 优先推进近期自我心理活动
3. 主动探索
4. 小步巩固某些背景工程线
5. 固定保留新陈代谢支路

这会导致两个后果：

1. 即使任务已经完成或意义变弱，账本仍不断把调度重心推回“主动探索 / 自我活动 / 代谢”
2. planner 会把“空闲时可做”逐渐误当成“默认应该优先做”

### 5.3 planner 的输入和 fallback 过宽

当前 heartbeat 任务体系不是只读工作任务，还显式保留三类宽口：

1. `主动想干`
2. `整理清洁`
3. `闲时无聊`

这些类别本来是为了让系统在空闲时仍能做低风险推进，但在任务池已经污染、账本又带偏置时，就容易演变成：

1. 没有硬 DDL
2. 没有强约束任务
3. 系统自动回落到“主动探索 / 清理 / 逛外部世界 / 自我活动”

于是用户看到的就不再是“稳态后台维护”，而是一些动机模糊、价值感不高的执行。

## 6. 问题分层

截至 0317，可以把问题明确拆成三层。

### 6.1 第一层：运行态脏

表现：

1. stale PID
2. stale run state
3. watchdog cooldown 与 run state 不一致
4. status 展示不真实

结果：

- 心跳会出现“应该重启但没重启”“看起来在跑但其实没跑”

### 6.2 第二层：任务池脏

表现：

1. 任务重复
2. 任务过时但未归档
3. 对话原文直接入板
4. 清理/闲聊/主动探索任务过多
5. 运行临时文件进入任务目录

结果：

- planner 拿到的“真源任务”本身就混杂、稀释、失真

### 6.3 第三层：调度策略偏向主动探索

表现：

1. 没有硬 DDL 时默认倾向主动探索
2. `manager_state.reason` 持续强化这一倾向
3. `HeartbeatOrchestrator` 的 planning context 吃到的内容过宽

结果：

- 即便没有故障，系统也可能稳定地去做一些用户感知上“不知所谓”的任务

---

## 7. 当前结论

0317 的问题已经可以明确表述为：

1. “经常停跳”主要不是 planner 不会规划，而是 heartbeat 运行态和 watchdog 状态已经不可信
2. “执行不知所谓任务”主要不是单条任务写错，而是任务池污染 + 账本偏置 + 调度 fallback 过宽
3. `memory_manager.py` 目前仍掌握 heartbeat 启停、watchdog、执行主链路，是这条问题链上的最大集中点
4. `HeartbeatOrchestrator` 负责规划执行编排，但 planner 输入的真源质量已经不足够干净

---

## 8. 修复顺序建议

如果后续继续修，不建议一起大改，建议顺序如下：

### 8.1 先修运行态真实性

先处理：

1. stale PID 判定
2. stale run state 清理
3. watchdog cooldown 与真实进程对齐
4. `status` 输出改成真实反映 `disabled / running / cooldown / stale`

### 8.2 再清任务真源

重点处理：

1. 合并重复任务
2. 归档明显过时任务
3. 限制长对话原文直接入板
4. 把临时文件与正式任务文件分开
5. 收紧 `主动想干 / 整理清洁 / 闲时无聊` 的规模

### 8.3 最后再调调度策略

再处理：

1. 无硬 DDL 时降低主动探索权重
2. 增加空转保护
3. 让“闲时可做”不再等价于“默认应做”
