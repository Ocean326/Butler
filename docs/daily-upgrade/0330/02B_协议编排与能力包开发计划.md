## 一句话裁决

`02B` 负责把外部 harness 能力翻译成 Butler 可编译、可验证、可执行的协议对象。  
先冻结 `template -> ir -> vm -> session` 主链，再谈更复杂的 graph、crew、flow、subagent 玩法。

## 本文边界

- 主层级：`L3 Multi-Agent Protocol`
- 次层级：`Domain & Control Plane`、`L4 Multi-Agent Session Runtime`
- 主代码目录：
  - `butler_main/runtime_os/multi_agent_protocols/`
  - `butler_main/runtime_os/process_runtime/templates/`
  - `butler_main/orchestrator/workflow_ir.py`
  - `butler_main/orchestrator/workflow_vm.py`
  - `butler_main/orchestrator/compiler.py`
- 默认测试：
  - `test_orchestrator_workflow_ir.py`
  - `test_orchestrator_workflow_vm.py`
  - `test_agents_os_protocols.py`
  - `test_orchestrator_framework_compiler.py`

## 当前已对齐能力

1. `WorkflowTemplate`、`Workflow IR`、`workflow_vm` 已形成最小 compile 与执行入口。
2. `approval / verification / recovery / handoff` 已具备字段骨架。
3. `framework_compiler` 已能把 mapping/profile/mission payload 合并成 compiler inputs。
4. 协议层与 session/runtime 已开始分层，不再把运行中的 session state 误写成 protocol 真源。

## 当前缺口

1. `WorkflowTemplate / Workflow IR / compiler inputs` 的稳定版本策略仍不够清晰。
2. capability package、governance package、workflow package 仍偏概念化，缺少统一 attach point。
3. `handoff / join / acceptance / recovery` 的字段存在，但编译期约束与错误提示还不够强。
4. 外部框架映射虽然已有 compiler 支撑，但“哪些字段允许来自 mapping、哪些必须由 mission override”还没有完全写死。

## P0 开发计划

1. 冻结最小协议对象集合：
   - `WorkflowTemplate`
   - `Workflow IR`
   - `RoleSpec`
   - `HandoffSpec`
   - `Approval / Verification / Recovery contract`
2. 明确 compiler merge 顺序：
   - `framework mapping`
   - `framework profile`
   - `mission/node override`
3. 定义 capability package / governance package / workflow package 的最小引用形态与挂载位置。
4. 为 `handoff / approval / verification / recovery` 增加 compile-time validation，避免运行期才发现结构缺损。

## P1 开发计划

1. 增加协议版本和兼容策略说明，避免 `runtime_os` 与兼容目录长期漂移。
2. 增加 role binding、artifact contract、input/output contract 的交叉校验。
3. 把 operator surface 所需的 compile preview、patch diff、risk hints 明确成 compile 输出的一部分。
4. 为外部 harness 映射增加“允许吸收 / 明确不吸收”的编译级注释位。

## P2 开发计划

1. 为外部映射输入增加更强的自动校验样板。
2. 为 `subgraph / parent graph / remote agent adapter` 预留稳定扩展位。
3. 在不反向引入 vendor DSL 的前提下，为图式 authoring 提供更强的 compile preview。

## 关键合同

1. `WorkflowTemplate`
  - 表达静态编排结构，不表达运行时会话状态。
2. `Workflow IR`
  - 表达编译后、可执行的中间语义，是 L3 对 L4/L2/L1 的主要下游输入。
3. `compiler inputs`
  - 只允许承载编排、治理、运行绑定提示，不承载产品投影状态。
4. `capability / governance / workflow package`
  - 是 Butler 自己的 package 语言，不是外部 vendor 对象名的直接镜像。

## 验收口径

1. `test_orchestrator_workflow_ir.py` 验证 IR 序列化、反序列化与合同保真。
2. `test_orchestrator_workflow_vm.py` 验证 step routing、gate、resume_from 等执行语义没有偏离协议。
3. `test_agents_os_protocols.py` 验证协议对象与归一化常量。
4. `test_orchestrator_framework_compiler.py` 验证 framework inputs 到 Butler compile 结果的合并顺序与默认值。

## 明确不做

1. 不把 LangGraph、CrewAI、Dify 等外部 DSL 直接当 Butler 协议真源。
2. 不让 UI 节点结构反向定义 `Workflow IR`。
3. 不把运行时 event log 或 projection 混写回协议层。
