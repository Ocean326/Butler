## 一句话裁决

`02G` 负责结构化治理、可审计观测和 acceptance 闭环。  
Butler 要优雅地发挥 agent 能力，前提不是“让它更自由”，而是让 `risk_level / autonomy_profile / approval / trace / receipt / action ledger` 成为第一类事实。

## 本文边界

- 主层级：`Domain & Control Plane`
- 次层级：`Product Surface`、`L1`、`L2`
- 主代码目录：
  - `butler_main/domains/campaign/`
  - `butler_main/orchestrator/campaign_service.py`
  - `butler_main/orchestrator/query_service.py`
  - `butler_main/orchestrator/feedback_notifier.py`
  - `butler_main/agents_os/skills/`
- 默认测试：
  - `test_orchestrator_campaign_service.py`
  - `test_orchestrator_campaign_observe.py`
  - `test_skill_exposure.py`
  - `test_console_server.py`

## 当前已对齐能力

1. `OperatorActionRecord`、`OperatorPatchReceipt`、`RecoveryDecisionReceipt` 已进入 `0330/01` 正式对象层。
2. `trace_id / receipt_id / recovery_decision_id` 已开始贯穿 query、feedback、console。
3. `risk_level / autonomy_profile / approval_state` 已在部分 surface 合同与 planning/runtime metadata 中出现。
4. skill exposure 已具备 registry、injection、prompt policy 等基础治理面。

## 当前缺口

1. 这些治理字段目前存在于多处合同中，但语义归口和字段解释仍不够统一。
2. tracing 覆盖还未在 route decision、workflow compile、session transition、execution receipt、recovery action 之间完全打通。
3. acceptance 证据仍容易停留在“测试跑过了”层面，缺少对 receipt、trace、operator action、文档回写的统一收口。
4. governance package 与默认策略注册体系还没有稳定提升为现役真源。

## P0 开发计划

1. 冻结以下字段的统一含义与归属：
   - `risk_level`
   - `autonomy_profile`
   - `approval_state`
   - `trace_id`
   - `receipt_id`
   - `recovery_decision_id`
2. 定义 `OperatorActionRecord -> OperatorPatchReceipt -> RecoveryDecisionReceipt -> execution/durability receipt` 的关联链。
3. 把 tracing 的最小覆盖面写死到真源里，至少包括：
   - route decision
   - workflow compile
   - session transition
   - execution receipt
   - recovery/approval action
4. 把 acceptance 明确成“测试 + receipt + trace + 文档回写 + 必要时重启复验”的组合闭环。

## P1 开发计划

1. 建立 `governance policy package` 与 `default policy registry`。
2. 把 query、feedback、console 三个面板统一到同一治理字段解释口径。
3. 增加 operator runbook 与审计板的固定展示结构。
4. 把 skill exposure 与治理默认项接通，避免 skill 只被当作 prompt 注入片段。

## P2 开发计划

1. 增加 runtime SLO、告警、指标分级与 postmortem 模板。
2. 增加多租户或多 workspace 的治理策略层。
3. 增加对 A2A/MCP 外部接入的风险与审批分级。

## 关键合同

1. `risk_level / autonomy_profile`
  - 是结构化治理参数，不是 prompt 文案提示。
2. `approval_state`
  - 是运行状态机的一部分，不是产品备注字段。
3. `trace_id / receipt_id / recovery_decision_id`
  - 是跨层证据链主键，query/feedback/console 必须共享。
4. `OperatorActionRecord / OperatorPatchReceipt / RecoveryDecisionReceipt`
  - 是人机治理操作的正式审计对象，不是调试日志。

## 验收口径

1. `test_orchestrator_campaign_service.py` 验证治理字段、campaign/projection、operator action 相关合同。
2. `test_orchestrator_campaign_observe.py` 验证 observe/query 面不把 trace 与 projection 混层。
3. `test_skill_exposure.py` 验证 skill exposure contract 与注入边界。
4. `test_console_server.py` 验证 console 对治理字段与 audit receipt 的外显。

## 明确不做

1. 不把 trace spans 当作产品投影状态本身。
2. 不把 acceptance 收口成“口头说已支持”或“只贴一段聊天结论”。
3. 不把治理字段散落到 prompt、UI、临时 metadata 而没有统一归口。
