## 一句话裁决

从今天起，Butler Desktop 的现役前台口径继续收紧为：

> **Desktop 不是 team dashboard，也不是多页面 workbench；它是一个以 `Manager` 为唯一一级前台代理对象的 mission conversation shell。**

补充裁决：

- 左侧是 thread continuity，不再把 `History` 显式做成与 mission 分裂的主对象。
- 右侧永远优先是当前 thread 的主对话，不再让 `Supervisor / Agent focus / Templates` 抢主位。
- `Supervisor / Recovery / Studio / Team` 只作为同一条 Manager 对话里的内部模式、来源或可展开细流存在。

## 今日新增结论

1. `0405` 的 `thread-home / manager-thread / supervisor-thread / agent-focus / template-team` 仍是当前分支的代码基线，但它不再是最终产品叙事。
2. Desktop 的前台一级对象只保留 `Manager`；thread、history、modes 都是围绕 Manager conversation 组织。
3. 左 rail 不再承担“功能切页”职责，而是承担：
   - 工作区与配置上下文
   - 新线程入口
   - 线程列表与连续历史
4. 右侧不再把 `Supervisor` 和 `Agent focus` 做成独立大页，而是统一折回当前 thread 内的内联结构块与可展开细流。
5. 顶栏固定收缩为 1 到 2 行必要状态与少量动作，避免工具栏化。
6. 页面设计继续向 Codex 靠拢，但不直接复制命名；Butler 仍保留自己的 runtime 语义与命令路径。
7. 根 `AGENTS.md` 当前只服务仓库内本地 agent 的轻量路由协议；网页端入口继续归根 `README.md`。
8. 根 `AGENTS.md` 不是 butler-flow 的 ambient runtime authority；repo contract 仍只在 `control_profile.repo_contract_paths` 显式绑定后生效。
9. `thread-home` 的 linked history 现役合同不仅要同时保留 `manager + supervisor` 双 summary，还要把同一 linked flow 的顺序固定为相邻的 `manager -> supervisor`，以保证 manager-first shell 的连续性。
