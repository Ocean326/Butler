# Daily Upgrade

`daily-upgrade/` 用于按日期归档阶段性改动文档。

**索引页：[INDEX.md](./INDEX.md)**（由 `gen_index.ps1` 自动生成；新增日期目录或文档后，在仓库根目录执行 `.\docs\daily-upgrade\gen_index.ps1` 即可刷新。）

## 约定

1. 每次新增日更文档先创建 `docs/daily-upgrade/<MMDD>/`
2. 同一天的现状、计划、排查、总结放在同一日期目录
3. 自 `0322` 起，每日目录默认维护 `1+N` 文档结构：
   - `00_当日总纲.md`：当天所有推进事项的大纲和索引
   - `01_...md`、`02_...md`：二级计划文档，按两位序号递增
4. 同一二级主题从“先计划 -> 落实 -> 再计划”都继续写在同一份二级计划文档里，不再按时间反复拆成多篇同主题文档。
5. 二级计划文档前几行先写主线，后面再按时间块追加细节、落地记录和下一步计划。
6. `0315` 到 `0321` 的历史目录继续保留原有 `HHmm_标题.md` 风格；`0322` 起的新目录默认改用 `00_`、`01_`、`02_` 这类序号命名。
7. 文档内优先使用相对路径，不写本机绝对路径
8. 长期有效内容不要放这里，应迁入 `docs/concepts/` 或 `docs/project-map/`

## 规则

1. 从 2026-03-15 19:00 起，Butler 的变更说明文档默认写具体更新时间，至少精确到分钟，格式至少为 `yyyy-MM-dd HH:mm`，如有必要可写到 `yyyy-MM-dd HH:mm:ss`。
2. 日更文档头部默认补充一行 `时间标签：MMDD_HHmm`，用于标识首个落点时间；后续追加记录继续写明确更新时间。
3. `00_当日总纲.md` 负责维护当天所有事项的入口、状态、优先级和对应二级计划链接。
4. 当日持续变更，优先追加到对应的 `01_...md` / `02_...md` 文档，并按时间块记录，而不是只改“最终结论”不留时间。
5. `00_当日总纲.md` 需要增加一节 `影响的 project-map 条目`，说明当天哪些当前导航条目受到影响。
6. 后续代码改动若影响主链路、架构边界、测试结果、运行方式或维护规范，必须同步更新对应文档时间戳。
7. 已经稳定且被反复引用的结论，不要长期只留在日更文档里，应同步提升到 `docs/project-map/` 或 `docs/runtime/`。
