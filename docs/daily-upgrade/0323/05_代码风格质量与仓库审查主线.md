# 05 代码风格质量与仓库审查主线

日期：2026-03-23
时间标签：0323_0005
状态：长期进行中

## 定位

这是一条长期治理主线，不直接承担产品功能交付，而是持续约束：

1. 代码风格
2. 代码质量
3. 仓库结构卫生
4. review / 审查纪律
5. 测试与回归门槛

它的角色更接近：

**Engineering Quality / Repo Governance Owner**

## 为什么要单独设这条线

Butler 现在已经不是单模块项目，而是多平面、多主线、多 worker 并行推进的系统。

如果没有一条独立的质量与仓库审查主线，后果会很直接：

1. 每条 worker 都会按自己的节奏写代码，风格越来越散。
2. 目录迁移、兼容壳、临时 adapter 会不断堆积，但没人持续清账。
3. 新旧体系交错时，review 标准会失去统一口径。
4. 文档、代码、测试、目录语义容易逐步漂移。

所以这条线不是“可有可无的整理工作”，而是长期系统建设里的工程治理主线。

## 长期职责

这条主线长期主要负责：

1. 统一代码风格和模块边界
2. 控制兼容壳、临时桥接层、历史残留的数量和寿命
3. 建立 repo review / code review 的固定口径
4. 推动测试、回归、最小 smoke 验证形成门槛
5. 持续审查：
   - 哪些对象层次放错了
   - 哪些目录语义漂移了
   - 哪些 runtime 真源还没收口
6. 保证“长期架构文档”和“仓库实际结构”不会长期背离

## 明确不负责

这条主线不直接负责：

1. 单个产品能力的功能设计
2. 单个 runtime 的执行语义设计本身
3. 单个前台入口的产品交互体验
4. 代替 `00/01/03/04` 这些主线做具体功能开发

它更像所有主线之上的：

**质量闸门 + 仓库卫生监督线**

## 当前应重点盯的对象

截至 2026-03-23，这条线最该持续盯的对象包括：

1. `chat` 新 frontdoor 与旧 body 的混合区
2. `heartbeat` 冻结后仍残留的 legacy 兼容区
3. `orchestrator` 中临时 bridge / VM / store 的边界是否失控
4. `multi_agents_os` 是否被误扩成第二执行器
5. `agents_os` 是否继续混入定义层 / 展示层语义
6. `skill / capability / package / tool` 的术语和目录归位
7. 测试命名、覆盖口径、回归入口是否持续可用

## 长期检查维度

建议这条主线长期按六类问题审查：

### P0 层次审查

1. 代码是否放在正确层
2. 是否有定义层、执行层、协作层、治理层混写

### P1 仓库卫生

1. 是否出现过多临时兼容壳
2. 是否出现目录真源漂移
3. 是否存在“新旧双真源”

### P2 风格一致性

1. 命名风格是否统一
2. 模块出口是否清晰
3. 数据模型和 contract 是否自洽

### P3 审查纪律

1. review 是否优先找边界错误、行为回归、缺失测试
2. 是否仍有人以“先写进去再说”的方式绕过长期架构

### P4 测试与回归

1. 是否为新主线建立最小验证
2. 是否保留稳定 smoke path

### P5 文档-代码一致性

1. 长期规划是否仍能解释当前代码
2. 日计划是否仍和实际施工同步

## 与其他主线的关系

1. 对 `00 agents_os`
   - 审查 execution kernel 是否持续收口
2. 对 `01 Chat`
   - 审查 frontdoor / runtime / prompt / compat 是否继续解耦
3. 对 `02 Heartbeat`
   - 审查 legacy 是否真的冻结，而不是口头冻结
4. 对 `03 Orchestrator`
   - 审查 control plane 是否被临时桥接层拖乱
5. 对 `04 multi_agents_os`
   - 审查 substrate 是否守住边界，不被误当执行器

## 一句话记忆版

`05` 不是功能线，而是：

**Butler 的代码质量、仓库卫生与架构审查长期主线。**

## 最新判断（2026-03-23）

当前这条主线不能再按“全面细抠所有旧目录”的方式推进。

基于仓库现状，质量线今天更准确的职责是：

1. 先做 repo hygiene 和边界隔离。
2. 再限制临时目录、编辑器残留、OS 垃圾文件继续进入仓库。
3. 对计划丢弃的旧目录，只做冻结和隔离，不再追加长期治理投资。

## 当前范围修正

已明确：

1. `butler_bot_agent/`
2. `butler_bot_code/`
3. `space/`

这三个目录按“计划丢弃”处理。

因此，`05` 主线对它们的策略不是：

- 继续做深度风格统一
- 继续补长期架构真源
- 继续在旧目录里沉淀新的治理壳层

而是：

1. 阻止继续长出新的真源。
2. 阻止继续向 repo 根部扩散临时文件和兼容残留。
3. 把真正长期有效的治理动作尽量放在仓库级规则和文档口径，而不是放进待废弃目录内部。

## 2026-03-23 仓库审查结论

这轮按仓库层面复核后，当前最明显的卫生问题是：

### P0 当前硬问题

1. repo 根部已经出现多个 `_tmp_*` 临时目录和探针产物。
2. 仓库里已经出现 `.DS_Store` / `._*` / `.claude/` / `docs/Untitled.md` 这类本地残留风险对象。
3. 当前 `.gitignore` 对这些对象的阻断还不完整。

### P1 当前判断

1. 这不是“代码风格”问题，而是更基础的仓库卫生问题。
2. 如果不先把 repo hygiene 收紧，后续任何架构治理都会被临时残留持续稀释。
3. 对计划丢弃目录做深度重构收益很低，当前更应该做的是：
   - 停止新增债务
   - 记录范围边界
   - 为后续删除或迁移创造干净外部环境

1. 是否明确了质量线的真实职责边界。
2. 是否把 repo hygiene 的最基本阻断补上。
3. 是否把计划丢弃目录从“继续投入治理”切换为“冻结、隔离、等待后续迁出或删除”。
