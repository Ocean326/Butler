2. 从今天开始，`0323-0325` 的总轴线统一改成：

## 今日重新排序后的主优先级

### 优先级 1：长期目标与统一语言

1. 固化 Butler 长期目标，不再用“支持任意形式多 Agent 流”这类口号描述目标。
2. 固化 Butler 的统一内部语言：
   - `Workflow IR`
   - `Artifact Contract`
   - `Handoff / Decision / Verification / Approval / Recovery`
   - `Mission / Workflow / Session / Branch` 边界
3. 固化 Butler 的独立路线：
   - 借鉴 `OpenFang` 的 Agent OS 视角
   - 保留 Butler 自己的 `mission / workflow / receipt / checkpoint` 体系
   - 不复制任何单一外部项目的术语和壳层

### 优先级 2：运行时主线

1. 明确 `agents_os` 是未来 `Runtime Core`，而不是协议占位层。
2. 明确 `orchestrator` 是 `Mission / Control Plane`，而不是 workflow VM。
3. 明确 `multi_agents_os` 是 `Collaboration Substrate`，而不是第二套后台主线。

### 优先级 3：外部框架吸收主线

1. 今天起不再讨论“兼容某个外部框架”。
2. 统一转为：
   - 外部框架画像
   - Butler 内部映射
   - Framework Catalog
   - Framework Compiler
3. 外部框架吸收结果必须优先沉淀为：
   - package
   - contract
   - runtime binding
   - governance policy

### 2026-03-23 03:30:00 合龙原则修正

1. 当前不再把“多 worker 成功合龙”理解为：每条 worker 线最后彼此全部直接打通。
2. 正确推进方式应固定为：
   - 下层形成稳定 contract
   - 逐层上交给上一层消费
   - 最终验收由上层控制面与入口面承接
3. 对应到当前 Butler：
   - `agents_os` 向上交执行 contract
   - `multi_agents_os` 向上交协作状态 contract
   - `research` 向上交场景解释与结构化产物
   - `orchestrator` 负责后台主验收
   - `chat` 负责前台用户验收
4. 因此，后续协调重点不是“每条线彼此直连”，而是：
   - `01` 交干净入口与交付面
   - `03` 交控制面与验收主线
   - `04` 交 collaboration substrate contract
   - `agents_os` 交 execution kernel contract

### 2026-03-23 03:34:00 `agents_os` 长期维护线补充

1. Butler 长期需要一条常驻 `agents_os` worker 线。
2. 这条线的角色应固定为：
   - `Kernel / Runtime Owner`
   - 不是产品总包方
   - 不是最终验收层
3. 它长期主要负责：
   - execution contract
   - runtime binding
   - capability invocation
   - checkpoint / resume
   - approval / verification / recovery 的执行侧接口
4. 它的成功标准不是“自己做完整产品闭环”，而是：
   - 稳定向上交付 execution kernel contract
   - 被 `orchestrator` 稳定消费
   - 最终通过 `chat` 完成用户可见交付
