# 微信多用户升级方案

## 一、目标定义

本方案优先解决的“微信多用户”定义为：

- 一个已登录的微信 bot 账号
- 同时接收并回复多个不同私聊用户
- 每个用户的会话、记忆、执行状态彼此隔离
- 不因为某一个用户的长任务阻塞其他用户

本方案第一阶段不把“群聊多 participant”混进来，避免一开始把边界做乱。

## 实施进度（2026-03-24）

当前已落地：

- `Phase 1A`：微信输入层显式写入 `weixin.chat_type` 与 `weixin.conversation_key`
- `Phase 1B`：新增微信会话注册表，维护活跃会话、执行中会话、最近错误
- `Phase 1D`：`getupdates` 批量消息改为“同会话串行、跨会话并行”分发
- `core /health` 与首页已可见微信活跃会话数、执行中数量与最近会话
- 微信 bridge client 已补回 `on_reply_sent` 钩子，回包后会触发统一记忆持久化

当前仍未落地：

- `Phase 1C` durable inbox journal
- `Phase 1E` outbox retry / resend / 更完整诊断
- recent / memory 的真正 `conversation_key` scope 化改造

### Phase 1 范围

- 多私聊用户并发服务
- 每个用户独立 session / actor / memory 边界
- 支持多用户同时发消息、排队、回复
- 支持核心状态页看到当前活跃微信用户会话

### Phase 2 范围

- 群聊
- 群里的多 participant 语义
- @bot / 非 @bot 过滤
- 群级记忆与个人记忆分层

## 二、当前代码现实

当前微信链路其实已经具备“多用户最小基础”，但还没有被正式设计成产品能力。

### 已具备的基础

1. 微信轮询是 bot 级全局 cursor，不是按单用户绑定。
   `getupdates` 能收到该 bot 账号下的全部消息。

2. 当前输入层已经能从消息里提取用户和会话边界：

- `actor_id` 来自 `from_user_id / open_id`
- `session_id` 来自 `session_id / conversation_id / source_event_id`
- `receive_id` 也来自发消息用户

3. 当前发送层已经是“回给消息来源用户”，不是写死单用户。

因此，从协议上说，当前实现已经不是“只能服务一个用户”，而是“可以收到多个用户的消息并逐条回复”。

### 真正缺的能力

1. **没有显式的多用户会话模型**

当前只是把 `actor_id/session_id` 塞进 invocation，没有建立“微信会话注册表”。

2. **没有并发调度模型**

当前 `poll_weixin_bridge_once()` 是串行遍历消息，一条一条处理。  
一个用户的长任务会阻塞其他用户。

3. **没有 durable inbox / outbox**

当前是“拉到消息 -> 直接处理 -> 直接发回”。  
一旦后面转并发或 core 重启，中间状态就不够稳。

4. **没有多用户状态诊断**

core 状态页目前只知道“微信 listener 是否活着”，不知道有哪些活跃用户、哪个用户正在执行、哪个用户失败。

5. **群聊边界尚未定义**

当前 private DM 基础较明确，但群聊 / participant 还没有正式语义。

## 三、推荐方案

推荐采用：

**一个 bot 账号 + 一个全局微信接入 runtime + 一个会话注册表 + 一个 durable inbox + 按会话串行、按全局并行的执行模型**

这是最适合当前代码基座的演进路线。

## 四、核心设计

### 1. 分两层 session

#### A. BotRuntimeSession

这是 bot 登录态和拉消息游标，不按用户拆。

建议字段：

- `account_id`
- `bot_token`
- `bridge_base_url`
- `cdn_base_url`
- `cursor`
- `longpoll_timeout_ms`
- `login_state`
- `last_poll_at`

#### B. ConversationSession

这是用户会话态，按“一个用户 / 一个会话边界”拆。

建议主键：

- `platform = weixin`
- `account_id`
- `conversation_key`

建议字段：

- `actor_id`
- `receive_id`
- `receive_id_type`
- `raw_session_ref`
- `chat_type`
- `last_message_id`
- `last_seen_at`
- `pending_request_id`
- `last_error`
- `memory_scope`

### 2. 明确 `conversation_key`

建议规则：

#### 私聊

`conversation_key = weixin:{account_id}:dm:{actor_id}`

解释：

- 当前第一阶段只做“一个用户一个私聊会话”
- 不需要再把 message_id 级别抬成会话边界

#### 群聊（Phase 2）

`conversation_key = weixin:{account_id}:group:{chat_id}`

群里单个说话人的 `actor_id` 单独保留，但 session 边界属于 group thread。

### 3. 引入 durable inbox journal

当前最大结构缺口不是“识别不到多用户”，而是“没有接入队列和落盘缓冲”。

建议流程改成：

1. `getupdates` 拉到一批消息
2. 每条消息先写入 inbox journal
3. journal 落盘成功后再推进全局 cursor
4. dispatcher 从 journal 拉待处理消息
5. 执行完成后写结果状态

这样做的收益：

- core 重启不丢正在处理/待处理消息
- 可以安全转成并发模型
- 可以做失败重试和状态诊断

### 4. 调度模型：按会话串行，按全局并行

推荐调度策略：

- 同一个 `conversation_key` 内串行执行
- 不同 `conversation_key` 之间并行执行

这样可以同时满足：

- 单个用户不会出现回复乱序
- 多个用户不会互相阻塞

建议实现：

- 全局 worker pool
- 每个 `conversation_key` 一个轻量队列
- dispatcher 负责把新消息按 `conversation_key` 分发

### 5. delivery 保持按消息来源回包

当前 delivery 逻辑本身方向是对的，不需要推翻。

应保留：

- `target = weixin.receive_id`
- `thread_id = weixin.raw_session_ref`
- `context_token = weixin.context_token`

需要补的是：

- 在 delivery plan / log 里带 `conversation_key`
- 在 outbox 里记录对应入站消息与发送请求的映射关系

### 6. memory 继续统一真源，但 scope 改清楚

多用户不等于多套 memory 系统。

应该保持：

- recent memory 真源统一
- long-term memory 真源统一
- skills 真源统一

但要明确 scope：

- session scope：按 `conversation_key`
- actor scope：按 `actor_id`
- channel scope：按 `weixin`

也就是说：

- 记忆系统还是同一套
- 只是读取 / 写入时用更清楚的 session key

### 7. core 状态页增加微信多用户视角

建议在 core 的 `/health` 或后续 `/status` 里增加：

- 当前微信活跃会话数
- 最近活跃的 `conversation_key`
- 正在执行中的微信会话数
- 最近失败会话
- 最近一条失败原因

## 六、分阶段实施建议

### Phase 1A：先把多用户语义显式化

目标：

- 不改执行模型，先把 key 和状态补清楚

动作：

- `normalize_event()` 增加 `weixin.chat_type`
- 计算并写入 `weixin.conversation_key`
- core 状态里显示当前微信活跃会话
- 日志打印 `conversation_key`

收益：

- 低风险
- 可先验证“多用户现实流量长什么样”

### Phase 1B：引入 session registry

目标：

- 形成“微信 bot 下面有哪些活跃用户会话”的明确视图

动作：

- 新增 `session_registry.py`
- 每次入站更新 `last_seen_at / last_message_id / actor_id / raw_session_ref`
- 提供 status snapshot

### Phase 1C：引入 durable inbox

目标：

- 为并发处理和重启恢复做准备

动作：

- 新增 inbox journal
- `getupdates` 拉到消息先落 inbox，再推进 cursor

### Phase 1D：引入 per-conversation dispatcher

目标：

- 同会话串行、跨会话并行

动作：

- 新增 dispatcher
- 同一个 `conversation_key` 单独串行
- 不同用户消息并行处理

### Phase 1E：补 outbox / retry / diagnostics

目标：

- 多用户场景下看得见、重试得动

动作：

- 发消息请求记录到 outbox
- 失败支持 retry
- `/status` 可见失败会话

## 七、群聊方案（后续）

群聊不要和私聊第一阶段一起做。

建议后续单开：

### 群聊关键问题

1. session 是群还是人？
2. 群里是否只响应 @bot？
3. 群级记忆与个人记忆如何分层？
4. 群里一个用户的长任务是否占整个群会话？

建议默认规则：

- `session_id = group thread`
- `actor_id = participant`
- 默认仅在 @bot 或显式命令时触发

## 八、验收标准

达到以下条件，可视为“微信支持多用户”第一阶段完成：

1. 一个 bot 账号能同时服务多个私聊用户。
2. 多个用户可以并行发消息，互不阻塞到不可接受程度。
3. 同一用户的消息保持顺序，不乱序回复。
4. 每个用户的 recent / memory scope 正确隔离。
5. core 状态页可看到活跃微信会话与失败状态。
6. core 重启后不会因为 cursor / 队列设计导致明显丢消息。

## 九、最终建议

**推荐做法不是“直接宣称微信已多用户”，而是按 Phase 1A -> 1D 逐步升级。**

原因：

- 当前代码已经有多用户基础，不需要推倒重来
- 真正要补的是会话模型、调度模型和 durable inbox
- 先把私聊多用户做稳，再碰群聊，风险最低
