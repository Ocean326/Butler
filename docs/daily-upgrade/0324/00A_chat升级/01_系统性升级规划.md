# chat 系统性升级规划

## 一、升级背景

本轮升级来自三个连续信号：

1. 飞书、微信、CLI 已经不是一个单一入口，而是稳定存在的多入口 chat 产品。
2. 微信上已经暴露出“如果 prompt / presentation 不解耦，就会天然带飞书味”的结构问题。
3. 本地使用方式已经开始向“常驻核心 + 本地客户端 + 多渠道监听”演进，旧的单入口心智已经不够。

因此，本轮不应再以“补 bug / 补分支”推进，而应明确转为一次 chat 架构和交付方式的系统性升级。

## 二、升级目标

### 目标 1：把 chat 从“单入口工程”升级为“统一 chat 产品内核”

统一项包括：

- invocation
- routing
- prompt assembly
- memory access
- runtime execution
- output bundle

这些能力必须与具体渠道解耦。

### 目标 2：把飞书 / 微信 / CLI 从“入口条件分支”升级为“独立对话入口”

每个入口都应具备明确的：

- input adapter
- prompt policy
- presentation policy
- delivery adapter
- runtime bootstrap

### 目标 3：把 chat 从“启动单个入口”升级为“启动一个常驻核心”

目标形态：

- `butler-core` 负责启动统一核心
- 统一核心自动拉起飞书 / 微信监听
- 本地 `butler` 优先连接已运行核心
- 后续可扩展 Web / Telegram / 企业微信 / API gateway

### 目标 4：为后续多用户、多会话、多渠道共存提供结构准备

需要做到：

- 会话边界明确
- 渠道策略独立
- memory 与 channel 解耦
- 新渠道接入不需要回改公共逻辑

## 三、本轮升级范围

### 本轮纳入

- prompt 侧飞书 / 微信 / CLI 对话提示词拆分
- 本地常驻 `chat core` 与 `butler` 客户端
- 微信链路稳定性修复
- 微信输出去飞书化
- 渠道策略继续下沉的规划与执行准备

### 本轮不纳入

- Web 前端对话入口
- 群聊 / 多 participant 对话治理
- 多租户配置隔离
- 远程 API 服务化鉴权

## 四、升级原则

1. 记忆、能力、模型调用保持统一真源，不因渠道分叉。
2. prompt、presentation、delivery 等渠道特性下沉到渠道层，不留在公共层做大 `if/else`。
3. core 只承载与渠道无关的 chat 业务流。
4. supervisor 与 core 业务语义分离，避免“常驻进程逻辑”重新污染 chat 业务层。
5. 每次升级都应同步补测试与升级文档，避免结构性漂移。

## 五、升级产出定义

本轮升级完成后，应至少得到：

- 一套清晰的 chat 升级文档
- 一版可工作的常驻 core
- 一条明确的结构收口路线
- 一份可以按日推进的实施计划

## 六、阶段划分

### Phase A：入口表达与本地核心成型

- 飞书 / 微信 / CLI 对话提示词拆分
- `butler-core` / `butler` 第一版打通
- 微信链路稳定

### Phase B：渠道策略对象化

- 拆 `channel_profiles.py`
- 把 prompt / presentation / normalization 分发到渠道 policy

### Phase C：目录与模块边界收口

- 引入 `chat/channels/`
- 引入纯业务 `chat/core/`
- 让 `mainline` 只依赖 channel registry

### Phase D：多用户 / 多入口扩展准备

- 群聊与共享线程语义梳理
- 多渠道统一 session 模型
- 新入口接入模板化

## 七、成功标准

达到以下条件，可视为本轮系统性升级成功：

- 新渠道不需要再改公共 if/else 才能接入
- 微信不会再默认吃飞书表达
- `butler` 本地调用优先走常驻核心
- chat memory / skills / runtime 仍然只有一份真源
- 文档、代码、测试三者一致
