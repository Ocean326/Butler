# chat 功能升级方案

## 一、功能升级目标

本轮功能升级不是“加一个新按钮”，而是提升 chat 的整体产品能力边界。

要解决的核心问题：

- 微信、飞书、CLI 三个入口的体验边界不清
- 本地使用方式还不够像一个长期运行的产品
- 后续扩渠道、扩会话形态时没有稳定骨架

## 二、本轮已显性的功能升级项

### 1. 微信回复去飞书化

目标：

- 微信不再带飞书 Markdown、卡片、附件面板心智

落地：

- 微信对话提示词单独拆分
- 微信正文输出增加纯文本归一化

### 2. 本地 `butler` / `butler-core`

目标：

- 本地使用从“直接起一个一次性 chat 进程”升级为“优先连接常驻核心”

落地：

- `butler-core` 启动核心
- `butler` 优先连接已运行核心，不在则自动拉起

### 3. 常驻 core 自动启动监听服务

目标：

- 核心启动后自动拉起飞书 / 微信相关监听

落地：

- core 默认 channels 为 `cli,feishu,weixin`

## 三、后续应补齐的功能升级

### 功能 1：core 状态页 / 健康页增强

当前已有基础 HTTP health。

后续应补：

- 当前激活渠道
- 各监听线程状态
- 最近错误
- 微信登录状态
- 当前配置版本

### 功能 2：本地 core 管理命令

建议补：

- `butler status`
- `butler stop`
- `butler restart`
- `butler logs`

### 功能 3：统一会话视角

本地、飞书、微信都应可明确看到：

- 当前 session id
- 当前 actor id
- 当前渠道
- 当前模型 / CLI

### 功能 4：渠道级能力差异可见化

例如：

- 微信：纯文本优先
- 飞书：可文件、可图片、可后续补充
- CLI：终端纯文本优先

这些差异不仅要体现在 prompt，还应体现在状态页与运行时诊断中。

### 功能 5：后续新入口接入模板

当要新增 Web Chat / Telegram / 企业微信时，应提供：

- channel 模板目录
- 最小 adapter 协议
- 示例测试

## 四、与多用户对话相关的功能升级

### 当前可做

- 多个用户分别和 Butler 单聊
- 多个 session 并行存在

### 后续要做

- 群聊 / 多 participant 对话模型
- 共享 thread 的角色边界
- 群级记忆与个人记忆分层
- 多入口同一用户映射策略

## 五、功能升级优先级

### P0

- 微信对话正确表达
- 本地 `butler` 优先连常驻 core
- core 自动启飞书 / 微信监听

### P1

- core status / stop / restart
- 渠道级状态可见化
- 统一会话诊断

### P2

- 新渠道接入模板
- 群聊 / 多 participant 能力
- 多租户 / 多工作区变体

## 六、功能升级验收

- 本地 `butler` 使用方式足够稳定
- 微信与飞书的对话体验明显区分
- 常驻 core 能作为统一使用入口
- 后续新入口接入不需要再手工拼大量条件分支
