# chat 代码升级方案

## 一、当前代码现实

当前 `butler_main/chat` 已经具备多入口能力，但代码层仍处于“统一业务流 + 中央渠道分支”的过渡态。

典型表现：

- `channel_profiles.py` 仍承接大量渠道差异
- `mainline.py` 仍显式依赖飞书 / 微信 adapter
- `runtime.py` 仍在公共层直接调用渠道归一化
- `core.py` 已存在，但本质是 supervisor，不是纯业务 core

因此，本轮代码升级重点不是再加功能，而是收边界。

## 二、代码升级目标

### 目标 A：缩小公共层里的渠道知识

要逐步把以下逻辑从公共层移走：

- prompt block 分支
- reply requirement 分支
- output bundle normalization 分支
- presentation / delivery 分支

### 目标 B：建立清晰的模块职责

建议职责切分如下：

- `chat/core/`
  - invocation -> runtime request
  - prompt assembly orchestration
  - execution
  - output bundle
- `chat/channels/<channel>/`
  - input
  - prompt policy
  - presentation policy
  - delivery
  - bootstrap
- `chat/supervisor/` 或保留 `core.py`
  - 常驻服务
  - 入口监听
  - 本地 HTTP / IPC

### 目标 C：保证兼容迁移

升级过程要允许：

- 老接口可继续运行
- 老导入路径先保留 alias
- 逐步替换，不一次切断

## 三、建议改造顺序

### 第一步：把 `channel_profiles.py` 行为逻辑对象化

做法：

- 保留最薄的 `ChannelProfile` 数据结构
- 新建渠道 policy 协议
- 让飞书 / 微信 / CLI 各自实现 policy

收益：

- 风险小
- 直接降低中心化 if/else
- 为后续目录迁移铺路

### 第二步：把 `mainline.py` 的 adapter 依赖改为 registry

做法：

- 引入 channel registry
- `mainline.py` 只按 channel key 获取 input / delivery / presentation 组件

收益：

- 新渠道接入点统一
- 公共层不再直接 import 某个具体渠道

### 第三步：把 `prompting.py` 中的渠道提示逻辑继续下沉

当前已经拆出三份对话提示词资产，但还不是策略对象。

下一步应把：

- 对话资产路径
- reply requirement
- decide 能力
- normalization 要求

统一放入渠道 prompt / presentation policy。

### 第四步：把 `core.py` 与纯业务 core 分离

当前 `core.py` 同时承担：

- 常驻服务
- 启监听
- HTTP 接口
- chat 执行入口

后续应拆成：

- `chat/core/`：纯业务
- `chat/supervisor.py`：常驻与监听
- `chat/core_client.py`：本地客户端

## 四、建议新增的核心抽象

### 1. ChannelRuntime

建议字段：

- `channel_name`
- `input_adapter`
- `prompt_policy`
- `presentation_policy`
- `delivery_adapter`

### 2. ChannelPromptPolicy

建议职责：

- 渠道对话说明
- 渠道 reply requirement
- 是否允许 decide send
- 是否需要文本压缩

### 3. ChannelPresentationPolicy

建议职责：

- OutputBundle 归一化
- Markdown / rich output 降级
- 文件 / 图片 / 卡片处理

### 4. ChannelRegistry

建议职责：

- 注册渠道 runtime
- 统一获取 channel policy / adapter
- 提供默认 channel fallback

## 五、兼容与风险

### 风险 1：结构迁移导致老入口失效

应对：

- 保留 alias
- 先补测试再迁移

### 风险 2：多线程监听中共享状态污染

应对：

- 保持 turn state 为 thread-local
- 不在 channel policy 中放可变共享状态

### 风险 3：常驻 core 与单次 CLI 行为不一致

应对：

- 所有本地客户端都走统一 core API
- 不再保留第二套并行的 chat 本地执行链

## 六、代码升级验收标准

- `channel_profiles.py` 不再承担行为分支中心
- `mainline.py` 不再直接 import 具体渠道 adapter
- 新增渠道可通过 registry 接入
- 常驻 core 与本地 `butler` 行为一致
- 全部相关测试通过
