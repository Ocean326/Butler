# chat 架构升级方案

## 一、目标架构

目标不是“让代码看起来更整齐”，而是形成一套能长期承载多入口 chat 产品的架构。

目标结构建议为：

```text
butler_main/chat/
  core/
    invocation.py
    routing.py
    runtime.py
    execution.py
    output.py
  channels/
    feishu/
      input.py
      prompt_policy.py
      presentation.py
      delivery.py
      bootstrap.py
    weixin/
      input.py
      prompt_policy.py
      presentation.py
      delivery.py
      bootstrap.py
    cli/
      input.py
      prompt_policy.py
      presentation.py
      delivery.py
      bootstrap.py
  registry/
    channels.py
  supervisor.py
  core_client.py
```

## 二、分层职责

### 1. Core 层

只负责：

- invocation -> runtime request
- prompt assembly orchestration
- memory access
- runtime execution
- output bundle

Core 不应知道：

- 飞书卡片怎么发
- 微信怎么降级 Markdown
- CLI 怎么渲染本地终端

### 2. Channel 层

只负责：

- 输入协议转 Invocation
- prompt / presentation / delivery 的渠道特性
- 渠道侧 bootstrap

Channel 不应重复实现：

- 记忆系统
- 技能系统
- 统一 runtime 调度

### 3. Supervisor 层

只负责：

- 启停核心
- 拉起监听
- 提供 health / local API
- 管理本地常驻生命周期

Supervisor 不应承载：

- 渠道表达逻辑
- prompt 规则
- delivery 细节

## 三、当前架构与目标架构差异

### 当前

- `channel_profiles.py` 是中心化渠道策略点
- `mainline.py` 直接 import 飞书 / 微信 adapter
- `core.py` 混合 supervisor 和业务入口
- prompt 资产已拆，但 policy 仍未对象化

### 目标

- channel 行为全部由注册的 policy / runtime 提供
- `mainline.py` 只依赖 registry
- core 纯业务化
- supervisor 单独承载常驻服务语义

## 四、架构升级路线

### Route 1：先策略对象化，再做目录迁移

优点：

- 风险低
- 改动可控
- 回归面较小

缺点：

- 过渡期会有“旧目录 + 新对象”并存

### Route 2：直接搬目录再补对象

优点：

- 结构变化直观

缺点：

- 风险高
- 容易在迁移期引入回归

建议采用 Route 1。

## 五、关键接口建议

### ChannelPromptPolicy

建议方法：

- `build_prompt_block()`
- `build_reply_requirements()`
- `dialogue_asset_path()`

### ChannelPresentationPolicy

建议方法：

- `normalize_output_bundle(bundle)`
- `render_visible_reply(bundle)`

### ChannelRuntime

建议字段：

- `name`
- `input_adapter`
- `prompt_policy`
- `presentation_policy`
- `delivery_adapter`

### ChannelRegistry

建议方法：

- `get(channel_name)`
- `resolve(channel_alias)`
- `default()`

## 六、架构升级收益

### 收益 1：新增渠道成本下降

新增渠道时，不再需要回改：

- `channel_profiles.py`
- `mainline.py`
- `prompting.py`

### 收益 2：渠道体验更清晰

飞书、微信、CLI 可以真正独立演进，而不是共享一个中间态表达。

### 收益 3：多用户、多会话治理更清楚

当后续做群聊、多 participant、多入口统一用户映射时，边界更容易定义。

### 收益 4：core 可服务化

后续若要继续扩成：

- 本地 HTTP API
- Web Chat
- 远程 gateway

不会再被渠道逻辑卡住。

## 七、架构升级成功定义

- core 层没有飞书 / 微信 / CLI 特有表达逻辑
- channels 层不重复实现公共 chat 逻辑
- supervisor 单独存在，不污染业务层
- registry 成为唯一的渠道接入点
