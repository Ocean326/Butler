# 07 工作区再失控复盘与源头治理

日期：2026-03-24
状态：已完成一轮止血；历史脏源仍待清理

## 结论

这次“工作区又乱了”不是因为没人写守则，而是因为守则没有被固化成默认路径策略。

直接乱源有两类：

1. 新上的 skill 管理链路把默认输出硬编码到了 `工作区/skill-*`
2. 测试辅助继续在 repo 根制造 `_tmp_*`

更深层原因有三类：

1. 工作区守则停留在文档，没有统一路径解析层
2. 脚本各自维护 `--output-dir` 默认值，没有经过中心化治理
3. 临时目录只有“建议”，没有统一 tmp 命名空间和测试守护

## 本轮已做的源头修复

### 1. 新增统一路径解析真源

新增：

统一提供：

2. `工作区/Butler/runtime/skills/<skill-name>` 路径

4. 自定义或默认输出目录的统一解析

### 2. skill 运行结果与 skill 内部工作目录已分层

现在默认路径分成两类：

1. 对外运行结果：`工作区/Butler/runtime/skills/<skill-name>`

也就是说：

1. skill 自己的工作目录先约束在 skill 真源旁边
2. 不再把 skill 治理草稿默认写到 `工作区/` 展示层
3. 只有真正需要对外展示或长期汇总的结果，才升格到 `工作区/Butler/...`

并且已经同步更新：

1. 相关脚本
2. `skill_manager_agent` 的 talk prompt

4. skill governance 相关 `SKILL.md`

## 为什么还会“看起来又乱”

因为这轮修的是“继续制造新脏源的代码入口”，不是自动清空所有历史遗留。

目前仍残留两类历史脏源：

1. repo 根现存 `_tmp_*` / `.tmp_*` / 散文件
2. `工作区/` 根层已有的 `skill-*`、`temp_xhs_capture_*`、`memory-curation-smoke`

这些目录是否删除、迁移到 `Recycle`，还是转入正式命名空间，还需要按内容判断，不能无脑扫掉。

此外，live prompt / live docs 中仍有一批 `工作区/temp` 旧约定，代表旧心智还没完全退出。例如：

1. `butler_main/butler_bot_agent/agents/heartbeat-planner-agent.md`
2. `butler_main/butler_bot_code/butler_bot/memory_manager.py`
3. `butler_main/sources/skills/pool/chat/feishu_chat_history/SKILL.md`

这部分会影响更广的行为语义，建议作为下一轮专项治理，而不是在这轮混改。

## 下一步治理建议

### P0：把“允许写哪里”变成硬约束

下一阶段应继续做：

1. talk / orchestrator / heartbeat 分支不再接受自由字符串 `output_dir`
2. 改为 `artifact_slot -> path resolver -> physical path`
3. 默认只允许：
   - `task_workspaces`
   - `工作区/Butler/...`
   - 显式声明的项目目录

不做这一步，其他分支迟早会继续复发。

### P0：补一个工作区污染巡检

建议新增一个低成本巡检脚本或测试，扫描：

1. repo 根是否新出现 `_tmp_*` / `.tmp_*`
2. `工作区/` 根层是否新出现非法的 Butler 运行目录

一旦命中，直接给出 fail 或告警。

### P1：清理历史遗留目录

建议按三类处理：

1. 可删：明显一次性 probe / test / html 抓取残留
2. 可迁：有价值但路径不合规的 `skill-*` 目录，迁到 `工作区/Butler/...`
3. 待审：来源不清、内容不清的目录，先移入 `工作区/Recycle/`

### P1：冻结旧心智入口

后续还应继续收敛：

1. prompt 中“默认写 `工作区/temp`”
2. 文档中“默认写裸 `工作区/<主题>`”
3. 一次性脚本里随手写根层 `_tmp_*`

否则新守则会持续被旧文档带偏。
