## 一句话裁决

Butler 现改为区分“控制根目录”和“Codex 实际执行根目录”：

- `coding_flow` 固定 `execution_context=repo_bound`，继续在项目仓库内执行
- 非 `coding_flow` 默认 `execution_context=isolated`
- 隔离执行根默认落在 `~/.butler/codex_exec_roots/<workflow_id>/`
- `CODEX_HOME` 仍保持 per-flow/per-session 语义，不再把它误当成执行根隔离手段

同日把 Manage Center manager 收口到以下口径：

- manager chat 持久化 `manage_session + draft + pending_action + turns`
- 第一次给出 draft 时不再自动创建 template / flow
- 只有存在待确认动作且用户输入是纯确认时，才允许 `action_ready=true`
- 真正落盘改为走 `draft_payload -> manage_flow()`，不再依赖模型自由文本去决定最终写入
- 首轮纯文本默认绑定当前 manage focus asset；同 session 后续轮次默认沿用 session sticky target，除非显式改 target
- 悬空 `$` / 无效 mention 前缀会在 TUI 提交前清理，避免污染 manager instruction
- manager 非 JSON 回复会原样透传，并把 `parse_status / raw_reply / error_text` 记入 manage turn；parse failed 时不主动清空既有 `pending_action`
- manager chat 继续保持 `Codex-only`；若已有 manager session 的 `resume` 命中典型 resume/timeout/reconnect 故障，会自动改为 fresh Codex exec 重试当前轮，不切 Cursor，且成功后切换到新 `manager_session_id`
- shared asset / instance static asset 新增 `supervisor_profile + run_brief + source_bindings`，并同步编译 `derived/supervisor_knowledge.json`
- manager chat 的后续稳态继续收口为 `skill-style` 机制，而不是继续堆长 prompt：代码侧新增 `manager skill registry`、按需 `references/` 渐进加载、轻量 `asset/session/pending_action` 摘要编译，以及 `draft ownership + action validator`，使职责边界与提交门控更多落在机制层
- Butler Flow Desktop 当前不再只有单一 `FlowSummaryDTO`，而是补齐了 `workspace / single flow / manage center` 所需的 shared surface DTO 与 builder/service，并让 TUI controller 开始复用这层
- `tui/app.py` 的 runtime poll 也同步收口为“只在 flow 主视图触发全量刷新”，避免 `history / settings / manage` 视图切换期间的意外 snapshot 刷新
- Butler Flow Desktop 现补了仓库内直达入口 `tools/flow-desktop`；若默认 `butler_bot.json` 不存在，`butler-flow` 会自动从 `butler_bot.json.example` 生成 machine-local bootstrap config 到 `~/.butler/bootstrap_configs/`，并把 `workspace_root` 指向当前仓库根，避免新 clone 或新机器第一次启动就被缺配置挡住

同日继续把 supervisor 治理收口到以下口径：

- `control_profile` 成为实例级运行治理合同，默认覆盖 `packet_size / evidence_level / gate_cadence / repo_binding_policy`
- `project_loop / managed_flow` 默认 `medium + role_bound`
- `repo_bound` 只表达执行位置，不再自动等于“显式绑定 repo contract”
- repo contract 改为显式绑定：只有进入 `control_profile.repo_contract_paths` 才生效
- supervisor 的控制调整会真实回写 `flow_state.control_profile`
- packet 编译时以实例态控制画像优先，避免资产态与实例态双份分叉
