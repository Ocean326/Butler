# 0319 项目未来构想与落实计划：原子功能 + 业务复用

更新时间：2026-03-19 01:19
时间标签：0319_0119

## 一、目标

本文件回答一个核心问题：未来 Butler 增加多个科研业务、多个项目业务、更多专职 agent 时，应如何继续生长，才能避免继续围绕 `talk`、`heartbeat`、临时任务链路自然堆叠。

本轮主结论：

- `talk` 只是接口层的一种 channel，不再视为系统中心。
- `heartbeat` 只是 loop 驱动与运行时调度机制，不再视为业务本体。
- 后续系统应拆成：原子能力层、编排模式层、业务域层、对外接口层、工作区/产物层。
- 新增科研/项目能力时，优先复用前两层，而不是每个业务各长一套独立 agent 系统。

## 二、总纲：Runtime First

### 2.1 总纲定义

Butler 下一阶段的最高层原则可以收敛成一句话：

> 一个与 agent 解耦的 runtime；agent 是 runtime 上可替换的工人；workflow 是 runtime 上可重组的流程；domain 为这些工人和流程施加业务语义、验收标准与产物定义；interface 只是外部接入口。

这句话对应的工程含义是：

- 系统中心不再是 `talk`，也不再是 `heartbeat`。
- `agent` 不再被视为系统本体，而是 runtime 调度的可替换执行单元。
- `workflow` 不再写死在某一条 prompt 或某一条业务链路里，而是 runtime 可以装配和切换的流程模板。
- `domain` 负责回答“为什么做、做到什么算完成、产物沉淀到哪里”。
- `interface` 只负责输入输出，不承载核心业务判断。

### 2.2 runtime 的核心部件

如果顺着这个原则继续往下拆，runtime 至少应显式包含以下核心部件：

- `task_ledger`
  - 任务真源、状态流转、重试信息、依赖关系
- `workflow_engine`
  - workflow 装配、步骤推进、阶段切换、完成判定
- `scheduler / trigger`
  - 心跳、定时器、事件触发、优先级与并发控制
- `worker_adapter`
  - 统一承接 agent、sub-agent、agent-team 这类执行单元
- `tool_gateway`
  - 工具调用、外部能力接入、限权与审计入口
- `memory_context_service`
  - 任务上下文、短期记忆、长期记忆的读取与回写
- `artifact_store`
  - 文档、实验结果、报告、代码 patch 等产物沉淀
- `event_bus`
  - 发布 / 订阅、回执、广播、状态变更通知
- `gate_policy`
  - 高风险动作审批、预算控制、自动/人工切换策略
- `observer_recovery`
  - 日志、trace、健康检查、失败恢复、断点续跑

一句话说：runtime 不只是一个 loop 壳，而是一套负责“任务、流程、工人、工具、记忆、产物、门控、观测”的运行底座。
## 三、重新定位 `talk` 与 `heartbeat`

### 3.1 `talk`

- 属于接口层。
- 负责对话、汇报、反馈、人类介入。
- 它不是业务层，也不是唯一入口。

### 3.2 `heartbeat`

- 属于运行时 / 编排触发层。
- 负责 tick、循环、巡检、重试、调度、保活。
- 它不是科研 agent，也不是项目 agent。

一句话：`talk` 是入口，`heartbeat` 是引擎，真正的业务应生长在独立的 domain 层。

## 四、目标架构：五层结构

### 4.1 原子能力层（Kernel / Primitives）

建议逐步沉淀为以下原子能力：

- `agent_loop`
- `sub_agent`
- `agent_team`
- `task_ledger`
- `artifact_store`
- `event_bus`
- `memory_read` / `memory_write`
- `tool_call`
- `gate_check`
- `report_emit`

要求：与业务解耦、可重复复用、可被多种编排模式组合。

### 4.2 编排模式层（Orchestration Patterns）

建议先显式定义几类标准编排模式：

- `single_loop`
- `planner_executor`
- `planner_executor_reviewer`
- `manager_worker`
- `pub_sub`
- `watcher_supervisor`
- `human_in_the_loop`

说明：`heartbeat` 只是这些编排模式的触发器或驱动器，不应再与某个具体业务一一绑定。

### 4.3 业务域层（Domains）

建议未来按业务域拆分，而不是按入口拆分：

- `research`
- `project`
- `secretary`
- `governance`

每个 domain 自己定义：任务类型、角色分工、验收标准、产物类型、默认编排模式、是否允许自动推进。

### 4.4 对外接口层（Interfaces）

当前与未来可包含：

- `talk`
- `heartbeat`
- `cli`
- `feishu`
- `api`
- `webhook`
- dashboard

这一层只负责输入事件接入、输出结果投递、权限/身份入口，不直接承载业务逻辑。

### 4.5 工作区 / 产物层（Workspace / Artifacts）

每个科研课题或项目实例都应映射成独立 workspace，内部存放：brief、notes、plans、experiments、code patches、run records、results、reports、decision memos。

一句话：domain 层定义怎么做，workspace 层保存做出了什么。

## 五、未来科研 / 项目能力该如何复用

### 5.1 不是两套系统，而是两类 domain pack

更稳的方式是：

- 共用一套原子能力层
- 共用一套编排模式层
- 在业务域层分化为 `research` 与 `project`

也就是说，科研 agent 和项目 agent 都不是新底座，而是同一底座上的不同 domain pack。

### 5.2 科研业务的推荐形态

科研 domain 初期建议围绕：

- 文献检索
- 文献整理
- baseline 复现
- 代码小改
- 实验监督
- 结果分析
- 汇报与听取意见

推荐优先采用：

- `planner_executor_reviewer`
- `manager_worker`
- `human_in_the_loop`

中后期再逐步扩展到：hypothesis design、method design、ablation planning、paper packaging。

### 5.3 项目业务的推荐形态

项目 domain 可围绕：

- 需求拆解
- 任务编排
- 实现推进
- 质量检查
- 回执与汇报

推荐优先采用：

- `planner_executor`
- `manager_worker`
- `pub_sub`

## 六、统一抽象对象

建议逐步把以下对象升为一等公民：

- `Task`
- `Workflow`
- `AgentRole`
- `Event`
- `Artifact`
- `Workspace`
- `Run`

它们的关系可以概括为：接口层输入事件，编排层选择 workflow，workflow 调用原子能力，domain 层定义任务与验收，workspace 层沉淀结果。

## 七、建议的目录演进方向

```text
butler_main/
  butler_bot_code/
    butler_bot/
      kernel/
      orchestration/
      domains/
        research/
        project/
        secretary/
        governance/
      interfaces/
        talk/
        heartbeat/
        cli/
        feishu/
      runtime/
```

这里最关键的变化是：

- `talk` 从默认中心退到 `interfaces/`
- `heartbeat` 从混合业务退到 `interfaces/ + runtime/`
- `research / project` 进入 `domains/`
- 原子能力与编排模式被单独显式化

## 八、落实计划

### Phase 0：冻结概念

动作：

1. 补这份“原子功能 + 业务复用”方案文档
2. 补“真源矩阵”
3. 补“统一任务协议”
4. 补“编排模式注册表”

验收：新增能力时，先能回答它属于哪一层。

### Phase 1：先抽原子能力

动作：

1. 抽出 `task_ledger` 读写协议
2. 抽出 `artifact_store` 协议
3. 抽出 `event` 与 `report` 协议
4. 抽出 `gate_check` 与高风险动作门控点
5. 抽出最小 `agent_loop`

### Phase 2：把编排模式显式化

动作：

1. 明确 `single_loop`
2. 明确 `planner_executor`
3. 明确 `planner_executor_reviewer`
4. 明确 `manager_worker`
5. 明确 `human_in_the_loop`

### Phase 3：先落地一个 `research` domain

建议的科研 MVP：

1. 课题任务列表
2. 文献检索与整理
3. baseline 复现任务
4. 小步实验代码修改
5. 实验运行监督
6. 结果总结与汇报

验收：能在有人 / 无人两种模式下推进一个科研课题。

### Phase 4：再复制到 `project` domain

动作：

1. 用同样的原子能力和编排模式实现项目域
2. 只替换任务类型、角色和产物定义
3. 不再重造一套任务引擎

验收：`research` 与 `project` 共享底座而不共享混乱。

## 九、边界句

- `talk` 是接口，不是系统中心。
- `heartbeat` 是 loop 驱动，不是业务本体。
- `research`、`project` 是 domain，不是新的底层。
- 先做原子能力与编排复用，再做业务扩张。
- 未来 Butler 应从功能堆叠切换到“内核 + 编排 + domain + 接口”的增长方式。
