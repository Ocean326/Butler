# Butler 工程化目标架构

更新时间：2026-03-15

## 一、目标

从 0315 起，Butler 不再按“能跑就继续堆”推进，而按长期维护系统推进。

目标不是一次性重写，而是把现有真实主链路收口成：

1. 架构边界清晰
2. 协议真源单一
3. 状态真源稳定
4. 重构可回归
5. 长期升级不再反复打散系统

## 二、目标架构分层

### 1. 主机制层

位置：`butler_main/butler_bot_code/butler_bot/` 根目录

只保留主链路入口与总编排：

- `agent.py`
- `butler_bot.py`
- `memory_manager.py`
- `heartbeat_orchestration.py`
- `heartbeat_service_runner.py`
- `governor.py`
- `butler_paths.py`
- `memory_cli.py`

要求：

- 根目录不继续堆业务碎模块
- 这里只负责入口、总编排、生命周期和跨层协调
- 复杂逻辑必须继续外提到职责子目录

### 2. 服务层

位置：`butler_main/butler_bot_code/butler_bot/services/`

职责：

- 任务账本
- 验收回执
- 记忆读写
- 提示词装配
- 请求分析
- 潜意识/总结等可独立服务

要求：

- 以纯服务能力为主
- 少依赖入口层全局状态
- 尽量具备独立测试入口

### 3. 运行时层

位置：`butler_main/butler_bot_code/butler_bot/heartbeat/`

说明：文中“运行时层”语义对应实现目录 `heartbeat/`（承载 CLI 运行、路由与日志等运行时职责）。

职责：

- CLI 执行
- Runtime 路由
- 运行日志

要求：

- 把“选哪个执行器”和“怎么调用执行器”收口
- 高权限执行参数必须可治理、可审计

### 4. 协作执行层

位置：`butler_main/butler_bot_code/butler_bot/execution/`

职责：

- sub-agent / team 执行
- 团队协作的串行/并行控制

要求：

- 协作失败不能拖垮全局主链路
- 并行异常必须结构化回执

### 5. 注册与资产层

位置：`butler_main/butler_bot_code/butler_bot/registry/`

职责：

- skills
- sub-agents
- teams
- 公共能力目录

要求：

- 只做注册、发现、渲染
- 不承担运行时执行副作用

### 6. 工程规范层

位置：`butler_main/butler_bot_code/butler_bot/standards/`

说明：文中后续如出现 `standards/`，均指该目录（不是仓库根目录）。

职责：

- 架构清单
- 长期有效协议
- 规范性 prompt 真源

这是 0315 之后必须新增的一层。它的意义是：

以后 Butler 的任务协作、自我更新、自我认识协议，不再散落在 `memory_manager.py`、`agent.py`、`heartbeat_orchestration.py` 的硬编码字符串里，而是有单一真源。

### 7. 过时归档层

位置：`butler_main/butler_bot_code/butler_bot/obsolete/`、仓库根 `过时/`

职责：

- 隔离已经退出主链路的旧代码
- 保留参考价值但不再参与当前系统认知

要求：

- 旧链路不再被 README、测试默认套件、观察脚本视为现役组件

## 三、长期稳定的三套协议

### 1. 任务协作协议

目标：

- 对话、heartbeat、self_mind 不再维护三套任务池
- `task_ledger.json` 成为执行状态真源
- 任务工作区承担可读投影、过程记录和最终报告

最低要求：

- 新任务必须有目标、边界、完成判据、阻塞
- 执行结果必须能回写验收摘要和过程角色

### 2. 自我更新协作协议

目标：

- 让 Butler 可以持续升级，但不以牺牲稳定性为代价

最低要求：

- 升级先出方案，再审批，再改动，再验证
- 身体层高风险改动必须经过统一 upgrade / governor 闸门
- 升级必须留下回执、风险与回滚点

### 3. 自我认识协作协议

目标：

- 让 self_mind 成为长期稳定的认知层，而不是漂移的第二调度中心

最低要求：

- self_mind 负责认知和续思，不直接替代执行系统
- 认知线索、执行任务、观察性线索必须分流
- 可执行事项交接到 heartbeat / task ledger

## 四、长期架构约束

1. 根目录仅保留主机制代码。
2. 协议文本必须有单一真源。
3. 高风险升级不得绕过 governor。
4. `task_ledger` 是执行状态真源。
5. self_mind 不维护平行执行系统。
6. 每次重构必须同步更新测试与文档。

## 五、0315 之后的主改造顺序

1. 先建立标准层与协议真源。
2. 再把散落在 prompt 里的协议逐步收口到标准层。
3. 再拆 `memory_manager.py` 的重职责区块。
4. 再收观测、CI、依赖锁、跨平台启动等工程化外围。

## 六、当前结论

Butler 现阶段不该再继续靠“多加几段 prompt”和“在大文件里补几百行函数”推进。

0315 后的正确方向是：

先把架构与协议变成可维护真源，再在这个骨架上持续拆重、补稳、补测试。

## 七、外部 Agent 架构对照（2026-03-17 小步更新）

### 1. 外部大厂 / 开源 Agent 架构常见共性

- **LLM 作为“决策/规划内核”，外围显式分层**  
  - 通常分为：前台入口（对话/HTTP）、调度与任务编排层（planner/supervisor）、执行层（tool runtime / worker）、状态与记忆层（task store / memory store）、观测与治理层（telemetry/governor）。
- **统一的任务与状态真源**  
  - 统一 task/flow/graph 结构，作为执行与回执的唯一 machine-readable 真源，对话/文档只是镜像与说明。
- **工具与子 Agent 注册中心**  
  - 工具、子 Agent、Team 通过 registry/catalog 暴露能力与协议，而不是散落在业务脚本里，支持可发现、可治理、可观测。
- **显式的执行循环与角色分工**  
  - 以“planner → executor(s) → evaluator/critic → accepter/summarizer”的循环为主流，支持多轮、分阶段推进，而不是单轮大 prompt。
- **多层记忆与检索体系**  
  - recent context / short-term memory / long-term knowledge / workflow state 分层；LLM 在每次决策前通过检索装配上下文，而不是一次性塞满。
- **治理与安全边界**  
  - 高风险动作通过 governor / approval flow 控制；自升级、自修改路径与普通工具调用分离。

### 2. Butler 已经对齐的关键点（对照当前设计）

- **分层架构与角色分工已成型**  
  - 已有前台入口层（飞书 Bot + `request_intake_service.py`）、调度与经理层（`heartbeat_orchestration.py` + `runtime_router.py` + `task_ledger_service.py` + `acceptance_service.py`）、执行层（team executor / runtime）、记忆与索引层（`memory_manager.py` + `memory_backend.py` + `local_memory_index_service.py`）、运行与监视层（`manager.ps1` / `guardian` / `watch_stack.ps1`），整体形态与外部“入口 / 调度 / 执行 / 记忆 / 观测”分层高度接近。
- **统一任务真源：task_ledger 已上线并被心跳接入**  
  - `task_ledger_service.py` 已作为任务与状态真源落地，heartbeat `apply_plan()` 已改为通过 task ledger 落账，符合“任务真源单一”的共性。
- **执行循环中已有 planner / executor / evaluator / acceptance 角色**  
  - 通过 `process_role`、`runtime_profile`、心跳中的计划与验收回执，已经形成 planner（经理）、executor（执行者）、test/evaluator（评估）、acceptance（收口）的结构化分工，而不是单轮大 prompt。
- **记忆层从单文件走向 backend + index 分层，并引入离线 heartbeat**  
  - 统一 recent、多流 companion、local memory 的 L0/L1/L2 设计，加上 heartbeat 作为“离线巩固/自我整理”机制，与外部“短期/长期/离线巩固”多层记忆架构方向一致。
- **自升级与高风险变更已经引入 governor 与升级通道约束**  
  - `governor.py`、`heartbeat_upgrade_request.json` 路径及工程化目标架构中的“自我更新协作协议”，对齐了外部 Agent 项目中对自升级/高权限动作的显式闸门设计。

### 3. Butler 仍有差距或需要演进的方向

- **planner/supervisor 服务的物理解耦与职责收口仍在路上**  
  - 当前 `HeartbeatOrchestrator` 仍偏重，`planner_service.py / supervisor_service.py` 尚未真正独立成服务层模块；与外部常见的“独立 planner/supervisor 服务”相比，还需要继续拆分与瘦身 orchestrator，使其更像 facade 而非大总控。
- **标准层与协议真源仍在迁移过程中**  
  - 0315 之后已明确引入 `standards/` 作为协议真源，但大量历史 prompt/协议仍散落在核心代码与 Soul/role 中，需要按“先建新真源、再迁移旧协议”的节奏持续收口，向外部“大厂统一协议中心 + 运行时按需装载”的模式靠拢。
- **记忆与任务在更高层语义上的对齐与可视化不足**  
  - 相比于部分 Agent 平台提供的“图形化 workflow + 语义记忆浏览器”，当前 Butler 在任务/记忆的可视化、跨层追踪与 schema 级治理上仍偏工程文档 + 索引，需要后续在标准层与工作区工具上补充更直观的视图。
- **多 Agent / 多 worker 协作模式还未完全标准化**  
  - 已有 team executor 与 capability registry，但与外部项目中“统一的 multi-agent 协作协议 + 角色模板库”相比，仍需在 `butler_main/butler_bot_code/butler_bot/registry/` 与 `butler_main/butler_bot_code/butler_bot/execution/` 层抽象更稳定的 Team/role 协议，减少一次性 Team prompt。

### 4. 本次小步整理的定位

- **范围**：仅更新工程化目标架构文档中的“外部架构对照”认知，不做代码级重构，不调整现有运行时行为。
- **用途**：为后续拆分 `HeartbeatOrchestrator`、完善 `standards/` 协议层、强化 task_ledger/记忆的可视化与治理提供对照视角，避免在局部重构时偏离外部主流 Agent 架构演化路径。

### 5. Agent 架构共性综述（可复用原则增量）

下面原则用于“后续任何重构/新加协议/新增服务”时的快速自检，目标是把共性从“描述”变成“可复用决策规则”（原则本身可直接复制到团队标准层）。

1. 分层原则：把系统拆为 `入口/调度/执行/状态(记忆)/观测与治理`，并为每层明确输入输出与责任边界；不在层间混用“真源”和“镜像”。
2. 真源原则：为每个关键对象选定唯一 machine-readable 真源（例如 `task/run/workflow` 的状态），对话/文档只能是投影或说明，不能替代真源。
3. 角色分工原则：用“planner（计划/选择）→ executor（执行）→ evaluator（评估/批判）→ accepter（验收/收口）”表达演化循环；不要只用单轮大 prompt。
4. 执行器抽象原则：用 `worker` 抽象执行器，把 `agent` 视为“带角色语义的 worker 特化”，而不是把 agent 当成系统中心。
5. 能力注册原则：工具与子能力通过 `registry/catalog` 暴露能力与协议（包含可治理的元信息），避免把发现逻辑散落在业务脚本或 prompt 中。
6. 记忆装配原则：把记忆分层为“检索前 context / 短期记忆 / 长期知识 / workflow state”，检索发生在每次决策前而不是一次性塞满。
7. 风险治理原则：把高风险动作（写代码/自升级/敏感工具）从普通工具调用中隔离出来，使用 `governor/approval flow` 做强约束。
8. 可观测原则：把 tracing 当作基础设施而不是事后补丁；至少覆盖 run/workflow 选择、worker 调用、工具调用、失败与恢复点。
9. 恢复与复试原则：把“暂停等待人工/失败重试/从断点恢复”视为流程的一部分，并让这些状态可落盘、可审计。
10. 协作协议原则：多 agent / 多 worker 协作必须有“可落地的最小协议”（队伍/角色模板/handoff/并发异常如何回执），减少临时拼 prompt 的不可控性。

- 将“外部共性”从概括条目扩展为 10 条可复用原则（可直接搬到 standards/）。
- 将 Butler 差距方向从 4 条扩展为 12 条对照条目（覆盖真源 schema 治理、tracing 事件集、恢复点协议、协作协议等）。

### 7. 下一步还差什么（待补清单，5 项）

1. 继续把 planner/supervisor 从 `HeartbeatOrchestrator` 的总控责任里拆出来，形成明确服务接口与落盘产物契约。
2. 在 standards/ 中固化最小对象协议：`run/workflow/worker/context/trace/artifact` 的字段与约束（至少做到可校验）。
3. 给 task ledger 真源补上 schema 级可视化索引与回放入口，让状态能被后续任务引用（而不仅是人工阅读）。
4. 统一 evaluator/critic 与 accepter 的输出/准入条件，形成可自动化验收的规则集。
5. 将多 agent 协作的最小协议写进 standards：handoff 证据字段、并发异常回执格式、产物契约。
