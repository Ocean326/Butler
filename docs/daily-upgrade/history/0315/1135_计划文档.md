# 0315 计划文档

更新时间：2026-03-15

## 一、当前判断

到 2026-03-15，Butler 的状态已经和 0314 升级前不同，今天的重点不应再是继续堆新链路，而是把已经接上的链路收口成稳定、可控、可验证的运行结构。

结合当前代码与现状文档，可以先明确三点：

1. `guardian` 已不再是 Butler 运行时的必要前提，主对话进程已经接管 `heartbeat` 与 `self_mind` 的启动、看护与部分重启控制。
2. 0314 提出的 heartbeat planner 汇报协议、planner prompt 真源、任务分组执行思路已经进入现有结构，不再是纯想法。
3. 当前最大问题已经从“有没有新能力”转为“结构是否稳定、真源是否统一、旧代码和旧文档是否继续制造混乱”。

所以，0315 的主方向不是盲目全面重写，而是：

- 先收口运行真源
- 再压缩历史残留和结构混乱
- 最后判断哪些模块值得继续拆、哪些模块需要局部重写

## 二、当前真实现状

### 1. 运行架构现状

- 主入口：`butler_main/butler_bot_code/butler_bot/butler_bot.py`
- 后台控制与生命周期：`butler_main/butler_bot_code/butler_bot/memory_manager.py`
- heartbeat planner prompt 真源：`butler_main/butler_bot_agent/agents/heartbeat-planner-prompt.md`

现状上已经形成：

- `talk` 负责接收用户消息与显式后台控制命令
- `heartbeat` 由主进程直接拉起 sidecar，并由主进程 watchdog 看护
- `self_mind` 由主进程内线程维护
- 用户可以直接通过对话发送 `重启后台 / 重启心跳 / 重启意识循环 / 停止心跳` 等指令

这意味着 0315 的计划重点应从“去 guardian”切换为“把新主链路打磨稳定”。

### 2. 任务、汇报与记忆现状

0314 文档里仍然有效、且应继续沿用的部分包括：

- 任务入口优先收口到 `heartbeat_tasks.md` 与分类看板，而不是新增临时入口
- planner 负责产出 `task_groups / branches`，执行器按组串行、组内并行推进
- `user_message` 用于 heartbeat 窗口状态说明
- `tell_user_candidate` 只保留值得打到主对话窗的候选意图，不等于最终用户文案
- recent -> summary -> local memory 的沉淀链路已经存在，不宜再新增平行记忆机制

换句话说，0315 不应该再重新发明一套任务协议，而应基于 0314 已形成的协议继续收口。

### 3. 当前仍然存在的核心问题

1. `memory_manager.py`、`heartbeat_orchestration.py` 仍然过重，运行控制、心跳调度、记忆维护、审批与发送链路耦合较深。
2. 安全稳定运行机制虽然比以前前进一步，但还没有完全形成统一真源，尤其是 watchdog、状态展示、脚本状态识别、残留 PID/旧状态这类问题还会制造误判。
3. 旧代码、旧文档、旧路径残留较多，已经开始反过来影响判断，属于认知噪音来源。
4. 部分能力已经落地，但对应的“现状文档 / README / 真源说明”还没有完全同步，后续很容易再次进入“代码是真的，文档是旧的”状态。

## 三、0315 核心目标

### 目标 A：把后台运行真源收口到单主进程架构

目标不是继续依赖 `guardian` 补洞，而是明确 Butler 当前的真实后台结构就是：

- `talk` 主进程
- `heartbeat` sidecar
- `self_mind` 主进程内循环

围绕这个结构，把启动、停止、重启、状态识别、失败回执、日志观察都统一到同一套真源上。

### 目标 B：把 heartbeat 从“能跑”推进到“稳定可控”

重点不是继续加功能，而是验证并稳定以下链路：

- planner 的运行时选择是否稳定
- `user_message / tell_user_candidate` 协议是否真的在工作
- 分支执行、失败复试、阶段汇报、完成收口是否形成闭环
- 连续多轮 heartbeat 下是否还能保持稳定，不靠手动补救

### 目标 C：把 0314 的任务与记忆协议继续沉到底层真源

今天不再另起一套任务系统，而是继续收口现有协议：

- 任务入口收口到任务看板与账本
- 汇报入口区分 heartbeat 窗口与主对话窗口
- 记忆入口继续沿用 recent / summary / local memory 的既有机制
- 升级与重启继续走审批链，不让后台自行做高风险改动

### 目标 D：开始结构性减负，而不是无边界重写

原始判断“应进行一次全面重构，必要时重写”方向没错，但今天更合理的落法应是：

1. 先收口真源
2. 再拆重模块
3. 最后只对真正阻塞稳定性的模块做局部重写

避免一边运行、一边大面积重写，导致再次进入不可控状态。

同时增加一条明确约束：

- 代码目录要边减负边梳理，逐步结束当前过于扁平的平铺结构
- `butler_bot_code/butler_bot/` 根目录只保留主机制代码
- 其余按职责归类到子目录，如 `services / runtime / registry / execution / utils / obsolete`
- 归类时必须同步修正导入与 README，不能只挪文件不收口真源

## 四、0315 执行计划

### P0. 后台运行与控制链收口

本优先级最高，先解决“系统到底怎么活着”的问题。

- 以主进程接管后的结构为准，确认 `heartbeat` 与 `self_mind` 的启动、停止、重启入口
- 继续淡化 `guardian` 在现状文档和控制逻辑中的主地位，只保留历史说明或兼容痕迹
- 收口 `manager.ps1`、watchdog 状态、PID 文件、日志状态之间的关系，减少“实际已运行但状态展示滞后”的错觉
- 明确以后后台控制的真入口优先是对话命令与主进程生命周期，而不是外部守护绕行

### P1. heartbeat planner / executor / 汇报协议稳定化

- 保持 0314 已确定的 planner prompt 真源，不再回退到散乱规则
- 延续“开始执行 / 中途确认 / 完成收口 / 按需汇报任务列表”的协议
- 继续区分“用户直接任务”和“planner 为完成目标拆出的内部执行任务”
- 验证 planner 输出与 executor 执行之间的 branch contract 是否足够稳定，减少靠兜底补齐字段的情况

### P2. 任务真源与记忆真源继续统一

- 任务继续优先进入任务看板与 task ledger，不新增临时平行入口
- 记忆继续沿用 recent、summary、local memory 写入流水，不另造新层
- 升级请求继续写入审批文件，由主对话进程在用户批准后执行，不把高风险动作留给后台自行决定

### P3. 代码与文档减负

- 标记哪些文档仍是当前真源，哪些只是历史设计或阶段记录
- 清理“代码已经切走，但文档仍在暗示旧架构”的残留说明
- 将 `butler_bot_code/butler_bot/` 的扁平模块按职责归类，根目录只保留主机制代码
- 为后续拆分 `memory_manager`、`heartbeat_orchestration` 做边界准备，但今天不追求一次拆完

## 五、今天的验收口径

满足以下条件，可视为 0315 方向判断与计划收口正确：

1. 对当前后台结构的描述已经统一为“主进程 + heartbeat sidecar + self_mind”，不再把 `guardian` 写成必要主链路。
2. 0314 的任务、汇报、记忆协议被明确继承下来，而不是被新的临时说法覆盖。
3. 计划重点已经从“继续堆功能”转向“稳定运行、统一真源、减少混乱”。
4. “全面重构 / 必要时重写”被收敛成有顺序的工程策略，而不是泛泛口号。

## 六、0315 一句话结论

0315 的正确方向不是继续扩张链路，而是基于“guardian 已下线、主进程已接管后台”的新现状，把运行控制、任务协议、汇报机制与记忆真源真正收口，然后再决定局部拆分与重写。
