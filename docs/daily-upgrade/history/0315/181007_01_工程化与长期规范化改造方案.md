# 2026-03-15 工程化与长期规范化改造方案

## 一、本轮目标

根据 `1350_现状文档` 和 `1423_deep-research-report`，本轮不再继续做零散修补，而是落实第一批工程化基础设施：

1. 给 Butler 明确目标架构
2. 给任务 / 自我更新 / 自我认识建立长期协议真源
3. 让协议开始脱离硬编码散落状态
4. 顺手处理一批已知稳定性问题

## 二、分阶段改造方案

### Phase 1：标准层落地

范围：

- 新增 `standards/`
- 建立架构清单
- 建立协议注册表
- 把长期协议以独立文件维护

验收：

- 协议可以从单一入口读取
- prompt 构造层开始引用标准层，而不是继续复制文本

### Phase 2：主链路接线

范围：

- `agent.py`
- `heartbeat_orchestration.py`

动作：

- 维护协议改为由协议注册表提供
- 任务协作、自我更新、自我认识协议开始进入真实 prompt 链路

验收：

- maintenance / heartbeat 执行链路能看到统一协议块
- 协议后续修改不需要再同时改多个硬编码副本

### Phase 3：稳定性补强

范围：

- `execution/agent_team_executor.py`
- `services/task_ledger_service.py`

动作：

- team 并行成员异常不再拖垮整个 team
- task ledger 改为原子写，并增加备份回退读取

验收：

- 并行成员抛异常时仍返回结构化 team report
- ledger 主文件损坏时可回退到最近备份读取

### Phase 4：重模块继续拆分

范围：

- `memory_manager.py`

优先拆分：

1. self_mind loop
2. upgrade request / governor flow
3. delivery / reply flow

说明：

这一步风险最高，不适合在协议和稳定性真源都还没建立前直接大拆。

## 三、本轮已落实的第一批改造

### 1. 工程规范层已建立

新增：

- `butler_bot/standards/architecture_manifest.py`
- `butler_bot/standards/protocol_registry.py`
- `butler_bot/standards/protocols/*.md`

意义：

- 协议开始从散落 prompt 硬编码转为单一真源
- 架构边界开始被显式记录，而不是只靠口头约束

### 2. 协议已开始接入主链路

当前已接入：

- `agent.py`
- `heartbeat_orchestration.py`

已经纳入真源管理的协议：

- 任务协作协议
- heartbeat 执行协议
- 统一维护入口协议
- 自我更新协作协议
- 自我认识协作协议

### 3. 稳定性第一批补强已落实

已处理：

- team 并行成员异常结构化兜底
- task ledger 原子写与备份读取

### 4. memory_manager 第一层拆分已落实

本轮不是直接重写 `memory_manager.py`，而是先把三块高耦合职责外提成服务，再由主类兼容委托：

- `services/self_mind_runtime_service.py`
  - 承接 self_mind loop 启动、重启、循环调度
- `services/upgrade_governance_service.py`
  - 承接升级申请标准化、审批判断、governor 写入闸门
- `services/message_delivery_service.py`
  - 承接 Feishu 私聊发送与启动通知发送

当前效果：

- `memory_manager.py` 仍保持对外接口稳定
- 现有测试不需要大规模改写
- self_mind loop / 升级治理 / 发送链路已经从主类实现里抽出第一层

这一步的目标不是“拆完”，而是先建立可持续继续拆的服务边界。

### 5. memory_manager 第二层拆分已落实

在第一层拆分基础上，本轮继续把 self_mind 的状态与 prompt 相关职责往服务层下沉：

- `services/self_mind_state_service.py`
  - 承接 self_mind 路径解析、状态读写、bridge 读写、talk target 与 delivery override
- `services/self_mind_prompt_service.py`
  - 承接 self_mind cycle prompt 和 heartbeat handoff receipt 文本组装

当前效果：

- `memory_manager.py` 对 self_mind 的路径/状态/prompt 已开始从“直接实现”切到“服务委托”
- self_mind 相关长字符串 prompt 不再继续堆在总编排类里
- 后续继续拆 cognition、bridge、context refresh 时，已经有了稳定落点

### 6. 代码健康约束已进入标准层

本轮新增：

- `standards/code_health_manifest.py`
- `tests/test_code_health.py`
- `docs/concepts/代码健康约束_20260315.md`

已经落地的健康约束包括：

- `services/` / `runtime/` / `execution/` 不允许反向依赖 `memory_manager.py`
- 关键新模块有文件体量预算
- `runtime/cli_runtime.py -> memory_manager.py` 的反向依赖已切断，改为独立 `runtime/cursor_runtime_support.py`

这部分的意义是：

后续工程化不只靠“继续拆文件”，而是靠自动化健康检查持续约束独立性和可维护性。

## 四、接下来要继续做的事

1. 补 `memory_manager.py` 的拆分设计图与真实切分。
2. 继续把 self_mind cognition / bridge / context refresh 从 `memory_manager.py` 拆出。
3. 继续把更多散落 prompt 协议迁入 `standards/`。
4. 建立依赖锁、CI、覆盖率门禁。
5. 收口 runtime 高权限参数治理。
6. 继续清理文档入口与 legacy 残留。

## 五、当前判断

这一轮不是最终完成，而是 Butler 从“经验驱动的脚本系统”转向“可长期维护系统”的第一阶段起点。

只要标准层和状态真源持续收口，后续拆 `memory_manager.py`、补 CI、补安全治理时，系统就不会再反复失控。
