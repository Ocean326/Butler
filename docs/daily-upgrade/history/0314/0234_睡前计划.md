# 0314 睡前升级计划

更新时间：2026-03-15

## 本轮目标

1. 废弃 guardian 作为主调度依赖，收口到单主进程架构：`talk + heartbeat + self_mind`。
2. 升级 heartbeat planner 的“直接面向用户汇报”机制：
   - 任务开始执行时可汇报
   - 中途遇到不确定细节、风险或需要决策时可汇报
   - 任务完成或阶段完成时可汇报
   - 在开始执行与完成收口节点，支持按需汇报当前任务列表
3. 任务列表在表达上区分两类：
   - 用户直接添加/明确交办的任务
   - planner 为达成目标而拆出的内部执行任务
4. 汇报逻辑不做死规则硬编码，而是通过 planner role / prompt 协议 + 稳定 `user_message` / `tell_user_candidate` 机制完成。
5. 心跳 planner 运行时切换到 `codex + gpt-5.2`，并补齐代理配置；其它执行链默认继续走 `cursor`。

## 执行方案

### A. 运行时与模型

- 新增 heartbeat planner 独立 `planner_cli` 配置，默认使用 `codex`
- `planner_model` 固定为 `gpt-5.2`
- `codex` provider 打开，并补 `HTTP_PROXY / HTTPS_PROXY`
- executor 继续默认 `cursor`

### B. planner 真源与汇报协议

- 补齐 `heartbeat-planner-prompt.md` 作为 planner prompt 真源
- 在 prompt 中明确：
  - `user_message` 是 heartbeat 窗口状态说明，默认由 planner 主动组织
  - `tell_user_candidate` 是主对话候选意图，不是最终文案
  - 对“开始执行 / 中途需确认 / 完成收口 / 任务列表”做协议化要求
  - 任务列表按“用户任务 / planner 拆分任务”两类表达

### C. 上线与验证

- 验证 planner 运行态已不再被硬编码为 `cursor`
- 验证 `planner_cli=codex`、`planner_model=gpt-5.2` 能进入运行态
- 验证 prompt 真源文件存在且可被 heartbeat planner 装载
- 重启主进程并核查状态日志

## 本轮已落地

- 已补 planner 独立 CLI 选择，不再硬编码 `cursor`
- 已把 heartbeat planner 配置切到 `codex / gpt-5.2`
- 已给 codex provider 补代理配置
- 已新建 planner prompt 真源文件并写入 0314 汇报协议

## 验收口径

满足以下条件即视为本轮升级完成：

1. 运行态日志/状态能显示 planner 使用 `codex`
2. planner 默认模型为 `gpt-5.2`
3. executor 默认仍为 `cursor`
4. planner prompt 真源文件存在并进入装载链
5. heartbeat planner 的提示词中已经包含“开始/确认/完成/任务列表”汇报协议
