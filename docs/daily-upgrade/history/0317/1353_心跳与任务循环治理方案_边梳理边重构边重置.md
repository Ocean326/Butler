# 0317 心跳与任务循环治理方案：边梳理、边重构、边重置

> 更新时间：2026-03-17
>
> 本文不是现状报告，而是基于 0317 现状，给出一套可持续推进的治理方案。目标不是“先忍着跑”，而是把 heartbeat 从脏运行态、脏任务池、脏调度偏置中拉回到可长期维护的稳定系统。

## 1. 目标

这轮治理同时追三个结果：

1. 心跳状态可被真实判断，不能再靠猜
2. 任务真源收敛成少量可维护入口，不能再持续污染
3. 后台默认行为回到“稳态维护”，而不是“自动找事做”

对应策略不是一次性大翻修，而是：

1. 边梳理：先把真边界、真状态、真数据源画清楚
2. 边重构：只拆最大脏点，不在脏地基上继续补丁
3. 边重置：对已经失真的状态文件、任务文件、账本理由做定向清零和归档

---

## 2. 治理总原则

### 2.1 运行态只允许一个真相源

当前 `heartbeat_run_state.json`、`heartbeat_watchdog_state.json`、PID 文件、真实进程互相分叉，这是第一故障源。

后续必须收敛为：

1. `heartbeat_watchdog_state.json` 负责“进程级健康状态”
2. `heartbeat_run_state.json` 负责“单轮执行阶段状态”
3. PID 文件只作为辅助，不单独代表存活
4. `manager.ps1 status` 只展示经代码校验后的真实状态，不直接复述静态归属语

### 2.2 任务系统只允许显式任务驱动

heartbeat 不再把“想到、聊到、顺手想探索”默认视为应执行任务。

后续真源只保留：

1. 工作任务
2. 明确到期的长期/定时任务
3. 少量明确授权的治理任务

以下内容不再作为默认调度真源：

1. 大量 self_mind 式主动起念
2. 对话原文直接入板
3. 无明确完成条件的探索类条目
4. 运行临时产物和调试文件

### 2.3 重置不是删库，而是“可追溯地清空脏状态”

这轮需要删除和归档的不是用户真实资产，而是：

1. 已失真的运行时状态
2. 重复、陈旧、无法执行的任务项
3. 仅用于兼容、已经造成噪音的旧镜像和旧逻辑

所有重置动作都应满足：

1. 可重建
2. 可回溯
3. 不误删用户业务内容

## 3. 先梳理：把边界收清

### 3.1 运行链路边界

从代码看，当前 heartbeat 主链路仍集中在 [`memory_manager.py`](/C:/Users/Lenovo/Desktop/Butler/butler_main/butler_bot_code/butler_bot/memory_manager.py)：

1. `_use_external_heartbeat_process()` 当前直接返回 `False`，说明系统口径上说 sidecar，但主实现仍偏向内嵌总控
2. `_heartbeat_process_watchdog_loop()` 同时处理存活检测、崩溃恢复、cooldown
3. `_heartbeat_loop()` 负责周期调度
4. `_run_heartbeat_once()` 负责单轮 plan/execute/apply
5. `run_heartbeat_service_subprocess()` 又单独处理 sidecar 单例锁、PID 和首次启动

结论：

1. `memory_manager.py` 现在承担了过多角色，是当前最大脏点
2. sidecar 模式和内嵌模式的边界并不硬，后续必须统一

### 3.2 状态展示边界

[`manager.ps1`](/C:/Users/Lenovo/Desktop/Butler/butler_main/butler_bot_code/manager.ps1#L376) 的 `Start-HeartbeatSidecar` 仍直接输出“现由 butler 主进程内置拉起与看护”，这只是归属表述，不是健康状态。

后续 `status` 输出必须拆成：

1. 配置状态：`enabled / disabled`
2. 进程状态：`running / stopped / stale / crashed`
3. watchdog 状态：`running / restarting / cooldown / degraded`
4. 单轮状态：`idle / planning / executing / apply / failed`

### 3.3 任务真源边界

当前任务主目录是 [`heartbeat_tasks/`](/C:/Users/Lenovo/Desktop/Butler/butler_main/butler_bot_agent/agents/local_memory/heartbeat_tasks)，但目录内部已经混入：

1. 运行临时文件 `03_定时.md.38800.32860.tmp`
2. 超大流水 `task_change_log.jsonl`
3. 大量 conversation 原文
4. 大量 self_mind 式主动起念任务

后续必须明确：

1. `heartbeat_tasks/` 只放正式看板文件
2. `_quarantine/` 只放待人工复核的损坏/污染文件
3. 大流水不再长期裸放在调度目录
4. 运行时临时文件禁止写入任务真源目录

## 4. 边重置：先做一次硬清场

这一步不是“优化”，而是恢复系统可判断性的必要前置。

### 4.1 重置运行态

建议第一批直接执行的动作：

1. 删除失真的 `run/heartbeat_run_state.json`
2. 删除失真的 `run/heartbeat_watchdog_state.json`
3. 删除 `run/butler_bot_heartbeat.pid`
4. 删除 `run/heartbeat_service.lock`
5. 在重启 heartbeat 前重新生成干净状态

执行前提：

1. 先确认旧 heartbeat 进程已不存在
2. 若存在真实存活进程，先按 PID 和命令行交叉核验，再停进程

重置后新规则：

1. `heartbeat_watchdog_state.json` 缺失时视为 `unknown`，不是 `running`
2. `heartbeat_run_state.json` 超时未更新且 PID 不活时，自动标记 `stale`
3. 任何“跳过启动”必须先校验 PID 对应命令行是否真是本 heartbeat 进程

### 4.2 重置任务池

建议第一批直接执行的动作：

1. 将 `03_定时.md.38800.32860.tmp` 移入 `_quarantine/`
2. 将 `task_change_log.jsonl` 按日期切分归档到单独 `archives/` 或 `logs/heartbeat_tasks/`
3. 对 `02_主动想干.md` 做去重，只保留最近一条有效条目
4. 对 `05_整理清洁.md` 清除“整段对话原文式任务”
5. 对 `06_闲时无聊.md` 做冻结，不再作为默认 planner 输入

重置判定规则：

1. 连续 7 天未被触发且无明确 DDL 的探索任务，改 `obsolete`
2. 与现有 task 语义重复的，只保留一条主任务，其他并入 `source_refs`
3. 明显是聊天原话、汇报文本、执行回声的，不作为 task 保存
4. 没有完成条件的自由感想，不进 heartbeat 看板

### 4.3 重置 task ledger

[`task_ledger.json`](/C:/Users/Lenovo/Desktop/Butler/butler_main/butler_bot_agent/agents/state/task_ledger.json) 当前核心问题不是字段缺失，而是 `manager_state.reason` 长期写入同一种偏置话术。

建议第一批动作：

1. 对所有 `done / obsolete / duplicate` 任务移出活跃账本，归档到独立 history 文件
2. 清空历史条目里模板化的 `manager_state.reason`
3. 活跃任务只保留最近一次真实调度原因
4. `reason` 改为结构化字段，不再长句复写 ideology

建议新结构最少包含：

1. `decision_basis.has_explicit_task`
2. `decision_basis.has_due_schedule`
3. `decision_basis.user_online`
4. `decision_basis.allow_autonomous_explore`
5. `decision_basis.fallback_source`

## 5. 边重构：把脏点拆掉

### 5.1 重构运行态服务，拆出 HeartbeatRuntimeStateService

建议把下面职责从 `memory_manager.py` 拆出去：

1. PID 文件读写
2. `heartbeat_run_state.json` 读写
3. `heartbeat_watchdog_state.json` 读写
4. stale 判定
5. cooldown 判定
6. 启动前自清理

目标：

1. `memory_manager.py` 不再直接散写状态文件
2. 所有状态写入经过统一校验和状态迁移规则
3. `status`、watchdog、sidecar 入口都复用同一套状态判定

### 5.2 统一 heartbeat 模式，只保留 sidecar 真模式

当前 `_use_external_heartbeat_process()` 返回 `False`，但代码里又保留 sidecar 子进程入口，属于“口径和实现都半截”。

长期建议：

1. 明确 heartbeat 只以 sidecar 运行
2. 主 talk 进程不再直接承担 heartbeat loop
3. 主进程只负责：
   - 启动授权
   - 停止授权
   - 健康查看
   - 重启申请

这样可以直接删除或冻结一批模糊逻辑：

1. 主进程内嵌 heartbeat 的旧判断分支
2. 同时依赖 `self._heartbeat_process` 和外部 PID 文件的双轨判断
3. “主进程统一管理”这类不具运行态意义的展示文案

### 5.3 重构 planner 输入，默认只读收敛后的 task board

[`heartbeat_orchestration.py`](/C:/Users/Lenovo/Desktop/Butler/butler_main/butler_bot_code/butler_bot/heartbeat_orchestration.py#L223) 当前会同时拼：

1. `heartbeat_tasks.md`
2. task workspace
3. recent
4. local memory hits

这会让 planner 输入过宽。

建议改成三级输入：

1. 一级硬输入：显式工作任务、到期定时任务、显式治理任务
2. 二级辅助输入：最近执行摘要、必要局部记忆
3. 三级禁用输入：闲逛类、感想类、长对话原文

### 5.4 收紧 fallback 策略

[`heartbeat_orchestration.py`](/C:/Users/Lenovo/Desktop/Butler/butler_main/butler_bot_code/butler_bot/heartbeat_orchestration.py#L498) 当前本地兜底会：

1. 优先捞短任务
2. 再捞长期任务
3. 最后执行固定新陈代谢支路

问题是：在任务池已污染时，fallback 会把脏任务继续执行下去。

建议改成：

1. 先检查是否存在“白名单任务类别”
2. 不存在就进入 `status-only`
3. 固定新陈代谢支路改成显式低频维护任务，不再每次默认附带

## 6. 删除什么，保留什么

### 6.1 建议删除或停用的脏代码

以下逻辑建议进入删除名单或至少先停用：

1. `memory_manager.py` 中主进程内嵌 heartbeat 的旧分支
2. 任何仅用于兼容旧 `heart_beat_memory.json` 镜像、但已反向污染真源的自动同步逻辑
3. 将对话原文直接 upsert 到 heartbeat 看板的宽松入口
4. planner fallback 默认附带“固定新陈代谢支路”的策略
5. `manager.ps1` 中不区分真实状态的静态 heartbeat 说明文案

### 6.2 建议归档但不直接删除的数据

以下内容不应继续进入活跃目录，但不建议直接物理清空：

1. `task_change_log.jsonl`
2. 历史 `done / obsolete / duplicate` 任务
3. 旧 heartbeat 执行摘要
4. 旧 restart / cooldown 记录

建议位置：

1. `agents/local_memory/heartbeat_tasks/archive/`
2. `agents/state/archive/`
3. `run/archive/`

### 6.3 必须长期保留的真源

必须保持简洁并长期保真的只有：

1. 活跃任务板
2. 活跃 task ledger
3. 当前 watchdog state
4. 当前单轮 run state
5. 最近一次 heartbeat output 摘要

## 7. 长稳定维护方案

### 7.1 每次启动前自动做运行态自检

启动 heartbeat 前固定执行：

1. 校验 PID 文件对应进程是否真实存活
2. 校验锁文件 owner 是否真实存活
3. 校验 run state 更新时间是否超 stale 阈值
4. 若 watchdog 为 cooldown，展示剩余时间和原因
5. 若发现 stale 状态，先重置再启动，而不是直接跳过

### 7.2 每天一次任务池代谢

固定由维护脚本执行：

1. 去重主动任务
2. 将 `done / obsolete` 移出活跃板
3. 将 `.tmp`、`.bak`、孤儿文件隔离
4. 压缩或切分变更流水
5. 输出一次“任务池健康报告”

### 7.3 每周一次账本瘦身

固定规则：

1. 活跃 ledger 只保留近 30 天仍活跃任务
2. 历史结果挪入 archive
3. 统计重复率、过时率、无完成条件率
4. 当某类任务污染率过高时，直接冻结该类别输入

### 7.4 调度策略长期约束

长期硬规则建议写死：

1. 无显式任务时，默认 `status-only`
2. `explore` 只能在显式开关开启时进入
3. `cleanup` 只能执行白名单治理动作
4. `initiative` 类任务不能直接从 self_mind 自动灌入 heartbeat 真源
5. planner 输出若缺少明确 `selected_task_ids`，直接降级

### 7.5 指标化维护

至少维护以下指标：

1. 心跳 stale 次数/日
2. 自动重启次数/日
3. cooldown 次数/周
4. 活跃任务数
5. 重复任务率
6. obsolete 任务率
7. fallback 触发率
8. status-only 占比

只要这些指标开始漂移，就说明系统又在脏化。

## 8. 推荐执行顺序

### 第一阶段：当天止血

1. 重置运行态文件与锁文件
2. 修正 `manager.ps1 status` 的真实状态展示
3. 隔离任务目录中的 `.tmp`
4. 冻结 `06_闲时无聊.md` 输入
5. 将 fallback 改为“无白名单任务则 status-only”

### 第二阶段：两天内收口

1. 拆出 heartbeat runtime state service
2. 将 task ledger 活跃区和 archive 区分离
3. 任务板去重和过时归档
4. 停掉对话原文直接入板
5. 收紧 planner context

### 第三阶段：一周内定型

1. 统一 heartbeat 为 sidecar 真模式
2. 删除内嵌 heartbeat 旧路径
3. 建立 daily task hygiene 脚本
4. 建立 weekly ledger compact 脚本
5. 增补 stale/restart/task hygiene 回归测试

## 9. 对当前仓库的直接判断

基于 0317 当前仓库状态，可以直接下结论：

1. 现在最该删的不是 planner，而是脏运行态和脏任务输入
2. 现在最该拆的不是 prompt，而是 `memory_manager.py` 里混杂的 runtime/watchdog/loop 状态职责
3. 现在最该重置的不是所有历史，而是 run state、watchdog state、PID/lock、重复任务、模板化 reason
4. 长期最该坚持的不是“让心跳更主动”，而是“让 heartbeat 先成为一个可验证、可回收、可维护的后台执行器”

---

## 10. 最终建议

这轮不要继续在现有脏状态上打小补丁，建议按下面原则执行：

1. 先做一次硬重置，恢复运行态真实性
2. 再做一次任务真源瘦身，切掉污染入口
3. 然后拆 runtime state service，收走 `memory_manager.py` 的混乱职责
4. 最后再谈 planner 能力和主动探索

顺序不能反。否则只会继续出现：

1. 心跳看起来在跑，其实没跑
2. 系统看起来在做事，其实在重复脏任务
3. 每次修一个点，过两天又被旧状态和旧数据拖回去

这套治理的核心不是“更聪明”，而是“先变干净、再变稳定、最后再变强”。
