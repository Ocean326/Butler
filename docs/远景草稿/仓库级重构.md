## 1. 一句话远景

Butler 的长期仓库形态，不再是“若干目录平铺 + agent 自己猜边界”，而是：

> 三条并列产品线建立在共享 platform 之上，并由 repo-level governance plane 约束 agent 的读写、落位、文档回写和收口动作。

其中三条并列产品线固定为：

1. `butler-flow`
2. `chat`
3. `campaign + orchestrator`

---

## 2. 本文与当前真源的关系

本文不是“当前结构说明”，而是“未来多轮重构的上位目标图”。

当前必须保持的现役事实：

- 根目录 `runtime_os/` 仍是现役 compatibility surface，不能被本文直接写成“已退出”
- 根目录 `tools/` 仍是现役 CLI / 运维 / 审计入口，不能被本文直接写成“已改名为 scripts”
- `工作区/`、`过时/` 仍是当前现役目录名；`workspace/`、`retired/` 只允许作为远景候选命名
- `butler_main/butler_bot_code/` 仍承载当前 Butler 运行体与测试面，不能在远景稿里直接被抹掉

因此，本文只做三件事：

1. 固定 Butler 长期仓库的主叙事
2. 明确“产品 / platform / repo governance”三层分工
3. 给后续多轮物理搬迁提供同一张目标图

---

## 3. 为什么必须重写仓库级叙事

当前仓库的真实问题，不是目录名不够英文，也不是树长得不够整齐，而是以下三类边界经常被混写：

1. **产品边界混写**
   - `butler-flow`、`chat`、`campaign/orchestrator` 的主次与关系在不同文档里不稳定
2. **平台边界混写**
   - runtime、execution、durability、skills、shared assets 与产品壳经常混在同一层叙事里
3. **agent 治理边界缺位**
   - vibecoding 后哪些文件能新增、哪些运行产物不能进 git、哪些文档必须回写，往往要靠人临时兜底

如果不先把这三类边界讲清楚，后续任何“重构文件树”都会变成局部搬目录，而不是长期可维护的仓库收口。

---

## 4. 长期主结构：三产品 + Platform + Repo Governance

## 4.1 产品层

Butler 的长期产品叙事固定为三条并列产品线：

### `butler-flow`

定位：

- 前台 workflow shell
- 本地 attached runtime
- TUI / future Desktop
- manage center / flow workbench

核心对象：

- flow instance
- single flow workspace
- supervisor / workflow 双流
- shared assets / manage center

### `chat`

定位：

- chat / frontdoor 轻入口
- 对话式解释面
- routing / query / feedback 的产品入口
- session continuity 与轻量桥接层

核心对象：

- chat session
- frontdoor compile
- mode / lane / continuity
- query / feedback 入口

### `campaign + orchestrator`

定位：

- 长任务产品面
- mission / campaign 控制与推进
- operator-facing control surface
- console / query / projection 的后台产品壳

核心对象：

- campaign ledger
- mission orchestration
- turn receipt / acceptance
- operator console / observability surface

当前裁决：

- 第三条产品线不是“legacy campaign”
- `campaign` 与 `orchestrator` 在远景叙事里作为一条产品线统一表达
- `console` 默认视为该产品线的 operator surface，而不是单独第四产品

---

## 4.2 Platform 层

Platform 不再与产品线抢主叙事。它只承担三条产品线复用的共享能力。

长期应收口的 platform 能力包括：

- runtime
- execution adapters
- durability / writeback / recovery
- protocol / template / contract
- skill exposure
- storage
- shared assets
- host runtime / service packaging
- common libraries

当前与远景的对应关系大致如下：

- `butler_main/runtime_os/` -> platform runtime
- `butler_main/agents_os/`、`butler_main/multi_agents_os/` -> compat-first platform capability
- `butler_main/sources/skills/` -> platform skill sources
- `butler_main/butler_bot_code/` -> platform host runtime / service packaging

---

## 4.3 Repo Governance 层

Butler 需要的“agent 自治能力”，不是第四条产品线，而是 repo-level governance plane。

它的职责不是生成功能，而是约束 agent 的工作轨道，避免 vibecoding 后仓库再次失控。

Repo governance plane 的长期固定组成：

- 根 `AGENTS.md`
  - 改前读取协议
  - 术语映射
  - 文档冲突裁决
  - 默认收尾动作
- `docs/project-map/`
  - 功能地图
  - 真源矩阵
  - 改前读包
- `tools/vibe-close`
  - `analyze -> doc_targets -> test -> apply` 的默认收口链路
- 仓库分发边界
  - `.gitignore`
  - `.env.example`
  - `.codex/config.template.toml`
  - runtime artifact / private overlay 裁决

一句话：

> Butler 的 agent 自治，优先由仓库治理协议保证，而不是靠单个产品里的 prompt 自觉。

---

## 5. 目标文件树的正确表达方式

未来的目标文件树，不应只按“代码分层”描述，还要同时表达“产品”、“平台”和“治理”。

一个更稳的长期目标图如下：

```text
Butler/
├─ README.md
├─ AGENTS.md
├─ docs/
├─ butler_main/
├─ runtime_os/        # 现阶段根级 compatibility surface
├─ tools/             # 现阶段根级现役脚本入口
├─ 工作区/             # 现阶段活跃工作台
└─ 过时/               # 现阶段仓库级归档区

butler_main/
├─ products/
│  ├─ butler_flow/
│  ├─ chat/
│  └─ campaign_orchestrator/
├─ platform/
│  ├─ host_runtime/
│  ├─ runtime/
│  ├─ execution/
│  ├─ durability/
│  ├─ protocols/
│  ├─ skills/
│  ├─ storage/
│  ├─ shared_assets/
│  └─ common/
├─ compat/
│  ├─ runtime_os/
│  ├─ agents_os/
│  └─ multi_agents_os/
└─ incubation/
   └─ research/
```

重要说明：

- 上图是**远景分层图**，不是当前物理目录现状
- 根目录 `runtime_os/`、`tools/`、`工作区/`、`过时/` 当前仍保留
- `workspace/`、`scripts/`、`retired/` 等英文命名，当前只允许作为后续波次的候选，不得提前写成现役事实

## 6. 当前目录到远景结构的主映射

为了避免后续实施时反复猜测，长期主映射固定如下：

| 当前目录 | 远景归属 | 说明 |
| --- | --- | --- |
| `butler_main/butler_flow/` | `products/butler_flow/` | 前台 flow 产品线 |
| `butler_main/chat/` | `products/chat/` | chat/frontdoor 产品线 |
| `butler_main/orchestrator/` | `products/campaign_orchestrator/` | 第三产品线的控制壳 |
| `butler_main/domains/campaign/` | `products/campaign_orchestrator/` | 第三产品线的领域真源 |
| `butler_main/console/` | `products/campaign_orchestrator/` | operator-facing console surface |
| `butler_main/butler_bot_code/` | `platform/host_runtime/` | 后台运行体、配置、run/tests/ops |
| `butler_main/runtime_os/` | `platform/runtime/` | 现役 runtime 主体 |
| `butler_main/agents_os/` | `compat/agents_os/` | 兼容期目录 |
| `butler_main/multi_agents_os/` | `compat/multi_agents_os/` | 兼容期目录 |
| `butler_main/sources/skills/` | `platform/skills/` | skill 真源 |
| `butler_main/research/` | `incubation/research/` | 研究/孵化能力，不与主产品并列 |

其中：

- `butler_bot_code` 不视为独立第四产品
- `research` 不视为现役并列产品，而是 incubation / scenario /研究区
- `console` 不单列产品，归 `campaign + orchestrator` 产品线

---

## 7. 根目录的长期职责

根目录长期只承载四类职责：

1. 仓库入口
   - `README.md`
   - `AGENTS.md`
2. 正式文档
   - `docs/`
3. 主工程与 compat surface
   - `butler_main/`
   - `runtime_os/`
   - `tools/`
4. 工作与归档
   - `工作区/`
   - `过时/`

这意味着根目录未来治理的重点不是“尽快删光”，而是：

- 不再新增新的业务代码根目录
- 不再把临时运行输出留在根目录
- 不再让 agent 在根目录自由长临时文档和脚本
- 不再让目录命名迁移先于事实迁移

---

## 8. 仓库级硬规则

1. 新产品能力必须先归类到三条产品线之一，再决定代码落位。
2. 共享能力必须先证明其为三产品复用，才能进入 platform。
3. `agents_os/`、`multi_agents_os/`、根 `runtime_os/` 在 compat 清退前不得被写成“已退出”。
4. agent 自治能力必须优先固化到根 `AGENTS.md`、`project-map/` 和 `tools/vibe-close`，而不是只写在产品 prompt 或临时文档里。
5. runtime artifact、machine-local overlay、私有配置不能再被写回版本库。
6. 任何目录重命名都必须晚于边界裁决、导入兼容和文档导航收口。

---

## 9. 分阶段实施原则

仓库级重构的正确顺序，不是“先搬目录”，而是：

1. 先固定产品 / platform / repo governance 叙事
2. 再固定当前目录到远景结构的映射
3. 再做分波次物理迁移
4. 最后才清 compat surface 与候选命名

尤其是以下动作都不应在第一波直接执行：

- 根目录 `runtime_os/` 退出
- `tools/` 直接改名为 `scripts/`
- `工作区/` 直接改名为 `workspace/`
- `过时/` 直接改名为 `retired/`

这些都必须放到兼容清理的后段波次。

---

## 10. 远景验收口径

当以下条件同时成立时，可视为 Butler 仓库级重构进入稳定态：

1. 三条并列产品线的代码、文档与叙事长期稳定
2. 共享 runtime / execution / durability / skills / shared assets 已稳定归入 platform
3. `agents_os/`、`multi_agents_os/`、根 `runtime_os/` 的 compat 地位已被清晰标注并逐步收敛
4. repo governance plane 已稳定约束 agent 的改前读取、文档回写、收尾与 git 边界
5. runtime artifact / private overlay / cross-machine bootstrap 边界长期稳定，不再频繁反复
6. 根目录不再长出新的业务根目录或一次性临时资产

---

## 11. 最终口径

Butler 的长期仓库结构，不再是“代码树自己讲故事”，而是三层共同成立：

1. **三条并列产品线**
   - `butler-flow`
   - `chat`
   - `campaign + orchestrator`
2. **共享 Platform**
   - runtime / execution / durability / protocols / skills / storage / host runtime
3. **Repo Governance Plane**
   - `AGENTS.md`
   - `docs/project-map/`
   - `tools/vibe-close`
   - 分发边界与落位纪律

这三层稳定下来后，目录搬迁才有长期意义；否则只会把混乱从一个目录名搬到另一个目录名。
