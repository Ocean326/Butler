# 0314 升级前现状：任务调度、对话入队与 recent memory 链路

## 1. 这份文档的用途

这份文档不是写“理想架构”，而是把 0314 升级前 Butler 当前真实在跑的链路写清楚，重点回答五件事：

1. talk 现在是怎么把任务放入 heartbeat 的
2. talk 现在有没有稳定的“取消 heartbeat 任务”协议
3. planner 当前是怎么获取任务、分发、监督、验证、向用户汇报、最后交付的
4. recent memory 现在是怎么从活跃窗口走到 summary、再走到 local memory 的
5. 还有哪些容易被忽略、但已经影响系统行为的侧链和机制

---

## 2. 当前总图

当前的真实链路可以先压缩成一张脑图：

- talk 前台
  - 接消息、分诊、同步回复
- 对话后记忆落盘
  - 从 user + assistant 本轮对话里提炼 recent、heartbeat task candidate、long-term candidate
- heartbeat 任务入口层
  - `heartbeat_tasks.md` 分类看板
  - `task_ledger.json` 执行账本
- planner / manager
  - 从任务看板、task ledger、recent、local memory、workspace 上下文做计划
- branch 执行层
  - executor / test / acceptance / manager 风格的 branch
- 回执层
  - `user_message`
  - branch `tell_user_markdown`
  - 最终 heartbeat 私聊回执
- 记忆归档层
  - talk recent
  - beat recent
  - recent summary pool / archive / ladder
  - local memory / memory backend

要特别注意：

- 当前系统不是“talk 直接操作 task ledger”。
- 当前系统也不是“planner 只看一个 JSON 任务队列”。
- 它现在是“对话提炼 + 任务看板 + task ledger + recent + local memory”联合驱动。

---

## 3. talk 放入 heartbeat 任务的现状

## 3.1 当前不是显式协议 prompt，而是“对话后提炼 + 入队”

当前 talk 把任务放入 heartbeat，主链路不是用户在前台说一个固定命令后立即写任务，而是：

1. 前台收到消息，先走对话 prompt 和分诊
2. 对话回复先发给用户
3. 回复发出后，`on_reply_sent_async()` 异步启动 recent 持久化
4. `TurnMemoryExtractionService.extract_turn_candidates()` 同时从 `user_prompt + assistant_reply` 提炼：
   - recent entry
   - `heartbeat_tasks`
   - `heartbeat_long_term_tasks`
   - `long_term_candidate`
5. `memory_manager._merge_heartbeat_tasks_from_entry()` 再把任务写入：
   - 短期任务：heartbeat memory
   - 长期任务：heartbeat long tasks
   - 同步到 `task_ledger.json`
   - 同步到 `heartbeat_tasks` 分类看板

也就是说，当前 talk 放任务是“对话结束后的语义提炼入队”，不是“前台单独一条 task API”。

---

## 3.2 当前 talk 侧有两层任务提炼

当前有两层：

### 第一层：模型提炼

`memory_service.py` 会要求模型输出 JSON，其中直接包含：

- `heartbeat_tasks`
- `heartbeat_long_term_tasks`
- `long_term_candidate`

这意味着对话模型可以显式把某轮内容解释成：

- 一条后台短期任务
- 一条长期 reminder / 定时任务
- 一条应沉淀到长期记忆的事实

### 第二层：启发式兜底

即使模型提炼失败，`_extract_heartbeat_candidates()` 也会兜底抓几类信号：

- 出现“后台 / 心跳 / 每天 / 提醒 / 后续继续”等词
- 出现 reminder 类时间表达，如“每天 9 点提醒我”
- 出现工作时间窗口表达

所以当前 talk 放入 heartbeat 的本质是：

- 模型提炼优先
- 规则兜底补洞

---

## 3.3 当前 talk 放任务的协议边界

当前其实已经有“放任务”的隐式协议，但它不是用户可见、可控、可验证的显式协议 prompt。

现状更接近：

- 用户自然说一句“以后每天 9 点提醒我”
- 或“这个你后台继续推进”
- 对话回复后，系统把这轮语义提炼成 heartbeat task

这有两个结果：

### 优点

- 用户说话自然，不需要学命令
- 后台任务能无缝从对话延续出来

### 缺点

- 什么时候会被提炼成任务，用户并不总是可预期
- 是否是短期任务、长期任务、还是只留在 recent，当前缺少显式确认层

---

## 3.4 当前“取消 heartbeat 任务”基本没有前台稳定协议

这是 0314 前一个非常关键的现状缺口。

当前系统里：

- 有“放入 heartbeat 任务”的隐式提炼链
- 有 planner 在运行后通过 `complete_task_ids / defer_task_ids / touch_long_task_ids` 修改任务状态
- 也有 upgrade request 的批准/取消协议

但是：

- **没有一个稳定、显式、用户可学会的 talk 协议，用来取消某条已经入队的 heartbeat task**

也就是说，当前“取消任务”主要还是依赖：

- planner 运行时把任务标记为 defer / waiting_input / done
- 手工改 task board / task ledger
- 后续升级再补明确的前台取消协议

这点建议在 0314 升级时单列，不然“放任务”与“撤任务”会长期不对称。

---

## 4. planner 当前链路：获取任务 -> 分发 -> 监督 -> 验证 -> 汇报 -> 交付

## 4.1 planner 当前从哪里拿任务

当前 planner 不是只看一个列表，而是同时读：

- `heartbeat_tasks.md`
  - 作为统一任务看板读口
- task ledger 导出的任务工作区上下文
- talk recent + beat recent
- recent summary / recent summary archive ladder
- local memory 命中
- role / soul / skills / subagents / teams / public agent library

其中最重要的当前真相是：

- `heartbeat_tasks.md` 偏任务入口和分类视图
- `task_ledger.json` 偏执行状态、生命周期、工作区、验收回执

所以当前不是单真源，但已经形成了“看板入口 + 执行账本”的双层结构。

---

## 4.2 planner 当前怎么做计划

`HeartbeatOrchestrator.plan_action()` 当前输出的 plan 结构已经接近经理制：

- `chosen_mode`
- `execution_mode`
- `reason`
- `user_message`
- `tell_user_candidate / reason / type / priority`
- `selected_task_ids / deferred_task_ids`
- `task_groups`
- `updates`

其中 `task_groups[].branches[]` 已经支持：

- `process_role=executor|test|acceptance|manager`
- `runtime_profile`
- `selected_task_ids`
- `complete_task_ids`
- `defer_task_ids`
- `touch_long_task_ids`
- `depends_on`

这意味着当前 planner 虽然还没正式拆成独立 `planner_service / supervisor_service`，但 branch schema 已经具备多阶段 manager 编排骨架。

---

## 4.3 planner 当前不只是拿显式任务，还会自动补固定支路

当前 plan 不完全由 planner 自己自由输出。

`_ensure_fixed_branches()` 还会按配置自动插入：

- `heartbeat-metabolism`
  - 固定新陈代谢支路
- `heartbeat-background-growth`
  - 空闲时的受控自我提升支路

所以当前 heartbeat 不只是“执行用户任务”，还带有：

- 轻治理
- 轻维护
- 背景成长

这也是为什么有时候即使没有显式任务，heartbeat 仍然会干一些“整理/巡检/补说明”的事情。

---

## 4.4 planner 当前如何分发与执行

当前分发是：

1. planner 产出 `task_groups`
2. 每组内分 `parallel_ready` 和 `serial_queue`
3. `run_branch()` 为每个 branch：
   - 解析 `process_role`
   - 经 `runtime_router.py` 选择 `runtime_profile`
   - 组装执行 prompt
   - 注入角色协议、执行协议、回执协议
   - 调用 team executor 或普通 CLI 执行

当前运行时上已经支持：

- 按 branch 选择不同 runtime
- 按 branch 标记不同 process role
- branch 输出中抽取 `tell_user_markdown`

这已经不是旧式“一轮 planner 文本 + 单路执行器”了。

---

## 4.5 planner 当前如何监督与验证

当前监督和验证还没有完全抽成独立 supervisor 服务，但已有三层事实机制：

### 第一层：branch 级状态

每个 branch 会回写：

- `ok`
- `output`
- `error`
- `process_role`
- `runtime_profile`

### 第二层：task ledger 应用层

`TaskLedgerService.apply_heartbeat_result()` 会根据：

- `selected_task_ids`
- `complete_task_ids`
- `defer_task_ids`
- `touch_long_task_ids`
- branch 成败

把任务推进成：

- `done`
- `in_progress`
- `waiting_input`

并生成：

- `acceptance_status`
- `acceptance_summary`
- `manager_state`
- `runtime_profile`
- `process_roles`

### 第三层：acceptance receipt

`acceptance_service.py` 会从 branch results 里整理：

- 有没有 acceptance branch
- 是否应该 accept / complete / deferred / in_progress
- 最终 acceptance summary

所以当前系统里已经存在“验证/验收收口”，只是还没有完全独立成一整条 acceptance 流水线。

---

## 4.6 planner 当前怎样向用户主动汇报进度

这里要区分三层，不然很容易混淆：

### 层 A：`user_message`

这是 plan 自带的对用户说明，代表：

- 本轮准备做什么
- 为什么这样做

它会进入 heartbeat 最终私聊回执。

### 层 B：branch `tell_user_markdown`

每个 branch prompt 都被要求：

- 若本分支有值得同步给用户的结果
- 就在输出末尾追加 `【heartbeat_tell_user_markdown】...`

heartbeat 汇总时会把这些内容渲染进“值得同步”区块。

### 层 C：heartbeat 私聊回执

每轮 heartbeat 结束后，会给用户发一条总回执，通常包含：

- `user_message`
- 执行摘要或值得同步
- 失败项 / 延后项
- 本轮耗时分段

另外还有两种补充回执：

- 规划起始回执
- 规划进度回执

但这两种当前受 debug receipt 开关控制，不是默认总会发。

所以当前“主动汇报进度”已经有机制，但不是完全实时流式，而更偏：

- 轮次级进度同步
- 分支摘要汇总

---

## 4.7 planner 当前最后如何交付任务完成情况

当前的交付不是只发一条 heartbeat 消息就完。

实际交付分为三层：

1. 用户可见回执
   - heartbeat 私聊消息
2. recent 层留痕
   - beat recent snapshot
3. task ledger / workspace 收口
   - `task_ledger.json`
   - `agents/state/task_workspaces/...`
   - `final_report.md`

这意味着“任务完成”当前已经不只是聊天回复，而是会沉淀到执行账本和工作区。

---

## 5. recent memory 现状：过时、summary、最终归入 local memory

## 5.1 当前有 talk recent 和 beat recent 两个池

当前不是一个 recent。

系统至少有两个池：

- talk recent
  - 对话连续性
- beat recent
  - heartbeat 计划和执行连续性

两者相互可读，但不是完全混写。

这使得：

- 对话上下文不会被 heartbeat 日志完全淹没
- heartbeat 也能保留自己的运行连续性

---

## 5.2 每轮对话的 recent 是怎么写入的

当前 per-turn 流程是：

1. `begin_pending_turn()`
   - 先写一条 `replying` 的 provisional recent
2. 回复发出后，`on_reply_sent_async()`
   - 先写 fallback completion
   - 再异步启动 `_finalize_recent_and_local_memory()`
3. `_finalize_recent_and_local_memory()`
   - 用记忆提炼服务生成正式 recent
   - 用 `subconscious_service` 做主条目 + companion 条目整合
   - 可能同步抽出：
     - heartbeat tasks
     - long-term candidate
     - self_mind cognition signal
   - 再 compact、保存、同步 memory backend

所以当前 recent 不是“用户一发消息立即完成”，而是：

- 先 provisional
- 再 completion fallback
- 最后异步精炼成正式 recent

---

## 5.3 当前 recent 的“过时”机制

当前 recent 会按规则分成：

- active entries
- stale entries

判定依据包括：

- 是否是最近保留的 talk turn
- 是否是 interrupted
- 是否是 companion stream
  - mental
  - relationship_signal
  - task_signal
- 是否已经 promoted 到 local memory
- active_window 是否还是 current

stale entry 不会直接删除，而是：

1. 归档到 `recent_memory_stale_*.json`
2. 往 `recent_archive.md` 追加一条人类可读摘要

所以当前 recent 的过时机制是“移出活跃上下文，但保留可追溯归档”。

---

## 5.4 当前 recent summary 机制

当前 recent summary 已经是三层：

### 第一层：summary pool

把“最近窗口之外”的 talk recent，按 chunk 聚合成窗口 summary。

### 第二层：summary archive

如果 summary pool 超出活跃上限，就把更早的 summary 滚入 archive。

### 第三层：summary ladder

再从 summary pool + archive 里聚合出时间阶梯小结：

- 最近 10 天
- 最近 4 个月
- 最近 1 年

然后在 prompt 注入时：

- 最近窗口走 `recent_memory`
- 窗口外相关小结走 `recent_summary`
- 更久历史走 `recent_summary_archive / ladder`

这意味着当前 recent 已经不只是“截断最近 N 条”，而是开始有窗口压缩和历史阶梯抽象。

---

## 5.5 当前 recent 最终归入 local memory 的机制

当前 recent 进入 local memory 主要有两条路：

### 路 1：per-turn 直接 promotion

如果本轮对话提炼出的 `long_term_candidate.should_write=true`，会在 `_finalize_recent_and_local_memory()` 中直接尝试：

- `_upsert_local_memory()`
- 并标记 recent entry 已 promoted

### 路 2：维护轮次补 promotion

即使 per-turn 没升上去，后续 recent maintenance 也会再扫一遍：

- `_promote_recent_long_term_candidates()`

所以当前从 recent 到 local memory 不是一次性动作，而是：

- per-turn 先试一次
- maintenance 再补一次

---

## 5.6 local memory 不是只有一份文档

当前 local memory 已经不是单一 `semantic_memory.md`。

它至少包括：

- summary / detail 文件
- index
- relations
- write journal
- memory backend semantic event

也就是说，“最终归入 local memory”已经是：

- 生成或追加 summary
- 必要时落 detail
- 更新 index / relations
- 写 journal
- 同步 memory backend

---

## 6. 你可能还没显式想到，但已经在影响系统的链路

下面这些都是当前已经存在、但很容易被忽略的机制。

## 6.1 self_mind 不是旁路，它会反向交办 heartbeat

当前 `self_mind` 不只是自己续思。

它还可能：

- 直接 talk
- 或把事项 handoff 给 heartbeat，生成 `self_mind` 来源的 heartbeat task

所以 heartbeat 的任务来源并不只来自 talk。

---

## 6.2 heartbeat 任务来源已经是多入口

当前看板文本里写得很明确：

- 对话
- 潜意识
- self_mind
- heartbeat 自修正

都可以写入 heartbeat 任务看板。

所以如果后面要做 0314 升级，“任务入口协议”不能只考虑 talk。

---

## 6.3 task board 和 task ledger 不是一回事

当前这两者经常被混成一个概念，但它们分工不同：

- `heartbeat_tasks.md`
  - 入口、分类、planner 读口
- `task_ledger.json`
  - 生命周期、状态、运行回执、workspace、final report

这两个层面后续如果不继续收口，很容易出现“看板意图”和“执行事实”偏离。

---

## 6.4 任务 workspace 已经是重要真源

当前 planner 不只读任务列表，也会读 task workspace 上下文。

这意味着：

- 任务的过程文档
- 最近动作
- final report

已经开始进入下一轮调度输入。

后续如果升级 manager/supervisor，不应该再绕开 task workspace。

---

## 6.5 heartbeat 现在自带固定治理支路

当前 heartbeat 并不是“没任务就空转”。

即使 planner 给出 status，本轮仍可能自动插入：

- metabolism
- background_growth

所以后续升级时必须先明确：

- 哪些是固定系统代谢
- 哪些是用户任务
- 哪些是主体成长/自修正

不然会持续混在一个回执里。

---

## 6.6 当前“主动向 user 汇报”分成 talk lane 和 heartbeat lane

现在至少有两条对用户输出的主动链：

- heartbeat lane
  - `user_message`
  - branch `tell_user_markdown`
  - heartbeat 私聊回执
- self_mind talk lane
  - direct talk
  - tell_user 审计日志

如果后面不明确区分，很容易出现：

- heartbeat 的汇报口吻
- talk 的人格口吻

互相打架。

---

## 6.7 当前还有升级/重启审批侧链

heartbeat 现在还承担一个额外职责：

- 发现升级/重启需求
- 只发审批，不直接执行
- 等聊天主进程接收批准后处理

这条链和普通任务链不同，但也会影响 heartbeat 的对用户回执内容。

---

## 6.8 当前监视和运行状态也已经是系统的一部分

当前不是“看日志就行”。

实际上已经有：

- main state
- heartbeat run state
- watchdog state
- `watch_stack.ps1`

这意味着运行监视本身已经是 Butler 运行事实的一部分，不应该再只靠人工翻日志。

---

## 7. 当前最关键的结论

0314 升级前，Butler 当前最真实的状态不是“还没有任务链路”，而是：

- 已经有 talk -> heartbeat 的隐式入队链
- 已经有 planner -> branch -> acceptance 的最小经理制骨架
- 已经有 recent -> summary -> local memory 的分层链
- 已经有 self_mind / subconscious / heartbeat 自修正等多入口写任务

但同时也有三个必须正视的缺口：

1. talk 侧缺少显式的“放入 / 取消 heartbeat 任务”协议
2. planner / supervisor / acceptance 还没有完全服务化拆开
3. task board、task ledger、talk lane、heartbeat lane 之间的边界仍然需要再收紧

如果 0314 要继续升级，最值得先做的不是再加功能，而是把这三条边界写成明确协议。
