# 0313 升级对照总结（2026-03-14）

## 1. 背景

本轮整理覆盖从 2026-03-13 的升级设计，到 2026-03-14 的恢复、补模块、重启、监视和上线修正。

期间发生过一次明显事故：

- 一批新增模块在历史清理/回滚过程中丢失
- 导致主进程和 heartbeat 出现 `ModuleNotFoundError`
- 后续通过本地缓存、现有调用链和 0313 方案，按“最小可运行骨架”重建

所以这轮不是一次“纯升级”，而是：

- 0313 方案落地
- 丢失模块恢复
- 运行链修正
- 架构继续收口

---

## 2. 对照 0313：已落地项

## 2.1 飞书前台化

已落地：

- 新增 `request_intake_service.py`
- 前台可以把请求分为讨论 / 同步快任务 / 同步后转异步 / 异步 program
- 主入口不再默认把所有复杂任务都塞进当前轮直接做完

仍未完全落地：

- 会话状态卡片还未完全独立成专门服务
- “前台状态工作台”体验仍主要靠现有回复和运行状态回执支撑

## 2.2 经理制多角色调度

已落地：

- branch 开始支持 `process_role`
- 已具备 `manager / executor / test-evaluate / acceptance` 的最小角色骨架
- `acceptance_service.py` 已接入任务回执与 task ledger

仍未完全落地：

- `planner_service.py / supervisor_service.py` 尚未真正拆出
- 当前仍由 `heartbeat_orchestration.py` 承担主要经理编排职责

## 2.3 动态 runtime 路由

已落地：

- 新增 `runtime_router.py`
- 运行时路由从 `cli_runtime.py` 的 provider 适配层中上移
- 已支持 branch 级别 `runtime_profile`
- 已补 `codex` 侧的可执行性/偏好兜底，避免自动分支强绑定到不稳定 runtime

新增的线上修正：

- 自动分支当前默认更偏向 Cursor 稳定路径
- 对 Codex 保留配置能力，但不再强制作为 specialized branch 的默认依赖

## 2.4 记忆分层与解耦

已落地：

- 新增 `memory_backend.py`
- 新增 `local_memory_index_service.py`
- `prompt_assembly_service.py` 承担更多 prompt 组装职责

仍未完全落地：

- `memory_manager.py` 仍偏重
- `self_mind` 尚未真正外挂化

## 2.5 执行与能力层

已落地：

- 恢复并接入 `agent_team_executor.py`
- 恢复并接入 `agent_capability_registry.py`
- 恢复并接入 `prompt_assembly_service.py`

这使得团队执行、能力注册、prompt 装配不再散落在单点入口里。

## 3. 本轮恢复与修正的关键动作

这轮真正影响上线稳定性的，不只是“做了哪些新模块”，还包括以下恢复动作：

1. 重建丢失模块

- `request_intake_service.py`
- `runtime_router.py`
- `acceptance_service.py`
- `cli_runtime.py`
- `memory_backend.py`
- `local_memory_index_service.py`
- `agent_team_executor.py`
- `agent_capability_registry.py`
- `prompt_assembly_service.py`

2. 修正 task ledger 行为

- 清理任务工作区的重复目录，避免 bucket 迁移后遗留旧工作区

3. 修正 Cursor 运行环境

- 为 Cursor CLI 补环境隔离
- 使用独立 session 目录承载 `APPDATA / LOCALAPPDATA / TEMP / PROFILE`
- 降低 heartbeat 并发调用时共享运行目录带来的冲突

4. 修正 heartbeat 可观测性

- 新增 `watch_stack.ps1`
- 可以直接观察主进程、heartbeat、watchdog 和日志尾部

5. 修正心跳回执体验

- 调整本地兜底 `user_message`
- 让心跳兜底回执更偏“运行说明”，减少和 talk 消息的语气重叠

6. 修正飞书发送链

- 私聊发送不再吃环境代理
- 避免 heartbeat 因坏代理导致初始化消息或回执失败

## 4. 当前架构事实

截至 2026-03-14，当前系统的真实状态可以概括为：

- 前台分诊骨架已建立
- 经理制调度骨架已建立
- branch 运行时路由已建立
- 验收回执链已建立
- 记忆 backend/index 分层已开始形成
- 监视入口已补齐

但同时也要承认：

- 这还不是 0313 方案的终态
- `HeartbeatOrchestrator` 还没有完全让位给独立 manager/planner/supervisor 服务
- `self_mind` 还没有彻底外挂化
- guardian 自动恢复链还需要继续收口

所以当前阶段最准确的判断不是“升级完成”，而是：

0313 的核心骨架已经落地，系统已经从旧单体路径推进到新分层路径；但完整的服务化解耦仍在第二阶段。

---

## 5. 当前新架构描述

当前架构可以简写为：

- 飞书 Bot：前台入口、讨论、澄清、同步回执
- Request Intake：把请求分成同步和异步两类
- Heartbeat Manager：负责计划、回执、阶段推进和兜底
- Runtime Router：为 branch 选择 CLI / model / runtime profile
- Executor / Team Executor：实际执行任务
- Acceptance：生成验收收口与最终回执
- Task Ledger：作为任务真源承接生命周期与最终报告
- Memory Backend + Local Memory Index：作为共享记忆底座
- Watch Stack：作为快速运行观测入口

这套结构已经比旧状态更接近 0313 里提出的：

- 前台 / 后台分离
- 经理 / 执行 / 验收分离
- runtime 路由集中化
- memory 从大文件重耦合走向分层

---

## 6. 建议的后续阶段

如果继续按 0313 方案推进，下一阶段优先级建议是：

1. 正式拆出 `planner_service.py` 与 `supervisor_service.py`
2. 将 heartbeat 中“经理决策”和“监督重试”彻底从 facade 中抽离
3. 将 `self_mind` 拆成独立可启停 app
4. 收紧 guardian 的单实例恢复链，避免 sidecar 残留
5. 将 `watch_stack.ps1` 升级成更强的运行事实仪表盘

---

## 7. 一句话结论

相对 0313 方案，Butler 目前已经从“单入口混杂执行”升级到“前台分诊 + 经理调度 + runtime 路由 + 验收回执 + 记忆分层”的新骨架；但它仍处于骨架完成、服务彻底解耦尚未收尾的过渡阶段。
