﻿﻿# 外部机制提炼：其他机器人/角色为何看起来有“灵魂”

目的：将他人系统表现出“有灵魂”的机制，提炼成可执行规则，方便迁移到 Guardian 并持续迭代。

概览（可直接落地的规则）：

1. 稳定的“灵魂真源”（Canonical Soul）:
- 规则：将核心身份/价值/约束写成单一、版本化的长期记忆文档；运行时作为权威参考。
- 执行：把文档作为只在受治理允许下变更的 artifact（例如 guardian/docs/外部机制_灵魂要素.md）。agent 启动或构建提示时检索并注入。

2. 分层表达（Values → Policies → Voice）:
- 规则：把“价值观/优先级/行为边界”与“语气/措辞集合”分开管理。
- 执行：在 soul 文档中明确三层内容；运行时仅在需要时注入对应层级（决策用价值观，输出用语气模板）。

3. 明确的决策规则（Tie-breakers & Heuristics）:
- 规则：当不确定或多选时，使用 soul 中的优先级来打平冲突。
- 执行：实现简单规则集，例：`如果用户目标与安全约束冲突→优先安全；如果多目标并列→按价值顺序评分并选择最高得分方案`。

4. 可检索与可注入（Retrieval + Prompt Injection）:
- 规则：运行时有稳定的检索路径，将 soul 的摘要注入到“开放式/咨询式”对话与建议场景的 prompt 中。
- 执行：prompt-builder 判断场景（open-ended/advisory/role-play/long-running），在条件满足时加载并缓存 soul 摘要并注入前置系统说明。

5. 可审计的变更与回滚（Governance）:
- 规则：任何对 soul 的修改都需具备证据、评审记录、变更日志与回滚路径。
- 执行：修改必须经过 N 人/自动验证或多次一致性事件（心跳/审计/用户反馈），并在 ledger 记录：修订原因、证据、批准人、回滚指令。

6. 可测试的行为基线（Regression Tests & Monitors）:
- 规则：为 soul 定义行为测试用例，防止行为漂移。
- 执行：把若干“语气样例 + 决策样例”做为单元/回归测试；定期对话抽样比对期望输出与实际输出。

7. 可解释性（Explainability）:
- 规则：当作出有影响的决策或建议时，简短说明“基于 soul 的理由”。
- 执行：在响应中自动附带一句短说明，例如“按照Guardian的价值：优先可用性与安全，因此……”。

8. 个性化与边界（Personalization & Opt-in）:
- 规则：允许用户选择/微调 voice palette，但不得改变核心价值与安全约束。
- 执行：把可变的“语气配置”存为用户偏好档，应用时先验证不违反 soul 的高阶约束。

9. 标识与一致性信号（Signatures）:
- 规则：使用少量一致的信号（短语、格式、签名样式）增强连续性与可识别性。
- 执行：在消息尾或元信息中包含可配置的签名或口头习惯用语，确保跨会话保持一致。

10. 退化与容错（Fallbacks）:
- 规则：当 soul 内容不可用或不一致时，采用中性安全的默认行为，避免意外“人格漂移”。
- 执行：定义默认中性 persona 文本与低风险行为集，作为运行时缺失时的备援。

迁移与放置建议：
- 把“灵魂真源”作为 Guardian 的长期文档放在 guardian/docs/（或 guardian 的 long-term local_memory 路径），并在 Guardian 启动或 prompt-builder 中增加“检索并注入”步骤。
- 推荐路径：guardian/docs/外部机制_灵魂要素.md（本文件）；如需运行时快速访问，可复制到 guardian_bot 的 local_memory 目录并纳入版本化管理。

短期落地清单（可直接执行）：
- 在 Guardian 的 prompt-builder 中加入：场景分类 → 满足条件则检索 guardian/docs/外部机制_灵魂要素.md 的摘要 → 注入系统/助手提示前缀。
- 建立 soul 修改流程：PR → 证据附带 → 两人 review → 列入 ledger 并在指定心跳后生效（阶段性启用）。
- 添加 10 个回归样例（语气/决策）并纳入 CI 或定期检测任务。

备注：此规则集合以可执行为优先，可根据 Guardian 的具体架构把“检索/注入/治理”部分具体化为代码/配置项。
