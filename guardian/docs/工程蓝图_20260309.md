# Guardian 工程蓝图

## 1. 工程定位

guardian 是独立维护 bot，不直接复用现有 `restart_guardian_agent.py` 作为最终形态。

现有脚本可以继续承担过渡期 watchdog 角色，但第一版正式工程以 `guardian_bot_code` 为主。

## 2. 核心流程

### 2.1 请求进入

来源：

1. heartbeat 轻量修复备案
2. heartbeat 升级申请
3. butler 升级申请
4. 用户直接修复任务
5. guardian 自主巡检发现

### 2.2 审阅

输出：

1. `approve`
2. `reject`
3. `need-info`

### 2.3 执行

1. 低风险请求可直接执行
2. 代码修改前必须先生成 patch 预案
3. 动态选择测试集合
4. 验证后自动重启并上线

### 2.4 归档

每次请求必须形成：

1. request 文件
2. review 文件
3. execution 文件
4. test 文件
5. rollback 文件（若发生）

## 3. 并发约束

1. 审阅可以并发
2. 执行必须串行
3. 同一时刻只允许一个 active execution request

## 4. 建议事件目录

1. `requests/pending`
2. `requests/reviewed`
3. `requests/executing`
4. `requests/done`
5. `requests/rejected`
6. `ledger/reviews`
7. `ledger/executions`
8. `ledger/tests`
9. `ledger/rollbacks`

## 5. 第一版最小可运行目标

1. 能吃到 heartbeat 的 request 文件
2. 能判断低风险还是高风险
3. 能写 review 结果
4. 能把执行过程写 ledger
5. 能对低风险请求执行基础动作
